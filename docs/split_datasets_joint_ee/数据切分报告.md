# 数据切分报告（Joint / EE）

本报告对应脚本：`docs/split_datasets_joint_ee/split_ee_or_joint_dataset.py`

目标：将一个 LeRobot Dataset v3.0（AgileX 录制）拆分为两份“低维状态空间不同”的数据集：

- `*_joint`：只包含关节角（joint space）
- `*_ee`：只包含末端位姿（end-effector space）

输出仍然保持 v3.0 数据集结构（`meta/ data/ videos/`），可直接用于后续训练/解析。

---

## 1. 输入数据的关键事实（以 v3.0 schema 为准）

以你的数据集为例：

- 原始路径：`/home/agilex/.cache/huggingface/lerobot/cqy/agilex_vla_demo_ee_pinocchio_Bowl_on_Plate_Placement_test`
- `meta/info.json` 显示：
  - `observation.state`：26 维
    - 14 维关节：左右臂各 7（含 gripper）
    - 12 维末端：左右臂各 6（`x,y,z,roll,pitch,yaw`）
  - `action`：14 维（关节）
  - 视频 keys：`observation.images.camera_left/right/front`

也就是说：原始数据里 **状态同时包含 joint + ee**，但 **动作只有 joint**。

---

## 2. 切分目标与最终产物

### 2.1 joint 数据集（`*_joint`）

目标：训练“关节空间策略”，避免把 EE 信息混进 state（有些训练只希望 joint state）。

结果：

- `observation.state`：从原 `observation.state` 中抽取 14 维关节
- `action`：保持原始 14 维关节 action 不变
- `videos/`：完整保留（可 hardlink/symlink/copy）
- `meta/`：保持 v3.0 结构，同时更新 `meta/info.json` 的 shape/names 与对应统计

### 2.2 ee 数据集（`*_ee`）

目标：训练“末端空间策略”，只暴露 EE pose 作为低维状态。

结果：

- `observation.state`：抽取 `left_ee.*` + `right_ee.*` 共 12 维
- `action`：原始数据没有 EE action，所以脚本定义为“同一 episode 内下一帧 EE state”
  - `action[t] = observation.state[t+1]`
  - episode 边界与最后一帧：`action[t] = observation.state[t]`（防止跨 episode 串联）
- `videos/`、`meta/` 同样保留并更新

备注：这个 `ee action` 是一种 **可复现的定义**（next-state imitation），并不等价于机器人真实下发命令。

---

## 3. 脚本实现流程（核心步骤）

### 3.1 用 schema 自动找维度（不写死下标）

脚本以 `meta/info.json` 为唯一事实来源：

1. 读取 `features["observation.state"]["names"]` 与 `features["action"]["names"]`
2. `joint` 维度：用 `action.names` 在 `observation.state.names` 里逐个匹配，得到 joint indices
3. `ee` 维度：在 `observation.state.names` 里筛选前缀为 `left_ee.` / `right_ee.` 的字段，得到 ee indices

这样做的好处：

- 录制顺序变了也不怕（以 names 对齐）
- 兼容不同相机 keys（脚本不依赖固定 camera 名）

### 3.2 复制 meta 与 videos（保持 v3.0 目录）

输出目录会包含：

- `meta/info.json`：写入更新后的 features（shape/names）
- `meta/tasks.parquet`：直接拷贝
- `meta/episodes/*/*.parquet`：直接拷贝（后续只更新 stats 列）
- `videos/`：按 `--video-mode` 选择 hardlink / symlink / copy（默认 auto）

### 3.3 处理 data parquet（逐文件）

#### joint 版本

对每个 `data/chunk-xxx/file-yyy.parquet`：

- `observation.state = observation.state[:, joint_indices]`
- `action` 不变
- 其它列 `timestamp/frame_index/episode_index/index/task_index` 全部原样保留
- 用 HF datasets 的 feature schema 写回 parquet（保证 list/array 列的 arrow schema 一致）

#### ee 版本

对每个 `data/chunk-xxx/file-yyy.parquet`：

- `ee_state = observation.state[:, ee_indices]`
- 计算 `ee_action`：
  - 同 episode 内做 1-step shift
  - episode 边界处不跨 episode
  - 最后一帧用自身 state 填充
- 同时做了 **跨 parquet 文件边界** 的修补：
  - v3.0 的 `data/file-xxx.parquet` 可能把同一个 episode 切到下一文件
  - 脚本会缓存上一个 parquet 的最后一行，如果发现下一 parquet 的第一行仍是同一 episode，则把上一文件最后一行的 action 修正为下一文件第一帧的 state

## 4. 统计（stats）与 episode 元数据更新

切分后，`observation.state` 与 `action` 的维度发生变化，因此必须更新两处：

1. `meta/stats.json`
2. `meta/episodes/*/*.parquet` 每行 episode 的 `stats/<feature>/*` 列

做法：

- 重新遍历输出数据集的 `data/*/*.parquet`，按 `episode_index` 聚合出每个 episode 的：
  - `observation.state`（数组）
  - `action`（数组）
- 使用仓库内的 `compute_episode_stats()` 计算每个 episode 的统计（min/max/mean/std/quantiles/count）
- 把每个 episode 的统计写回到对应 `meta/episodes` parquet 的 stats 列
- 再用 `aggregate_stats()` 聚合出整库统计，写回 `meta/stats.json` 中这两个 key：
  - 覆盖 `observation.state`
  - 覆盖 `action`
- 其它 key（例如视频统计、timestamp/index 等）保持原值不变

一个实现细节：

- `count` 是形状 `(1,)` 的数组。pandas 在单元格赋值时可能把它变成 0-d array，导致 parquet 写入报错。
- 脚本对 `stats/*/count` 做了 `np.atleast_1d` 的强制修正，确保 parquet 可写。

## 5. 校验结果（本次已验证）

对输出的两个数据集，用 `LeRobotDataset(root=...)` 加载并抽样验证：

- `*_joint`：`observation.state.shape == (14,)`，`action.shape == (14,)`
- `*_ee`：`observation.state.shape == (12,)`，`action.shape == (12,)`
- 视频 keys 保留：camera_left/right/front 均可被识别

备注：如果本机 HuggingFace datasets cache 目录不可写，读取时可设置：

`HF_DATASETS_CACHE=/tmp/hf_datasets_cache`

---

## 6. 已知限制与建议

- `*_ee` 的 `action` 是“下一帧 EE state”的定义，适合做 next-state imitation；如果你需要“真实控制指令”的 EE action，需要从控制命令链路重新构建。
- 脚本目前只在 `AgileX` 这类 `observation.state` 同时包含 joint 与 EE 的 schema 上验证过；其它机器人只要符合 v3.0 schema 且 names 可对齐，也可以使用，但建议先小规模抽样检查。
