# 零基础学运控 4：用 FK 录 EE、用 IK 回放 EE（Pinocchio 版，结合本仓库代码）

本文面向**零基础运控/控制工程师**：不要求你掌握机器人运动学推导，但希望你看完能：

1. 读懂“FK/IK 到底在解什么问题”
2. 看懂本仓库里 `record_ee_pinocchio.sh` / `replay_ee_pinocchio.sh` 的端到端链路
3. 理解为什么“能回放但有误差”
4. 知道把这套方案迁移到**下一台机器人**时，哪些代码能复用、哪些必须改（尤其 6 轴 vs 7 轴）

相关核心文件（建议边看边对照）：

- 录制脚本：`record_ee_pinocchio.sh`
- 回放脚本：`replay_ee_pinocchio.sh`
- 录制端 FK 写入：`src/lerobot/robots/agilex/agilex.py`（`AgileXRobot.get_observation()`）
- 回放端 IK 解算：`src/lerobot/scripts/lerobot_replay_ee.py`（`replay_ee()`）
- Pinocchio FK/IK 纯计算模块：`my_ik/piper_pinocchio_kinematics.py`（`PiperPinocchioFK`/`PiperPinocchioIK`）
- 机器人配置（关节/末端配置项）：`src/lerobot/robots/agilex/config_agilex.py`

> 术语约定：
> - “EE”：End-Effector（末端），本方案里指记录在数据集里的 `left_ee.* / right_ee.*`
> - “TCP”：Tool Center Point（工具坐标系），可理解为“你真正关心的末端点/末端坐标系”
> - “URDF”：机器人模型文件，决定连杆长度、关节轴、关节限位等
> - “q”：关节角（或关节位移）向量
> - “T”：4x4 齐次变换矩阵（位置 + 姿态）

## 1. 先用一句话讲清楚：你现在的方案在干什么？

你的方案分成两条链路：

1. **录制（Record）**：每一帧读到从臂关节角 `q` → 用 **FK** 计算末端位姿 `T(q)` → 把 `T` 写入数据集的 `observation.state`（也就是 `left_ee.* / right_ee.*`）。
2. **回放（Replay）**：每一帧从数据集读出末端位姿 `T*` → 用 **IK** 解出关节角 `q_hat` 使得 `T(q_hat) ≈ T*` → 把 `q_hat` 下发给机器人关节控制。

这就是“用 FK 录末端，用 IK 放末端”。

**注意：数据集里的 `action` 仍然是关节角（不是末端动作）**。末端位姿只体现在 `observation.state` 里。

---

## 2. 零基础必懂的三件事：坐标系、位姿表示、URDF

### 2.1 为什么必须讲“坐标系”？

机器人运动学的本质是：**在不同坐标系之间表达同一个点/姿态**。

至少会出现：

- `base`：机械臂基座坐标系（URDF 根）
- `link_i`：每个连杆自己的坐标系
- `ee` 或 `tcp`：末端坐标系

你记录的 `left_ee.x/y/z/roll/pitch/yaw` 不是“绝对世界坐标”，而是 **URDF 根坐标系到末端坐标系** 的相对位姿。

### 2.2 位姿常用表示：4x4 变换矩阵（建议你把它当“最终真相”）

一个刚体位姿通常写成：

```
T = [ R  p ]
    [ 0  1 ]
```

- `p`：3D 位置向量（m）
- `R`：3x3 旋转矩阵（无单位）

代码里无论你输入/输出是 rotvec、rpy、quat，本质都会在内部变成 `R` 或 `T` 来算 FK/IK。

### 2.3 本方案用的姿态表示：RPY（roll/pitch/yaw）

Pinocchio EE 数据格式是：

```
xyzrpy = [x, y, z, roll, pitch, yaw]
```

并使用约定：

```
R = Rz(yaw) · Ry(pitch) · Rx(roll)
```

对应实现可在这些文件里看到：

- `my_ik/piper_pinocchio_kinematics.py`（`_xyzrpy_to_matrix` / `_matrix_to_xyzrpy`）
- `src/lerobot/scripts/lerobot_replay_ee.py`（`_xyzrpy_to_matrix`）
- `src/lerobot/robots/agilex/agilex.py`（`_xyzrpy_to_matrix`）

> 工程提醒：RPY 有“万向节锁”问题（pitch 接近 ±90° 时不稳定）。做训练可以继续用它当输入；做精确几何误差/插值更建议转成 rotation matrix 或 quaternion。

### 2.4 URDF 是什么？为什么它决定了你 IK/FK 的“上限”

URDF 描述了：

- 连杆之间的连接关系（树/链）
- 每个关节的轴、零位、限位
- 每段连杆的位姿偏置（`origin xyz/rpy`）

**FK/IK 只能在“URDF 描述的几何模型”内自洽**。只要真实机器人和 URDF 不完全一致，IK 回放就会有残差（第 6 节会展开）。

## 3. FK（Forward Kinematics）是什么？本仓库怎么做 FK？

### 3.1 FK 的定义（一句话）

> FK：已知关节角 `q`，求末端位姿 `T(q)`。

你可以把 FK 理解成：

```
T_base^ee(q) = T_base^1(q1) · T_1^2(q2) · ... · T_{n-1}^n(qn) · T_n^ee
```

每一段变换来自 URDF 的“固定偏置 + 关节运动”。

### 3.2 录制端 FK 在哪里算？算完写到哪里？

在 `src/lerobot/robots/agilex/agilex.py`：

- `AgileXRobot.get_observation()` 每帧读取从臂（puppet）关节角
- 若 `record_ee_pose=True`：
  - 把关节角喂给 Pinocchio FK（`PiperPinocchioFK`）
  - 得到 `[x,y,z,roll,pitch,yaw]`
  - 写入：
    - `left_ee.x/y/z/roll/pitch/yaw`
    - `right_ee.x/y/z/roll/pitch/yaw`

这些字段最终进入数据集 `observation.state`，并在 `meta/info.json` 里体现为 `observation.state.shape=[26]`。

### 3.3 `PiperPinocchioFK` 做了哪些关键工程动作？

在 `my_ik/piper_pinocchio_kinematics.py`（`class PiperPinocchioFK`）里：

1. **加载 URDF**（Pinocchio build model）
2. **锁定不参与求解的关节**（`lock_joints=["joint7","joint8"]`）  
   这一步把“手指/夹爪内部关节”锁死，只剩机械臂 6 个关节参与 FK/IK。
3. **给模型添加一个末端 frame：`ee`**  
   通过 `ee_rotation_rpy` 与 `ee_translation`，把“你关心的末端坐标系”明确写进模型。
4. 每帧调用：
   - `pin.forwardKinematics(...)`
   - `pin.updateFramePlacements(...)`
   - 从 `oMf[ee_frame_id]` 拿到末端位姿

> 你可以把“添加 `ee` frame”理解成：  
> **我不一定要把末端定义为 URDF 里现成的某个 link（比如 gripper_base），我也可以在最后一个关节上定义一个更符合控制习惯的 TCP 坐标系。**

### 3.4 本仓库的“ROS -> URDF 关节映射”为什么重要？

真实系统里，读到的关节角（ROS topic）不一定和 URDF 的关节零位/正方向完全一致。

所以 `AgileXRobot.get_observation()` 会做映射：

```
q_urdf = q_ros * signs + offsets
```

对应配置项在 `src/lerobot/robots/agilex/config_agilex.py`：

- `left_joint_signs/right_joint_signs`（长度 6）
- `left_joint_offsets_rad/right_joint_offsets_rad`（长度 6）

如果你把这些配错了，你录制出来的 `*_ee.*` 也会是“错的但自洽”的，回放误差会非常大。

## 4. IK（Inverse Kinematics）是什么？本仓库怎么做 IK？

### 4.1 IK 的定义（一句话）

> IK：已知目标末端位姿 `T*`，求关节角 `q` 使得 `T(q) ≈ T*`。

IK 通常：

- 没有解析解（或解析解太复杂）
- 有多个解（同一个末端位姿可能对应多种“肘上/肘下”姿态）
- 甚至没有解（目标超出工作空间或触碰关节限位）

所以工程上大多使用**数值 IK**：迭代/优化求解。

### 4.2 本仓库 Pinocchio IK 的核心思想：把 IK 写成一个带约束的优化问题

在 `my_ik/piper_pinocchio_kinematics.py`（`class PiperPinocchioIK`）中：

1. 变量：`q`（长度 = 机械臂 DOF，这里是 6）
2. 参数：目标位姿 `T_target`（4x4）
3. 误差：使用 `log6( T_current^{-1} · T_target )` 得到 6 维误差向量  
   - 前 3 维近似对应位置误差
   - 后 3 维近似对应姿态误差
4. 目标函数（简化理解）：

```
min_q  pose_weight * ||pose_error(q)||^2  +  regularization_weight * ||q||^2
```

5. 约束：关节上下限（从 URDF 读取）
6. 求解器：`casadi.Opti()` + IPOPT
7. 工程增强：
   - 上一帧解作为初值（warm start）
   - 若解跳变过大（>30°）则重置初值，避免发散
   - 求解失败返回上一帧解（`history_data`），并标记 `success=False`

这就是你回放时“每一帧做 IK”的来源。

### 4.3 回放端 IK 在哪里调用？输入是什么？输出是什么？

在 `src/lerobot/scripts/lerobot_replay_ee.py`：

- 从数据集 `observation.state` 读取 `left_ee.* / right_ee.*`
- 拼成 `xyzrpy = [x,y,z,roll,pitch,yaw]`
- 调 `ik.inverse_kinematics(xyzrpy, q_init)` 得到 `q_urdf_rad`
- 再做一次“URDF -> ROS”映射：

```
q_ros = (q_urdf_rad - offsets) / signs
```

- 把 `q_ros` 写回 action 字典，调用 `robot.send_action(action)`

## 5. 端到端流程：record_ee_pinocchio / replay_ee_pinocchio 做了什么？

### 5.1 录制脚本 `record_ee_pinocchio.sh`（你录到的“末端数据”到底是什么）

你打开了这些关键参数：

- `--robot.record_ee_pose=true`
- `--robot.kinematics_backend=pinocchio`
- `--robot.kinematics_urdf_path=...piper_description.urdf`
- `--robot.pinocchio_lock_joints=["joint7","joint8"]`
- `--robot.ee_rotation_rpy=[0.0,-1.57,-1.57]`
- `--robot.ee_translation=[0.0,0.0,0.0]`

因此录制的每一帧里：

- `observation.state[:14]`：双臂关节（含 gripper）
- `observation.state[14:26]`：双臂 EE pose（由 FK 算出来的 xyzrpy）
- `action`：仍是双臂关节目标（用于训练/监督学习）

### 5.2 回放脚本 `replay_ee_pinocchio.sh`

它最终运行：

`python -m lerobot.scripts.lerobot_replay_ee ... --ik.backend=pinocchio`

脚本/代码里几个对安全和效果很关键的开关：

- `--dry_run=true/false`：不发动作，仅验证 IK 能否解算、速度是否过大
- `--dataset.fps=...`：回放时的下发频率（**时间缩放**，不会下采样帧）
- `--ik.arms=left/right/both`：只控制一只手时，另一只手保持初始姿态（更安全）
- `--ik.max_iter/--ik.tol/...`：IK 求解器参数
- `--preflight=true`：回放前对末端轨迹做速度粗检查（防止危险轨迹）

## 6. 为什么“能回放但有误差”？误差从哪来？怎么定位？

你现在观察到的“回放误差”，基本都可以归到三类：**模型误差、解算误差、执行误差**。

### 6.1 模型误差（URDF/TCP/映射不准确）

这是最常见、也是影响最大的部分：

1. **URDF 和真实机械尺寸不一致**（连杆长度、关节零位、装配偏差）
2. **EE/TCP 定义不准确**（工具装上去了，但 `ee_translation/rotation` 没更新）
3. **signs/offsets 映射不准确**（ROS 角与 URDF 角不一致）
4. **左右臂基座外参没有建模**  
   当前实现对左右臂各自用同构 URDF 做 FK/IK，并不提供“两臂在同一全局坐标系的外参”。  
   如果你肉眼看“左右臂协作”会更容易感受到误差。

定位建议（从快到慢）：

- 先做“数学闭环”验证：用同一个 URDF
  - 随机取一些 `q` 做 FK 得 `T`
  - 再把 `T` 喂给 IK 得 `q_hat`
  - 检查 `q_hat` 与 `q` 或 `FK(q_hat)` 与 `T` 的误差  
  这能排除“代码实现错误”，把问题缩小到“模型与真实不一致”。

### 6.2 解算误差（IK 数值迭代的残差/局部最优/不收敛）

数值 IK 一定有残差和失败概率，典型原因：

- 目标位姿接近奇异点（雅可比病态）
- 目标姿态 rpy 表示在该区域不稳定
- 求解器迭代次数不够或容差太松
- 初值不好导致跳到另一个解支（肘上/肘下切换）

对应你可以调的参数（`replay_ee_pinocchio.sh`）：

- 增大 `--ik.max_iter`
- 收紧 `--ik.tol`
- 调整 `--ik.pose_weight`（更强调贴近目标位姿）
- 调整 `--ik.regularization_weight`（更强调姿态连续/关节不过度摆动）

### 6.3 执行误差（机器人控制跟不上/限速/延迟）

即使 IK 算得“数学上很好”，机器人也未必跟得上：

- 电机/驱动器是带宽有限的系统（跟踪误差）
- `AgileXConfig.max_relative_target` 会对单步关节变化做裁剪（安全限速器）
- 回放频率过高（30Hz）时，硬件可能追不上快速轨迹

对应你可以做的事情：

- 回放先用更慢的 `--dataset.fps`（比如 10 或 15）
- 适当调大 `--robot.max_relative_target`（安全评估后再做）
- 轨迹本身速度太快就不要强行回放（先做平滑/降速）

## 7. 迁移到“下一台机器人”：哪些代码能复用？迁移清单是什么？

把方案拆成三层，你会更容易迁移：

1. **设备层（通讯与执行）**：怎么读关节、怎么发关节
2. **运动学层（模型与解算）**：URDF + FK/IK
3. **数据层（记录/回放/训练接口）**：怎么把东西写进 dataset、怎么从 dataset 读出来

### 7.1 强复用（基本可以原封不动搬走）

- `src/lerobot/scripts/lerobot_replay_ee.py` 的“总体回放结构”  
  流程（读 dataset → 取 EE → IK → 发动作）非常通用。
- `my_ik/piper_pinocchio_kinematics.py` 的“Pinocchio + Casadi IK 模板”  
  这套“建模 + addFrame + log6 误差 + 约束优化”的写法对很多串联机械臂都通用。
- 录制端“把 FK 结果写入 observation”这件事本身  
  LeRobot 的 dataset 结构不会因为换机器人而改变。

### 7.2 必改（跟具体机器人强绑定）

1. **URDF 与 package_dirs**
   - `--robot.kinematics_urdf_path`
   - `--robot.pinocchio_package_dirs`
2. **参与 IK/FK 的关节集合**
   - 哪些关节要锁（gripper/finger），`pinocchio_lock_joints`
3. **EE frame 的定义**
   - `ee_rotation_rpy`
   - `ee_translation`
   - EE 挂在哪个父关节上（目前 `PiperPinocchio*` 写死挂在 `"joint6"`）
4. **关节顺序与映射**
   - 真实硬件/SDK 报的关节顺序 ≠ URDF 的 joint 顺序很常见
   - signs/offsets 必须重新标定
5. **机器人接口层**
   - `get_observation()` 从哪里读关节角
   - `send_action()` 怎么发关节角（控制模式、限速、限位）

### 7.3 推荐迁移步骤（最少踩坑版本）

1. 用新机器人的 URDF 在纯 Python 脚本里把 pinocchio model build 起来（先不碰硬件）
2. 确认你能对给定 `q` 做 FK，输出 `T`
3. 做 FK→IK→FK 的闭环数值验证（不用上机）
4. 再接入硬件读 `q_ros`，标定 signs/offsets，使 FK 输出“看起来像真实末端”
5. 最后上机做回放：先 `--dry_run=true`，再 `--ik.arms=single`，再 `--dataset.fps` 降速

## 8. “这个是 6 轴的，如果是 7 轴怎么办？”（非常关键）

### 8.1 概念差异：6 轴 vs 7 轴 IK 的本质区别

- **6 轴（6-DOF）机械臂**：末端位姿有 6 个自由度（位置 3 + 姿态 3），理论上“方程数≈变量数”，常见是“多个离散解”。
- **7 轴（7-DOF）机械臂**：变量比约束多 1 个，是“冗余机械臂”，满足同一个末端位姿的关节解**有无穷多组**。必须引入“偏好”（例如靠近某个舒适姿态、远离限位、保持肘部朝向等），才能选出一个解。

你当前的 Pinocchio IK 本质是优化：

```
min pose_error + regularization
```

其中 `regularization` 这项就是用来在“多解/冗余”里选解的。对 7 轴尤其重要。

### 8.2 这套代码里哪些地方“写死了 6 轴”？

如果你要迁移到 7 轴（或更多自由度），先找这些“6 的假设”：

1. **回放脚本只控制前 6 个关节**
   - `src/lerobot/scripts/lerobot_replay_ee.py`：`ARM_JOINT_NAMES = JOINT_NAMES[:6]`
2. **录制端 FK 只取前 6 个关节做解算**
   - `src/lerobot/robots/agilex/agilex.py`：`q_left_ros = left_state.position[:6]`（右臂同理）
3. **配置里的 signs/offsets/tcp_offset 长度固定为 6**
   - `src/lerobot/robots/agilex/agilex.py`：`_validate_len(... expected_len=6)`
   - `src/lerobot/robots/agilex/config_agilex.py`：`left_joint_signs` 等默认长度是 6
4. **Pinocchio 模型里 EE frame 挂在 `joint6` 上**
   - `my_ik/piper_pinocchio_kinematics.py`：`getJointId("joint6")` + `addFrame("ee", ...)`

> 结论：现在这套实现是“6 轴臂 + gripper（另算）”。  
> 要支持 7 轴，需要把这些地方的“6”改成“机械臂自由度 N”。

### 8.3 7 轴怎么改？（工程上推荐的改法）

这里给一套“尽量少改、但能长期维护”的改法思路。

#### 8.3.1 先统一一个概念：N = 机械臂参与 IK/FK 的关节数

- 6 轴：`N=6`
- 7 轴：`N=7`
- gripper/finger 不属于 IK/FK（通常锁死或独立控制）

#### 8.3.2 对 Pinocchio FK/IK：让它变成“对任意 N 都成立”

你当前的 `PiperPinocchioFK/IK` 已经具备“优化法 IK 能处理冗余”的基础，但需要把“父关节名”和“输入维度”参数化：

建议你在迁移时做两处抽象（不一定要立刻改代码，但要按这个思路走）：

1. **把 `ee` frame 的父关节从写死 `"joint6"` 改成可配置**
   - 6 轴时：父关节 = 最后一个关节（`joint6`）
   - 7 轴时：父关节 = 最后一个关节（例如 `joint7`）
2. **用 `self.nq` 驱动所有维度判断，不要写死 6**
   - FK/IK 输入 q 的长度应该等于 `reduced_robot.model.nq`

> 重点：对 7 轴来说，IK 不是“解不出来”，而是“解太多”。  
> 你现在的优化目标里有 `regularization_weight * ||q||^2`，它会偏好关节角整体更小的解。  
> 工程上更常用的是 `||q - q_nominal||^2`（靠近一个舒适姿态），这样解更稳定、也更像人类习惯姿态。

#### 8.3.3 对录制端：FK 输入从 `[:6]` 改成 `[:N]`

需要你做到：

- 读取从臂关节角 `q_ros` 时取前 `N` 个（而不是 6）
- `signs/offsets` 配置也变为长度 `N`
- FK 仍输出 6 维末端位姿（位置 3 + 姿态 3），不随 N 变化

> 注意：`observation.state` 里 joint 部分会变长（每个手臂多一个关节）。这会改变数据集 schema。

#### 8.3.4 对回放端：IK 输出 `N` 个关节角，并全部下发

对应改动：

- `ARM_JOINT_NAMES = JOINT_NAMES[:N]`
- `PiperPinocchioIK.inverse_kinematics()` 返回长度 `N` 的 `q_urdf_rad`
- 映射到 ROS 后，循环写入 `action[f"{side}_{jn}.pos"] = ...` 也要覆盖 `N` 个关节

#### 8.3.5 7 轴 IK 要额外注意的“选解策略”（不然会抖/跳解）

冗余臂最怕两件事：

1. **解在不同冗余维度之间来回跳**（末端看起来跟得上，但肘部疯狂摆）
2. **靠近关节限位时突然换解支**（造成突变）

工程上常用的“稳定选解”策略（你可以逐个加）：

- 用上一帧解作为初值（你现在已经做了）
- 正则项用 `||q - q_nominal||^2` 或 `||q - q_prev||^2`（比 `||q||^2` 更稳）
- 增加“远离限位”代价（例如惩罚接近 lower/upper 的关节）
- 加权正则（某些关节更不希望动）

## 9. 补充：除了“优化法 IK”，你还应该知道 Jacobian 迭代法（帮助你理解本质）

很多运控工程师最常见的 IK 心智模型是 Jacobian：

1. 在当前关节角 `q` 下算出末端位姿 `T(q)`，以及雅可比矩阵 `J(q)`（末端速度对关节速度的线性近似）。
2. 设定末端误差 `e`（位置+姿态）
3. 用最小二乘求一个关节增量：

```
Δq = J^T (J J^T + λI)^-1 e
q ← q + Δq
```

这个就是“阻尼最小二乘（DLS）”IK 的核心。

你现在的 Casadi+IPOPT 优化法可以理解为：

- Jacobian 法在做“局部线性化 + 迭代更新”
- 优化法在做“把问题一次性写成带约束最小化”，求解器内部也会用导数信息迭代

优化法的优点是：**关节限位、正则、偏好姿态**更容易加进去；缺点是：依赖求解器、速度可能慢一些。

## 10. 给零基础同学的“最小自检清单”（你拿去给新人最有用）

当你怀疑 FK/IK 不对时，按下面顺序排查：

1. **单位检查**
   - 关节角是否都是 rad？
   - EE 的 xyz 是否是 m？
2. **关节顺序检查**
   - 你读到的第 i 个关节，是否真对应 URDF 的第 i 个关节？
3. **signs/offsets 检查**
   - 把机器人停在一个你“肉眼知道末端大概朝向”的姿态，看看 FK 输出是否合理
4. **ee_rotation_rpy/ee_translation 检查**
   - 你记录的末端坐标系是否真是你想要的 TCP？
5. **FK→IK→FK 闭环检查（不用上机）**
   - `T = FK(q)`
   - `q_hat = IK(T)`
   - `T_hat = FK(q_hat)`
   - 看 `T_hat` 和 `T` 的差（这一步能把“代码问题”和“真实模型误差”分开）
6. **回放安全**
   - 先 `--dry_run=true`
   - 再 `--ik.arms=right` 单臂
   - 再降低 `--dataset.fps`
   - 最后才 both arms + 提速

## 11. 附：你现在这套方案最核心的参数该怎么理解？

### 11.1 录制端（FK）参数

在 `record_ee_pinocchio.sh`：

- `--robot.kinematics_urdf_path`：URDF 文件路径（模型真不真，全靠它）
- `--robot.pinocchio_lock_joints`：锁住哪些关节（通常是手指）
- `--robot.ee_rotation_rpy / --robot.ee_translation`：定义 `ee` frame（TCP 定义的一部分）
- `--robot.left_tcp_offset_xyzrpy / right_tcp_offset_xyzrpy`：在 FK 输出基础上再叠加 TCP（本仓库默认 0）

### 11.2 回放端（IK）参数

在 `replay_ee_pinocchio.sh`：

- `--ik.max_iter`：最大迭代次数（大一点更稳但更慢）
- `--ik.tol`：收敛容差（小一点更准但更慢）
- `--ik.pose_weight`：更强调贴合末端位姿（大则更贴合但更可能抖/不稳定）
- `--ik.regularization_weight`：更强调关节姿态“温和/不乱跑”（大则更稳但末端跟踪误差可能更大）
- `--dataset.fps`：回放频率（越小越安全、越容易跟踪）
