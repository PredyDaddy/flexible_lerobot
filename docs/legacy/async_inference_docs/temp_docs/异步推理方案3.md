# 异步推理方案3：PolicyServer/RobotClient（动作分块 + 解耦推理与执行）

## 1. 背景：你现在的推理方式（同步、单臂）

你目前是用单进程同步循环做推理与控制（推理时机器人控制环会被阻塞），典型用法如下：

```bash
# 左臂推理
python agilex_infer_single_cc_vertical.py \
  --checkpoint outputs/act_agilex_left_box/checkpoints/030000/pretrained_model \
  --arm left --fps 30 --binary-gripper

# 右臂推理
python agilex_infer_single_cc_vertical.py \
  --checkpoint outputs/act_agilex_right_black_cup/checkpoints/last/pretrained_model \
  --arm right --fps 30 --binary-gripper
```

在 30Hz（33.3ms/step）下，如果「取图/预处理/推理/发指令」任一环节耗时接近或超过 33ms，就会出现：
- FPS 抖动、控制不稳定（miss deadline）
- 推理偶发慢一帧就“卡一下”
- GPU/CPU 互相抢占导致更难稳定到目标频率

## 2. 是否能“加速”？能，但要看瓶颈在哪里

`src/lerobot/async_inference/` 的核心价值不是“把单次 forward 变快”，而是两点：

1) **动作分块（Action Chunking）降低推理调用次数**  
很多策略（包含 ACT）一次前向可以输出一个 action chunk（多个未来动作）。如果你把每步推理改成“每 N 步推理一次，拿到 N 个动作放队列里慢慢执行”，平均到每步的推理成本会显著下降。

2) **推理与执行解耦（异步）消除 wait-for-inference 空档**  
Robot 端按固定频率执行队列里的动作；与此同时，PolicyServer 端在后台算下一段动作。只要“算下一段”的总耗时小于队列消耗速度，就不会卡顿。

注意：如果你当前主要瓶颈在 ROS 相机取图/网络传输/机器人侧阻塞，而不是模型推理，那么异步方案通常带来的是**更稳的控制频率**，而不是显著提高上限 FPS。

## 3. 方案3：LeRobot 异步推理架构（gRPC）

方案3直接复用现有实现：
- `lerobot.async_inference.policy_server`：加载 policy，在 GPU/高算力机器上做推理，输出 action chunk
- `lerobot.async_inference.robot_client`：连接真实机器人，维持控制环与 action queue，按需发送 observation

数据流（简化）：

```text
AgileXRobot(取图/读关节) -> RobotClient(发obs) ==gRPC==> PolicyServer(GPU推理,出chunk)
AgileXRobot(执行动作)   <- RobotClient(action queue) <==gRPC==  PolicyServer(回传chunk)
```

关键机制：
- **队列只保留“最新 observation”**（PolicyServer 端 `Queue(maxsize=1)`），旧的会被丢弃，避免堆积导致延迟越来越大。
- **client 侧有 action queue**：队列里还有动作时继续执行；当队列比例低于 `chunk_size_threshold` 时开始发送新观测，提前“补货”。
- **must_go**：当队列空了，下一帧观测会强制送去推理（不做“相似观测过滤”），避免停住。

## 4. 落地到你的 AgileX 单臂 ACT checkpoint：需要先解决“特征对齐”

你现在的 checkpoint 是单臂模型：  
- 观测 state 只有 7 个关节（例如 `left_*` 或 `right_*`）
- action 也只有 7 维（含 gripper）

但 `AgileXRobot` 默认是双臂：  
- `observation.state` 是 14 维（左 7 + 右 7）
- `action` 也是 14 维

异步推理这套代码要求：**RobotClient 送给 PolicyServer 的 observation/state 维度必须与 policy 训练时一致**；否则一定 shape mismatch。

因此有两条落地路线：

### 路线 A（最省改代码）：使用“双臂策略”（14 维）

如果你后续能训练/导出一个同时控制双臂（14 维 action）的策略，那么基本可以直接用 `AgileXRobot` 接入 RobotClient（相机 key 对齐即可）。

适合：你本来就想双臂协同，或者你不介意重训。

### 路线 B（贴合你当前用法）：保留“单臂策略”，在 client 侧做一个“单臂适配器”

思路和你现有脚本一致：  
- policy 只输出单臂 7 维
- 另一只臂保持当前位置（或保持初始位）

做法是在 RobotClient 侧提供一个 **Single-Arm Wrapper**：
- `observation_features` / `action_features` 只暴露一个臂的 7 维（+ 对应相机，如 `camera_front` + `camera_left/right`）
- `send_action()` 内部把 7 维动作补成 14 维：另一臂从 `ros_bridge.get_puppet_state()` 读当前值填回去
- 可在 `send_action()` 里复用你现在的 `binary-gripper` 逻辑（阈值化夹爪）

实现落点建议（任选其一）：
1) 新增一个机器人类型（推荐更“正统”）：`agilex_single_arm`（RobotConfig + Robot 实现），让 `python -m lerobot.async_inference.robot_client ...` 直接可用。  
2) 新写一个脚本（最快落地）：参考 `examples/tutorial/async-inf/robot_client.py`，直接在脚本里构造 wrapper robot + 复用 `RobotClient` 的核心逻辑（或复刻一份简化 client）。

如果你希望我把这条路线的 wrapper + 可运行 client 脚本也补上，我可以按你目前 `agilex_infer_single_cc_vertical.py` 的行为做一个 `scripts/agilex_async_client_single_arm.py`。

## 5. 参数怎么调才“加速/更稳”（核心就三项）

### 5.1 `actions_per_chunk`：一次推理输出多少步

经验值（30Hz）：
- 先从 **30~60** 开始（覆盖 1~2 秒动作）
- 如果 server 推理偶发慢：增大到 80~100
- 如果你希望更“实时”（更依赖新观测）：减小到 10~30

一个简单的约束：  
`server_端生成一个chunk的耗时 < actions_per_chunk * dt * 安全系数`  
其中 `dt = 1/fps`，安全系数建议 0.6~0.8（留出网络/抖动余量）。

### 5.2 `chunk_size_threshold`：队列消耗到多少比例就开始发新观测

client 侧触发条件是：`queue_size / action_chunk_size <= chunk_size_threshold`

经验值：
- **0.7~0.9**：更保守，提前补货，卡顿风险低（推荐先用）
- **0.3~0.6**：更省带宽/更少观测发送，但对 server 性能更敏感

### 5.3 `aggregate_fn_name`：动作重叠时怎么融合

默认 `weighted_average` 更平滑；如果你想更“跟手”（尤其是单臂抓取），可以尝试：
- `latest_only`：永远用最新动作（更激进）
- `conservative`：更保守（更稳）

## 6. 推荐的落地步骤（减少踩坑）

1) 先测 baseline：记录你当前同步脚本里每步 `get_observation` / `predict_action` / `send` 的耗时分布，确认瓶颈。  
2) 先本机跑通方案3（localhost）：PolicyServer 和 RobotClient 都在同一台机器上，先验证“队列不断粮”。  
3) 再拆分到两台机器：robot 侧只负责 ROS/相机/控制环，server 侧用 GPU 推理（延迟更稳定）。  
4) 最后再考虑双臂：两条单臂模型可以开两个 server（不同 port）+ client 侧融合；或者直接训练一个双臂策略。
