# LeRobot 数据网页端可视化最终方案（融合方案1 & 方案2）

> **状态说明 (lerobot 0.4.3 baseline)**  
> 本文档描述的是“新增独立 `vis/` 模块”的方案。当前仓库 0.4.3 baseline 并不包含 `vis/` 实现与 `vis.sh`，因此本文不提供可直接执行的步骤。  
> 在路线 A 下，建议优先使用内置的 `lerobot-dataset-viz` 完成数据集可视化与质检。  

> 结论：需要一个独立 `vis/` 模块，提供“数据集浏览器（概览/列表/筛选/QC）+ episode 详情（多相机视频裁剪播放 + timeseries 曲线）”。实现成败的关键是：**LeRobotDataset v3.0 的视频是“多 episode 拼接到同一 mp4”**，必须用 episodes 元数据里的 `from_timestamp/to_timestamp` 在前端裁剪播放区间。

## 1. 背景与痛点（基于仓库现有实现）

当前可视化主要依赖 `lerobot-dataset-viz`（`src/lerobot/scripts/lerobot_dataset_viz.py`）+ `vis.sh` 的工作流：

- 每次只能 `--episode-index N` 看一条轨迹
- 以 Rerun Viewer 为中心，缺少“数据集级别入口”（episode 列表/筛选/缩略图/异常标记）
- 远程查看不便（虽然 Rerun 支持 `--mode distant` 提供 Web Viewer，但仍然是“单 episode 深入”，无法高效扫完数据集）

因此需要补齐一个面向“数据集浏览/质检/QC”的产品形态，而不是替代 Rerun 的深度分析能力。

## 2. 目标 / 非目标 / 约束

### 2.1 目标（网页端必须做到）

- 浏览器访问：`http://HOST:PORT/` → 选数据集 → 选 episode → 连续播放（支持自动下一条）
- 数据集概览：info/stats/tasks + episode 列表（分页/筛选/排序）
- episode 详情：多相机视频同屏 + action/state/reward/done/success 曲线（支持下采样）
- 质检（QC）：把 `docs/train_agilex/数据检查指南.md` 的关键项产品化成 badge 与筛选条件

### 2.2 非目标（先不做）

- 不复刻 Rerun 的全部交互能力（保留“导出 .rrd”作为可选增强）
- 不做在线转码/切片（优先复用原始 mp4 + Range seek）
- 不引入复杂前端构建链（MVP 优先原生 JS + 模板）

### 2.3 硬约束

- 不修改 `src/lerobot/` 与数据落盘格式；新增独立模块 `vis/`
- 离线可用（支持 `HF_HUB_OFFLINE=1`），默认读取本地 HuggingFace cache（`HF_LEROBOT_HOME`）
- 视频播放走浏览器原生 `<video>`（必须正确支持 HTTP Range）

## 3. 数据结构与关键事实（决定实现路线）

### 3.1 目录结构（LeRobotDataset v3.0）

```
dataset_root/
  meta/
    info.json
    stats.json
    tasks.parquet
    episodes/
      chunk-000/file-000.parquet
      ...
  data/
    chunk-000/file-000.parquet
    ...
  videos/
    <video_key>/
      chunk-000/file-000.mp4
      ...
```

关键模板/常量见 `src/lerobot/datasets/utils.py`：

- `DEFAULT_DATA_PATH = "data/chunk-{chunk_index:03d}/file-{file_index:03d}.parquet"`
- `DEFAULT_VIDEO_PATH = "videos/{video_key}/chunk-{chunk_index:03d}/file-{file_index:03d}.mp4"`

### 3.2 episodes 元数据必须使用的字段

`meta/episodes/*.parquet`（通过 `load_episodes` 加载）至少包含：

- `episode_index`、`length`、`tasks`（字符串列表）
- `dataset_from_index`、`dataset_to_index`
- 对每个 `video_key`：
  - `videos/<video_key>/chunk_index`、`videos/<video_key>/file_index`
  - `videos/<video_key>/from_timestamp`、`videos/<video_key>/to_timestamp`

### 3.3 最关键事实：多 episode 拼接到同一个 mp4

视频写入逻辑见 `src/lerobot/datasets/lerobot_dataset.py:_save_episode_video`：

- episode 先编码成临时 mp4
- 再拼接到“当前 chunk/file 的 mp4”里（达到大小阈值才换新文件）
- 把该 episode 在 mp4 内的时间偏移写到 episodes 元数据：`from_timestamp/to_timestamp`

因此网页端必须：

- 播放“整段 mp4 文件”（canonical media URL 以 `chunk_index/file_index` 为主，利于浏览器缓存复用）
- 用 `from_timestamp/to_timestamp` 在前端裁剪播放区间（并跨相机同步）

## 4. 总体架构（Browser ↔ FastAPI ↔ Dataset Root）

```
Browser (HTML/JS)
  - 列表页：缩略图/筛选/QC badge
  - 详情页：多相机 <video> + 曲线
        │  HTTP/JSON + HTTP Range
        ▼
FastAPI (vis/web)
  - 读取 meta/data（只读）
  - 提供 JSON API（episodes/timeseries/QC）
  - 静态暴露 mp4（必须 Range）
        │  filesystem
        ▼
dataset_root/ (meta/, data/, videos/)

可选离线预计算：
  vis/scripts/precompute.py → vis/.cache/<dataset_id>/{thumbs,qc,summary}.*
```

## 5. 技术选型（从方案1吸收工程化，从方案2保证正确性）

- 后端：FastAPI + Uvicorn（类型友好、易部署）
- 数据访问：
  - 元数据：复用 `lerobot.datasets.utils.load_info/load_stats/load_tasks/load_episodes`
  - timeseries：优先 `pyarrow.dataset` 做列裁剪 + episode 过滤；也可复用 `load_nested_dataset(..., episodes=[...])`
- 视频：不转码，直接 `FileResponse`/静态文件服务 + Range（必要时按方案1补 206 实现）
- 前端：Jinja2 模板 + 少量原生 JS（MVP 不引入构建链）
- 图表：Chart.js 或本地 vendored Plotly/ECharts（离线、无需 CDN）
- 缓存：`vis/.cache/`（缩略图、episode summary、QC 结果）

## 6. `vis/` 模块目录与职责（建议）

```
vis/
  README.md
  requirements.txt
  web/
    main.py               # CLI：启动服务（root/repo-id/host/port）
    dataset_access.py     # 识别 dataset root、加载 meta、读 timeseries（列裁剪/下采样）
    api.py                # 路由：pages + JSON API + media
    qc.py                 # QC 规则（从 数据检查指南 抽象）
    cache.py              # 读写 vis/.cache（thumbs/summary/qc）
    templates/
      index.html          # 数据集选择/概览/列表入口
      dataset.html        # 某数据集 episode 列表页（可选：也可合并到 index）
      episode.html        # episode 详情页
    static/
      app.css
      app.js
      vendor/             # plotly.min.js / echarts.min.js（可选）
  scripts/
    precompute.py         # 预生成缩略图 + summary + qc（强烈推荐）
```

## 7. 后端设计（参数、路由、API）

### 7.1 启动参数与 dataset root 识别

入参：`--root` + 可选 `--repo-id`

- `--root` 既可以是：
  - dataset root（包含 `meta/info.json`）
  - HF cache root（包含 `<user>/<repo>/meta/info.json`）
- 识别逻辑：
  - 若 `root/meta/info.json` 存在：dataset root = `root`
  - 否则若 `root/<repo-id>/meta/info.json` 存在：dataset root = `root/<repo-id>`
  - 否则报错并提示正确路径

默认 root 建议用 `HF_LEROBOT_HOME`（见 `src/lerobot/utils/constants.py`）。

### 7.2 页面路由（用户直接访问）

- `GET /`：数据集列表或默认数据集入口
- `GET /datasets/<user>/<repo>`：该数据集 episode 列表页
- `GET /datasets/<user>/<repo>/episodes/<ep>`：episode 详情页

> 也可做成单页应用（hash 路由），但 MVP 用模板更省心。

### 7.3 媒体路由（canonical URL，利于缓存复用）

- `GET /media/datasets/<user>/<repo>/videos/<video_key>/chunk-<chunk_index>/file-<file_index>.mp4`
  - 只做安全映射到 `dataset_root/videos/<video_key>/chunk-xxx/file-yyy.mp4`
  - 必须支持 Range（`206 Partial Content` + `Accept-Ranges`）
  - 必须防目录穿越：`resolve()` 后验证仍在 dataset root 内

### 7.4 JSON API（MVP 最小集合 + 可扩展）

#### 数据集

- `GET /api/datasets`
  - 返回 root 下可发现的数据集列表（`<user>/<repo>`）
- `GET /api/datasets/<user>/<repo>/info`
  - 返回 `meta/info.json` 的关键字段 + 推导出的 `video_keys/camera_keys`
- `GET /api/datasets/<user>/<repo>/stats`（可选）
  - 返回 `meta/stats.json`

#### Episodes

- `GET /api/datasets/<user>/<repo>/episodes?offset&limit&task&has_issue&min_len&max_len`
  - 返回 episode 摘要列表（用于列表页）
  - 建议字段：`episode_index,length,duration_s,tasks,thumbnail_url,qc_summary`
- `GET /api/datasets/<user>/<repo>/episodes/<ep>`
  - 返回该 episode 的视频裁剪信息（核心）：
    - `fps`
    - `videos[video_key] = {url, from_timestamp, to_timestamp, duration_s}`

#### Timeseries（曲线）

- `GET /api/datasets/<user>/<repo>/episodes/<ep>/timeseries?keys=action,observation.state,next.reward,next.done&stride=5&dims=...`
  - 必须支持 `stride/downsample`（避免 JSON 过大）
  - `keys`：逗号分隔；`dims`（可选）用于 vector 特征维度选择（例如 `action:0-6,observation.state:0-13`）
  - 返回建议包含：`timestamp/frame_index` + 选择的 keys（数组或按维度拆分）

#### 缩略图（推荐缓存）

- `GET /api/datasets/<user>/<repo>/episodes/<ep>/thumbnail?video_key=...&w=320`
  - 优先命中 `vis/.cache/.../thumbs/episode-xxxxxx.jpg`
  - 未命中时可懒生成并写入缓存（或直接返回 404 提示先跑 precompute）

## 8. 数据访问与性能策略（落地要点）

### 8.1 启动阶段只读元数据（快）

启动只加载：

- `meta/info.json`（fps、features、video_path 等）
- `meta/episodes/*.parquet`（episode 列表与视频裁剪信息）
- `meta/tasks.parquet`（可选）
- `meta/stats.json`（可选）

并用 LRU 缓存减少重复 IO。

### 8.2 timeseries 读取（列裁剪 + 下采样）

推荐实现顺序：

1. 用 `pyarrow.dataset.dataset(root/"data")`：
   - filter：`episode_index == ep`
   - columns：仅取 `timestamp, frame_index` + `keys` 请求列
2. 下采样：
   - 最简单：stride（`table.take(range(0, N, stride))`）
   - 可选增强：min/max 聚合下采样（保峰值，后续再做）
3. JSON 输出控制：
   - 默认自动 stride（例如把点数限制到 2k～5k）
   - 对 vector 列支持 `dims` 参数减少体积

### 8.3 缩略图与 QC 建议预计算（列表页体验关键）

列表页如果每条 episode 都临时解码首帧，会导致首屏慢/CPU 高。建议提供：

- `vis/scripts/precompute.py`：
  - 抽帧：优先 `ffmpeg`（参考 `docs/train_agilex/数据检查指南.md` 的用法），必要时回退 `decode_video_frames(...)`
  - 输出：
    - `vis/.cache/<dataset_id>/thumbs/episode-000123.jpg`
    - `vis/.cache/<dataset_id>/episodes_summary.json`
    - `vis/.cache/<dataset_id>/qc.json`

## 9. 前端交互设计（让“刷完整个数据集”真的高效）

### 9.1 列表页（数据集级入口）

- 信息密度：episode_index、tasks、length/duration、缩略图
- QC badge：timestamp/NaN/静止/视频等，一眼定位异常
- 筛选：task、异常、长度范围
- 快捷：点击进入详情；支持键盘上下切换

### 9.2 详情页（多相机裁剪播放 + 曲线）

**多相机视频**：

- 每个 `video_key` 一个 `<video>`
- `loadedmetadata` 后 seek：`video.currentTime = from_timestamp`
- 播放边界：监听 `timeupdate`，超过 `to_timestamp` 自动 pause；若开启“自动下一条”则跳转

**同步策略**（简单可靠）：

- 选一个 master（默认第一个相机）
- 计算 `t_rel = master.currentTime - from_master`
- 其他相机校正到 `from + t_rel`（允许小 drift，避免频繁强制 seek）

**曲线**：

- 默认展示 `action/observation.state/next.reward/next.done/next.success`
- API 强制 stride；前端支持选择维度/隐藏曲线

**快捷键建议**：

- `J / ←`：上一条 episode
- `K / →`：下一条 episode
- `Space`：播放/暂停
- `A`：自动下一条开关

## 10. QC（质检）规则（从“数据检查指南”抽象）

参考 `docs/train_agilex/数据检查指南.md`，建议 MVP 先做这几类：

- 时间戳稳定性：`diff(timestamp)` 超过 `2*(1/fps)` 计异常
- NaN/Inf：action/state 任意维度出现即异常
- 静止段：`diff(action)` 绝对值小于阈值的连续段过长（例如 >60 帧）
- 视频可播/可解码：可选离线用 `ffprobe` 扫描（或前端能否正常加载）

`qc.json` 建议结构：

- `episode_index -> {status: ok|warn|bad, reasons: [...], metrics: {...}}`

## 11. 与现有 Rerun 可视化的融合（可选增强）

网页端负责“浏览/筛选/QC”，Rerun 负责“单 episode 深入”：

- 在 episode 详情页提供“导出 .rrd”按钮：
  - 后端复用 `lerobot.scripts.lerobot_dataset_viz.visualize_dataset(..., save=True)`
  - 生成 `.rrd` 后返回下载链接

## 12. 里程碑（交付节奏）

- Phase 1（P0）：可用的浏览器刷数据集（列表 + 详情 + 多相机裁剪播放 + 自动下一条）
- Phase 2（P1）：预计算缩略图 + timeseries 曲线 + QC badge/筛选
- Phase 3（P2）：Rerun 导出、逐帧/标注、多数据集管理

## 14. 风险与对策（提前规避踩坑）

- Range 不正确导致无法 seek：优先使用框架内置的静态文件/`FileResponse` Range 支持；若版本不支持，再按方案1实现 206。
- episode 时长与帧数微不一致：播放边界用 `to-from`；曲线以 parquet 的 `timestamp/frame_index` 为准。
- JSON 过大：timeseries API 强制 stride/downsample + dims；列表页依赖预计算 summary。
- 静态文件泄露：严格限制仅暴露 `videos/` 与 `vis/.cache/`，并做 `resolve()` 校验。
