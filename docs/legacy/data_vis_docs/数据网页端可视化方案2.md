# 数据网页端可视化方案2（不改源码，新增 `vis/` 模块）

> **状态说明 (lerobot 0.4.3 baseline)**  
> 本文档描述的是“新增独立 `vis/` 模块”的方案。当前仓库 0.4.3 baseline 并不包含 `vis/` 实现与 `vis.sh`，因此本文不提供可直接执行的步骤。  
> 在路线 A 下，建议优先使用内置的 `lerobot-dataset-viz` 完成数据集可视化与质检。  

> 目标：把目前 `vis.sh` 里“每次只能 `lerobot-dataset-viz --episode-index N` 打开一条轨迹”的工作流，升级成**浏览器可点 URL** 的数据集浏览器：`http://localhost:PORT/` → 选 episode → 连续看完整个数据集（并支持快速质检/QC）。

---

## 1. 现状与痛点（基于仓库现有实现）

当前仓库已经提供了数据可视化工具：

- CLI 入口在 `pyproject.toml`：
  - `lerobot-dataset-viz="lerobot.scripts.lerobot_dataset_viz:main"`
- 实现位于 `src/lerobot/scripts/lerobot_dataset_viz.py`
  - 使用 Rerun（`rerun-sdk`）逐帧写入相机图像、action/state/reward 等标量
  - 通过 `LeRobotDataset(repo_id, episodes=[episode_index], root=..., tolerance_s=...)` 加载单个 episode
  - 通过 `EpisodeSampler` 用 `dataset.meta.episodes["dataset_from_index"/"dataset_to_index"]` 精确取该 episode 的帧范围

你现在的 `vis.sh`（仓库根目录）本质是：

```bash
lerobot-dataset-viz --repo-id ... --root ... --episode-index 0
```

痛点：

1. **只能“一条一条”**：要看完整个数据集，需要手动改 `--episode-index`，反复打开/关闭 viewer。
2. **缺少“目录/索引页”**：看数据集时通常需要“快速浏览→定位异常→深入查看”，当前流程没有数据集级别的总览入口（episode 列表、筛选、缩略图、异常标记）。
3. **复用成本高**：`lerobot-dataset-viz` 的 viewer 主要面向“单 episode 深入”，而不是“数据集浏览/审核”。

> 备注：`lerobot-dataset-viz` 支持 `--mode distant` 并调用 `rr.serve_web_viewer(...)`（见 `src/lerobot/scripts/lerobot_dataset_viz.py`），确实能用浏览器看 Rerun Web Viewer，但仍然是“单 episode”，无法解决“点 URL 看完整个数据集”的核心诉求。

---

## 2. 约束与设计原则

### 2.1 约束

- **不修改 `src/lerobot/` 源码**（包括不改 `lerobot-dataset-viz`、不改 dataset 存储格式）。
- 新增一个独立模块：`vis/`（仓库根目录已存在空目录）。
- 在 `(my_lerobot)` conda 环境中开发/运行（与仓库约定一致）。
- 数据通常在本地 HuggingFace cache：`~/.cache/huggingface/lerobot/<repo_id>/...`（见 `src/lerobot/utils/constants.py: HF_LEROBOT_HOME`）。

### 2.2 设计原则

- **以仓库现有数据结构为准**：完全遵循 LeRobotDataset v3.0 的落盘格式与语义。
- **优先“直接利用 mp4 + episodes 元数据”**，避免每次请求都 Python 解码视频帧（性能/并发更好）。
- **浏览优先**：先实现“快速扫完数据集”的效率工具（缩略图、快捷键、自动下一条、异常标记），再补充高级功能（逐帧细查、Rerun 导出）。
- **离线可用**：沿用你 `vis.sh` 的思路（`HF_HUB_OFFLINE=1`），并允许显式传 `--root`。

## 3. 必须理解的数据集结构（结合仓库实现）

网页端方案的关键在于：**LeRobotDataset 的视频不是“一集一个 mp4”**，而是**多个 episode 串到同一个 mp4** 里（按文件大小切分）。

### 3.1 目录结构（LeRobotDataset v3.0）

在 `src/lerobot/datasets/lerobot_dataset.py` 的 docstring 里给了典型布局；路径模板在 `src/lerobot/datasets/utils.py`：

- `DEFAULT_DATA_PATH = "data/chunk-{chunk_index:03d}/file-{file_index:03d}.parquet"`
- `DEFAULT_VIDEO_PATH = "videos/{video_key}/chunk-{chunk_index:03d}/file-{file_index:03d}.mp4"`
- episodes 元数据：`meta/episodes/chunk-xxx/file-yyy.parquet`
- 总信息：`meta/info.json`（`create_empty_dataset_info` 定义字段）
- 任务表：`meta/tasks.parquet`

### 3.2 episodes 元数据里有哪些“网页必须用”的字段

`LeRobotDatasetMetadata.load_episodes()`（`src/lerobot/datasets/utils.py:load_episodes`）加载的是 HF Dataset（Arrow/Parquet），并且会保留不以 `stats/` 开头的列，典型包含：

- 基本：
  - `episode_index`
  - `tasks`（列表）
  - `length`
  - `dataset_from_index`
  - `dataset_to_index`
  - `data/chunk_index`
  - `data/file_index`
- 对每个视频 key（例如 `observation.images.camera_front`）：
  - `videos/<video_key>/chunk_index`
  - `videos/<video_key>/file_index`
  - `videos/<video_key>/from_timestamp`
  - `videos/<video_key>/to_timestamp`

### 3.3 “多 episode 串一个 mp4”的关键逻辑在哪里

写视频的逻辑在 `src/lerobot/datasets/lerobot_dataset.py:_save_episode_video()`：

- 每个 episode 先编码成临时 mp4（`_encode_temporary_episode_video` / `encode_video_frames`）
- 然后：
  - 若当前视频文件未满：`concatenate_video_files([latest_path, ep_path], latest_path)`
  - 若满了：换新文件（更新 chunk/file index）
- 最终把偏移写入 episode 元数据：
  - `from_timestamp = latest_duration_in_s`
  - `to_timestamp = latest_duration_in_s + ep_duration_in_s`

读取视频帧的逻辑在 `LeRobotDataset._query_videos()`：

- `from_timestamp = ep[f"videos/{vid_key}/from_timestamp"]`
- `shifted_query_ts = [from_timestamp + ts for ts in query_ts]`
- `decode_video_frames(video_path, shifted_query_ts, tolerance_s, backend)`

**结论：网页端要正确播放某个 episode 的视频，必须使用 episodes 元数据提供的 `from_timestamp/to_timestamp` 做切片/偏移。**

## 4. 方案概述：`vis/` 模块提供“数据集浏览器”

### 4.1 总体架构

推荐采用“Python 后端 + 浏览器前端”的一体化本地服务：

- 后端：FastAPI/Starlette（只读访问 dataset root）
  - 负责读取 `meta/info.json`、`meta/episodes/*.parquet`、`meta/tasks.parquet`
  - 负责提供 mp4 静态文件（必须支持 HTTP Range，方便网页 `<video>` seek）
  - 提供 JSON API：episodes 列表、episode 详情、时间序列（action/state/reward/done 等）、质检结果
- 前端：Jinja2 模板 + 少量原生 JS（尽量不引入前端构建链）
  - 首页：episode 列表（缩略图 + 关键指标 + 筛选）
  - episode 页：多相机视频 + 时间轴 + action/state 曲线 + 异常提示 + 快捷键（上一条/下一条）

> 为什么不直接“全用 Rerun Web Viewer”：因为 Rerun 对“单 episode 深入”非常强，但缺少“数据集导航/缩略图/筛选/批量 QC”的产品形态；同时每次切换 episode 的数据推送也需要额外编排。

### 4.2 `vis/` 目录建议结构（不改 `src/lerobot`）

```
vis/
  README.md                      # 使用说明
  requirements.txt               # 仅 vis 需要的额外依赖（不动项目主依赖）
  web/
    __init__.py
    main.py                      # 启动入口：python -m vis.web.main ...
    dataset_access.py            # 读取 LeRobotDataset 元数据/数据
    qc.py                        # 数据质检（复用 docs/train_agilex/数据检查指南.md 的检查项）
    cache.py                     # 缓存缩略图/episode summary
    templates/
      index.html                 # episode 列表页
      episode.html               # 单 episode 详情页
    static/
      app.css
      app.js
      vendor/                    # 可选：离线 JS 库（如 plotly.min.js）
  scripts/
    precompute.py                # 预生成缩略图/QC/summary
```

## 5. 后端数据访问设计（强绑定仓库数据结构）

### 5.1 Dataset 识别与 root 规则

你现在习惯传完整 root：

- `--root /home/agilex/.cache/huggingface/lerobot/<repo_id>`

但 `LeRobotDataset` 的 docstring 写的是 “root/repo_id”（与代码 `self.root = Path(root)` 有出入，见 `src/lerobot/datasets/lerobot_dataset.py`）。

**建议在 `vis` 里做兼容：**

- 若 `root/meta/info.json` 存在：认为 `root` 就是 dataset root
- 否则若 `root/<repo_id>/meta/info.json` 存在：自动补齐成 `root/<repo_id>`
- 否则报错并提示用户给出正确路径

### 5.2 只读加载元数据（不必加载全量数据）

启动时只需：

- `info = load_info(root)`（`src/lerobot/datasets/utils.py:load_info`）
- `episodes = load_episodes(root)`（`src/lerobot/datasets/utils.py:load_episodes`）
- `tasks = load_tasks(root)`（`src/lerobot/datasets/utils.py:load_tasks`）

这些都能在不解码视频、不读全量 parquet 的情况下完成，启动很快。

### 5.3 episode 列表 API（首页）

建议提供：

- `GET /api/info`：返回 `meta/info.json` 的关键字段（fps、features、total_episodes、total_frames、robot_type）
- `GET /api/episodes`：返回 episode 列表（分页/筛选）
  - 每个 episode 返回：
    - `episode_index, length, duration_s=length/fps, tasks`
    - `dataset_from_index, dataset_to_index`
    - 对每个 `video_key`：
      - `video_relpath = meta.get_video_file_path(ep, video_key)`（等价逻辑见 `LeRobotDatasetMetadata.get_video_file_path`）
      - `from_timestamp, to_timestamp`
    - 可选：QC 结果摘要（见 5.6）

### 5.4 mp4 静态服务（浏览器播放的关键）

浏览器 `<video>` 为了 seek/拖动，通常会发 `Range` 请求；后端必须正确支持：

- `Accept-Ranges: bytes`
- `206 Partial Content`

FastAPI/Starlette 的 `FileResponse` 一般可以满足；建议把视频文件暴露在：

- `GET /media/videos/{video_key}/{chunk_index}/{file_index}.mp4`

并在实现中通过 episodes 元数据映射到真实路径，避免用户拼路径造成目录穿越：

1. 只允许访问 `root/videos/` 下文件
2. 使用 `path.resolve()` 验证仍在 `root` 目录内

### 5.5 episode 详情页所需 API

episode 详情页需要两类数据：

1) 视频播放信息（来自 episodes 元数据）：

- `GET /api/episodes/{ep}`：
  - 返回该 ep 的 `video_key -> {url, from, to, duration}` 映射

2) 标量/向量时间序列（来自 `data/*.parquet`）：

- `GET /api/episodes/{ep}/timeseries?keys=action,observation.state,next.reward,next.done,next.success&stride=1`
  - 数据来源建议两种实现路线（都不改源码）：
    - A) 直接用 PyArrow dataset 读 parquet（推荐，能做列裁剪/过滤）
      - `pyarrow.dataset.dataset(root/"data", format="parquet")`
      - filter：`episode_index == ep`
      - columns：只取 `timestamp, frame_index, action, observation.state, next.reward, next.done, ...`
    - B) 复用仓库的 `load_nested_dataset(..., episodes=[ep])`（见 `src/lerobot/datasets/utils.py:load_nested_dataset`）
      - 再从 HF Dataset 取列

`stride`/`downsample` 是必须的：否则 action/state 长度动辄几百～几千帧，JSON 体积很大。

### 5.6 QC（质检）能力：把“数据检查指南”变成网页 badge

仓库已有一套手工检查清单：`docs/train_agilex/数据检查指南.md`。建议在网页里做成自动化结果：

- 时间戳稳定性：
  - `intervals = diff(timestamps)`
  - 异常阈值可以沿用文档：`interval > expected_interval * 2`
- NaN/Inf 检查：
  - `np.isnan(action).sum()` / `np.isinf(...)`
- 动作/状态范围：
  - 每维 min/max
- 静止帧检测：
  - `diff(action)` 小于阈值的连续长度
- 视频可解码（可选）：
  - 直接在前端播放能否正常加载；或离线 `ffprobe` 扫描（见指南第六部分）

实现建议：

- 在 `vis/scripts/precompute.py` 里离线预计算：
  - `episode_qc.json`（每个 episode 一个摘要）
  - 首页直接展示 “✅/⚠️/❌” 标记，支持点击跳转到异常 episode
- 预计算的原因：避免用户浏览时每点一次都做 parquet 全扫描

## 6. 前端交互设计：让“看完整个数据集”变轻松

### 6.1 首页（episode 列表页）

核心目标：**快速扫一遍，定位异常**。

建议信息密度（每行一条 episode）：

- 缩略图（默认取 `meta.camera_keys` 的第一个视频 key 的首帧）
- episode_index、task、length、duration
- QC badges：timestamp、NaN、静止、视频
- 快捷操作：
  - 点击进入 episode 页
  - “上一条/下一条” 快捷键提示
  - “自动播放下一条” 开关（用于快速刷完整个数据集）

### 6.2 episode 详情页

核心目标：**一眼看视频 + 一眼看曲线 + 快速跳转下一条**。

#### 视频区域（多相机）

对每个 `video_key` 显示一个 `<video>`：

- `src` 指向整个 mp4 文件（不是 per-episode mp4）
- 使用 episodes 元数据的 `{from_timestamp, to_timestamp}` 实现 episode 内播放：
  - 页面加载后：`video.currentTime = from_timestamp; video.play()`
  - 监听 `timeupdate`：超过 `to_timestamp` 自动 pause，并触发“下一条”（可选）

多相机同步策略：

- 以一个“主相机”为 master（通常 front）
- 维护相对时间 `t_rel = master.currentTime - master.from`
- 其他相机每次把 `currentTime` 轻微拉回 `from + t_rel`（允许小 drift）

> 为什么不用后端切片出单独 mp4：因为 LeRobotDataset 本来就按文件大小合并 mp4；网页端直接复用原始文件最省 IO/存储，且能利用浏览器硬件解码。

#### 曲线区域（action/state/reward/done）

曲线建议至少包含（与 `lerobot-dataset-viz` 对齐，见 `src/lerobot/scripts/lerobot_dataset_viz.py`）：

- `action`（每维一条线，或支持选择维度）
- `observation.state`
- `next.reward`
- `next.done`
- 可选：`next.success`

图表库建议：

- 若需“离线可用”且不想引入复杂前端：把 `plotly.min.js` 放到 `vis/web/static/vendor/`（不走 CDN）
- 或用更轻量的 ECharts（同样本地 vendor）

#### 快捷键与导航（提升效率的关键）

- `J / ←`：上一条 episode
- `K / →`：下一条 episode
- `Space`：播放/暂停
- `A`：自动下一条开关

## 7. 缩略图与缓存策略（让首页真的“好用”）

### 7.1 为什么必须缓存

如果首页每一行都临时解码视频首帧，会导致：

- CPU 占用高
- 首屏加载慢
- 多用户/多 tab 时卡顿

### 7.2 推荐做法：预生成缩略图

在 `vis/scripts/precompute.py`：

1. 遍历 `episodes`（使用 `load_episodes`）
2. 对每个 episode：
   - 取一个 video_key（比如 front）
   - 取 `video_path + from_timestamp`
   - 用 `ffmpeg` 或仓库的 `decode_video_frames(...)` 抽一帧
3. 输出：
   - `vis/.cache/<repo_id>/thumbs/episode-000123.jpg`
   - `vis/.cache/<repo_id>/episodes.json`（episode 摘要）
   - `vis/.cache/<repo_id>/qc.json`（QC 摘要）

> `ffmpeg` 在 `docs/train_agilex/数据检查指南.md` 已经被用来抽帧，说明你的环境链路可行。

## 8. 与现有 Rerun 可视化的融合（可选增强，不改源码）

虽然网页端是主入口，但 Rerun 仍适合做“某条 episode 的深度检查”（多 stream/time）。

可以在 episode 详情页提供一个按钮：

- “导出 Rerun 记录（.rrd）”

实现思路（不改源码）：

- 在 `vis` 后端 import `lerobot.scripts.lerobot_dataset_viz.visualize_dataset`
- 复用它的 `save=1, output_dir=...` 生成 `.rrd`
- 给用户返回一个下载链接

这样保留原工具价值，同时把“浏览/筛选/跳转”能力放到网页端。

## 9. 里程碑（建议按 3 个阶段交付）

### Phase 1：可用的“浏览器刷完整个数据集”（1～2 天）

- 后端：能加载 meta、列出 episodes、提供 mp4 静态服务
- 前端：首页列表 + episode 详情页（视频播放 + 上/下一条）
- 支持 `--repo-id` + `--root` + `HF_HUB_OFFLINE=1`

### Phase 2：把“质检”产品化（2～4 天）

- precompute 缩略图 + QC
- 首页筛选（task、异常、长度）
- episode 页 action/state 曲线 + 异常提示（timestamp gap、NaN、静止段）

### Phase 3：高级体验（按需）

- 逐帧模式（frame-by-frame + 标注）
- 导出 `.rrd` / 导出关键帧 / 导出报告
- 多数据集管理（一个服务浏览多个 repo_id）

## 11. 风险与对策（提前规避踩坑）

1. **视频文件 seek 依赖 Range**
   - 对策：后端务必用支持 Range 的 file response；上线前用浏览器拖动进度条验证。
2. **episode 时长与帧数轻微不一致**
   - 原因：视频编码与 timestamp tolerance（`tolerance_s`）机制，详见 `LeRobotDataset._query_videos`
   - 对策：前端以 `to_timestamp-from_timestamp` 作为播放边界；曲线用 parquet 的 `timestamp` 为准。
3. **数据量大导致前端 JSON 过大**
   - 对策：timeseries API 必须支持 stride/downsample；首页用预计算摘要。
4. **安全（误把任意文件暴露成静态资源）**
   - 对策：严格限定静态文件根目录为 dataset root 下的 `videos/` 和 `vis/.cache/`，并做 `resolve()` 校验。

## 12. 结论

在不改 `src/lerobot` 的前提下，依托仓库已实现的 LeRobotDataset v3.0 文件结构（尤其是 episodes 元数据里的 `from_timestamp/to_timestamp`），完全可以在 `vis/` 单独实现一个“数据集网页端浏览器”：

- 用浏览器 URL 一次性浏览整个数据集（episode 列表 → 点击 → 自动下一条）
- 直接播放原始 mp4（无需反复跑 `lerobot-dataset-viz`）
- 把 `docs/train_agilex/数据检查指南.md` 的检查项变成网页 QC 结果，显著提升数据审核效率
