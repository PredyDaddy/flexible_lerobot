# 服务端异步推理文档2（GPU 服务器推理，本机/机器人端接收执行）

## 1. 结论：仓库里已经支持“推理放服务器、机器人本机接收”

`src/lerobot/async_inference/` 这套设计本来就是 “PolicyServer（推理端） + RobotClient（执行端）” 的远端架构：

- PolicyServer：跑在有 GPU/算力的服务器上，负责加载模型、做推理、返回 action chunk
- Client：跑在机器人本机（可以没有显卡），负责 ROS 取观测、维护动作队列、按固定频率下发动作

两端通过 gRPC 通信，client 用 `--server-address=<server_ip:port>` 就可以连远端 server。

## 2. 架构与数据流（最小理解）

```text
机器人本机(ROS + Cameras)                          GPU 服务器(推理)
AgileXRobot.get_observation()  --obs(gRPC)-->   PolicyServer 推理 -> action chunk
AgileXRobot.send_action()      <--act(gRPC)--   返回 chunk（多步动作）
```

关键点：
- Client 有 action queue：只要队列里还有动作，机器人就继续动，不会“等推理”
- Server 一次推理返回多步动作（chunk）：降低推理调用频率与抖动

## 3. 部署前提（必须满足）

### 3.1 服务器侧

- 安装同版本代码与依赖（建议和机器人端同一仓库版本）
- 安装异步依赖：

```bash
(my_lerobot) pip install -e ".[async]"
```

- 服务器能访问模型（两种方式任选其一）：
  - 把 checkpoint 复制到服务器本地路径
  - 或者 NFS/共享盘挂载，让 server 能通过路径直接读到
  - 或者使用 hub id（需要服务器可联网）

### 3.2 机器人侧（本机）

- 不需要 GPU；需要 ROS + 相机话题正常
- 同样安装异步依赖（client 也需要 grpcio）：

```bash
(my_lerobot) pip install -e ".[async]"
```

## 4. 推荐用法（以你当前 AgileX 单臂脚本为例）

你现在已经有异步版脚本：
- `agilex_scripts/agilex_async_infer_single_cc_vertical.py`
- `agilex_scripts/agilex_async_infer_single_cc_horizal.py`

它本质上就是：
- `server`：启动 PolicyServer
- `client`：连接机器人并控制

### 4.1 服务器启动 PolicyServer

在服务器上：

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py server \
  --host 0.0.0.0 --port 8080 --fps 30
```

为什么这里没有 `--checkpoint`（这点最容易困惑）：
- 这个 PolicyServer **启动时是“空的”**，不会预先加载任何模型。
- 模型的 checkpoint 是在 client 启动时通过 gRPC 的 `SendPolicyInstructions` 发给 server 的（也就是你在 client 里传的 `--checkpoint`）。
- server 收到指令后才会执行 `from_pretrained(<checkpoint>)` 把模型加载到 server 的 GPU/CPU 上。

说明：
- `--host 0.0.0.0`：对外监听（否则默认 localhost 只能本机连）
- 确保服务器防火墙放行 `8080/tcp`，或在同一内网可直连

### 4.2 机器人侧启动 Client（连接服务器推理）

在机器人本机：

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py client \
  --checkpoint /server/path/to/pretrained_model \
  --checkpoint-on-server \
  --arm left --fps 30 --binary-gripper \
  --server-address <SERVER_IP>:8080 \
  --actions-per-chunk 100 --chunk-size-threshold 0.3
```

如果你需要“水平初始位置”，把 client 脚本换成：

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_horizal.py client \
  --checkpoint /server/path/to/pretrained_model \
  --checkpoint-on-server \
  --arm left --fps 30 --binary-gripper \
  --server-address <SERVER_IP>:8080 \
  --actions-per-chunk 100 --chunk-size-threshold 0.3
```

说明：
- `--checkpoint` 这里填写的是 **服务器可访问的路径（或 hub id）**
- `--checkpoint-on-server` 表示不检查本机是否存在该目录，直接把字符串发给 server 让 server 去加载

如果服务器和机器人端共享了同一套路径（例如 NFS 挂载后路径一致），也可以不加 `--checkpoint-on-server`。

补充：`--checkpoint-on-server` 到底“传了什么”
- 它不会把模型文件从机器人传到服务器；它只是不让 client 在本机校验 `--checkpoint` 是否存在。
- 真正会发给服务器的是 `--checkpoint` 这个字符串（路径或 hub id）。
- server 收到后会执行 `from_pretrained(<checkpoint>)` 去加载模型，所以你给的路径必须是 **server 能读到的**。

常见两种传法（机器人端怎么写 `--checkpoint`）：

1) 模型只在服务器上：传服务器绝对路径（推荐最省心）

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py client \
  --checkpoint /data/models/act_left/pretrained_model \
  --checkpoint-on-server \
  --server-address <SERVER_IP>:8080 \
  --arm left --fps 30 --binary-gripper
```

2) 服务器和机器人共享同一套相对路径结构：传相对路径（要求 server 在仓库根目录启动）

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py client \
  --checkpoint outputs/act_agilex_left_box/checkpoints/030000/pretrained_model \
  --checkpoint-on-server \
  --server-address <SERVER_IP>:8080 \
  --arm left --fps 30 --binary-gripper
```

补充：关于路径“到底在谁的机器上解析”
- 你传给 client 的 `--checkpoint` 最终是 **由 server 去打开/读取** 的。
- 如果你给的是相对路径（比如 `outputs/...`），它是相对于 **server 启动时的工作目录** 来解析的。
- 为了省心，远端部署时建议：
  - 要么给 server 的绝对路径（比如 `/data/models/.../pretrained_model`）
  - 要么确保 server 在仓库根目录启动，并且 server 上存在同样的相对路径结构

## 5. 关键参数解释（远端部署时怎么影响效果）

### 5.1 `--actions-per-chunk`

含义：server 每次返回给 client 的动作步数。

影响：
- 大：更少通信/更少推理调用、更稳；但响应新观测更慢
- 小：更跟手；但更频繁推理/更吃带宽

对 ACT 的提醒：模型每次 forward 固定会算出 `chunk_size`（你这个 checkpoint 是 100），
`actions-per-chunk` 只决定“返回/使用多少步”，不一定等比例降低 forward 耗时。

### 5.2 `--chunk-size-threshold`

含义：当队列剩余比例低于阈值时，client 发送新观测去补货。

远端建议：保守一点（0.3~0.6），确保补货时队列里还有足够时间覆盖：
- 网络抖动（RTT）
- server 推理耗时

经验公式（只用来直观理解）：

`缓冲时间 ≈ actions_per_chunk * threshold * (1/fps)`

缓冲时间越大，越不容易“断粮”。

### 5.3 `--fps`

含义：client 控制环频率（例如 30Hz）。

影响：
- 越高：越依赖“取图/网络/推理”稳定性
- 远端网络不稳时，不建议盲目上更高 fps

### 5.4 Server 侧 `--host/--port`

- `--host` 用 `0.0.0.0` 才能被远端访问
- `--port` 端口要放行（防火墙/安全组）

## 6. 安全注意事项（务必看）

当前实现特点：
- gRPC 默认是 **明文不加密**（insecure channel）
- 观测/动作用 **pickle** 序列化（不适合不可信网络）

建议：
- 只在可信内网/VPN 使用
- 不要把端口直接暴露到公网
- 如需跨公网，建议用 SSH 隧道或 VPN（实现加密与访问控制）

## 7. 常见问题与排查

- client 连不上 server：
  - server 是否用 `--host 0.0.0.0`
  - 端口是否放行、是否能 `ping`/`telnet <ip> <port>`
- server 报 “找不到 checkpoint”：
  - `--checkpoint` 是否是服务器可访问路径
  - 如 checkpoint 不在本机，client 要加 `--checkpoint-on-server`
- 行为不对/报维度错：
  - 相机 key 必须与训练一致（你的模型是 `camera_left` + `camera_front`）
  - 单臂模型必须只给 7 维 state（本脚本已做裁剪）
