## 1. 概述

### 1.1 什么是异步推理

异步推理是一种将**观测采集**、**策略推理**和**动作执行**解耦的架构模式。传统的同步推理中，这三个步骤串行执行，导致控制频率受限于最慢的环节（通常是模型推理）。

**同步模式流程：**
```
[采集观测] → [模型推理] → [执行动作] → [采集观测] → ...
     ↑_____________等待______________↑
```

**异步模式流程：**
```
客户端:  [采集观测] → [发送] → [执行动作] → [采集观测] → [发送] → ...
                        ↓                                    ↓
服务器:            [接收] → [推理] → [返回动作块]
```

### 1.2 核心优势

| 特性 | 同步模式 | 异步模式 |
|------|----------|----------|
| 控制频率 | 受推理延迟限制 (5-15 FPS) | 接近目标频率 (30 FPS) |
| GPU 利用率 | 间歇性使用 | 持续高效利用 |
| 网络延迟影响 | 直接影响控制周期 | 通过动作缓冲区吸收 |
| 扩展性 | 单机运行 | 支持分布式部署 |

### 1.3 适用场景

- 需要高频控制的精细操作任务
- GPU 资源与机器人控制器分离的部署场景
- 多机器人共享推理服务器
- 需要稳定控制频率的实时任务

### 2.2 核心组件

#### PolicyServer (策略服务器)

位置: `src/lerobot/async_inference/policy_server.py`

职责:
- 加载和管理策略模型 (ACT, Diffusion, Pi0, SmolVLA 等)
- 接收客户端发送的观测数据
- 执行策略推理，生成动作块 (Action Chunk)
- 通过 gRPC 返回动作给客户端

支持的策略类型:
```python
SUPPORTED_POLICIES = ["act", "smolvla", "diffusion", "tdmpc", "vqbet", "pi0", "pi05"]
```

#### AgilexAsyncClient (AgileX 异步客户端)

位置: `agilex_infer_async_client.py`

职责:
- 连接 AgileX 机器人硬件
- 采集相机图像和关节状态
- 将观测数据打包发送给服务器
- 管理动作队列，按时间戳执行动作
- 支持单臂模式 (left/right)

#### 辅助类 (helpers.py)

| 类名 | 说明 |
|------|------|
| `TimedObservation` | 带时间戳的观测数据封装 |
| `TimedAction` | 带时间戳的动作数据封装 |
| `RemotePolicyConfig` | 远程策略配置，用于客户端-服务器握手 |
| `FPSTracker` | FPS 统计工具 |

### 2.3 通信协议

使用 gRPC 进行客户端-服务器通信，主要接口:

| 方法 | 方向 | 说明 |
|------|------|------|
| `Ready()` | Client → Server | 客户端就绪通知 |
| `SendPolicyInstructions()` | Client → Server | 发送策略配置 |
| `SendObservations()` | Client → Server | 发送观测数据流 |
| `GetActions()` | Client ← Server | 获取动作块 |

## 3. 快速开始

### 3.1 环境准备

确保已安装 LeRobot 及相关依赖:

```bash
conda activate my_lerobot
pip install -e ".[all]"
```

### 3.2 启动策略服务器

在 GPU 机器上启动策略服务器:

```bash
python -m lerobot.async_inference.policy_server \
    --host=0.0.0.0 \
    --port=8080 \
    --fps=30 \
    --inference_latency=0.033 \
    --obs_queue_timeout=2
```

**参数说明:**
- `--host`: 服务器绑定地址，`0.0.0.0` 表示接受所有网络接口的连接
- `--port`: 服务端口
- `--fps`: 目标控制频率
- `--inference_latency`: 推理延迟控制 (秒)
- `--obs_queue_timeout`: 观测队列超时时间 (秒)

### 3.3 启动 AgileX 客户端

在机器人控制器上启动客户端:

**左臂模式:**
```bash
python agilex_infer_async_client.py \
    --arm left \
    --server_address <GPU_SERVER_IP>:8080 \
    --fps 30 \
    --binary-gripper
```

**右臂模式:**
```bash
python agilex_infer_async_client.py \
    --arm right \
    --server_address <GPU_SERVER_IP>:8080 \
    --fps 30 \
    --binary-gripper
```

**客户端参数说明:**
- `--arm`: 控制臂选择 (`left` 或 `right`)
- `--server_address`: 策略服务器地址
- `--fps`: 控制频率
- `--binary-gripper`: 启用二值化夹爪控制
- `--gripper-threshold`: 夹爪二值化阈值 (默认 0.7)
- `--mock`: 模拟模式，用于测试

### 3.4 完整使用示例

**场景**: 在本地 GPU 机器上运行服务器，控制 AgileX 左臂

**终端 1 - 启动服务器:**
```bash
cd /home/agilex/cqy/my_lerobot
conda activate my_lerobot
python -m lerobot.async_inference.policy_server \
    --host=0.0.0.0 \
    --port=8080 \
    --fps=30
```

**终端 2 - 启动客户端:**
```bash
cd /home/agilex/cqy/my_lerobot
conda activate my_lerobot
python agilex_infer_async_client.py \
    --arm left \
    --server_address localhost:8080 \
    --fps 30 \
    --binary-gripper
```

## 4. 配置参数详解

### 4.1 服务器配置 (PolicyServerConfig)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `host` | str | "localhost" | 服务器绑定地址 |
| `port` | int | 8080 | 服务端口 (1-65535) |
| `fps` | int | 30 | 目标帧率 |
| `inference_latency` | float | 0.033 | 推理延迟控制 (秒) |
| `obs_queue_timeout` | float | 2.0 | 观测队列超时 (秒) |

### 4.2 客户端配置 (ClientConfig)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `arm` | str | - | 控制臂 ("left" / "right") |
| `server_address` | str | - | 服务器地址 |
| `fps` | int | 30 | 控制频率 |
| `binary_gripper` | bool | False | 二值化夹爪 |
| `gripper_threshold` | float | 0.7 | 夹爪阈值 |
| `actions_per_chunk` | int | 50 | 每块动作数量 |
| `chunk_size_threshold` | float | 0.5 | 触发新推理的队列阈值 |

### 4.3 关键参数说明

#### actions_per_chunk

每次推理返回的动作数量。ACT 等策略天然支持动作分块 (Action Chunking)，一次推理可以预测未来多步动作。

- **较大值 (如 50-100)**: 减少推理频率，但动作可能过时
- **较小值 (如 10-20)**: 动作更新更频繁，但增加推理负担

#### chunk_size_threshold

当动作队列剩余比例低于此阈值时，触发新的观测发送。

```
触发条件: queue_size / action_chunk_size <= chunk_size_threshold
```

- **0.5 (默认)**: 队列消耗一半时请求新动作
- **0.3**: 更激进，等队列快空时才请求
- **0.7**: 更保守，提前请求新动作

#### aggregate_fn (聚合函数)

当新旧动作块有重叠时间戳时，如何合并动作:

```python
AGGREGATE_FUNCTIONS = {
    "weighted_average": lambda old, new: 0.3 * old + 0.7 * new,  # 偏向新动作
    "latest_only": lambda old, new: new,                         # 只用新动作
    "average": lambda old, new: 0.5 * old + 0.5 * new,          # 平均
    "conservative": lambda old, new: 0.7 * old + 0.3 * new,     # 偏向旧动作
}
```

## 5. 性能对比

### 5.1 同步 vs 异步性能数据

基于 AgileX 双臂机器人 + ACT 策略的测试数据:

| 指标 | 同步模式 | 异步模式 |
|------|----------|----------|
| 单帧延迟 | 40-180 ms | 33 ms (目标) |
| 实际 FPS | 5-15 | 25-30 |
| GPU 利用率 | 30-50% | 70-90% |
| 控制平滑度 | 有明显卡顿 | 流畅 |

### 5.2 延迟分解

**同步模式瓶颈分析:**
```
相机读取: 10-30 ms
图像预处理: 5-10 ms
模型推理: 20-100 ms (取决于模型大小)
动作后处理: 2-5 ms
─────────────────────
总计: 37-145 ms → 7-27 FPS
```

**异步模式优化:**
- 相机读取与推理并行
- 动作缓冲区吸收推理延迟波动
- 网络传输与本地控制解耦

### 5.3 优化建议

1. **网络优化**: 使用有线连接，减少 WiFi 延迟抖动
2. **GPU 预热**: 首次推理前运行几次空推理
3. **批量大小**: 根据 GPU 显存调整 `actions_per_chunk`
4. **队列阈值**: 网络不稳定时增大 `chunk_size_threshold`

## 6. 注意事项

### 6.1 适用场景限制

**适合异步推理的场景:**
- 动作分块策略 (ACT, Diffusion, Pi0 等)
- 相对稳定的网络环境
- 任务允许一定的动作延迟

**不适合异步推理的场景:**
- 需要即时反馈的反应式控制
- 网络极不稳定的环境
- 单步策略 (无动作分块)

### 6.2 常见问题

**Q1: 客户端连接失败**
```
grpc.RpcError: failed to connect to all addresses
```
解决: 检查服务器地址和端口，确保防火墙允许连接

**Q2: 动作队列频繁为空**
- 增大 `actions_per_chunk`
- 增大 `chunk_size_threshold`
- 检查网络延迟

**Q3: 推理速度慢**
- 确认 GPU 正确加载 (`nvidia-smi`)
- 检查模型是否在 CUDA 设备上
- 考虑使用更小的模型或量化

**Q4: 机器人动作不连贯**
- 调整 `aggregate_fn` 为 `weighted_average`
- 减小 `chunk_size_threshold` 提前请求新动作
- 检查 FPS 设置是否匹配

### 6.3 安全注意事项

1. **单臂模式安全**: `send_single_arm_action` 确保未控制的臂保持当前位置
2. **急停处理**: 使用 `Ctrl+C` 可安全停止客户端
3. **初始位置**: 启动前会自动移动到预设初始位置
4. **动作限幅**: `max_relative_target` 限制单步最大位移
