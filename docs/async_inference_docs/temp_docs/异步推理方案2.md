# 异步推理方案2：双服务器 + 单控制进程（面向 AgileX 单臂模型）

## 结论与加速点

能加速，主要是“隐藏推理耗时”，不是让单次推理更快。关键收益如下：

- 推理与控制解耦，控制环不再被模型阻塞，动作更连续。
- 动作分块 (`actions_per_chunk`) 降低 gRPC 往返频率，吞吐更稳。
- 观测去重与队列策略会丢弃“相似帧”，减少无效推理。

## 方案概览

你的左右臂是两个独立模型，最稳定的方式是双服务器 + 单控制进程合并动作，避免两个进程同时向机器人发指令。

```
AgileX Robot
   ▲   |
   |   v
单控制进程(采集观测/下发双臂动作)
   |           |
   v           v
Policy Server L   Policy Server R
```

## 适配当前用法的关键改造点

- 把 `agilex_infer_single_cc_vertical.py` 的模型加载与 `predict_action` 移到 `PolicyServer`。
- 客户端只负责：采集观测、发送到服务器、接收动作块、合并双臂动作并下发。
- 单臂模型只接受 7 关节 + 2 相机，需要构造“裁剪后的” `lerobot_features`，避免 14 关节维度不匹配。
- 二值化夹爪(`--binary-gripper`)放在客户端的动作后处理中完成。

## 运行步骤（示例）

1. 启动左右臂策略服务器（同机或不同机均可）

```bash
# 左臂 server
(my_lerobot) python -m lerobot.async_inference.policy_server \
  --host 127.0.0.1 --port 8081 --fps 30 --inference_latency 0.033 --obs_queue_timeout 1

# 右臂 server
(my_lerobot) python -m lerobot.async_inference.policy_server \
  --host 127.0.0.1 --port 8082 --fps 30 --inference_latency 0.033 --obs_queue_timeout 1
```

2. 启动单控制进程（建议新建脚本，例如 `agilex_async_client_dual.py`）

核心逻辑是两套 client 通道 + 两个动作队列 + 合并下发。伪代码要点如下：

```python
# 1) 初始化 AgileXRobot 与相机（camera_front + camera_left/right）
# 2) 为左/右臂分别构造 lerobot_features（只包含各自 7 关节 + 对应相机）
# 3) 分别向 8081/8082 发送 RemotePolicyConfig（包含各自 checkpoint）
# 4) 启动接收线程：循环 GetActions -> 更新左/右动作队列
# 5) 控制循环：取左右动作 -> 合并成 14 关节 -> 发送 send_joint_commands
# 6) 动作后处理：二值化夹爪、限幅（复用 send_single_arm_action 的裁剪逻辑）
```

## 构造单臂 `lerobot_features` 的建议方式

`async_inference` 默认把所有关节拼成一个 `observation.state`。为避免单臂模型维度不匹配，需要手动裁剪关节与相机：

```python
from lerobot.datasets.utils import hw_to_dataset_features
from lerobot.utils.constants import OBS_STR

def build_arm_features(robot_obs_features, arm: str):
    keep = {}
    for k, v in robot_obs_features.items():
        if k.startswith(f"{arm}_"):
            keep[k] = v
        if arm == "left" and k in ["camera_left", "camera_front"]:
            keep[k] = v
        if arm == "right" and k in ["camera_right", "camera_front"]:
            keep[k] = v
    return hw_to_dataset_features(keep, OBS_STR, use_video=False)
```

如果你的模型使用的相机名与机器人观测不一致，使用 `RemotePolicyConfig.rename_map` 做键名映射即可。

## 参数建议（起步值）

| 参数 | 建议 | 作用 |
| --- | --- | --- |
| `actions_per_chunk` | 8~16 | 过大响应慢，过小吞吐低 |
| `chunk_size_threshold` | 0.4~0.6 | 越小越频繁发观测，响应更快 |
| `inference_latency` | `1 / fps` | 维持目标推理节奏 |
| `obs_queue_timeout` | 0.5~1.0s | 避免 GetActions 长时间阻塞 |
| `aggregate_fn_name` | `weighted_average` | 动作更平滑 |

## 注意事项

- 每个 `PolicyServer` 只服务一个客户端，新连接会重置服务器状态。
- 双臂同时跑时建议单控制进程合并动作，避免两个进程同时发送 ROS 指令。
- 只跑单臂时仍可用该方案，只启一个 server + 一个动作队列即可。
- 模型输入必须与训练时一致（关节维度与相机名称），这是最常见的“无动作/错动作”原因。
