# 异步推理最终报告（AgileX 单臂 ACT，方案3）

## 1. 你现在的同步推理是什么

你原来用的是同步单臂推理脚本（每个控制周期里顺序执行：取观测 -> 推理 -> 下发动作）：

```bash
python agilex_infer_single_cc_vertical.py \
  --checkpoint outputs/act_agilex_left_box/checkpoints/030000/pretrained_model \
  --arm left --fps 30 --binary-gripper
```

同步的直观问题：如果“取图/预处理/推理/发指令”任何一步慢了，就会把 30Hz 控制环卡住。

## 2. 异步推理原理（最简单版）

异步推理把系统拆成两部分，并且让它们并行：

- Client（机器人侧）：一直按固定频率执行动作；需要时把最新观测发出去“补货”
- Server（推理侧）：收到观测就推理，返回一段 **动作 chunk（多步动作）**

核心收益不是“单次 forward 更快”，而是：
- **推理和执行解耦**：推理慢一点时，机器人还能继续执行队列里的动作，不会立刻卡住
- **动作分块**：一次推理给出多步动作，减少推理调用次数，降低抖动

## 3. 为什么你感觉提升不大（这点很关键）

ACT 本身就是“动作分块”的策略。

你的 checkpoint 配置（见 `outputs/act_agilex_left_box/checkpoints/030000/pretrained_model/config.json`）里：
- `chunk_size = 100`：模型每次 forward 会算出 100 步动作
- `n_action_steps = 100`：同步执行时会把这 100 步动作放进队列里，**接下来 100 步都不需要再次 forward**

这意味着你原来的同步脚本其实已经很“接近异步”的好处了：**它本来就不是每一步都推理**。

你 server 日志里也显示单次推理很快（20ms 左右），本身就远小于 30Hz 的 33ms，所以体感提升不明显是正常的。

## 5. 新脚本做了什么（你可以把它当“异步版推理入口”）

脚本有两个子命令：

1) `server`：启动 PolicyServer（推理端）
- 复用仓库自带的 `lerobot.async_inference.policy_server`
- 通过 gRPC 对外提供：接收观测、返回 action chunk

2) `client`：启动 RobotClient（机器人控制端）
- 连接 `AgileXRobot`，维持 30Hz 控制环
- 后台线程持续向 server 拉取 action chunk，放到本地 action queue
- 控制环每步：
  - 有动作就执行（不等推理）
  - 队列剩余不足就发送新观测去“补货”
- 单臂模型只输出 7 维动作：脚本会自动把另一臂补成“保持当前值”
- 支持 `--binary-gripper`：在 client 侧做夹爪阈值化

另外，它做了一个对 AgileX 单臂模型很关键的适配：
- 只把 **单臂 7 关节 + 2 路相机** 作为 features 发给 server
- 避免 AgileX 默认双臂 14 关节导致维度不匹配

## 6. 怎么运行（两终端）

先安装异步依赖（grpcio 等）：

```bash
(my_lerobot) pip install -e ".[async]"
```

终端 A：启动 server

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py server \
  --host 127.0.0.1 --port 8080 --fps 30
```

终端 B：启动 client（左臂）

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_vertical.py client \
  --checkpoint outputs/act_agilex_left_box/checkpoints/030000/pretrained_model \
  --arm left --fps 30 --binary-gripper \
  --server-address 127.0.0.1:8080 \
  --actions-per-chunk 50 --chunk-size-threshold 0.8
```

如果你的数据/策略是“水平初始位置”，把脚本换成：

```bash
(my_lerobot) python agilex_scripts/agilex_async_infer_single_cc_horizal.py client \
  --checkpoint outputs/act_agilex_left_yellow_bottle/checkpoints/last/pretrained_model \
  --arm left --fps 30 --binary-gripper \
  --server-address 127.0.0.1:8080 \
  --actions-per-chunk 50 --chunk-size-threshold 0.8
```

## 7. 参数是什么意思？会怎么影响效果？

下面三个参数最重要：

### 7.1 `--actions-per-chunk`

含义：client 每次“用多少步动作”（server 返回的动作 chunk 长度）。

影响：
- 越大：推理调用更少、更稳、更省；但动作更“依赖旧观测”，响应新观测更慢
- 越小：更频繁刷新动作（更跟手）；但 server 推理更频繁、总体负载更高

对你这个 ACT checkpoint 的特殊点：
- 模型每次 forward 固定会算出 `chunk_size=100` 步
- 你设 `actions-per-chunk=50` 不会让 forward 变成一半，只是让你更频繁地重新推理

建议：
- 如果目标是“更稳/更省”：优先尝试 `--actions-per-chunk 100`
- 如果目标是“更跟手”：才考虑 10/20/50

### 7.2 `--chunk-size-threshold`

含义：队列还剩多少比例时开始发送新观测去“补货”。

影响：
- 阈值越大（比如 0.8）：更早补货，更不容易断粮（更稳），但观测发送更频繁
- 阈值越小（比如 0.3）：更省带宽/更少补货，但更容易队列见底（可能不连续）

建议：
- 先用 0.8 跑通、保证不断粮
- 再逐步降到 0.3~0.5 找到更省资源的点

### 7.3 `--aggregate-fn-name`

含义：如果新旧动作 chunk 在同一 timestep 上有重叠，怎么融合。

常用：
- `weighted_average`：更平滑（默认）
- `latest_only`：更跟手、更激进

## 8. 后续是否值得继续做？（怎么判断）

你现在 ACT 推理本身已经很快 + 同步版本又是 100 步 action queue，所以：
- “单机、只跑 ACT、30Hz” 场景：异步方案不会带来明显“速度提升”，更多是架构上的准备

异步方案更值得做的场景：
- 上更重的模型（推理时间接近/超过 33ms）
- 推理放到另一台 GPU 机器（client 侧只保控制环，抖动更小）
- 双臂两模型/多模型（更需要解耦与队列机制）

推荐你下一步先做两件事：
1) 把异步参数改成更贴合 checkpoint：`--actions-per-chunk 100`，再观察是否更稳、更省
2) 如果你追求“更跟手”，再把 `actions-per-chunk` 降到 20/10 去对比行为（注意这会增大推理负载）
