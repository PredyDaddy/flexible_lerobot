# GR00T TRT Async Server 整改实施细则

更新时间：2026-03-15

本文档补充 [第二轮复核后整改单_20260315.md](/data/cqy_workspace/flexible_lerobot/my_devs/groot_trt_async_server/docs/第二轮复核后整改单_20260315.md) 的执行细节，目的是减少实现子代理在协议面和验证面上的歧义。

## 1. 适用范围

本文档只约束两条主线：

- `protocol/control-plane`
- `TRT backend` 验证闭环

本文档不是新设计提案，而是对本轮整改必须满足的最低实现语义做补充。

## 2. protocol/control-plane 最低合同

### 2.1 server 资源所有权

整改后的 server 不应再直接把 client 下发的 `pretrained_name_or_path`、`engine_dir` 当作最终本地加载路径。

最低可接受方案：

- client 只传 `resource_id`、`deployment_id`、`profile_name` 或等价的逻辑标识
- server 在本地完成逻辑标识到真实资源路径的解析
- 若 client 仍需兼容旧字段，server 也必须把它们视为“候选信息”，而不是直接信任的加载路径
- 未注册、未授权或无法解析的资源请求必须显式失败

不接受的方案：

- 继续由 client 直接决定 server 本地 checkpoint 或 engine 目录
- 用文档声明替代运行时拒绝逻辑

### 2.2 sticky session 最低合同

整改后的 sticky session 必须从“拒绝型约束”升级成“可恢复模型”。

最低可接受语义：

- server 维护显式 `session_id`
- 至少支持以下三者之一：
  - 同 owner 重连恢复
  - 显式 release 后重新 claim
  - 带 owner token 的安全 takeover
- server 不能只依赖 `context.peer()` 作为唯一身份

最低需要覆盖的状态：

- `claimed`
- `released`
- `reconnected` 或 `taken_over`
- `rejected`

### 2.3 request/ack 最低合同

整改后的 observation / action 协议必须形成最小闭环。

最低可接受语义：

- 每个 observation 请求都带显式 `request_id`
- server 返回 action chunk 时回传对应 `request_id`
- client 只在收到匹配 `request_id` 的响应后才清除对应 pending
- 没有匹配 `request_id` 的旧包或晚到包必须显式丢弃

建议最小字段集合：

- request:
  - `request_id`
  - `observation_timestep`
  - `session_id`
  - `must_go`
- response:
  - `request_id`
  - `session_id`
  - `action_chunk_timestep_start`
  - `action_chunk_timestep_end`
  - `status`

`status` 的最低语义建议：

- `accepted`
- `stale`
- `rejected`

### 2.4 重复包与旧包处理

client 至少需要做到：

- 对 `request_id < latest_acked_request_id` 的响应直接丢弃
- 对不匹配当前 `session_id` 的响应直接丢弃
- 对重复 `request_id` 的响应只接收一次

server 至少需要做到：

- 明确拒绝已失效 session 的 observation
- 不向新 session 回放旧 session 的缓存 chunk

### 2.5 pure mock 最低合同

`run_mock_roundtrip.py` 若要作为联调前验证入口，必须提供不依赖真实 robot/camera 的模式。

最低可接受方案：

- 增加纯 mock observation provider
- 增加不初始化真实机器人硬件的参数开关
- 允许只验证协议收发和队列语义，不落到设备侧 IO

不接受的方案：

- 仍连接真实 robot/camera，但宣称其为“纯协议验证”

## 3. TRT backend 最低验证闭环

### 3.1 不可回退的已有保护

本轮整改中，以下保护不得回退：

- 共享 `TrtSession` 的串行化保护
- image-token slots 与 ViT tokens 的严格一致性校验
- 返回前的 CUDA stream 同步

### 3.2 最低验证目标

`TRT backend` 这轮不要求做完整实机验证，但必须从“代码看起来合理”提升到“存在最小可信证据”。

最低可接受证据之一：

- 轻量验证脚本，可在受限环境下执行静态/半动态检查
- 明确的自检入口，能覆盖关键不变量
- mock session 验证，证明串行化和关键断言确实在运行路径上生效

### 3.3 建议的验证层级

建议把验证分成三层：

1. `static`
   - 文件能 import
   - 关键函数能被解析
   - 关键断言存在
2. `guarded-runtime`
   - 用 mock 或最小替身验证锁、断言、自检逻辑
   - 不要求真实 TensorRT engine
3. `full-runtime`
   - 真实 engine
   - 真实 processor 样本
   - 并发压测

本轮最低要求达到：

- `static`
- `guarded-runtime`

可以明确留到下一轮的内容：

- `full-runtime`

但必须在回报中写清楚这是“未验证项”，不能写成“已关闭”。

### 3.4 结果表述规范

实现回报必须把结论分成两类：

- `已验证`
- `待联调验证`

不允许出现下面这种表述：

- “已完全关闭”但没有脚本或运行证据

## 4. 下一轮复核重点

下一轮复核不再重复讨论大方向，只检查：

1. server 是否收回资源所有权
2. session 是否真正具备可恢复语义
3. request/ack 是否成为协议事实而不是本地状态
4. mock 是否真的脱离 robot/camera
5. backend 是否给出了最小可信验证证据
