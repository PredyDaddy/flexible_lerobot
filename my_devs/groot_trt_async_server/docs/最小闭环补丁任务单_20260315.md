# GR00T TRT Async Server 最小闭环补丁任务单

更新时间：2026-03-15

本文档用于把仍未落地的两条整改主线，进一步压缩成下一轮实现子代理必须交付的最小补丁。本文档只做架构层任务收窄，不代替代码设计。

## 1. 适用背景

当前已完成：

- 第二轮独立复核结果收口
- 整改总任务单落地
- 环境核验脚本与环境引导文档落地

当前未完成：

- `protocol/control-plane` 最小协议闭环补丁
- `TRT backend` 最小可执行自检补丁

当前不允许进入联调准备阶段。

## 2. 微任务 A：最小协议闭环补丁

### 目标

只关闭一条最小主线：

- 把 `request_id` 从 client 本地状态升级为显式协议字段
- 给 action 回包增加最小 ack 或等价 request 对应关系
- 让 client 对旧 chunk / 晚到 chunk / 重复 chunk 做显式丢弃
- 给 `run_mock_roundtrip.py` 增加纯协议 mock 模式，不依赖真实 robot/camera

### 唯一写入边界

- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

### 明确不做

- 不处理 server 资源所有权边界
- 不处理 sticky session / takeover
- 不处理环境依赖
- 不扩散到其他文件

### 必须交付的最小结果

1. observation 侧有显式 `request_id`
2. action 回包侧有与该 `request_id` 对应的显式字段或等价 ack
3. client 收到旧 request、晚到 request、重复 chunk 时有明确丢弃策略
4. `run_mock_roundtrip.py` 提供一个不依赖真实 robot/camera 的纯协议 mock 模式

### 最低验证要求

至少给出以下任一验证：

- `py_compile` 通过
- 纯 mock roundtrip 命令可启动
- 本地最小协议 smoke 可运行

### 回报模板

1. `Implemented`
2. `Verification`
3. `Residual Risks`
4. `Files Changed`
5. `Ready For Re-Review`

## 3. 微任务 B：TRT backend 最小自检补丁

### 目标

只关闭一条最小主线：

- 把现有 `groot_trt_policy.py` 中已经存在的并发保护、token 校验、stream 同步，从“口头存在”提升为“可执行检查”

### 唯一写入边界

- `my_devs/groot_trt_async_server/groot_trt_policy.py`
- 新增的 `my_devs/groot_trt_async_server/` 下轻量验证脚本

### 明确不做

- 不大改模型数学边界
- 不引入新的运行时架构
- 不扩散到非 backend 文件

### 必须交付的最小结果

1. 保持当前已有保护不回退
2. 提供一个可执行的轻量验证脚本、自检入口或 mock 验证
3. 回报里明确区分：
   - 哪些项已验证
   - 哪些项仍待联调验证

### 最低验证要求

至少给出以下任一验证：

- `py_compile` 通过
- 自检脚本可运行
- mock 验证入口能触发关键断言或关键保护路径

### 回报模板

1. `Implemented`
2. `Verification`
3. `Residual Risks`
4. `Files Changed`
5. `Ready For Re-Review`

## 4. 执行原则

- 一次只跑一个微任务，不并行扩散
- 没有代码改动，不允许回报完成
- 不接受阶段判断、状态摘要、计划说明替代实现交付
- 每个微任务完成后先做一次主线验收，再决定是否启动下一个微任务

## 5. 当前建议顺序

1. 先执行“微任务 A：最小协议闭环补丁”
2. 回收并主线验收后，再执行“微任务 B：TRT backend 最小自检补丁”
3. 两个微任务都回收后，再决定是否发起下一轮准入复核
