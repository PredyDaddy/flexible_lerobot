## 二、本轮确认的阻断项

### 阻断 A: 控制面失败路径没有形成 request 终态闭环

独立复核结论：

- 现有控制面闭环只覆盖“正向 ack”，没有覆盖“负向终态”。
- 服务端在多条真实失败路径上直接返回匿名空响应，失败没有绑定到具体 `request_id/session_id`。
- 客户端 `_pending_request` 仅在收到显式 ack 后释放，导致单 inflight 窗口在失败场景下可能永久卡死。
- 当前 `--pure-mock` 主要覆盖 helper 和局部状态机，没有真正驱动 `start/send/receive` 级别闭环，因此不能证明 P0 风险已关闭。

架构判断：

- 这是协议状态机级阻断，不是测试薄弱这么简单。
- 如果不补失败终态闭环，restart/recover/takeover/stale-peer/server-error 中任一路径都可能造成 observation 发送永久停滞。

### 阻断 B: 后端自检对 sync-before-unlock 的证明强度不足

独立复核结论：

- 现有检查只能证明“锁内调用过一次 stream synchronize”，不能证明“最后一次 TensorRT 工作完成后再 unlock”。
- mock session 没有进入统一 timeline，session 执行与 sync 之间没有顺序证据。
- 被同步 stream 与实际承载 session 工作的 stream 没有绑定证明，存在假绿风险。
- CLI 仍会在 mock 检查通过后输出强正向结论，容易误导为后端闭环已被真正证明。

架构判断：

- 这是验证可信度级阻断。
- 如果不提升行为证据强度，当前自检不能作为准入依据，只能作为局部回归保护。

### 外部阻断 C: `lerobot_flex` 环境缺少 `grpc`

环境核验结论：

- `check_env.py --mode all` 显示 `grpc` 在 `lerobot_flex` 中缺失。
- 因此真实 gRPC 端到端联调仍无法在当前环境内完成。

架构判断：

- 这是外部依赖阻断，不影响纯 mock 和静态/局部行为整改推进。
- 但在 `grpc` 就位前，真实 RPC 传输路径仍不能做准入级闭环验收。

## 三、整改任务拆分

### 任务 T1: 控制面失败终态闭环整改

责任边界：

- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/run_client.py`
- `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

整改目标：

- 为失败路径定义明确的 request 终态语义，且必须绑定 `request_id/session_id`。
- 客户端在失败终态下必须能释放 inflight，或进入受控 retry / abort 路径，不能永久卡死。
- pure mock 必须覆盖真实客户端/服务端关键入口，而不是只测 helper。
- pure mock 必须至少覆盖：
  - stale observation
  - session 切换后丢弃 chunk
  - 服务端异常
  - inflight 释放或失败后重试
  - restart / recover / takeover 下的失败终态恢复

验收口径：

- `python my_devs/groot_trt_async_server/run_mock_roundtrip.py --pure-mock`
  必须可通过，且输出中能明确体现失败终态覆盖项。
- 独立审核员不能再指出“失败路径没有 request 终态闭环”这一阻断。

### 任务 T2: 后端行为自证强度整改

责任边界：

- `my_devs/groot_trt_async_server/groot_trt_policy.py`
- `my_devs/groot_trt_async_server/check_groot_trt_backend.py`

整改目标：

- 让 session 执行事件进入统一 timeline。
- 明确证明顺序为“最后一次 session 工作完成/登记 -> stream synchronize -> unlock”。
- 为 session 与 stream 建立可验证绑定，避免“同步错流仍然通过”的假绿。
- 自检输出必须区分：
  - 已被 mock 行为证明的事项
  - 尚未覆盖真实 TensorRT / CUDA stream 的 integration gap

验收口径：

- `python my_devs/groot_trt_async_server/check_groot_trt_backend.py`
  必须可通过，且输出能显示 session/sync/unlock 的顺序证据。
- 独立审核员不能再指出“只证明有 sync，没有证明最后一次 TRT 工作后再 unlock”的阻断。

## 五、当前阶段结论

当前项目不处于“收尾联调”阶段，而处于“阻断整改后再复核”阶段。

更具体地说：

- 控制面主路径已经从“明显不闭环”推进到“正向路径基本闭环，但失败终态仍断裂”。

- 环境层面除 `grpc` 外没有发现新的基础依赖阻断。

因此本轮工作的正确动作不是直接集成，而是继续做两条并行阻断整改，并在整改后再次做独立复核。
