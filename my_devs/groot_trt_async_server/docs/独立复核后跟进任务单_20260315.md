# GR00T TRT Async Server 独立复核后跟进任务单

更新时间：2026-03-15

本文档用于把独立上下文 reviewer 的复核结论，转成下一轮实现子代理的执行输入。本文档只做架构层收敛，不代替实现细节设计。

## 1. 已有结论

当前已完成并经过主线验收：

- 环境核验包
- 最小协议闭环补丁
- TRT backend 最小自检补丁
- server 资源边界收口补丁

当前仍未达到联调准入，原因不再泛化，而是收敛为 reviewer 明确指出的 4 个问题：

1. 协议仍兼容旧弱格式，`request_id/acked_request_id` 不是强约束
2. backend 自检仍是源码结构检查，不是行为级验证
3. sticky session 仍未形成恢复模型
4. `grpc` 环境阻塞仍在

## 2. 独立 reviewer 结论摘要

### 2.1 协议闭环补丁的 reviewer 结论

当前状态：

- 新协议字段已经存在
- `--pure-mock` 已经成立

但 reviewer 明确指出：

- client 仍接受裸 `list` action payload
- client 在缺少 `acked_request_id` 时仍回退到旧 `request_id`
- server 仍接受不带 `request_id` 的旧 observation payload

因此当前结论不是“协议已封口”，而是：

- `新链路可用`
- `旧弱协议仍被默许`

### 2.2 backend 自检补丁的 reviewer 结论

当前状态：

- 自检脚本可运行
- image-token mismatch 已提升成可执行检查

但 reviewer 明确指出：

- 锁保护检查只是 `inspect.getsource(...)` 字符串匹配
- stream sync 检查也只是源码字符串匹配
- 没有真实执行 `predict_action_chunk()`
- image-token 正例只校验 shape，没有校验 token 注入内容

因此当前结论不是“行为级证据已建立”，而是：

- `静态回归绊线已建立`
- `行为级验证仍未建立`

### 2.3 资源边界补丁的 reviewer 结论

当前状态：

- client 不再直接把 server 本地路径作为加载决策发过去
- server 已能按自己的 `resource_profile` 解析路径

但 reviewer 明确指出：

- `vit_dtype / llm_dtype / dit_dtype` 仍由 client 传给 server
- client 接口表面仍保留旧路径参数与默认值

因此当前结论是：

- `client -> server 本地路径字符串` 这条主线基本已关闭
- `resource_profile 成为唯一资源边界` 还未完全成立

该问题当前先降级为次级跟进项，不阻断下一步先做协议与 sticky session 封口。

## 3. 下一轮实现任务

### 微任务 F：协议强约束 + sticky session 恢复模型

目标：

- 把协议从“新字段存在”升级为“旧弱协议不再被默许”
- 同时补上 sticky session 的最小恢复模型

唯一写入边界：

- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/robot_client.py`
- 必要时可最小改动：
  - `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

必须交付的最小结果：

1. server 不再接受缺失 `request_id` 的 observation 主路径
2. client 不再接受缺失 `acked_request_id` 的 action 主路径
3. 裸 `list` action payload 不再作为主路径被默许
4. 至少补一组负路径验证：
   - 缺失 `request_id`
   - 缺失 `acked_request_id`
   - 裸 `list` action payload
5. sticky session 不再只依赖 `context.peer()` 做硬拒绝
6. 至少提供一条恢复路径：
   - release
   - reconnect
   - safe takeover

明确不做：

- 不处理 `grpc` 环境安装
- 不处理 backend 文件
- 不扩散到资源边界之外的其他模块

最低验证要求：

- `py_compile` 通过
- 至少一个纯逻辑 / pure mock 验证入口覆盖负路径
- 至少一个本地自检入口覆盖 session claim/release/reconnect/takeover 中的一条路径

### 微任务 G：backend 行为级自检

目标：

- 把 backend 自检从“源码结构检查”提升到“最小行为级检查”

唯一写入边界：

- `my_devs/groot_trt_async_server/groot_trt_policy.py`
- `my_devs/groot_trt_async_server/check_groot_trt_backend.py`

必须交付的最小结果：

1. 不再仅靠 `inspect.getsource(...)` 判断锁保护和 stream sync
2. 至少有一个可执行入口能真实触发：
   - 锁保护相关路径
   - stream sync 相关路径
   - image-token 正例内容校验
3. 回报中明确区分：
   - 行为级已验证项
   - 仍待联调验证项

明确不做：

- 不做真实 TRT engine 并发压测
- 不大改模型数学边界
- 不扩散到 protocol / server 文件

最低验证要求：

- `py_compile` 通过
- 自检脚本可运行
- 自检输出能区分行为级检查与静态检查
