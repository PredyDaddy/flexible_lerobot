# GR00T TRT Async Server 剩余阻断修正任务单

更新时间：2026-03-15

本文档用于处理独立上下文 reviewer 在强化补丁后仍指出的剩余阻断。本文档只做架构层收敛，不代替实现细节设计。

## 1. 当前状态

当前已确认：

- 环境核验入口已落地
- 最小协议闭环补丁已落地
- server 资源边界主线已基本关闭
- backend 行为级自检已落地
- 第一轮独立复核、第二轮独立复核、以及后续独立复核都已经给出了明确 findings

当前仍未达到联调准入，剩余阻断收敛为：

1. request/ack 状态机在慢响应下可能自我饥饿
2. sticky session 恢复尚未贯通到“客户端重启后恢复/接管旧 lease”
3. backend 自检仍缺两处行为级证据：
   - `stream sync` 是否发生在解锁前
   - `image-token mismatch` 是否在 `predict_action_chunk()` 入口层面被行为化验证
4. `grpc` 环境阻塞仍存在

## 2. 微任务 H：request window + restart/takeover 恢复

目标：

- 修掉 request/ack livelock
- 把 sticky session 从“同进程 reconnect”提升到“客户端重启可恢复 / 可接管”

唯一写入边界：

- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/policy_server.py`
- 必要时可最小改动：
  - `my_devs/groot_trt_async_server/run_mock_roundtrip.py`
  - `my_devs/groot_trt_async_server/run_client.py`

必须交付的最小结果：

1. request/ack 主路径不会因为单个未完成请求被覆盖而自我饥饿
2. 至少建立一个明确的 inflight 策略：
   - 单 inflight request
   - 或显式 bounded window
3. pure mock 能覆盖“慢响应不应饿死动作队列”的验证
4. client 有办法复用旧 `session_id`，或显式发起 `takeover`
5. pure mock 能覆盖：
   - restart 后同 `session_id` 恢复
   - takeover 正常路径
   - 冲突 session 的失败路径

明确不做：

- 不处理 `grpc` 环境安装
- 不扩散到 backend 文件
- 不重做资源边界问题

最低验证要求：

- `py_compile` 通过
- `run_mock_roundtrip.py --pure-mock` 通过，且输出中明确包含：
  - 负路径校验
  - 慢响应 / inflight 验证
  - restart/recover/takeover 验证

## 3. 微任务 I：backend 剩余行为级证据补齐

目标：

- 把 backend 自检剩余的两处 reviewer finding 封口

唯一写入边界：

- `my_devs/groot_trt_async_server/groot_trt_policy.py`
- `my_devs/groot_trt_async_server/check_groot_trt_backend.py`

必须交付的最小结果：

1. 自检能证明 `stream sync` 发生在共享锁释放之前，而不是只证明“发生过一次”
2. 自检能在 `predict_action_chunk()` 入口层面覆盖 `image-token mismatch` 行为
3. 输出中仍然明确区分：
   - 行为级已验证项
   - 仅静态 tripwire
   - 仍待联调项

明确不做：

- 不做真实 TRT engine 并发压测
- 不做真实数值一致性验证
- 不扩散到 protocol / server 文件

最低验证要求：

- `py_compile` 通过
- `check_groot_trt_backend.py` 可运行
- 输出中能直接看到这两处缺口已经被覆盖

## 4. 当前不做的项

当前这一轮先不处理：

- `grpc` 环境安装或依赖修改

原因：

- 这是环境层阻塞，不是当前两条代码缺口本身
- 先把 reviewer 指出的代码级阻断收口，再统一做环境准入

## 5. 执行顺序

建议顺序：

1. 并行执行 `微任务 H` 和 `微任务 I`
2. 两个微任务都回收后，再起独立上下文 reviewer
3. reviewer 通过后，再决定是否处理 `grpc` 环境阻塞
4. 最后才考虑下一轮准入复核
