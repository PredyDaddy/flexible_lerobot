# GR00T TRT Async Server 第二轮独立复核任务单

更新时间：2026-03-15

本文档用于把当前阶段判断和第二轮独立复核工作正式落成任务单。它不是实现文档，也不是最终验收报告，而是架构层的审核编排文件。

## 1. 当前阶段判断

基于 [临时工作进展_20260315.md](/data/cqy_workspace/flexible_lerobot/my_devs/groot_trt_async_server/docs/临时工作进展_20260315.md)，当前项目状态应统一定义为：

- `M2 返工完成，待二次独立复核`

这意味着：

- 已完成方案设计收敛
- 已完成第一轮实现落地
- 已完成第一轮独立审核
- 已完成第二轮返工交付
- 尚未完成第二轮独立复核

因此当前项目不应被表述为：

- 可联调
- 可进入真机阶段
- 关键风险已关闭

## 2. 本轮复核目标

本轮复核只回答三个架构问题：

1. 第二轮返工是否真正关闭了第一轮独立审核暴露的高优先级问题
2. 当前是否满足“进入联调准备阶段”的最低门槛
3. 哪些风险只是被缓解，仍未达到关闭标准

本轮复核不做的事情：

- 不扩需求
- 不追加新功能设计
- 不直接下场改实现
- 不把“文档声称已修复”等同于“风险已关闭”

## 3. 审核总原则

- reviewer 必须以第一轮 finding 为基准，对照第二轮返工声明做闭环判断
- reviewer 必须优先输出问题，不先写总结
- reviewer 必须给出证据，优先使用文件和行号
- reviewer 必须区分三种结论：`已关闭`、`已缓解但未关闭`、`未关闭`
- 未经过真实运行验证的修复，默认不能升级为“已关闭”

## 4. 审核工作包

### A. server/config 复核包

审核范围：

- `my_devs/groot_trt_async_server/configs.py`
- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/run_server.py`

审核目标：

- sticky session 是否已经从文档约束变成运行时硬约束
- `from_payload()` 是否彻底移除了 TRT 到 PyTorch 的静默降级
- `SendPolicyInstructions()` 是否恢复 upstream `from_pretrained(...)` 契约
- `run_server.py --dry-run` 是否能真实暴露 import / config 问题
- server 是否正确拥有 checkpoint / processor / engine 资源边界

交付要求：

- 按严重性排序列出 findings
- 每条 finding 给出文件和行号
- 分别标记：已关闭 / 已缓解但未关闭 / 未关闭
- 明确回答：该方向是否允许进入联调准备

### B. client/protocol 复核包

审核范围：

- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/run_client.py`
- `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

审核目标：

- `--robot-type` 是否真正驱动 robot 分支和标定目录选择
- client 是否仍存在任意形式的静默 fallback
- observation / action chunk 队列语义是否只是局部补丁，还是形成了清晰时序契约
- 当前协议是否仍停留在弱 request id，而缺少 ack / 去重 / 过期 chunk 丢弃机制
- mock roundtrip 是否仍然依赖真实 robot / camera，从而削弱其作为纯验证工具的价值

交付要求：

- 按严重性排序列出 findings
- 每条 finding 给出文件和行号
- 分别标记：已关闭 / 已缓解但未关闭 / 未关闭
- 明确回答：当前协议闭环程度如何，是否允许进入联调准备

### C. TRT backend 复核包

审核范围：

- `my_devs/groot_trt_async_server/groot_trt_policy.py`

审核目标：

- 共享 `TrtSession` 的串行化保护是否足以覆盖 gRPC 并发模型
- CUDA stream 同步和 execution context 复用保护是否明确
- image-token slots 与 ViT tokens 是否从隐式假设升级为显式校验
- 哪些结论只是“代码看起来合理”，哪些已经具备可验证证据

交付要求：

- 按严重性排序列出 findings
- 每条 finding 给出文件和行号
- 分别标记：已关闭 / 已缓解但未关闭 / 未关闭
- 明确回答：该方向是否仍需要并发压测或真实 processor 样本验证

### D. 环境与验收门槛复核包

审核范围：

- `lerobot_flex` 当前运行环境
- `grpc` 依赖缺失对本地联调和验收的影响
- 各入口脚本的最小可启动性

审核目标：

- 确认 `grpc` 缺失是否仍然阻止 server 真实启动
- 定义补齐最小联调环境所需依赖
- 定义当前阶段的准入门槛：满足什么条件才允许宣布“进入联调准备”

交付要求：

- 给出环境阻塞清单
- 给出阻塞解除后的最小验证路径
- 明确回答：在环境未补齐前，哪些修复只能视为“声称成立”

## 5. 统一输出模板

每个 reviewer 的输出必须使用下面的结构：

1. Findings
2. 已关闭问题
3. 已缓解但未关闭问题
4. 未关闭问题
5. 新增风险
6. 是否允许进入联调准备

其中：

- `Findings` 必须放在最前面
- 如果没有发现问题，也必须明确写“未发现新增问题，但存在以下验证缺口”
- 任何“允许进入联调准备”的结论，都必须附带前置条件

## 6. 准入门槛

只有在以下条件同时满足时，项目状态才允许从 `M2 返工完成，待二次独立复核` 升级到“进入联调准备”：

1. A/B/C 三个实现方向的高优先级 findings 均被复核为已关闭或可接受残余风险
2. `lerobot_flex` 环境补齐 `grpc` 等关键依赖，server 入口可真实启动
3. 至少完成一次可信的 client/server roundtrip 验证
4. 协议层不存在会直接导致旧 chunk 误执行或 action 队列自锁的未关闭风险

如果任一条件不满足，则结论只能停留在：

- `返工完成，但未达到联调准入`

## 7. 架构师收口规则

架构师最终只输出三段结论：

- 哪些高风险项已经关闭
- 哪些风险只是缓解但未关闭
- 当前是否准许进入联调准备阶段

在 reviewer 明确给出证据之前，不使用“已解决”“可联调”“可真机”这类措辞。
