## 二、独立复核确认的剩余阻断

### 阻断 1: 启动阶段 sticky session 泄漏

问题描述：

- `Ready()` 成功后，如果 `SendPolicyInstructions()` 失败，客户端会把 `_session_claimed` 置回 `False`
- 但此时服务端已经持有 sticky session，后续 `stop()` 不会再发 release
- 结果是 session 被占住，破坏 restart / recover / takeover 闭环

架构要求：

- 启动阶段必须把“claim 成功但后续 setup 失败”的路径视为一类显式清理场景
- 要么在失败时立即 release 已 claim session
- 要么保留本地 session-claimed 状态并保证 `stop()`/cleanup 能发出 release
- 不允许出现“客户端自认为未 claim、服务端实际仍持有 session”的分裂状态

### 阻断 2: 发送侧普通异常不会释放 inflight

问题描述：

- 当前发送侧只对 `grpc.RpcError` 做失败终态收口
- 普通异常，例如序列化异常、chunk 发送异常，会留下 `_pending_request`
- 之后 `_ready_to_send_observation()` 持续为 `False`，单 inflight 窗口重新卡死

架构要求：

- 发送 observation 的所有可预期失败路径都必须进入统一终态收口
- 不得只依赖 gRPC 异常类型
- 需要保证一旦发送侧失败，`_pending_request` 必须被安全释放或进入明确 retry/abort

### 阻断 3: 接收侧普通异常会杀死线程并留下 inflight

问题描述：

- 当前接收侧只对 `grpc.RpcError` 做闭环
- `pickle.loads()`、payload 解包、聚合、本地后处理等普通异常会直接退出接收线程
- 线程退出时 `_pending_request` 不清理，导致控制环永久停滞

架构要求：

- 接收侧对协议解包和本地处理异常必须有统一兜底
- 出现坏包、兼容性错误或本地处理错误时，必须终结当前 inflight，不能留下悬挂 request
- 线程异常退出不能成为 inflight 永久卡死的入口

### 阻断 4: foreign-session 路径的 pure-mock 证明不够硬

问题描述：

- 当前测试是在丢掉 foreign-session chunk 后，再脚本化注入一条 abort terminal 才释放 inflight
- 这只能证明“后续若再到达 terminal，客户端能释放 inflight”
- 不能证明“foreign-session 路径本身已闭环”

架构要求：

- 要么把 foreign-session chunk 本身纳入明确终态策略
- 要么强化协议约束与 pure-mock，使该路径的终态保证由真实状态机直接给出，而不是由测试脚本额外补一条 terminal
- 最终测试必须能直接证明：不会因为 foreign-session 路径留下悬挂 inflight
