## 4. 第一轮独立审核结论

第一轮审核已经完成，主要问题已经被识别出来。

### 4.1 server / backend 侧问题

审核指出的核心问题包括：

- `TRT backend` 在继承的 gRPC 并发模型下不安全，多个请求可能复用同一组 `TrtSession`
- image-token slots 与 ViT tokens 的映射校验不够严格，存在静默错误风险
- `GrootTrtRemotePolicyConfig.from_payload()` 存在静默降级成 `pytorch` 的问题
- `SendPolicyInstructions()` 对 `pretrained_name_or_path` 的处理回退了 upstream `from_pretrained` 契约

### 4.2 系统集成层问题

审核指出的核心问题包括：

- “单机器人单会话 / sticky session” 只是文档约束，还不是运行时约束
- queue / timestep 语义存在“丢一次 action chunk 就可能补货自锁”的风险
- 协议边界过于松散，当前是 “gRPC 外壳 + pickle 内核”
- server 资源所有权放错了边界，client 在决定 server 本地加载哪套资源
- `run_server.py --dry-run` 之前会给出误导性的可用信号
- `mock roundtrip` 仍依赖真实 robot/camera，mock 颗粒度还不够纯

### 4.3 client / 入口层问题

审核指出的核心问题包括：

- `--robot-type` 参数无效，SO101 可能实际走到 SO100 路径
- client 的兼容 fallback 会静默把 TensorRT 契约降级成 PyTorch

## 5. 第二轮返工已完成的内容

以下内容基于返工子代理的正式汇报。

### 5.1 server/config 返工结果

返工涉及文件：

- `my_devs/groot_trt_async_server/configs.py`
- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/run_server.py`

已声称修复的问题：

- 为单机器人单会话增加了运行时 sticky session 约束
- `from_payload()` 不再把 TRT 协议静默降级成 `pytorch`
- `SendPolicyInstructions()` 恢复了 upstream `from_pretrained(...)` 的输入契约
- `run_server.py --dry-run` 现在会真实暴露 import / config 问题，而不是伪成功

返工后仍明确存在的残余风险：

- sticky session 目前是按 `context.peer()` 绑定 owner，没有 release / takeover 机制
- 如果 peer 变化，当前实现更偏向“显式拒绝”，而不是“安全接管”
- 当前 `lerobot_flex` 环境缺少 `grpc`，server 在该环境下仍不能真实启动

### 5.2 client/protocol 返工结果

返工涉及文件：

- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/run_client.py`
- `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

已声称修复的问题：

- `--robot-type` 现在会真正生效
- 标定目录默认值会跟随 `robot_type`
- client fallback 不再静默把 TRT 协议降级成 PyTorch
- observation / action 队列语义已调整，尝试降低“丢一次 chunk 直接饿死”的风险

返工后仍明确存在的残余风险：

- 目前仍然只是“弱 request id” 语义，不是显式 request ack 协议
- 真正的时序抖动与旧 chunk 晚到问题，还需要和 server 一起做实机联调确认

### 5.3 TRT backend 返工结果

返工涉及文件：

- `my_devs/groot_trt_async_server/groot_trt_policy.py`

已声称修复的问题：

- 对共享 `TrtSession` 增加了显式串行化保护
- 在返回前同步 CUDA stream，降低 execution context 复用竞态
- 对 image-token slots 与 ViT tokens 做了严格一致性校验

返工后仍明确存在的验证缺口：

- 只做了语法级验证和轻量检查
- 还没有真实 gRPC 并发压测
- 还没有真实 processor 样本的正反例校验
