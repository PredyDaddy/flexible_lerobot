## 2. 本轮整改目标

本轮整改不再扩新功能，只处理联调准入前必须关闭的问题：

1. 关闭 `server/config` 控制面上的结构性 P0
2. 关闭 `client/protocol` 上的 request/ack 级别 P0
3. 补齐最小运行环境与启动核验路径
4. 为 `TRT backend` 补上最小可信验证闭环

本轮整改完成后，才允许启动下一轮准入复核。

## 3. 已确认的高优先级问题

### 3.1 P0：运行环境硬阻塞

- `lerobot_flex` 环境缺少 `grpc`
- 当前 server/client 入口无法在同一环境内完成真实启动验证

### 3.2 P0：server 资源所有权边界错误

- client 仍在决定 server 本地加载哪套 checkpoint / engine
- 这不符合稳定控制面的边界定义

### 3.3 P0：sticky session 仍是拒绝型约束

- 目前绑定更接近 `context.peer()`
- 没有 release / reconnect / takeover 机制

### 3.4 P0：client 协议仍是弱 request id

- `request_id` 没有真正进入协议边界
- 没有显式 ack
- 旧 chunk 晚到、错配清 pending、重复补货风险仍未关闭

### 3.5 P1：mock / backend 验证证据不足

- `mock roundtrip` 仍依赖真实 robot/camera
- `TRT backend` 当前更多是代码层补丁，缺少并发与样本验证证据

## 4. 整改工作包

### A. server/control-plane 整改包

负责人类型：

- 实现子代理 A

写入边界：

- `my_devs/groot_trt_async_server/configs.py`
- `my_devs/groot_trt_async_server/policy_server.py`
- `my_devs/groot_trt_async_server/run_server.py`

必须关闭的问题：

- server 不再直接信任 client 下发的本地资源路径作为最终加载决策
- sticky session 从“拒绝型约束”升级为“可恢复会话模型”
- `--dry-run` 至少能覆盖到真实启动前的关键依赖与关键构造路径

允许的实现方向：

- 用 server-side 资源描述符、资源注册名、白名单映射或等价机制替代 client 直接决定本地资源路径
- 用显式 session id / robot id / owner token 或等价模型替代单纯依赖 `context.peer()`
- 给出 release / reconnect / safe takeover 的明确策略

禁止事项：

- 不允许继续扩大 client 对 server 本地资源布局的控制权
- 不允许只补文档，不补运行时约束

交付物：

- 代码改动
- 简短整改说明
- 可执行的最小验证命令

### B. client/protocol 整改包

负责人类型：

- 实现子代理 B

写入边界：

- `my_devs/groot_trt_async_server/robot_client.py`
- `my_devs/groot_trt_async_server/run_client.py`
- `my_devs/groot_trt_async_server/run_mock_roundtrip.py`

必须关闭的问题：

- `request_id` 必须成为协议字段，而不是只存在于 client 本地
- 必须形成最小显式 ack 语义
- 必须定义旧 chunk / 晚到 chunk / 重复 chunk 的处理规则
- `mock roundtrip` 必须能提供纯协议级验证路径，不能强依赖真实 robot/camera

允许的实现方向：

- 在 observation / action 消息里显式携带 request 标识
- 由 server 返回 request 对应关系或等价 ack 信息
- 在 client 端显式丢弃过期或错配 chunk
- 为 roundtrip 增加纯 mock 模式，或提供单独的协议验证入口

禁止事项：

- 不允许继续用“本地推断 pending 状态”替代协议级闭环
- 不允许把“半实物冒烟”继续包装成“纯协议验证”

交付物：

- 代码改动
- 协议语义说明
- 至少一个纯 mock 验证入口或命令
