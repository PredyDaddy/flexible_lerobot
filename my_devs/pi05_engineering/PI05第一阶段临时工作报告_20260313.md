这份报告用于记录截至 2026-03-13，我围绕 PI05 本地 Chunk 实时运行时第一阶段所完成的工作。它不是最终验收报告，也不是上机结项文档，而是一份当前阶段的临时工作总结，重点回答以下问题：

## 2. 工作背景与执行原则

本轮工作严格遵循三份 PI05 工程文档的边界约束，核心原则如下：

1. 第一阶段所有新增代码默认放在 `my_devs/pi05_engineering/` 下。
2. 不改动 `src/lerobot/async_inference/` 的通用 server/client 体系。
3. 不修改 `src/lerobot/policies/pi05/` 的模型结构。
4. 不破坏 baseline `my_devs/train/pi/so101/run_pi05_infer.py` 的既有主行为。
5. 不做任何真实机器人、真实相机、真实串口相关上机动作。
6. 所有验证仅限离线 dry-run、offline load smoke、unit test、mock test。
7. 开发与测试命令统一使用 `lerobot_flex` 环境。

换句话说，本轮工作目标不是把 PI05 做成最终通用异步框架，而是先把第一阶段定义的本地双线程 chunk runtime 做到：

- 路径清晰
- 语义正确
- 指标可见
- 可离线验证
- 保留后续上机前的安全边界

## 3. 本轮工作的推进方式

本轮我是按照“架构师派工”的模式推进，而不是直接上手把所有代码从头写完。实际执行流程如下：

### 3.1 第一轮：阅读文档并建立要求清单

我先通读了以下三份文档：

1. `PI05本地Chunk实时运行时实施Plan.md`
2. `PI05本地Chunk实时运行时技术方案.md`
3. `PI05第一阶段开发与测试执行清单.md`

然后把文档中的硬要求拆成了可检查项，主要包括：

- baseline 共享加载语义是否保留
- 新入口是否走 `predict_action_chunk()`
- `original_actions` / `processed_actions` 边界是否清晰
- queue controller 是否具备 plain/rtc 双模式语义
- startup barrier、安全停机、empty queue 行为是否落地
- metrics/logging 是否满足第一阶段要求
- RTC plumbing 是否接上但默认关闭
- 是否存在离线可执行的 load smoke
- 是否存在足够的单测/入口测试

### 3.2 第二轮：派审核 Agent 做独立审查

我没有直接相信当前 `my_devs/pi05_engineering/` 的实现状态，而是先派独立上下文的 Agent 去做静态审核，重点审查：

1. 文档要求和当前实现之间是否存在语义偏差。
2. 现有测试是否足够覆盖第一阶段 contract。
3. 是否有高优先级 bug 或运行时设计缺口。

审核阶段得到的关键结论包括：

- `actions-per-chunk` 的实现曾经错误地把 `original_actions` 也裁短，破坏 leftover/merge 语义。
- queue controller 早期缺少线程同步，双线程 producer/actor 有竞态风险。
- runtime state 早期对 RTC 关键状态暴露不足。
- `action_index_delta` 与 wall-clock `real_delay` 的交叉校验缺失。
- actor/produder 频率指标口径不够准确。
- 缺少 `first_warmup_latency`。
- `RTC off` 时没有显式清空 checkpoint 里可能存在的 `rtc_config`。
- 缺少入口级 offline load smoke、startup timeout cleanup、worker exception cleanup 等关键测试。

这些结论直接形成了后续开发补齐的任务清单。

### 3.3 第三轮：派开发 Agent 做缺口补齐

在审核结果基础上，我将开发任务限定在第一阶段范围内，要求只补缺口，不做无关扩展。补齐内容主要集中在：

- `run_pi05_chunk_infer.py`
- `runtime/producer_loop.py`
- `runtime/queue_controller.py`
- `runtime/runtime_state.py`
- `runtime/metrics.py`
- 部分测试文件

### 3.4 第四轮：重新离线验证并收尾

在收集到开发补齐结果后，我又做了两轮收尾：

1. 跑完整的离线 `pytest` 集合，识别真实失败点。
2. 针对失败的入口级测试和总时延口径做最后修正。

最终把相关测试集收敛到全绿。

## 5. 已完成工作的详细说明

下面按模块详细说明本轮已经完成的工作。

### 5.1 新入口脚本：`run_pi05_chunk_infer.py`

该入口文件已经承担第一阶段主入口职责，核心能力包括：

1. CLI 参数解析。
2. runtime config 构建。
3. repo root、policy path、robot config 的统一初始化。
4. 加载 policy、preprocessor、postprocessor。
5. 构建本地 queue controller、metrics、runtime state。
6. 启动 producer / actor 双线程。
7. 统一 shutdown、异常停机和资源清理。

本轮在这个入口上已经明确补齐了以下关键点：

#### 5.1.1 `dry-run` 路径

`--dry-run true` 已经可以验证：

- repo root 解析
- policy path 解析
- policy type 读取
- task / FPS / queue 参数解析
- 不触发任何硬件连接

并且日志中明确说明：

- `dry-run` 只验证 CLI/config/path wiring
- 不证明 `predict_action_chunk()` 已经打通
- 不会连接机器人

#### 5.1.2 `offline-load-smoke` 路径

新增了 `--offline-load-smoke true` 的入口路径，用于补足文档中明确要求的 pure load smoke。

该路径会实际执行：

- 本地 tokenizer 目录检查
- `strict=False` 的 policy 加载
- checkpoint-local `policy_preprocessor.json` 加载
- checkpoint-local `policy_postprocessor.json` 加载

同时保证：

- 不构建 robot
- 不调用 `robot.connect()`
- 不触发硬件连接

这个能力是第一阶段非常关键的补点，因为它弥补了 `dry-run` 无法证明真实加载链路的缺口。

#### 5.1.3 `RTC off` 语义明确化

入口在 `enable_rtc=false` 时，已经显式设置：

- `policy.config.rtc_config = None`

这样可以避免 checkpoint 自带的 RTC 配置残留影响 plain chunk 模式。这一点是审核阶段明确指出的问题，现在已经修正。

#### 5.1.4 启动 barrier 与统一停机

入口已经具备：

- 等待首个 chunk ready 的 startup barrier
- startup timeout 后的统一 cleanup
- `KeyboardInterrupt` 触发的统一 stop
- 线程退出后的统一 `join`
- `disconnect` 清理
- 最终 observability 日志输出

此外，startup timeout 的异常消息和 `state.stop_reason` 已经统一成同一条语义，避免“抛错内容”和“运行时状态记录”不一致。

### 5.2 共享 helper：`runtime/common.py`

这个模块承接了从 baseline 提取出来的共享初始化逻辑，当前已经具备以下内容：

1. `resolve_repo_root(...)`
2. `parse_bool(...)`
3. `env_bool(...)`
4. `maybe_path(...)`
5. `ensure_local_tokenizer_dir(...)`

7. `build_so101_robot_config(...)`
8. `build_dataset_features(...)`
9. `load_policy_and_processors(...)`

这部分工作解决了文档 Phase 1 的一个核心要求：

- 把 baseline 中“已知可用但比较脆弱”的加载语义集中起来
- 避免新入口和旧入口各写一套加载路径

并且这里保留了 baseline 的关键语义：

- `policy_class.from_pretrained(..., strict=False)`
- checkpoint local pre/post processor
- repo root 下 tokenizer fallback

### 5.4 运行时状态：`runtime/runtime_state.py`

这部分是本轮重点增强的模块之一。当前 state 不再只是一个简单的 stop flag，而是扩展成了一个线程共享的运行态快照中心。

已经支持记录的关键状态包括：

- `running`
- `stop_reason`
- `first_chunk_ready`
- `producer_iterations`
- `actor_iterations`
- `last_producer_iteration_at_s`
- `last_action_sent_at_s`

- `last_action_index_delta`
- `last_inference_delay`

- `last_merge_mode`

- `last_queue_depth_after_merge`
- `last_trimmed_prefix_steps`

- `last_error`

这意味着第一阶段文档里要求的“RTC 状态链可见、可调试、可解释”，现在基本有了落点。

### 5.5 队列控制：`runtime/queue_controller.py`

该模块是第一阶段的关键模块之一。当前做法不是重写底层 queue，而是在 `src/lerobot/policies/rtc/action_queue.py` 之上做 runtime 层的封装。

已经具备的能力包括：

#### 5.5.1 双模式 merge

- `merge_plain(...)`
- `merge_rtc(...)`

#### 5.5.2 stale-prefix trimming

plain chunk 模式下已经支持基于 `real_delay` 做 stale prefix trimming，不再只是底层 queue 的简单 append。

#### 5.5.3 完整保留 `original_actions`

这个点是本轮修复重点：

- 现在 `processed_actions` 只会按 `actions_per_chunk` 裁短
- `original_actions` 继续保留完整 chunk，用于 leftover / merge / RTC bookkeeping

这符合文档要求的语义边界。

#### 5.5.4 线程同步

`QueueController` 当前已经加上线程锁，保护以下共享状态：

- `_queue`
- `_last_sent_action`
- `_original_backlog`
- `_counters`

因此 producer / actor 双线程下的竞态风险已经明显降低。

#### 5.5.5 empty queue 安全策略

当前支持：

- `hold-last-action`
- `skip-send`
- `raise`

并且在：

- 没有 `last_action`
- 需要退化到 `skip-send`

时已经会打 warning，而不是静默返回。

### 5.6 Chunk 推理链：`runtime/chunk_inference.py`

当前已经把 PI05 的 chunk 推理路径拆成了独立模块，用于承接：

1. raw observation
2. dataset frame build
3. batch prepare
4. `preprocessor(...)`
5. `policy.predict_action_chunk(...)`
6. `postprocessor(...)`

输出上明确区分了：

- `original_actions`
- `processed_actions`

并保证：

- 不会从 `processed_actions` 回推 `original_actions`
- `original_actions` 只服务于 RTC leftover / merge / delay bookkeeping
- `processed_actions` 只服务于执行队列和 actor rollout

### 5.7 Producer loop：`runtime/producer_loop.py`

producer loop 本轮补齐了很多关键 contract，当前主要负责：

1. 判断 queue 是否需要 refill
2. 获取 observation
3. 触发 chunk inference
4. 根据 plain / rtc 模式执行 merge
5. 记录 delay、leftover、chunk metrics
6. 标记 first chunk ready

本轮重点修复点包括：

#### 5.7.1 `actions_per_chunk` 语义纠正

现已改为：

- `original_actions` 保留完整 chunk
- `processed_actions` 按执行预算裁成前缀

从而满足文档里“执行前缀可控，但模型 chunk 语义不丢”的要求。

#### 5.7.2 action index 与 wall-clock delay 的交叉校验

producer 现在会记录：

- `action_index_before_inference`
- `action_index_after_inference`
- `action_index_delta`
- `real_delay`

并在两者不一致时记录 warning，同时在 metrics 中累计 `delay_mismatch_count`。

#### 5.7.3 producer loop 频率口径修正

现在 producer 每次循环都会：

- `state.note_producer_iteration(...)`
- `metrics.record_producer_iteration(...)`

而不再只在真正执行 refill 时计数。

这样 producer 侧的 loop step 和 rate 指标更接近文档要求。

#### 5.7.4 total latency 口径修正

本轮最后一项重要收尾，是把 `total_s` 的计时起点前移到了：

- observation frame build
- batch prepare

之前。

这意味着现在 `total_latency` 更接近文档所要求的整条 chunk path 总耗时，而不只是 preprocessor/model/postprocessor 的内部耗时。

### 5.8 Actor loop：`runtime/actor_loop.py`

actor loop 当前已经具备以下职责：

1. 等待 startup barrier
2. 从 queue 中取下一条 processed action
3. 做 empty queue 策略处理
4. 做 `max_action_delta` 校验
5. 转成 robot action dict
6. 调用 send_action
7. 记录 actor metrics

当前已经具备的关键 contract 包括：

- first chunk 未 ready 前不误发动作
- empty queue 时按配置处理
- `hold-last-action` 与 `skip-send` 策略可区分
- `max_action_delta` 越界会触发 stop
- send 失败会记录异常并停机


同时最终指标中还出现了几个非常关键的信号：

### 8.2 当前对问题的判断与代码证据

基于这次实机日志、后续代码路径检查，以及多名独立审查 Agent 的复核，目前已经不只是“初步判断”，而是已经形成了较明确的问题拆解。

#### 8.2.1 问题一：当前双线程 runtime 在并发访问同一个 robot 串口

机器人实现中：

- `get_observation()` 会走 `bus.sync_read("Present_Position")`
- `send_action()` 会走 `bus.sync_write("Goal_Position", ...)`

而当前 plain chunk runtime 是：

- producer 线程持续读取 observation
- actor 线程持续发送 action

如果这两个线程没有在 robot I/O 层做显式串行化保护，那么就很容易在同一个串口上形成并发访问。

这与本次错误：

- `[TxRxResult] Port is in use!`

是高度一致的。

进一步的代码检查已经确认：

- 当前 `my_devs/pi05_engineering/run_pi05_chunk_infer.py` 在 `enable_rtc=false` 时，只是把 `policy.config.rtc_config` 置空，并不会取消 producer / actor 双线程。
- `build_producer_target(...)` 直接把 `context.robot.get_observation` 交给 producer 线程。
- `build_actor_target(...)` 直接把 `context.robot.send_action` 交给 actor 线程。
- `src/lerobot/robots/so_follower/so_follower.py` 中，`get_observation()` 直接走 `bus.sync_read("Present_Position")`，`send_action()` 直接走 `bus.sync_write("Goal_Position", ...)`。
- 当前 PI05 本地 runtime 并没有在 `robot` 这一层额外包一层共享锁。

更关键的是，官方参考实现 `examples/rtc/eval_with_real_robot.py` 已经明确给出了规避方式：

- 参考实现定义了 `RobotWrapper`。
- `RobotWrapper` 内部使用 `Lock()`。
- `get_observation()` 与 `send_action()` 都在同一把锁下串行执行。

因此，这次 `Port is in use!` 更像是“当前进程内部对同一总线的并发访问”，而不是外部另一个进程长期占用了 `/dev/ttyACM0`。

因此，当前最可疑的第一问题不是 checkpoint，不是模型，而是：

> 同一个 `robot` 对象上的 read/write 在双线程下没有做好串口访问互斥。

#### 8.2.2 问题二：plain chunk 的 refill 时机偏晚，导致 actor 频繁 fallback 到 hold-last-action

当前日志里：

- `queue_empty_events = 32`
- `queue_held_last_action_events = 32`

这说明 actor 在不少周期里拿不到新的 processed action，只能重复上一条动作。

结合：

- `queue_low_watermark` 默认是 `0`
- `latest_real_delay ≈ 4`
- `actions_per_chunk = 8`
- `fps = 30`

可以推断出当前 runtime 的 refill 时机太保守：

- 它基本要等队列见底才触发下一轮推理
- 但单次推理延迟大约已经等于 4 个控制步
- 当前 chunk 只有 8 步，而真实延迟已经要吃掉其中约 4 步
- 因此 producer 一旦开始算下一块 chunk，actor 很容易在这段时间里先把现有队列耗空

这也是为什么会看到：

- queue 见底
- hold-last-action 次数上升
- `action_index_delta` 与 `real_delay` 不一致 warning 频繁出现

这里还有一个容易误判的点需要写清楚：

- `delay_mismatch_count = 12`
- `queue_empty_events = 32`

这两个值不需要一一对应，因为它们不是同一口径：

- `delay_mismatch_count` 是 producer 每次推理结束时，根据 `action_index_delta` 与 `real_delay` 的比较结果累计一次。
- `queue_empty_events` 是 actor 每次从空队列取动作时累计一次。

也就是说，这些指标不是互相矛盾，而是在同时指向同一件事：

> 当前 plain chunk 在真实时序下存在比较明显的 starvation 链路。

#### 8.2.3 问题三：当前问题不只是参数问题，还包含 runtime 结构边界问题

这轮架构审查还补充发现了一个不能忽略的边界风险：

- 当前入口在 shutdown 时会做线程 `join`
- 但如果 worker 仍然卡在机器人 I/O 或相机 I/O 上，过短的等待时间可能导致上层较早进入 `disconnect`
- 这会让“线程尚未彻底退出”和“底层资源已经开始 teardown”在时序上发生重叠

这说明当前最薄弱的环节不只是 queue 参数，而是：

1. robot I/O ownership 没有被显式串行化。
2. plain mode 的补货余量策略过于激进。
3. shutdown / teardown 的边界还不够硬。

因此，当前问题必须拆成两类来处理：

- 结构问题：robot I/O 串行化、退出时序、线程与资源 ownership
- 参数问题：`queue_low_watermark`、`actions_per_chunk`、`fps`

#### 8.2.4 问题四：当前 plain chunk 还不能视为“稳定”

虽然本次实机试跑说明：

- 模型加载没问题
- 基础 runtime 能跑起来
- actor loop 频率接近目标 FPS

但由于同时出现：

- 串口冲突错误
- delay mismatch 持续出现
- queue 频繁见底
- hold-last-action 频繁触发

所以现在不能得出“plain chunk 已稳定”的结论。

更不能在这个状态下直接把 RTC 打开后就投入正式工作。

这次试跑非常有价值，因为它把问题范围大幅缩小了。

### 8.5 2026-03-15 追加验证后的状态更新

#### 8.5.1 `Port is in use!` 不能再被当作当前版本的主结论

根据当前代码状态与后续 30 秒 plain chunk 运行结果，`Port is in use!` 更像是早期版本在 producer/actor 双线程下对同一 robot I/O 缺少串行化保护所触发的问题。

- 这个问题在当前版本上已经不能再简单作为“PI05 chunk runtime 仍然无法控机”的直接证据
- 但它仍然保留为第一阶段的重要历史问题，因为它明确暴露了运行时 ownership 设计的重要性

#### 8.5.2 `fps=10` 的相机失败，本质上是参数语义问题，不是模型问题

```text
OpenCVCamera(4) failed to set fps=10 (actual_fps=15.0).
```

核心原因不是 PI05 policy 无法运行，而是入口早期把：

- 控制循环频率
- 相机配置 FPS

错误地绑定为同一个 `--fps` 参数。

这个问题后续已经通过增加 `--camera-fps` 并与 control `--fps` 解耦来修正。因此：

- `fps=10` 曾经失败，不代表“PI05 在 10Hz 下不能工作”
- 它更准确地说明“控制频率”和“相机配置频率”在运行时语义上必须分开

#### 8.5.3 早期 `run_time_s=30` 几乎未运行就退出，是计时起点过早

后续验证还暴露了另一个容易误判的问题：

- 早期 `run_time_s=30` 的统计起点过早
- 模型加载和 connect 耗时会吞掉大部分运行预算
- 导致 actor 线程实际运行前，duration limit 就已经接近到期

这个问题后续已经通过在 `connect()` 成功后重置 runtime 起始时间来修正。

因此，早期那种：

- `producer_iterations=1`
- `actor_iterations=0`
- 最终显示 duration limit reached

的现象，不能再被视作“plain chunk 主循环跑不起来”，而应视作“duration 口径曾经定义错误”。

#### 8.5.4 当前 plain chunk 的最新状态应定义为“已跑通短时保守实机运行”

基于 2026-03-15 的补充验证，当前更准确的阶段判断应当是：

> plain chunk 已经跑通一次保守参数下的 30 秒实机运行，并表现出连续发送动作、无 queue starvation、按时收尾的基本稳定性；但这还不等于已经完成长期实机验收。

### 8.6 当前剩余问题的重新归类

在补充验证之后，当前剩余问题不应再混在一起讨论，而应明确拆成三类：

#### 8.6.1 connect 生命周期风险

当前仍然可能在 `connect/configure` 阶段偶发看到：

- `Torque_Enable`
- `Incorrect status packet`

这类错误更像：

- 总线初始化瞬态
- 接触/供电/串口波动
- connect 阶段缺少 retry/backoff

而不是 chunk queue 主循环本身的确定性 bug。

#### 8.6.2 plain 模式下的 delay 观测口径仍偏保守

当前 plain 模式里的：

- `real_delay = ceil(latency / dt)`

本身就是偏保守估计，因此：

- `action_index_delta`
- `real_delay`

之间出现 1 个控制步的偏差，未必代表真实控制异常，更可能只是线程调度与 actor tick 相位差。

因此，当前 `delay_mismatch_count` 仍然有参考价值，但已经不能直接作为“系统不稳定”的单一证据。

#### 8.6.3 `latest_leftover_length` 的语义仍然不够适合 plain 模式

从最新日志看，`latest_leftover_length` 会持续增大，但同时：

- `queue_empty_events=0`
- actor 在持续消费动作
- producer 虽慢于 actor，但尚未击穿队列

这更像说明当前 leftover 更偏向：

- `original_actions` 账本

而不是：

- 真实待执行 backlog

因此，这个指标当前更像“观测语义未收敛”，而不是立即的执行故障。
