## 2. 项目背景

当前 AgileX 机器人训练使用的是：

- 训练脚本：`my_devs/train/groot/agilex/run_groot_train.sh`
- 推理目标：在显卡更强的工作站上运行 GR00T 模型推理，再把动作下发到另一台 AgileX 机器人机器上执行。

此前该目录下存在一版未完成的异步原型代码，以及若干中间方案文档、评审文档和 MVP 草稿。结合实际需求，本次工作的目标不是继续修补旧异步方案，而是优先交付一套可直接上机的、稳定的、与 `my_devs/train/groot/so101/run_groot_infer.py` 执行边界一致的远程同步推理链路。

也就是说，新的设计目标是：

1. 机器人端读取实时观测。
2. 将观测发送到远端工作站。
3. 工作站完成单步 policy 推理。
4. 将单步动作返回机器人端。
5. 机器人端把动作发布到 AgileX。

## 3. 初始评审结论

在正式开发前，对 `my_devs/train/groot/agilex` 和 `my_devs/train/groot/so101` 做了对照评审，结论如下：

### 3.1 旧异步原型不适合直接继续开发

旧原型本质上沿用了 action chunk、队列重排、观测筛选和异步聚合的执行模型，这和 `run_groot_infer.py` 当前的单步闭环推理模式不是同一种控制链路。对于真实机器人控制来说，如果目标是先稳定上机，继续在旧异步原型上迭代，风险高于收益。

### 3.2 AgileX 实际 checkpoint 合同与旧假设不一致

对当前 AgileX checkpoint 进行了实际解析，得到的结果是：

- `control_scope=right`
- `state_dim=7`
- `action_dim=7`
- `camera_keys=['camera_front', 'camera_right']`

这说明当前模型是一个右臂单臂模型，不是双臂整机模型。

但旧实现默认按如下方式工作：

- 读取双臂 `14` 维 state
- 读取三路相机 `front/left/right`
- 按双臂 `14` 维 action 直接解码

这会导致两个问题：

1. 送给模型的 observation 语义已经和训练时不一致。
2. 即使服务端推理成功，客户端在按 `14` 维去解码 `7` 维 action 时也会出错。

### 3.3 客户端存在明确的运行期 bug

在真实客户端联调中，发现 `build_agilex_hold_action()` 漏导入 `ACTION_FEATURE_NAMES`，导致机器人端首次运行时报 `NameError`。该问题已在后续修复。

## 4. 本次实现范围

本次工作对 `my_devs/train/groot/agilex` 进行了重构，核心思路是：

- 删除旧的未完成异步 MVP 原型。
- 新增一套远程同步推理服务端/客户端。
- 通过握手阶段返回 checkpoint 合同，避免机器人端写死维度和相机假设。
- 让当前单臂 checkpoint 可以直接安全控制 AgileX。

最终落地的核心文件如下：

- `remote_sync_common.py`
- `run_groot_remote_policy_server.py`
- `run_groot_remote_robot_client.py`
- `run_groot_remote_policy_server.sh`
- `run_groot_remote_robot_client.sh`
- `README.md`

## 5. 关键实现说明

### 5.1 新增统一通信与合同解析层

`remote_sync_common.py` 负责以下能力：

- socket 消息封包与解包
- 远程握手消息定义
- checkpoint 基础资产校验
- 预处理器与后处理器加载
- AgileX 观测校验
- 根据 checkpoint 自动解析真实的 policy 合同

这里新增了 `AgileXPolicyContract`，用于明确描述：

- 当前模型控制范围：`left/right/both`
- 需要哪些 state key
- 需要哪些 camera key
- 期望 action 维度和含义

### 5.2 服务端按 checkpoint 真正能力返回合同

`run_groot_remote_policy_server.py` 的行为是：

1. 加载 GR00T checkpoint。
2. 加载预处理器和后处理器。
3. 解析 checkpoint 的真实输入输出合同。
4. 在 `hello` 握手阶段，把合同返回给机器人端。
5. 在 `infer` 阶段完成单步 `predict_action(...)`。

这使得机器人端不再需要依赖写死的 `14` 维双臂逻辑。

### 5.3 客户端改为“合同驱动”运行

`run_groot_remote_robot_client.py` 不再假设模型一定是双臂模型，而是在启动后：

1. 先连接远程服务端。
2. 获取 `policy_contract`。
3. 按合同裁剪本地实时观测。
4. 将裁剪后的 observation 打包并发送。
5. 按合同解码返回的 action。
6. 将未控部分机械臂维持在当前观测位置。
7. 合成完整整机动作后再下发到 AgileX。

这套机制解决了当前右臂单臂模型无法直接控制整机的问题。

### 5.4 单臂控制策略

由于当前 checkpoint 是 `right` 臂模型，因此客户端策略是：

- 右臂：使用模型返回的 `7` 维动作。
- 左臂：保持当前实时观测到的关节位置。

这样可以满足当前单臂训练模型的真实部署需求，同时不破坏 AgileX 双臂控制消息格式。

## 9. 已知限制

### 9.1 当前模型是右臂模型

当前可用链路已经支持：

- 右臂由模型控制
- 左臂保持当前位置

如果未来训练出左臂或双臂模型，需要重新验证合同是否变为：

- `left`
- `both`
- 或新的相机组合

### 9.2 `total_latency_ms` 仅作参考

服务端日志中的 `total_latency_ms` 使用的是跨机器 wall clock 差值，如果两台机器时钟没有严格同步，该数值只能作为粗略参考，不适合作为精确性能指标。

### 9.3 当前协议基于 pickle

当前服务端/客户端协议使用 Python `pickle`。这对内部同版本仓库联调是足够的，但要求两端代码结构保持一致。如果后续要长期跨机部署，建议未来升级为更稳定的显式序列化协议。
