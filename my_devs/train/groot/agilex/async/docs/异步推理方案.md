## 1. 核心结论

本方案冻结的核心结论如下：

- **要做异步，但不要直接照搬 upstream `src/lerobot/async_inference` 的完整控制语义。**
- **第一版采用的是 bounded-async，也就是“有边界的小窗口异步预取”，不是完全自由的流式 chunk 管线。**
- **第一版继续复用当前 AgileX socket、observation 构造、policy contract 和 `merge_agilex_action` 语义，只升级控制面。**
- **第一版不冻结拍脑袋数值默认值，尤其不把 `chunk_size=2`、`low_watermark=1` 直接当成通用默认值。**
- **第一版必须同时设计失败闭环、watchdog 和 reset 语义，否则不上真机。**

## 2. 项目背景

### 2.1 当前系统现状

你当前已经有一条可工作的同步远程链路：

- 服务端运行在带 GPU 的算力服务器上
- 客户端运行在机器人端，负责 ROS 侧观测采集和动作下发
- 当前任务是 AgileX 右臂抓取箱子类任务
- 当前控制模式主要是 `command_master`
- 同步 server/client 已经能完整跑通真机控制

这意味着当前工作不是“从零设计一个机器人异步平台”，而是：

- 在一个已经可用的同步闭环上做受控升级
- 保留同步链路作为 gold baseline
- 让异步方案先证明收益，再决定是否继续复杂化

### 2.2 当前同步链路的真实语义

当前同步链路的表层流程是：

1. 机器人端读取一帧 live observation
2. 构造 policy observation frame
3. 发送到远端 server
4. server 调用 `predict_action(...)`
5. 返回单步 action
6. client 将预测动作与 `hold_action` 合并
7. 下发给机器人
8. 进入下一轮循环

但它的底层真实语义更重要：

- server 表面返回单步 action
- `predict_action(...)` 底层实际会走 `policy.select_action()`
- `select_action()` 在内部 action queue 为空时才会真正触发一次 `predict_action_chunk()`

所以当前同步方案的低效点，不只在模型推理慢，还在于：

- 网络边界放得太细
- preprocess / postprocess / 序列化 / 反序列化在每一步都重复支付
- server 内部的 chunk 优势没有通过协议层暴露给 client

## 3. 第一版为什么仍然值得做异步

当前做异步的目标不是证明“异步更先进”，而是：

- 在不明显伤害抓取成功率的前提下，提高控制流畅度
- 尽量重叠机器人动作执行与远端推理
- 把每一步都等待远端返回的同步阻塞改成小窗口预取
- 让 chunk 推理优势真正穿过网络边界

第一版不追求：

- 通用平台化
- 多机器人调度
- 高并发 session 管理
- 激进的 overlap merge 或 action aggregation

## 4. 为什么不直接采用 upstream async 运行时

### 4.1 结论

冻结结论是：

- **upstream `src/lerobot/async_inference` 值得参考拆线程方式和调试思路**
- **但不适合作为当前 AgileX 第一版真机异步实现的直接运行时基础**

### 4.2 需要纠正的一个事实

这里必须纠正一个容易说错的点：

- upstream async server 本身已经直接走 `policy.predict_action_chunk(...)`
- 它并不是因为继续复用了 GR00T 的 `select_action()` 才天然不适合

因此，本方案否定 upstream async 的原因，不是“它字面上复用了 GR00T 的内部单步 queue”，而是下面这些更实际的问题：

- 它默认接受 chunk overlap / aggregation 的设计空间
- 它带有 observation similarity filtering
- 它用通用 `chunk_size_threshold` 语义驱动发包
- 它的时间戳与 timestep 归属方式更偏向通用研发骨架，不是为当前 AgileX 抓取任务定制

### 4.3 当前任务里真正危险的点

对当前 AgileX 抓取任务，最危险的不是“名字叫异步”，而是下面这些控制语义：

- 新旧 chunk 可以重叠并被聚合
- observation 可以因为“足够相似”而被过滤
- 发包时机主要由 queue 比例而不是动作年龄和安全余量驱动
- 晚到 chunk 仍有机会参与融合

这些语义在抓取任务里风险很高，因为：

- 抓取动作通常不是线性可平均控制量
- 接触前后的动作切换需要果断，不应做模糊融合
- 视觉变化可能先于关节变化，单看关节空间相似度不够
- 旧动作一旦晚到，可能已经失去因果意义

### 4.4 对 upstream async 的冻结复用边界

第一版可以参考和吸收：

- client 将“控制循环”和“响应接收”解耦
- server 使用 `Queue(maxsize=1)` 保留最新 observation 的思想
- `TimedObservation` / `TimedAction` 一类时间戳调试方式
- latency、queue size、avg fps、must-go 类诊断指标

这里的“参考”含义是：

- 可以阅读和对照现有实现
- 可以在 `async/` 下重新实现所需逻辑
- 不允许通过修改现有同步文件来达成第一版 async 目标

第一版明确不直接继承：

- action aggregation
- overlap patching
- observation similarity filtering
- 以 `queue_size / action_chunk_size` 为核心的泛化发包策略
- 多种 aggregate function 可配置的通用语义

## 5. 冻结的总体技术路线

第一版冻结路线如下：

- 基于当前 AgileX socket 同步链路的行为语义，在 `async/` 目录下单独实现一条 AgileX 专用 bounded-async 分支
- 保留现有 observation 构造、policy contract、动作解码和 `merge_agilex_action` 的行为兼容性
- server 侧显式切换到 `predict_action_chunk()`
- client 侧维护唯一一层显式本地 action queue
- 协议层引入 session / request / terminal state
- 失败时优先进入 hold / reset / 回退，而不是停留在半异步状态

这里的“保留现有”指的是：

- 兼容其行为语义
- 不修改现有同步入口文件
- 所有实现代码继续放在 `async/` 下

第一版不冻结具体 transport 重写计划。  
在 socket 没被证明是瓶颈之前，不引入 gRPC 改造。

## 6. 冻结的控制模型

推荐方案名称：

- **Bounded Async Remote Chunk Prefetch**
- 中文可以叫: **有边界的远程小窗口异步预取**

它的冻结语义只有一句话：

- **client 在执行本地 queue 中已有动作的同时，为下一小段动作发起一个且仅一个 inflight request；server 只为该 request 产出一个最新有效 response；client 只接受与当前 pending request 匹配的 response。**

### 6.1 第一版必须满足的设计原则

1. **单机器人、单会话**
2. **单 inflight request**
3. **client 端只有一层显式 action queue**
4. **server 返回 chunk，不再暴露单步返回语义**
5. **latest-valid only**
6. **旧包、晚到包、重复包、foreign session 包严格丢弃**
7. **每个 request 都必须进入显式终态**
8. **异步失败时优先降级，不允许卡死在中间态**

### 6.2 第一版 queue 明确不做什么

第一版 queue 不做：

- old/new action averaging
- same-timestep merge
- conservative blending
- overlap patching
- 基于时间戳的复杂重排

第一版 queue 只做：

- 顺序追加
- 顺序弹出
- 结合 pending request 校验后严格入队
- 断粮时走 hold / retry / reset 语义

## 7. 冻结的协议与状态机边界

### 7.1 消息类型

保留当前 `RemoteMessage(kind, payload)` 外层包装，新增异步专用 payload。

第一版至少需要这些消息：

- `hello`
- `ready`
- `infer_request`
- `infer_response`
- `reset`
- `reset_ack`
- `close`
- `close_ack`
- `error`

### 7.2 `infer_request` 冻结字段

`infer_request` 至少包含：

- `request_id`
- `session_id`
- `capture_ts`
- `must_go`
- `queue_size_at_capture`
- `latest_action_id_at_capture`
- `observation`

第一版不要求显式传 `action_timestep_start`。  
在单 inflight、单 queue 语义下，本地顺序追加即可。

### 7.3 `infer_response` 冻结字段

`infer_response` 至少包含：

- `request_id`
- `session_id`
- `status`
- `reason`
- `server_received_at`
- `server_sent_at`
- `predict_latency_ms`
- `actions`

`status` 只允许三种：

- `ack`
- `retry`
- `abort`

### 7.4 协议不变量

以下约束已经冻结：

1. 任意时刻最多只有一个 pending request
2. `infer_response.request_id` 必须匹配当前 pending request
3. `session_id` 不匹配的 response 必须直接丢弃
4. 已进入 terminal state 的 request 不能再次生效
5. client 在 `pending_request != None` 时不能继续发新 request
6. terminal state 既可以来自 server response，也可以来自 client watchdog 的本地裁决

第 6 条是本次冻结新增约束。  
它的意义是：**request 的终态不能只依赖服务端回包。**

## 8. 冻结的 timeout / watchdog 语义

这是第一版必须补上的关键语义。

### 8.1 为什么必须有 client watchdog

如果系统只有 server 回包才能终结一个 request，那么一旦发生下面任何情况，单 inflight 设计就会卡死：

- response 丢失
- TCP 半开
- server 已经异常退出
- server 算完但 client 没收到
- session 已失效但 client 仍然认为 request 存活

因此，第一版冻结如下规则：

- client 为每个 pending request 维护 `sent_at` 和 `deadline`
- 一旦超过 `deadline`，client 必须主动做本地裁决
- 本地裁决只能将请求推进到 `retry` 或 `abort`
- request 的终态不允许悬空

### 8.2 本地裁决规则

第一版冻结的本地超时规则如下：

1. 单次超时且连接仍可恢复时，优先本地转成 `retry`
2. 连续超时超过预算，或连接状态不可信时，转成 `abort`
3. `abort` 后必须进入 hold / reset / 重新握手，不能继续沿用旧 session

### 8.3 `response_timeout_s` 的冻结原则

第一版冻结的是计算原则，不是硬编码默认值：

- `response_timeout_s` 必须由同步基线和 async 实测时延共同反推
- 它不应在设计文档中被硬编码为固定秒数
- 任何实现都必须在日志中输出 timeout 取值和取值依据

## 9. 冻结的 buffer sizing 原则

这是本次冻结新增的第二个关键内容。

### 9.1 不把 `chunk_size=2` 直接冻结成默认值

此前把 `chunk_size=2`、`low_watermark=1` 当作第一版默认值的说法不够严谨。  
它只能作为候选起点，不能在没有基线测量前直接冻结成通用默认值。

原因很简单：

- 控制周期取决于真实 `control_fps`
- 可用安全余量取决于 `ack` 时延分布，而不是主观感觉
- 如果 `ack` 的 `p95` 已经明显大于一个控制周期，那么 `low_watermark=1` 会天然断粮

### 9.2 冻结的反推公式

第一版参数必须按以下原则反推：

- `control_period_s = 1 / control_fps`
- `required_buffer_steps = ceil(target_p95_ack_latency_s / control_period_s)`

冻结约束如下：

- `low_watermark` 不能小于 `required_buffer_steps`
- `chunk_size` 不能小于 `low_watermark + 1`
- 在未完成同步基线测量前，不冻结数值型默认值

### 9.3 数值试探的冻结边界

Phase 0 之后，如果测量表明系统满足预算，`chunk_size=2` 才能作为最保守试探值。  
如果预算不满足，应优先修时延问题，而不是直接上真机盲试。

## 10. 冻结的模型边界

### 10.1 server 必须显式切换到 `predict_action_chunk()`

第一版冻结如下：

- 同步链路允许继续使用 `predict_action()`
- 异步链路必须显式使用 `predict_action_chunk()`

这么做的理由是：

1. 避免继续暴露同步单步语义
2. 让网络边界和模型 chunk 边界对齐
3. 让 client 真正拥有 queue 消费与 refill 的控制权

### 10.2 对 `predict_action_chunk()[:K]` 的冻结解释

第一版可以只消费 server 返回 chunk 的前 `K` 步，但必须接受一个事实：

- 这不保证与当前同步单步 pop 语义完全等价

因此，第一版冻结如下门槛：

- 在真机联调前，必须先做离线 replay gate
- replay gate 不要求“动作逐点完全一致”
- replay gate 必须证明调度语义、动作年龄和 underflow 风险可解释

## 11. 冻结的 session / reset / episode 语义

这是第一版必须补上的第三个关键内容。

### 11.1 session 的目标

session 的目标不是做平台化管理，而是防止：

- client 重启后收到旧响应
- 多端误连
- server reload 后旧响应仍被接受

### 11.2 session 的冻结语义

第一版冻结如下：

- 单 active session
- client 启动时生成新的 `session_id`
- server `hello` 成功后绑定 active session
- reconnect 或 reset 后允许新 session 覆盖旧 session
- 旧 session 响应一律按 foreign session 丢弃

### 11.3 reset 的冻结语义

第一版必须支持显式 reset。  
reset 不是可选调试功能，而是 episode boundary 的正式控制语义。

以下场景必须触发 reset 或等价的新 session 建立：

- 新 episode 开始前
- 人工复位后
- `abort` 后
- reconnect 后
- server policy runtime 明显失去可信状态后

### 11.4 reset 时必须清理的状态

一旦进入 reset，以下状态必须被视为无效并清理：

- client 本地 action queue
- pending request
- latest live observation cache 中与旧 request 强绑定的状态
- server 端 policy internal state
- server 端 preprocessor / postprocessor state

reset 成功前，client 不得继续消费旧 queue 并推进行为。

## 12. 与 AgileX 现有链路的兼容要求

第一版异步实现必须继续保留以下已验证语义：

### 12.1 control arm contract

必须继续沿用现有 `policy_contract`：

- `left`
- `right`
- `both`
- `auto`

### 12.2 `merge_agilex_action` 语义

必须继续保留：

- policy 只预测受控 arm 动作
- 非受控部分由 `hold_action` 填充
- 真正下发前通过 `merge_agilex_action` 得到完整机器人动作

### 12.3 live observation cache

第一版允许维护 `latest_live_observation`，用于：

- 生成最新的 `hold_action`
- 在 queue 断粮或 reset 期间保持动作安全

第一版不允许为了刷新 hold action 单独引入高成本观测抓取链路。

## 14. 冻结的验收门槛

### 14.1 协议层

必须满足：

- 单 inflight request 不被破坏
- 旧 response 不会误入队
- foreign session response 不会误入队
- watchdog 超时后 request 能进入本地终态
- retry 后系统能恢复发新 request
- abort 后系统能进入 hold / reset / stop 状态

### 14.2 离线 gate

在上真机前，必须通过离线 replay gate。  
该 gate 至少证明：

- request / response 状态机稳定
- 没有悬空 pending request
- 没有非法 response 入队
- 动作年龄和 underflow 风险可解释

### 14.3 真机 gate

最关键门槛如下：

- **异步方案的真实抓取成功率不得明显低于同步方案**

不允许用“看起来更顺”替代真实 A/B 数据。

## 15. 第一版明确不做的事情

为了控制复杂度，第一版冻结不做这些事：

1. 多机器人共享一个 remote server
2. 多 inflight window
3. chunk overlap merge
4. action averaging / weighted average
5. observation similarity filtering
6. phase-aware chunk size
7. server 资源 profile 平台化
8. gRPC 重写

## 16. 最终冻结结论

对于当前这条 AgileX 远程 GR00T 真机链路，第一版异步的冻结结论是：

- **继续沿用当前同步方案里已经验证过的特征处理、policy contract、AgileX action merge 和 socket transport 语义**
