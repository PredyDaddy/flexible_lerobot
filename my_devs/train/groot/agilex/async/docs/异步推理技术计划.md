## 1. 计划目标

1. 按什么顺序落地冻结方案

## 2.1 冻结约束到执行任务的映射

为了避免 plan 漂成“有阶段、没 owner”，下面这些冻结约束必须有明确落地责任：

1. `response_timeout_s` 必须由同步基线、async live latency 和 replay 共同反推
   - owner: Phase 0、Phase 4、Phase 5、Phase 6
2. reset 必须覆盖 episode boundary / abort / reconnect / runtime 不可信
   - owner: Phase 1、Phase 2、Phase 4
3. bounded-async 发包语义必须显式实现，不得退化成泛化 queue-threshold 策略
   - owner: Phase 1、Phase 2、Phase 4
4. `close` / `close_ack` / `error` 必须作为正式协议路径落地
   - owner: Phase 1、Phase 2、Phase 3、Phase 4
5. 冻结方案要求的关键指标必须有记录 owner
   - owner: Phase 0、Phase 3、Phase 4、Phase 5、Phase 6、Phase 7

## 4. 分阶段实施计划

### Phase 0: 同步基线测量

### 目标

在不修改现有同步链路文件的前提下，于 `async/` 目录内侧挂实现 baseline 测量工具，测清时延分布和控制预算。

### 要做的事

1. 在 `async/` 下新增 `sync_baseline_probe.py` 和配套启动脚本。
2. 基于现有同步行为语义，在 `async/` 目录内实现 baseline probe：
   - client 侧等价 observation build 测量
   - socket send / receive 测量
   - end-to-end round trip 测量
   - server 侧 preprocess / infer / postprocess / serialize / total 测量
3. 如需对齐现有同步入口，只允许把原脚本作为黑盒对照，不允许改原脚本加日志。
4. 记录分位数而不只记录平均值：
   - `p50`
   - `p95`
   - `p99`
5. 输出基线报告，并给出第一版 async 的预算建议：
   - `response_timeout_s`
   - `required_buffer_steps`
   - 候选 `low_watermark`
   - 候选 `chunk_size`
6. 明确写出预算只来自同步链路，不作为最终 timeout 定值：
   - 只能给出初始上界
   - 不能替代 async live latency 标定

### 产物

- `async/` 下的 baseline probe 代码
- 一份 baseline 报告文档

### 通过标准

- 不修改任何只读同步文件
- 能稳定产出分位数统计
- 给出可解释的 buffer sizing 建议
- 明确标注这些建议仍需 async live latency 二次校准

### Phase 1: 异步协议公共模块

### 目标

先把异步协议和状态机写清楚，暂时不接真模型和真机器人。

### 要做的事

1. 新建 `remote_async_common.py`
2. 定义异步专用 dataclass / payload：
   - `AsyncHello`
   - `AsyncReady`
   - `AsyncInferRequest`
   - `AsyncInferResponse`
   - `AsyncReset`
   - `AsyncResetAck`
   - `AsyncClose`
   - `AsyncCloseAck`
   - `AsyncError`
3. 定义状态机约束：
   - pending request 生命周期
   - server terminal state
   - client watchdog terminal state
   - request 发包准入条件
   - `must_go` 转换条件
   - reset 触发矩阵
4. 封装通用 helper：
   - session id 生成
   - request id 递增
   - queue metric 计算
   - timeout deadline 计算
   - response 校验
   - `can_send_request(...)` 判定
   - `should_force_must_go(...)` 判定
   - reset state cleanup helper

### Phase 1 必须写死的控制语义

这里不是写 dataclass 名称就算完成，还必须把下面这些规则落成代码级状态机注释和 helper：

1. 只有在 `pending_request is None` 且 `queue_size <= low_watermark` 时允许发新 request
2. queue 断粮时必须触发 `must_go`
3. `infer_request` 必须携带：
   - `queue_size_at_capture`
   - `latest_action_id_at_capture`
   - `must_go`
4. response 只允许匹配当前 pending request 生效
5. 不允许退化成“按 queue 比例持续发 observation”的通用策略

### 产物

- `remote_async_common.py`

### 通过标准

- 协议字段与冻结方案一致
- terminal state 支持 server 和 client 两种来源
- reset 语义在公共模块层面可表达
- `close` / `close_ack` / `error` 路径可表达
- 发包准入和 `must_go` 语义有明确 helper 和注释

### Phase 2: Mock Roundtrip

### 目标

在不接真实 GR00T 和真机的情况下，先把 request / ack / retry / abort / reset 跑通。

### 要做的事

1. 新建 `mock_roundtrip.py`
2. 实现一个 mock server：
   - 可返回 `ack`
   - 可注入 `retry`
   - 可注入 `abort`
   - 可模拟慢响应
   - 可模拟丢响应
3. 实现一个 mock client：
   - 维护 pending request
   - 维护本地 queue
   - 实现 watchdog
   - 实现 reset
4. 覆盖关键场景：
   - 正常 ack
   - 响应超时后本地 retry
   - 连续失败后 abort
   - reset 后旧 response 丢弃
   - foreign session response 丢弃
   - `close` / `close_ack`
   - `error` 回传
   - queue 断粮触发 `must_go`
   - pending request 存在时禁止重复发包
   - abort 后进入 reset 或新 session

### 产物

- `mock_roundtrip.py`
- mock 协议日志

### 通过标准

- 没有悬空 pending request
- reset 后状态可重新进入稳定工作
- 旧 session / 旧 request 响应不会误入队
- `must_go` 和单 inflight 约束不会被破坏

### Phase 3: 真实 server 最小闭环

### 目标

让异步 server 用真实 GR00T runtime 跑 `predict_action_chunk()`，但先不接真机动作下发。

### 要做的事

1. 新建 `run_groot_remote_async_policy_server.py`
2. 对照现有实现，在 `async/` 目录内重建这些能力：
   - checkpoint 加载
   - processor 加载
   - policy contract 解析
   - socket framing
3. 改成显式 chunk 推理路径：
   - 不走 `predict_action()`
   - 不走 `select_action()` 暴露的单步语义
   - 直接调用 `predict_action_chunk()`
4. 显式记录：
   - preprocess
   - infer
   - postprocess
   - serialize
   - total
   - `queue_wait_ms`
   - `chunk_size`
5. 实现 reset：
   - policy reset
   - preprocessor reset
   - postprocessor reset
6. 实现协议补全：
   - `error`
   - `close`
   - `close_ack`
7. 输出 async server 侧尾延迟统计：
   - `p50`
   - `p95`
   - `p99`

### 产物

- `run_groot_remote_async_policy_server.py`

### 通过标准

- server 真实返回 chunk
- reset 后 runtime 状态可恢复
- 日志可区分 ack / retry / abort 来源
- server 侧尾延迟统计可用于 timeout 初步校准

### Phase 4: 真实 client 最小闭环

### 目标

让异步 client 接真实 observation、真实 server、真实 queue，但严格停留在 dry-consume，不允许任何 actuation / bringup 外扩。

### 要做的事

1. 新建 `run_groot_remote_async_robot_client.py`
2. 对照现有实现，在 `async/` 目录内重建这些能力：
   - AgileX observation 获取
   - observation frame 构造
   - `policy_contract`
   - `decode_agilex_policy_action`
   - `build_agilex_hold_action`
   - `merge_agilex_action`
3. client 内部拆成至少两个线程：
   - 控制循环线程
   - 响应接收线程
4. 实现关键状态：
   - local action queue
   - pending request
   - watchdog deadline
   - latest live observation cache
   - session state
   - `must_go`
   - `latest_action_id`
5. 先支持 dry-consume：
   - 出队
   - 统计
   - 生成将要发送的动作
   - 但不真正执行危险动作
6. 实现 reset 流程：
   - 清本地 queue
   - 清 pending request
   - 重建 session 或向 server 发 reset
7. 把 reset 触发矩阵真正接上运行时：
   - 新 episode
   - 人工复位
   - `abort`
   - reconnect
   - server runtime 不可信
8. 采集 async live 端到端时延：
   - request send 到 response receive
   - `p50`
   - `p95`
   - `p99`
9. 基于 live async latency 和同步 baseline 二次标定：
   - `response_timeout_s`
   - `required_buffer_steps`
   - 候选 `low_watermark`

### 产物

- `run_groot_remote_async_robot_client.py`

### 通过标准

- 没有 server 回包时也不会永久卡死
- timeout 后能进入 retry / abort
- reset 后旧 response 不会误生效
- Phase 4 结束时仍然不允许真实 actuation
- `response_timeout_s` 已结合 live async latency 完成二次校准

### Phase 5: 离线 Replay Gate

### 目标

在真机 bringup 前，用离线观测序列验证调度语义和风险边界。

### 要做的事

1. 新建 `replay_eval.py`
2. 输入一段记录好的 observation 序列
3. 分别运行：
   - 当前同步边界
   - 异步 chunk 边界
4. 输出对比指标：
   - `queue_underflow_rate`
   - `action_age_at_execution_ms`
   - `pending_request_timeout_count`
   - `late_response_drop_count`
   - `foreign_session_drop_count`
5. 基于 replay 结果产出参数建议：
   - 第一组候选 `low_watermark`
   - 第一组候选 `chunk_size`
   - 动作年龄上界建议

### 产物

- `replay_eval.py`
- replay 报告

### 通过标准

- 不要求动作逐点一致
- 必须证明状态机稳定
- 必须证明动作年龄和 underflow 风险可解释
- replay 结果只参与 buffer 和调度风险判断，不单独决定 timeout

### Phase 6: 真机受控 Bringup

### 目标

在最保守配置下上真机，但把风险控制放在第一位。

### 要做的事

1. 只使用 Phase 0 和 Phase 5 反推得到的候选参数
2. 必须同时吸收 Phase 4 的 async live latency 校准结果
3. 首轮只跑短时段
4. 首轮必须重点看：
   - `queue_underflow_rate`
   - `action_age_at_execution_ms`
   - `response_stale_or_drop_rate`
   - `retry_count`
   - `abort_count`
5. 一旦出现以下情况，立即停止推进：
   - abort 频繁
   - underflow 持续
   - action age 明显失控
   - 抓取成功率明显低于同步基线
   - stale / drop 明显失控

### 产物

- 真机 bringup 记录
- 第一轮参数结论

### 通过标准

- 没有协议不变量被破坏
- 没有失控动作
- 成功率未明显劣化
- timeout / retry / abort 边界与 live latency 数据一致

### Phase 7: A/B 与参数调优

### 目标

只在 bounded-async 已经站稳后，再做小步调参。

### 要做的事

1. 每次只改一个核心变量：
   - `chunk_size`
   - `low_watermark`
   - `control_fps`
   - `response_timeout_s`
2. 每轮都对照同步 baseline
3. 每轮都记录：
   - 成功率
   - underflow
   - action age
   - stale / drop
   - retry / abort

### 通过标准

- 有真实 A/B 数据支持
- 参数收益可解释
- 未引入新的安全问题

## 5. 文件级实施顺序

建议严格按下面顺序落地：

1. 先补同步日志，不先写异步 server/client
2. 先写 `remote_async_common.py`
3. 再写 `mock_roundtrip.py`
4. 再接真实 async server
5. 再接真实 async client
6. 再补 `replay_eval.py`
7. 最后上真机

这个顺序的目的很明确：

- 先测预算
- 再做协议
- 再做真实 runtime
- 最后才让真机承担风险

## 6. 参数策略

第一版参数策略不是直接写死数值，而是遵守以下规则：

1. 先由 Phase 0 测得同步链路时延分位数
2. 再由 Phase 4 测得 async live 端到端时延分位数
3. 再由 Phase 5 replay 结果验证 async 调度风险
4. 然后才确定第一组真机参数

只有在预算满足时，才允许把 `chunk_size=2` 作为最保守试探值。  
如果预算不满足，优先修链路时延，不在真机上硬顶。

这里的“修链路时延”同样只能通过 `async/` 目录内实现完成，不能回头修改只读同步文件。

## 7. 验证矩阵

每阶段至少覆盖以下验证：

1. 正常路径:
   - `hello -> ready -> infer_request -> ack`
2. 超时路径:
   - pending request 超时后本地 `retry`
3. 失败路径:
   - 连续失败进入 `abort`
4. reset 路径:
   - reset 后旧 queue 和旧 response 全失效
5. reconnect 路径:
   - 新 session 建立后旧 session response 全丢弃
6. 显式关闭路径:
   - `close -> close_ack`
7. 错误回报路径:
   - `error` 能传回并触发本地裁决

## 8. 指标 Owner

为了避免“日志说要记，但没有阶段真正实现”，关键指标 owner 明确如下：

1. Phase 0:
   - 同步链路时延分位数
2. Phase 3:
   - `queue_wait_ms`
   - server `chunk_size`
   - server preprocess / infer / postprocess / serialize / total
3. Phase 4:

   - `queue_size_before_send`
   - `queue_size_after_ack`

   - `late_response_drop_count`
   - `foreign_session_drop_count`
   - `duplicate_response_drop_count`

4. Phase 5:

5. Phase 6 / Phase 7:

## 9. Review Gate

本计划在进入 Phase 3 前必须经过一次交叉审核。  
审核重点不是文风，而是下面这些问题：

1. 冻结方案中的每条关键约束，是否都落到了具体实现任务
2. 是否遗漏了 client watchdog 这条失败闭环
3. 是否遗漏了 reset / episode boundary 的实现任务
4. 是否在没有 baseline 和 replay 的情况下偷偷写死了数值默认值
5. 是否仍然把 upstream async 的不适配原因说错
6. 是否把 `close` / `error` / reconnect 路径遗漏成了后补逻辑
7. 是否把 replay gate 错当成 timeout 的唯一来源
8. 是否存在通过复用 helper 把 upstream 的 `overlap patching`、`action averaging`、`similarity filtering` 重新带回第一版的风险

## 10. 执行结论

这个计划的执行原则很简单：

1. 先把方案约束变成可执行任务
2. 先通过 mock 和 replay gate
3. 再做受控真机 bringup
4. 任何时候都保留同步链路作为回退

如果上述 gate 任何一个没有通过，就不推进到下一阶段。
