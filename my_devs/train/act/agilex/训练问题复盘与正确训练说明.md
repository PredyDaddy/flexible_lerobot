# AgileX ACT 训练复盘与正确训练说明

日期：2026-03-06  
位置：`my_devs/train/act/agilex`

## 1. 这份说明要解决什么问题

这份说明用于回答两个问题：

1. 之前那几次训练里，问题到底出在哪里。
2. 接下来应该怎样发起一次“更正确、更能说明问题”的训练。

本次结论不是“训练程序报错”，而是“之前的训练设置和数据时序条件，会让结果容易被误判”。

## 2. 之前实际发生了什么

### 2.1 `right_arm7` 的 10k 续训并没有崩

训练日志：

- `logs/train_act_right_arm7_resume_plus10k_from375_20260305_20260305_163327.log`

这次训练最终正常结束到：

- `step 10375`
- `End of training`

说明它不是典型的：

- CUDA OOM
- NaN/Inf 爆炸
- Python 异常中断
- dataloader 卡死

也就是说，**训练过程本身是通的**。

### 2.2 这次 10k 训练只用了 `episode [0]`

这是之前最关键的问题之一。

从训练日志里可以确认：

- `dataset.episodes = [0]`
- `dataset.num_frames = 300`
- `dataset.num_episodes = 1`

这意味着那次所谓的“10k 训练”，本质上是在 1 条 episode、300 帧数据上反复训练很多轮。

这会带来两个直接后果：

1. loss 可能会很好看，因为模型会过拟合这一条轨迹。
2. 这个结果不能代表整个数据集是否真的适合训练。

所以，**之前那次结果不能作为 full dataset 训练效果的证据**。

### 2.3 更早的 20k 全量训练也没有程序错误

更早的一次日志：

- `logs/train_act_test_pipeline_clean_20k_bs8_20260304_214626.log`

它也正常完成到了：

- `step 20000`
- `End of training`

所以从“训练程序是否会跑崩”这个角度看，问题依然不突出。

## 3. 数据排查后得到的核心结论

### 3.1 `right_arm7` 数据集本身能训练，不是坏文件

已经确认：

- 目录结构完整
- `info.json` / `tasks.parquet` / 视频引用都可读
- `action` / `observation.state` 无 `NaN` / `Inf`
- `right_arm7` 的单臂切片逻辑符合方案设计

因此，**它不是“数据集直接坏掉导致完全不能训”的问题**。

### 3.2 检查脚本能过，不代表训练语义一定正确

检查脚本：

- `docs/data_clean_check/check_lerobot_v3_dataset_final.py`

这份脚本非常适合检查：

- 结构
- 元数据
- index 连续性
- 视频可读性
- task 映射
- 基本数据质量

但是它**不会严格逐项校验 `stats.json` 的数值是否和当前 parquet 完全一致**。

这意味着：

- “检查通过” 说明格式大体正确；
- 但它不能证明“训练语义完全没问题”。

### 3.3 真正更值得怀疑的是时序对齐

当前采集链路里，录制脚本是按固定频率去抓“各 topic 的最新值”，然后直接拼成同一帧。

这类实现的风险是：

- 相机帧、主臂动作、从臂状态并不是天然同一时刻；
- 即使每一路都在更新，也可能存在几十毫秒到上百毫秒的偏移；
- 转换成 LeRobot 以后，`qpos[i]` 和 `action[i]` 会被默认当作同一时刻监督信号。

实际排查里，右臂数据出现了明显的时序偏移特征：

- `action_t` 与 `state_t+3` 更接近；
- 偏移量大约是 `+3` 帧；
- 在 `30 FPS` 下约等于 `100ms`。

这说明更可能的真实问题是：

**数据不是“格式坏了”，而是“动作和状态/视觉之间存在系统性时间差”。**

这类问题非常容易导致：

- loss 还能下降；
- 但策略输出不跟手、发迟、或动作方向不稳定。

### 3.4 左臂数据不适合作为主训练目标

`left_arm7` 也能过结构检查，但它有一个更明显的问题：

- 左臂大部分时间几乎不动。

这类数据更容易学到“静止策略”，不适合作为主训练集判断系统好坏。

因此，当前更合理的主训练目标是：

- `test_pipeline_clean_12_19_removed_right_arm7`

## 4. 为什么还要进行这一次训练

这一次训练的意义，不是简单“再跑一次看看”。

它的作用是**建立一个不被错误设置污染的基线**。

之前的结果里混入了两个干扰项：

1. 训练只用了 `episode [0]`
2. 双臂原始数据里左臂几乎静止，容易干扰判断

因此，如果不重新做一次“正确配置的 full dataset right-arm training”，我们就无法回答下面这个关键问题：

> 模型效果差，到底是训练命令配错了，还是数据时序本身有问题？

这次训练的目的就是把第一个变量先排干净。

如果这次按照正确方式训练之后，策略效果仍然明显不理想，那么下一优先级问题就几乎可以明确落在：

- 采集时序对齐
- action/state/image 的同步关系
- 示教动作定义与监督目标之间的时延

## 5. 这次“正确训练”应当满足的原则

### 5.1 默认使用右臂单臂数据集

训练对象应当是：

- `datasets/lerobot_datasets/test_pipeline_clean_12_19_removed_right_arm7`

原因：

- 已经去掉左臂干扰；
- 相机组合与右臂训练目标一致；
- 比左臂版本更有学习价值。

### 5.2 默认使用全量数据，而不是单条 episode

正确训练的默认设置应当是：

- 不传 `--dataset.episodes=[0]`
- 直接使用全量 `112` 条 episode

这一步非常关键。

因为上一次“训了 10k step”但只喂了 1 条 episode，结论并不成立。

### 5.3 由 epoch 自动换算 step，避免手工算错

训练脚本应当支持：

- 先指定 `TARGET_EPOCHS`
- 再根据 `total_frames / batch_size` 自动换算 `steps`

这样可以避免：

- 口头说“训练 5 个 epoch”
- 实际命令却只对应半个 epoch，或者反过来过度训练。

### 5.4 暂时关闭 eval

当前本地环境里更关心的是：

- 训练是否按正确数据集进行
- loss 与 checkpoint 是否稳定落盘

所以默认：

- `eval_freq=-1`

这样可以避免把环境注册、机器人评估链路的问题混进这次训练判断里。

### 5.5 如果 resume，必须走正确的 resume 方式

当前仓库里 resume 的正确方式不是直接传 `checkpoint_path`，而是：

- `--config_path=<checkpoint>/pretrained_model/train_config.json`
- `--resume=true`

因此脚本里预留了：

- `RESUME_CONFIG_PATH`

用于指向 `train_config.json`。

## 6. 本次提供的脚本

脚本路径：

- `my_devs/train/act/agilex/train_right_arm7_correct.sh`

这个脚本做了下面几件事：

1. 默认训练 `right_arm7` 全量数据集。
2. 默认拒绝 `DATASET_EPISODES=[0]` 这类子集训练，避免重复上次错误。
3. 支持从 `TARGET_EPOCHS` 自动换算 `STEPS`。
4. 默认把日志写到 `logs/`。
5. 默认把训练输出写到 `outputs/train/`。
6. 自动检测 `cuda/cpu`。
7. 支持正确的 resume 方式。
8. 支持 `DRY_RUN=1` 先预演命令。

## 7. 推荐用法

### 7.1 先做 dry-run

```bash
DRY_RUN=1 bash my_devs/train/act/agilex/train_right_arm7_correct.sh
```

作用：

- 检查路径
- 检查 step 计算
- 检查最终 `lerobot-train` 命令是否符合预期

### 7.2 跑一个 5 epoch 的 full-dataset baseline

```bash
TARGET_EPOCHS=5 \
BATCH_SIZE=4 \
bash my_devs/train/act/agilex/train_right_arm7_correct.sh
```

说明：

- 这是当前最建议的第一条 baseline 命令；
- `BATCH_SIZE=4` 是更保守的设置；
- 如果显存更充足，可以继续调大。

### 7.3 从已有 checkpoint 继续训练

```bash
RESUME_CONFIG_PATH=/home/agilex/cqy/flexible_lerobot/outputs/train/<run>/checkpoints/<step>/pretrained_model/train_config.json \
STEPS=<new_total_steps> \
bash my_devs/train/act/agilex/train_right_arm7_correct.sh
```

注意：

- 这里的 `STEPS` 是**新的总步数**，不是“追加多少步”。

## 8. 这次训练完成后应该怎样解释结果

如果这次 full-dataset right-arm training 后：

- loss 正常下降；
- 推理效果也明显改善；

那么说明之前主要问题就是：

- 训练子集选错了；
- 基线建立方式不对。

如果这次训练后：

- loss 继续下降；
- 但实际动作仍然明显延迟、偏、或不稳定；

那么下一步最该怀疑的就不是训练脚本，而是：

- 采集时序同步
- 主臂动作与从臂状态之间的延时
- 图像与动作的时间对齐

换句话说，这次训练的目的，是把“训练命令本身是否正确”这个变量先单独排除掉。
