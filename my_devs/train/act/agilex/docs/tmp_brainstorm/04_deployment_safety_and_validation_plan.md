# Agilex ACT 实机上线安全与验证计划

## 1. 文档定位

这是一份发散式临时脑暴文档，目标不是直接定稿，而是从“把已经训练完成的 Agilex ACT checkpoint 上机”这个动作出发，先把风险、验证阶段、退出条件、日志建议和优先级边界讲清楚。

本文件基于以下事实：

1. 当前训练好的 checkpoint 为：
   - `outputs/train/20260313_194500_act_agilex_first_test_full/checkpoints/100000/pretrained_model`
2. checkpoint 配置显示：
   - `observation.state` 为 14 维
   - 输入图像为三路：`camera_front`、`camera_left`、`camera_right`
   - 图像尺寸为 `3 x 480 x 640`
   - 输出 `action` 为 14 维
   - `chunk_size=100`
   - `n_action_steps=100`
   - checkpoint 内置设备为 `cuda`
3. 当前仓库里的 Agilex ROS 桥接默认观测来自：
   - `/puppet/joint_left`
   - `/puppet/joint_right`
   - `/camera_f/color/image_raw`
   - `/camera_l/color/image_raw`
   - `/camera_r/color/image_raw`
4. Agilex 机器人配置支持两种控制模式：
   - `passive_follow`：只读，不发布命令
   - `command_master`：对命令 topic 发布动作
5. 参考脚本 `my_devs/train/act/so101/run_act_infer.py` 已经给出一个标准 LeRobot 推理主线：
   - 连接机器人
   - 获取观测
   - 走 checkpoint 的 pre/post processor
   - 调用 `predict_action`
   - 把动作发给机器人

## 2. 直接把训练好的 ACT 模型拿去上机的主要风险

### 2.1 特征对齐风险

这是第一优先级风险。

虽然 checkpoint 的输入输出 shape 和 Agilex 当前机器人实现表面上是对齐的，但“shape 对齐”不等于“语义对齐”。上机前至少要确认下面四件事：

1. `observation.state` 的 14 维顺序必须和训练时一致。
2. 三路相机 key 必须严格对应：
   - `camera_front`
   - `camera_left`
   - `camera_right`
3. 图像分辨率必须保持 `480 x 640`。
4. 输出 `action` 的 14 维顺序必须和动作下发侧一致。

如果这里有一项错位，模型不会显式报错，但会直接表现为动作异常，属于“最危险的一类静默错误”。

### 2.2 动作语义风险

当前 Agilex 接入里，训练链路和控制链路很可能不是一回事。

训练数据报告里写得很清楚：

1. `observation.state` 来自从臂 `puppet`
2. `action` 在录制链路中来自主臂 `master`

而实机推理时，`AgileXRobot.send_action()` 在 `command_master` 模式下会把动作发布到命令 topic。

这带来一个核心问题：

1. 训练出来的 14 维动作，到底语义上是“主臂目标位姿”还是“可直接给从臂控制器消费的目标关节位置”？
2. 当前下游控制链路是否真的期望接收这一组绝对关节位置？

如果动作语义没核死，上线就是高风险行为。

### 2.3 ACT 动作块风险

当前 checkpoint 的 `chunk_size=100`，`n_action_steps=100`。

这意味着默认部署行为是：

1. 模型一次前向预测一个 100 步动作块。
2. 后续循环中逐步消费这个动作块。
3. 直到当前块用完才触发下一次完整重规划。

这对实机来说有两个风险：

1. 如果第一步就错，错误会在一个动作块内持续一段时间。
2. 如果相机抖动、物体偏移、机器人起始位姿变化，默认 100 步承诺长度过长。

因此，默认训练配置未必等于默认上线配置。

### 2.4 设备与 processor 一致性风险

checkpoint 配置里的 device 是 `cuda`，参考推理脚本也直接加载 checkpoint 内的 pre/post processor。

这带来一个容易被忽略的问题：

1. 如果最终上机机器没有可用 CUDA，脚本层虽然可以覆盖 policy device，
2. 但 preprocessor/postprocessor 的 device 行为是否同步覆盖，必须单独验证。

否则会出现：

1. 推理主模型在 CPU
2. 预处理还在往 CUDA 搬 tensor
3. 运行时才炸

这不只是功能问题，也是上线稳定性问题。

### 2.5 观测 readiness 风险

Agilex 当前桥接逻辑要求：

1. 左右 JointState 都到位
2. 三路图像都到位

只要其中一路缺失，连接阶段就应该失败。

这个设计本身是对的，但在实机环境里会遇到两个常见问题：

1. 相机话题在线但图像卡住
2. 某一路偶发超时或分辨率不对

如果最终脚本不把 readiness 状态打印清楚，现场会很难判断到底是模型问题还是观测链路问题。

### 2.6 无保护闭环风险

参考 `run_act_infer.py` 是“连上就推理、推理就发动作”的闭环。

对于 SO101 这种单臂轻量平台，这种结构可接受；对 Agilex 双臂真机，直接套用会偏激进。

Agilex 方案里必须显式区分：

1. 只读观测模式
2. 推理不发命令模式
3. 低风险闭环模式
4. 正式闭环模式

否则脚本本身就不具备安全分级能力。

## 3. 建议的分级验证步骤与退出条件

这里建议把上线分成 5 个阶段，每个阶段都定义明确的通过标准和退出条件。

### 3.1 阶段 0：静态资产核对

目标：

1. 不连机器人
2. 只核对 checkpoint 与部署环境

检查项：

1. `POLICY_PATH` 是否指向 `pretrained_model`
2. `config.json` 中的输入输出 shape 是否符合 Agilex 当前 schema
3. 部署主机是否具备目标 device
4. `lerobot_flex` conda 环境是否可用
5. `policy_preprocessor.json` / `policy_postprocessor.json` 是否齐全

退出条件：

1. policy path 不存在
2. 缺少 processor 文件
3. 计划使用 CPU，但 processor/device 路径未验证
4. checkpoint 与 Agilex 当前特征设计不一致

通过标准：

1. 路径、设备、processor、shape 全部明确
2. 最终上线使用哪一个 checkpoint 已冻结

### 3.2 阶段 1：脚本级 dry-run

目标：

1. 只跑参数解析
2. 打印最终配置
3. 不连接机器人
4. 不加载实机数据

这一步应该至少打印：

1. policy path
2. policy device
3. task
4. fps
5. `chunk_size`
6. `n_action_steps`
7. `temporal_ensemble_coeff`
8. 控制模式
9. topic 绑定

退出条件：

1. 默认参数解析结果与预期不一致
2. policy runtime override 不生效
3. 控制模式默认就是闭环发布

通过标准：

1. dry-run 日志本身已经足够让现场工程师确认“这次到底要跑什么”

### 3.3 阶段 2：只读观测联通检查

目标：

1. 连接 Agilex
2. 只采观测
3. 不做 policy 前向
4. 不发布命令

推荐这一步必须使用 `passive_follow`。

检查项：

1. 左右从臂关节是否持续更新
2. 三路图像是否都在刷新
3. 图像 shape 是否为 `480 x 640 x 3`
4. 图像 key 是否正确映射到：
   - `camera_front`
   - `camera_left`
   - `camera_right`
5. readiness 超时是否能清楚报错

退出条件：

1. 任一话题缺失
2. 任一路图像维度不对
3. topic 名称正确但数据不刷新
4. 读取观测的循环频率明显低于目标 fps

通过标准：

1. 观测循环可稳定跑指定时间
2. 输出日志能明确显示 state 和 image 已就绪

### 3.4 阶段 3：影子推理检查

目标：

1. 连接 Agilex
2. 跑 live observation
3. 做 policy 前向
4. 不发布动作

这是正式闭环前最关键的一步。

检查项：

1. 前向是否稳定，无 NaN/Inf
2. 预测动作 shape 是否始终为 14
3. 左右臂动作值是否在合理范围内
4. 动作与当前观测之间的偏差是否离谱
5. 在连续运行过程中，平均循环时间是否可接受

退出条件：

1. 任意一步输出 NaN/Inf
2. 动作维度不符
3. 预测动作跳变明显超出机械臂可接受范围
4. 推理频率无法支撑预期控制频率

通过标准：

1. 模型能连续稳定输出动作
2. 日志上能看出动作统计值在合理区间
3. 现场工程师能接受“如果此时发布动作，不会立刻出现灾难性突变”

### 3.5 阶段 4：受限闭环验证

目标：

1. 开始发布动作
2. 但限制运行时间和测试工况
3. 操作员全程在场

建议策略：

1. 首轮只在安全初始位姿测试
2. 首轮缩短 `run_time_s`
3. 首轮不要默认使用 `n_action_steps=100`
4. 优先考虑更短的部署重规划长度，例如 `8` 或 `16`

原因很简单：

1. 更短的 action chunk 能缩短错误动作持续时间
2. 更适合首次上线阶段的人为观察与及时中断

退出条件：

1. 任一路相机丢失
2. 控制频率抖动明显
3. 预测动作出现连续大幅跳变
4. 机械臂出现明显异常趋势
5. 操作员主观判定存在碰撞风险

通过标准：

1. 在小时间窗内能稳定完成闭环运行
2. 没有出现控制发散、明显震荡或持续超调

### 3.6 阶段 5：正式任务闭环

目标：

1. 在受限闭环通过后，进入真实任务验证
2. 再逐步放开运行时间、动作块配置和任务复杂度

这里不建议一步到位。

建议按顺序放开：

1. 先放开运行时长
2. 再尝试更复杂起始位姿
3. 最后再评估是否保留默认 `n_action_steps=100`

退出条件：

1. 成功率明显不稳定
2. 小偏差被快速放大
3. 对场景变化敏感到不可用

通过标准：

1. 多轮重复测试下，行为稳定
2. 即使失败，也能以“停住或轻微偏离”为主，而不是危险性失控

## 4. 关键观测指标与日志建议

最终技术方案里，日志不是附属品，而是上线安全的一部分。

### 4.1 必须记录的指标

1. 当前控制模式
2. topic 绑定
3. 实际使用的 checkpoint 路径
4. policy device
5. `chunk_size`
6. `n_action_steps`
7. `temporal_ensemble_coeff`
8. 循环 step 数
9. 累计运行时间
10. 单步循环耗时
11. 有效控制频率
12. 每一路图像是否 ready
13. 当前 observation.state 的最小值、最大值或摘要
14. 当前预测 action 的最小值、最大值、均值或 L2 范数

### 4.2 强烈建议记录的指标

1. 观测时间戳与当前时间的偏差
2. 相机帧是否陈旧
3. 预测动作与当前关节位置的差值统计
4. 每次重新规划发生的频率
5. 运行期间是否发生 processor reset / policy reset

### 4.3 日志输出节奏建议

不建议每步都刷长日志。

推荐：

1. 启动时打印完整配置
2. readiness 通过时打印一次摘要
3. 循环中按 `log_interval` 打印摘要
4. 异常时立即打印详细上下文
5. 退出时打印结束原因

## 5. 异常处理建议

### 5.1 连接前异常

应直接 fail-fast，不要隐式降级。

包括：

1. checkpoint 不存在
2. processor 文件缺失
3. 设备不可用
4. topic 配置缺失

### 5.2 连接后但未闭环异常

应立即停止进入下一阶段。

包括：

1. 任一路观测未 ready
2. 图像 shape 不符
3. policy 前向报错
4. 输出动作出现 NaN/Inf

### 5.3 闭环中的异常

最终方案至少要遵循下面的原则：

1. 一旦发生关键异常，立即停止继续发布动作
2. 尽快断开 robot 连接
3. 在日志中保留异常前最后一次关键状态摘要

这里的关键异常至少包括：

1. topic 丢失
2. 推理异常
3. 输出动作非法
4. 主循环严重超时
5. 操作员主动中断

### 5.4 操作员层面的建议

实机首次上线时，不应把“按 Ctrl+C”当成唯一安全措施。

建议在最终技术方案里明确：

1. 哪个人负责观察机械臂运动
2. 哪个人负责盯日志
3. 哪个人负责执行物理急停或软件中断

至少在首次闭环验证阶段，要把人控流程写清楚。

## 6. 对最终技术方案的优先级划分

### 6.1 必须有

1. 明确的 `POLICY_PATH`
2. 明确的 Agilex topic 和 `control_mode` 配置
3. dry-run 模式
4. 只读观测检查模式
5. 影子推理模式
6. 闭环模式与运行时长限制
7. `policy_n_action_steps` runtime override
8. 关键日志输出
9. 缺失观测和非法动作的 fail-fast 机制
10. 安全上线流程和退出条件

### 6.2 强烈建议有

1. 预测动作统计摘要
2. 观测与动作的简单 sanity check
3. 默认使用更保守的首次部署参数
4. 把 `passive_follow` 作为默认起始模式

### 6.3 可以后置

1. 更复杂的在线可视化面板
2. 自动 rosbag 录制
3. 更细粒度的性能 profiling
4. 自动化成功率统计脚本
5. temporal ensembling 的系统化调参

## 7. 我对最终方案的倾向性结论

如果要把这次 Agilex ACT 上机方案做稳，而不是做快，我更倾向于最终技术方案明确采用“分级上线”架构，而不是单一脚本一把梭：

1. 先允许只读
2. 再允许影子推理
3. 最后才允许闭环发命令

在首次闭环阶段，不建议直接继承训练时的 `n_action_steps=100`，而应把更短的 runtime override 当成默认安全策略之一。

此外，最终方案必须把“动作语义是否与命令 topic 真正一致”列为上线前硬前提。这个问题如果没有被单独确认，其他工程工作做得再细，也仍然属于高风险上线。
