## 0. 背景与目标

现状：
- Cobot Magic 的数据采集与回放主要由两个脚本完成：
  - `my_devs/cobot_magic/collect_data/collect_data.py`：订阅 ROS 话题，时间对齐后写出 `episode_x.hdf5`
  - `my_devs/cobot_magic/collect_data/replay_data.py`：读取 `episode_x.hdf5`，发布回 ROS 话题用于回放
- 当前仓库 LeRobot 数据格式版本为 `v3.0`（`src/lerobot/datasets/lerobot_dataset.py` 的 `CODEBASE_VERSION = "v3.0"`）。

本方案要解决两件事：
1. **把现有录制/回放做成 Web 程序**：后续通过访问 URL 就能开始录制或回放（含查看状态/日志/预览）。
2. **给出 HDF5 -> LeRobotDataset v3.0 的转换技术方案**：明确字段映射、实现路径、校验与风险。

---

## 1. 约束与原则（必须满足）

### 1.1 绝对约束：不改原脚本

你明确要求：
- `collect_data.py` 与 `replay_data.py` **原本怎么工作就保持一致**。
- **不对它们做任何改动**（包含改参数默认值、改 bug、改日志等都不动）。

因此 Web 化的实现必须采用“旁路增强”的方式：
- 新增 Web 服务与管理代码放在 `my_devs/docs/hdf5_lerobot`。
- 原脚本保持原封不动，作为“黑盒能力”被调用（subprocess 或隔离进程）。

### 1.2 工程原则

- 最小化侵入：新增模块不影响现有 ROS/采集链路。
- 可回滚：任何时刻都能直接回到命令行跑原脚本。
- 可观测：Web UI 能看到实时日志、状态、输出路径、失败原因。
- 可扩展：后续能加“数据检查、自动转换、标注、上传 hub”等功能。

## 2. 现状接口梳理（基于脚本实际行为）

### 2.1 `collect_data.py`：录制行为与产物

关键参数（当前脚本）：
- `--dataset_dir`：根目录，默认 `./data`
- `--task_name`：子目录名，默认 `aloha_mobile_dummy`
- `--episode_idx`：episode 编号，默认 0
- `--max_timesteps`：采样步数，默认 500
- `--frame_rate`：循环频率，默认 30
- `--use_depth_image`：是否订阅并写 depth（脚本内部存在潜在空指针风险，但本方案不改它）
- `--use_robot_base`：是否记录底盘
- 一组 topic 参数：三路 RGB、三路 depth、主臂/从臂关节、里程计

输出路径：
- 最终文件：`{dataset_dir}/{task_name}/episode_{episode_idx}.hdf5`

HDF5 内容（主版）：
- `/observations/qpos` `[T,14]`
- `/observations/qvel` `[T,14]`
- `/observations/effort` `[T,14]`
- `/action` `[T,14]`
- `/base_action` `[T,2]`（脚本把 `base_vel` 写进了 `base_action` 这个 key，语义需要后续统一）
- `/observations/images/<cam>` `[T,480,640,3] uint8`
- （可选）`/observations/images_depth/<cam>` `[T,480,640] uint16`

### 2.2 `replay_data.py`：回放行为与接口

关键参数：
- `--dataset_dir`（必填）+ `--task_name` + `--episode_idx`
- `--frame_rate`
- 一组 topic 参数（发布到这些话题）
- `--only_pub_master`：只发布主臂轨迹（高频插值发布）

行为：
- 读取 HDF5 -> 循环 publish：
  - 主臂/从臂关节（`JointState`）
  - 三路相机图像（`Image`）
  - 底盘速度（`Twist`，发布到 `/cmd_vel` 默认）

## 3. 方案 A：录制/回放 Web 化（不改原脚本）

### 3.1 目标用户体验（URL 打开即可用）

访问 `http://<host>:<port>/` 后：
- 选择模式：`录制 Record` / `回放 Replay`
- 录制页：
  - 填参数（dataset_dir/task_name/episode_idx/max_timesteps/frame_rate/topic 等）
  - 实时预览（至少 RGB 三路）
  - 一键开始/停止
  - 实时日志（同步失败、帧数计数、保存成功等）
  - 保存完成后自动跳到“数据浏览”并可触发“转 LeRobot v3”
- 回放页：
  - 选择已有 episode（扫描 dataset_dir/task_name）
  - 设置回放发布话题（可选择不发某些话题）
  - 一键开始/停止
  - 可选：回放进度条、倍速、循环

### 3.2 总体架构（推荐）

核心思路：**Web 服务只做“编排与可视化”，录制/回放仍由原脚本在独立进程执行。**

组件划分：
- **Web 前端（Browser）**
  - 提供 UI、参数表单、预览窗口、日志窗口
- **Web 后端（Python / FastAPI）**
  - 提供 REST API + WebSocket（推送日志/状态）
  - 进程管理（启动/停止 `collect_data.py` / `replay_data.py`）
  - 数据目录索引（列举任务、episode、大小、时间）
  - 可选：触发转换任务（HDF5 -> LeRobot v3）
- **ROS 环境**
  - `roscore` + 各传感器/机械臂节点
  - 可选：`web_video_server` 或 `rosbridge_suite`（用于 Web 预览）

为什么推荐 FastAPI：
- 天然支持 WebSocket（日志/状态推送最省事）
- pydantic 校验参数，避免“Web 传参导致脚本崩溃”
- 和后续“转换任务队列、后台 worker”衔接自然

### 3.3 “不改原脚本”的落地策略

两条路线对比：

1) **子进程调用（推荐）**
- 后端通过 subprocess 启动：
  - `python my_devs/cobot_magic/collect_data/collect_data.py ...`
  - `python my_devs/cobot_magic/collect_data/replay_data.py ...`
- 捕获 stdout/stderr -> 推送到 WebSocket -> 前端显示
- 停止时发送 SIGINT/SIGTERM

优点：
- 完全不需要改原脚本，也不需要把脚本改造成可 import 的 library
- 避免 `rospy.init_node()` 在同进程冲突（Web 后端本身也可能要订阅话题做预览）

缺点：
- 需要做“进程生命周期管理 + 参数拼装 + 日志解析”

2) import 复用（不推荐作为第一版）
- 在后端 `import collect_data` 然后调用类/函数

风险：
- `rospy.init_node()` 同进程只能调用一次，和 Web 预览/其他 ROS 订阅容易冲突
- 线程/事件循环和 rospy 的 signal handler/阻塞模型耦合

结论：第一版必须走子进程调用。

### 3.4 Web 预览方案（两种可选）

目标：录制时在 Web 页面看到三路相机实时画面（用于确认相机正常、曝光对焦、三路同步大致 OK）。

方案 P1：ROS `web_video_server`（推荐做最省事的 MVP）
- ROS 侧单独启动 `web_video_server`，对每个 image topic 提供 HTTP MJPEG 流
- Web 页面用 `<img src="http://host:8080/stream?topic=/camera_f/color/image_raw">` 即可

优点：
- Web 后端完全不用写“订阅 ROS image -> 转 JPEG -> WebSocket”的代码
- 与“原脚本不改”完全独立

缺点：
- 依赖 `web_video_server` ROS 包是否可用

方案 P2：后端自建 ROS 订阅 + WebSocket 推送
- FastAPI 进程里用 `rospy` 订阅 `sensor_msgs/Image`
- 用 `cv_bridge` 转 numpy，再 jpeg 编码，通过 WebSocket 推送给前端

优点：
- 不依赖额外 ROS 包
- 可以做更强的逻辑：帧率限制、叠字、延迟统计、同步诊断

缺点：
- 需要解决 `rospy.init_node()` 与子进程录制脚本之间的 ROS 命名冲突（可通过匿名 node 名、或让预览进程独立）
- 实现成本更高

推荐落地顺序：
- MVP 用 P1（快速跑通 URL 预览）
- 稳定后再做 P2（增强功能与可控性）

### 3.5 后端 API 设计（建议）

录制：
- `POST /api/record/start`
  - body：`dataset_dir, task_name, episode_idx, max_timesteps, frame_rate, use_depth_image, use_robot_base, topics...`
  - 返回：`session_id, pid, start_time, output_path`
- `POST /api/record/stop`
  - body：`session_id`
- `GET /api/record/status?session_id=...`
  - 返回：运行中/已退出、当前 stdout tail、已写帧数（从日志解析或自定义 sidecar 统计）

回放：
- `POST /api/replay/start`
- `POST /api/replay/stop`
- `GET /api/replay/status`

数据浏览：
- `GET /api/datasets/tasks?dataset_dir=...`
- `GET /api/datasets/episodes?dataset_dir=...&task_name=...`

日志与状态推送：
- `WS /ws/sessions/{session_id}/logs`
- `WS /ws/sessions/{session_id}/status`

### 3.6 进程管理与并发策略

建议实现一个最小“会话管理器”：
- 录制与回放互斥（同一套话题同一时间一般不允许同时跑）
- 支持“一个录制会话 + 一个预览会话”（预览要限帧率）
- 任何会话退出后：
  - 释放锁
  - 写 `session.json`（包含参数、开始/结束时间、退出码、stdout 路径等）

停止策略（避免僵尸进程）：
1. SIGINT（模拟 Ctrl+C）
2. 等待 N 秒
3. SIGTERM
4. 再等 N 秒
5. SIGKILL（最后手段）

### 3.7 推荐目录组织（代码放 `my_devs/docs/hdf5_lerobot`）

建议后续实现时采用：
- `my_devs/docs/hdf5_lerobot/app/backend/`
  - `main.py`（FastAPI 入口）
  - `process_manager.py`（subprocess 生命周期 + 日志采集）
  - `models.py`（请求/响应 schema）
  - `storage.py`（扫描 episode、写 session.json）
  - `ros_preview/`（可选：自建 preview）
- `my_devs/docs/hdf5_lerobot/app/frontend/`
  - `index.html` + `app.js` + `styles.css`（建议先无构建，降低门槛）

### 3.8 风险点与规避

风险 1：ROS 环境变量/工作目录不一致导致脚本启动失败  
对策：Web 后端启动子进程时显式：
- 指定 `cwd=/home/agilex/cqy/flexible_lerobot`
- 继承当前环境（或允许在配置中指定 `source /opt/ros/noetic/setup.bash` 等）

风险 2：录制脚本本身存在一些参数默认值/深度处理 bug  
对策：本方案不改脚本；Web 侧通过：
- UI 强制用户明确选择 master/puppet topic（避免使用脚本默认值）
- 当用户勾选 depth 时提示“脚本可能不稳定”，并建议先用 RGB 跑通

风险 3：Web 预览占用 CPU/带宽，影响采集稳定性  
对策：
- 预览强制降帧（例如 5 FPS）
- 预览和录制隔离进程

## 4. 方案 B：Cobot Magic HDF5 -> LeRobotDataset v3.0 转换

### 4.1 转换目标与边界

目标：
- 把 `collect_data.py` 产出的 `episode_x.hdf5` 批量转为 **LeRobotDataset v3.0** 目录结构：
  - `meta/info.json`
  - `meta/stats.json`
  - `meta/tasks.parquet`
  - `meta/episodes/chunk-xxx/file-yyy.parquet`
  - `data/chunk-xxx/file-yyy.parquet`
  - `videos/<video_key>/chunk-xxx/file-yyy.mp4`（可选但推荐）

边界（第一版建议）：
- RGB 三路相机 + 关节状态 + action + base_action 必须转
- depth 数据：LeRobot 当前 image writer/视频编码链路对 depth（2D/uint16）支持不完善，**第一版建议先不转或转为可视化 8-bit**（见 4.6）

### 4.2 核心实现策略：直接复用 LeRobot 官方 writer API

LeRobot v3.0 的 writer 已经实现了：
- episode 级 parquet 写入与分片
- tasks.parquet 管理
- 统计量 stats.json 维护
- 图像落盘与 mp4 编码（针对 3 通道图像）
- episodes 元数据（含 videos 的 from/to timestamp）自动维护

因此转换器不应“手写 meta/data/videos”，而是走：
- `lerobot.datasets.lerobot_dataset.LeRobotDataset.create`
- `dataset.add_frame(...)`
- `dataset.save_episode()`
- `dataset.finalize()`

### 4.3 输入输出约定

输入：
- `dataset_dir/task_name/episode_*.hdf5`

输出（示例）：
- `output_root/<repo_id>/`（一个 LeRobotDataset v3.0 数据集目录）

`repo_id` 建议命名：
- 本地转换：`local_cobot_magic/<task_name>` 或 `cobot_magic_<task_name>`
- 后续要 push hub 时再替换成正式的 HF `repo_id`

### 4.4 字段映射（建议 v1）

以一个 episode 的帧为单位映射到 LeRobot “frame dict”：

HDF5 -> LeRobot features：
- `/observations/qpos` -> `observation.state`（dtype `float32`, shape `(14,)`）
- `/observations/qvel` -> `observation.qvel`（dtype `float32`, shape `(14,)`）
- `/observations/effort` -> `observation.effort`（dtype `float32`, shape `(14,)`）
- `/action` -> `action`（dtype `float32`, shape `(14,)`）
- `/base_action` -> `action.base` 或 `observation.base_vel`
  - 建议第一版直接映射成 `action.base`（dtype `float32`, shape `(2,)`）
  - 后续再统一语义（它到底是 odom 测得还是 cmd_vel 发出）
- `/observations/images/<cam>` -> `observation.images.<cam>`（dtype `video`, shape `(480,640,3)`）

任务字段：
- LeRobot 的 `add_frame` 要求每帧带一个 `task` 字符串
- 建议 `task = task_name`（目录名）或 UI/配置传入更自然语言的 prompt

时间戳：
- HDF5 没有保存显式 timestamp
- v1 建议 `timestamp = frame_index / fps`（fps 从采集配置传入，默认 30）

### 4.5 features 字典定义（示例）

转换器需要先定义 LeRobot features（不包含默认 features，它们会自动注入）：

示例（RGB + 关节 + 底盘）：
- `observation.state`: `float32 (14,)`
- `observation.qvel`: `float32 (14,)`
- `observation.effort`: `float32 (14,)`
- `action`: `float32 (14,)`
- `action.base`: `float32 (2,)`
- `observation.images.cam_high`: `video (480,640,3)`
- `observation.images.cam_left_wrist`: `video (480,640,3)`
- `observation.images.cam_right_wrist`: `video (480,640,3)`

注意：
- key 命名必须符合 LeRobot 层次命名约定（见 `lerobot_v3数据格式解析.md`）。
- video/image 目前仅支持 3 通道图像写入（LeRobot image writer 对深度图 TODO）。

### 4.6 深度图转换策略（必须明确）

现实限制（基于当前仓库实现）：
- `src/lerobot/datasets/image_writer.py:image_array_to_pil_image` 只支持 3 通道图像
- depth 通常是 2D `uint16`，直接写会报错
- mp4 编码链路也默认处理 8-bit 3 通道 png

可选方案：

方案 D0（推荐 v1）：**不转换 depth**
- 优点：最稳，先跑通训练链路
- 缺点：丢失深度信息

方案 D1：depth 可视化后作为普通 RGB 视频保存
- 将 `uint16` depth 归一化/截断到 `[0,255]`，再 colormap 成 3 通道 `uint8`
- 保存为 `observation.depth_viz.<cam>`（dtype `video`），并明确它不是“可度量 depth”

方案 D2：扩展 LeRobot 以支持 depth dtype（后续）
- 需要修改 LeRobot 的 image writer/feature validator/video pipeline
- 工作量较大，不建议与 Web MVP 同期做

建议：v1 选 D0，v2 选 D1，v3 再评估 D2。

### 4.7 转换器的处理流程（建议）

1) 扫描输入目录
- 找到所有 `episode_*.hdf5`
- 读取每个文件的 cam keys（`/observations/images/` 子 key）
- 校验每个 episode 的帧数 T、一致性

2) 创建空的 LeRobotDataset v3
- `LeRobotDataset.create(repo_id, fps, features, root=output_root, robot_type=...)`
- `use_videos=True`（推荐）或 `False`（极端情况下先落 PNG）
- `vcodec` 选项：
  - `libsvtav1`（默认，压缩率好但 CPU 重）
  - `h264`（更快，适合边录边转或小机器）

3) 逐 episode 转换
- 对每个 episode：
  - `dataset.episode_buffer = dataset.create_episode_buffer(episode_index=<idx>)`
  - for `i in range(T)`：构造 frame dict：
    - `timestamp=i/fps`
    - `task=<task>`
    - 向量特征（qpos/qvel/effort/action/base）转成 `np.float32`
    - 图像特征传入 `np.uint8` HWC
    - `dataset.add_frame(frame)`
  - `dataset.save_episode()`
  - `dataset.clear_episode_buffer()`

4) finalize
- `dataset.finalize()`（必须调用，否则 parquet footer/episodes meta 可能不完整）

### 4.8 验收与校验（转换完成后必须做）

最小验收：
- 用 `LeRobotDataset(repo_id, root=output_root/<repo_id>)` 能成功加载
- `len(ds) == total_frames`
- 取样 `ds[0]`：
  - 包含 `observation.state/action` 等向量
  - 图像 key 能正确解码成 tensor（说明 video 编码与 timestamp 对齐 OK）

数据一致性校验（建议自动化脚本实现）：
- 每个 episode：
  - `length == HDF5 T`
  - `timestamp` 递增，且相邻差值接近 `1/fps`
- 抽样对比：
  - HDF5 的某一帧图像与 LeRobot 解码结果在像素上近似一致（允许视频编码损失）

## 6. 开放问题（需要你确认的关键点）

为避免转换后“训练用错语义”，这里有 3 个需要你拍板的点：

1) `HDF5:/action` 的语义是什么？
- A. 主臂位置（示教）用于监督学习（更像 action label）
- B. 实际下发到从臂的控制命令（更像 executed action）

2) `HDF5:/base_action` 的语义是什么？
- A. 里程计测得的底盘速度（observation）
- B. 发送给底盘的 cmd_vel（action）

3) 相机图像在 HDF5 里是 RGB 还是 BGR？
