# 技术方案2

## 1. 目标与范围

本方案文档覆盖两件事：

1. 把现有 `my_devs/cobot_magic/collect_data/collect_data.py`（录制）和 `my_devs/cobot_magic/collect_data/replay_data.py`（回放）包装为 Web 程序能力。
2. 基于当前仓库 LeRobot v3.0 数据规范，设计 `Cobot Magic HDF5 -> LeRobot v3` 的转换方案。

本次方案的硬约束：

- **不修改原有两个脚本的业务逻辑与代码行为**。
- Web 化通过“新增模块/新增服务”完成，对原脚本采取调用式封装。
- 开发主目录放在 `my_devs/docs/hdf5_lerobot`。

## 2. 当前基线结论

### 2.1 录制脚本（collect_data）现状

- 通过 ROS 订阅多源 topic（图像、主从臂关节、底盘），在帧循环中按时间戳对齐并写入单 episode HDF5。
- 输出核心数据：
  - `/observations/qpos`, `/observations/qvel`, `/observations/effort`
  - `/observations/images/<camera>`（可选深度 `/observations/images_depth/<camera>`）
  - `/action`（主臂动作）
  - `/base_action`（底盘速度）
- 主要可配参数来自 CLI（`argparse`），如 `dataset_dir`、`task_name`、`episode_idx`、`frame_rate`、`camera_names`、各类 topic 名称、是否启用底盘与深度。

### 2.2 回放脚本（replay_data）现状

- 读取 HDF5 episode，按帧率回放图像与关节/底盘消息到 ROS topic。
- 具备 `--only_pub_master` 等开关能力。
- 依赖 HDF5 结构与录制脚本输出一致。

### 2.3 LeRobot v3.0 数据规范关键点

- 代码版本：`CODEBASE_VERSION = "v3.0"`。
- 目录形态：`meta/` + `data/chunk-xxx/file-yyy.parquet` + `videos/...`（或 `images/...`）。
- 必需元数据：`meta/info.json`, `meta/stats.json`, `meta/tasks.parquet`, `meta/episodes/chunk-xxx/file-yyy.parquet`。
- 帧级写入可复用 API：`LeRobotDataset.create(...)`、`add_frame(...)`、`save_episode()`、`finalize()`。

## 3. 方案A：录制/回放 Web 化（原脚本零改动）

## 3.1 设计原则

1. **原脚本零侵入**：`collect_data.py` 与 `replay_data.py` 不做任何修改。
2. **新增 Web 控制层**：Web 服务只负责参数校验、任务调度、状态查询、日志回传。
3. **任务进程隔离**：每次录制/回放任务在独立子进程执行，避免阻塞和资源串扰。
4. **失败可诊断**：任务状态与 stdout/stderr 落盘，支持 Web 页面查看。

## 3.2 总体架构

```text
Browser (URL)
   |
   v
Web UI (录制/回放页面)
   |
   v
Backend API (FastAPI)
   |
   +--> Job Manager (任务状态机 + 队列 + 日志)
   |
   +--> Script Runner (subprocess 调用原脚本)
             |
             +--> python my_devs/cobot_magic/collect_data/collect_data.py ...
             +--> python my_devs/cobot_magic/collect_data/replay_data.py ...
```

说明：

- Web 只作为“控制面”（control plane）。
- ROS 采集/回放逻辑仍在原脚本内执行。
- 后端通过参数模板将 UI 输入转成原脚本 CLI 参数。

## 3.3 目录建议（新增）

建议在 `my_devs/docs/hdf5_lerobot` 下新增：

```text
my_devs/docs/hdf5_lerobot/
  web_app/
    backend/
      app.py                  # FastAPI 入口
      api/
        record.py             # 录制接口
        replay.py             # 回放接口
        jobs.py               # 任务查询/停止/日志
      core/
        job_manager.py        # 任务状态管理
        runner.py             # subprocess 封装
        schema.py             # 参数模型与校验
        presets.py            # topic/设备参数模板
      data/
        jobs/                 # 任务状态与日志
    frontend/
      index.html              # 或后续 React/Vue
      record.html
      replay.html
```

## 3.4 API 设计（第一版）

1. `POST /api/record/start`
- 输入：录制参数（目录、任务名、episode、帧率、topic 配置等）
- 行为：启动 `collect_data.py` 子进程
- 输出：`job_id`

2. `POST /api/replay/start`
- 输入：回放参数（目录、任务名、episode、帧率、是否 only_pub_master 等）
- 行为：启动 `replay_data.py` 子进程
- 输出：`job_id`

3. `GET /api/jobs/{job_id}`
- 返回任务状态：`pending/running/succeeded/failed/stopped`
- 返回开始时间、结束时间、退出码

4. `GET /api/jobs/{job_id}/logs`
- 返回任务日志（stdout/stderr）

5. `POST /api/jobs/{job_id}/stop`
- 终止任务（先 `SIGINT`，超时再 `SIGTERM`）

## 3.5 URL 访问体验（你提到的“通过 URL 开始录制和回放”）

- 页面 1：`/record`
  - 填写录制参数
  - 点击开始 -> 显示任务状态与实时日志
- 页面 2：`/replay`
  - 选择已有 episode
  - 点击开始回放 -> 显示进度与日志
- 页面 3：`/jobs`
  - 查看历史任务，支持下载日志

## 3.6 参数策略（保证与原脚本一致）

- 参数字段名尽量与原 CLI 一致，减少语义偏差。
- Web 层只做“非业务性校验”，例如：
  - 路径存在性
  - episode 为非负整数
  - `camera_names` 与 topic 配置匹配
- 业务逻辑不在 Web 层重写，完全交给原脚本。

## 3.7 状态机与并发控制

- 状态机：`pending -> running -> succeeded/failed/stopped`
- 并发策略（建议第一版）：
  - 同一时刻只允许 1 个录制任务运行
  - 回放任务可配置并发上限（默认 1）
- 原因：ROS topic 与机器人设备通常不支持高并发抢占。

## 3.8 风险与规避

1. ROS topic 未就绪导致任务失败
- 启动前做 topic readiness precheck（可选调用 `rostopic list` 或脚本预检查）。

2. 子进程僵死或无输出
- 设计心跳与超时机制，超过阈值自动标记异常并可手动停止。

3. 参数配置复杂、易填错
- 用 preset 模板（例如“实验室默认相机与关节 topic”）降低输入错误。

## 4. 方案B：Cobot HDF5 -> LeRobot v3 转换

## 4.1 目标

把“每 episode 一个 HDF5”的数据，转换为 LeRobot v3 可直接读取/训练的数据集结构，并通过 `LeRobotDataset` API 验证可用性。

## 4.2 源/目标结构差异

1. 源（Cobot HDF5）
- 单文件单 episode
- 图像直接存 ndarray
- 动作与状态为固定数组
- 缺少 LeRobot v3 的 `meta/*` 体系

2. 目标（LeRobot v3）
- 分 chunk parquet + 视频/图像文件
- 有完整 `meta/info/stats/tasks/episodes`
- 帧记录中含默认字段与 feature 声明

## 4.3 字段映射建议（第一版）

建议最小可行映射如下：

- `/observations/qpos` -> `observation.state`
- `/action` -> `action`
- `/base_action` -> `observation.base_velocity`（或 `observation.base_action`，二选一并固定）
- `/observations/images/front` -> `observation.images.front`
- `/observations/images/left` -> `observation.images.left`
- `/observations/images/right` -> `observation.images.right`
- 若存在深度：`/observations/images_depth/*` -> `observation.images_depth.*`

关于 14 维关节（双臂）建议：

- 方案 B1（最小实现）：先保持 14 维统一向量，不拆左右臂。
- 方案 B2（增强实现）：拆分 `left_arm`/`right_arm` 两个 feature，便于后续多策略训练。

第一阶段建议先用 B1 打通全链路，第二阶段再演进 B2。

## 4.4 转换流程（最小可行）

1. 创建目标数据集骨架
- 调 `LeRobotDataset.create(...)`
- 定义 `fps`、`features`、是否写视频、chunk 参数

2. 遍历 HDF5 episode
- 逐帧读取状态/动作/图像
- 组装 frame dict（含 `task`、`timestamp`、各 features）
- 调 `add_frame(...)`

3. 每个 episode 完成后落盘
- 调 `save_episode()`
- 自动更新 `meta/episodes` 与 `stats`

4. 所有 episode 完成
- 调 `finalize()`
- 用 `LeRobotDataset(repo_id, root=...)` 回读验证

## 4.5 验证标准

1. 结构验证
- 目标目录包含 `meta/`, `data/chunk-*`, `videos/`（或 `images/`）
- `meta/info.json` 的 `codebase_version == "v3.0"`

2. 读取验证
- 能成功实例化 `LeRobotDataset`
- `len(dataset) > 0`
- 随机抽样 frame 的 `action/state/images` 形状正确

3. 训练前验证
- 运行最小训练/加载脚本，确保 dataloader 不报 feature mismatch

4. 语义验证
- 抽 1~2 个 episode 做可视化对比，确认转换前后动作与图像时间顺序一致

## 4.6 关键风险

1. 14 维动作在不同策略中的约定不一致
- 通过 feature schema 固化并在文档中明确。

2. 时间戳缺失或精度不足
- 若 HDF5 无稳定 timestamp，使用 `frame_index / fps` 生成单调时间轴。

3. 视频编码耗时
- 第一版可先落 `images`，二阶段再启用 `videos` 编码优化。
