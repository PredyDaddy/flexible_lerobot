## 0. 一句话结论

1) **Web 化**采用“控制面 + 子进程运行原脚本”的方式：FastAPI 负责参数校验、任务调度、日志/状态回传与数据目录浏览；录制/回放仍由 `collect_data.py` / `replay_data.py` 在独立进程中跑，满足“原脚本零改动”。  
2) **转换器**不手写 `meta/`、`data/`、`videos/`，而是严格复用仓库内 `LeRobotDataset.create/add_frame/save_episode/finalize` 写入 v3.0 数据集；默认不转 depth；相机通道顺序（RGB/BGR）、`/base_action` 语义与 `/action` 语义要在转换时显式配置并可被校验。

## 1. 硬约束与工程原则

### 1.1 硬约束（必须满足）

- **不修改原脚本代码与业务行为**：
  - `my_devs/cobot_magic/collect_data/collect_data.py`
  - `my_devs/cobot_magic/collect_data/replay_data.py`
- Web 化与转换器代码新增到 `my_devs/docs/hdf5_lerobot/` 范围内（不污染主包）。

> 注意：原脚本存在已知问题（详见第 2 节与第 7 节）。由于硬约束禁止改脚本，本方案采用“Web/转换侧规避 + 文档明确限制”的方式落地。

### 1.2 工程原则

- 最小侵入：新增模块不影响现有 ROS 采集链路；可随时回退到 CLI 直接运行原脚本。
- 可观测：Web UI 能看到实时日志、状态、输出路径、失败原因；任务信息与日志落盘可追溯。
- 可扩展：后续可加数据检查、自动转换、转换报告、上传 Hub、标注等能力。

## 2. 事实基线（以仓库代码为准）

本节只陈述“现在代码真实做什么”，作为后续方案的事实依据。

### 2.1 录制脚本 `collect_data.py` 的真实行为

文件：`my_devs/cobot_magic/collect_data/collect_data.py`

1) 输出目录与文件：
- 输出路径：`{dataset_dir}/{task_name}/episode_{episode_idx}.hdf5`
- 默认：`dataset_dir="./data"`，`task_name="aloha_mobile_dummy"`，`episode_idx=0`

2) HDF5 结构（主版本）：
- `/observations/qpos` `[T, 14]`
- `/observations/qvel` `[T, 14]`
- `/observations/effort` `[T, 14]`
- `/observations/images/<cam>` `[T, 480, 640, 3] uint8`
- （可选）`/observations/images_depth/<cam>` `[T, 480, 640] uint16`
- `/action` `[T, 14]`
- `/base_action` `[T, 2]`
- attrs：`sim=False`，`compress=False`

3) 关键语义（非常重要，影响转换映射）：
- `qpos/qvel/effort`：来自 **puppet（从臂）** JointState（脚本在 `process()` 中拼接 left+right）。
- `action`：来自 **master（主臂）** JointState（脚本在 `process()` 中拼接 left+right）。
  - 但脚本默认的 `--master_arm_left_topic/--master_arm_right_topic` 疑似配置错误（默认指向 `/puppet/joint_*`），Web 层必须强制用户显式填写或使用 preset 修正，否则 `/action` 可能并不是主臂示教。
- `base_action`：录制时来自 **Odometry 观测速度**：
  - `obs['base_vel'] = [odom.twist.twist.linear.x, odom.twist.twist.angular.z]`
  - 再写入 `/base_action`

4) 录制长度与“停止”语义：
- 脚本是 **固定长度采集**：循环到 `max_timesteps + 1` 才会返回并尝试保存。
- 如果提前中断（SIGINT 或 ROS shutdown），极大概率 **不会产出可用 HDF5**，因为主函数在保存前做了 `len(actions) < max_timesteps` 检查，不满足会直接 `exit(-1)`。
  - 因此 Web UI 的“Stop”对录制更像“Abort（中止且可能丢数据）”，必须在交互与验收里写清楚。

5) CLI 参数可靠性问题（Web 封装必须规避）：
- `--use_depth_image` / `--use_robot_base` 使用 `argparse` 的 `type=bool`，CLI 传参行为不可靠。
  - Web Runner 不能用 `--use_depth_image False` 这种方式关闭，因为 `"False"` 会被解析为真值。
  - 解决策略：Web 侧只在启用时传 `--use_depth_image True`（或只传 flag），禁用时不要传该参数；同时在 UI 做强提示。
- `--camera_names` 定义为 `type=str` 但默认值是 list，这意味着 Web 侧不应直接让用户用 CLI 覆盖 camera_names（会出现字符串被逐字符迭代等不可控行为）。

### 2.2 回放脚本 `replay_data.py` 的真实行为

文件：`my_devs/cobot_magic/collect_data/replay_data.py`

1) 节点与并发限制：
- `rospy.init_node("replay_node")` 未启用 `anonymous=True`，意味着同一 ROS Master 下并发跑多个回放会发生 node name 冲突（后启动者会顶掉前一个）。
- 因此 Web 层必须强制 “同一时刻最多 1 个 replay 任务”。

2) 数据读取与发布：
- 从 `{dataset_dir}/{task_name}/episode_{episode_idx}.hdf5` 读取 `qpos/qvel/effort/action/base_action/images/*`
- 发布：
  - master/puppet JointState
  - 三路相机 Image
  - `base_action` 作为 `Twist` 发布到 `--robot_base_topic`（默认 `/cmd_vel`）

3) 图像通道：
- 代码中对每路图像做了 `image = image[:, :, [2, 1, 0]]` 的 B/R 交换，并用 `"bgr8"` 发布。
- 这表明 HDF5 内图像通道顺序在实践中可能是 **RGB**（回放时换成 BGR 以适配 OpenCV/ROS 编码），但不能 100% 保证，转换器必须提供显式配置并用样本校验。

### 2.3 LeRobotDataset v3.0 的写入约束（仓库代码）

核心文件：
- `src/lerobot/datasets/lerobot_dataset.py`（`CODEBASE_VERSION = "v3.0"`）
- `src/lerobot/datasets/utils.py`（validate/paths/features）
- `src/lerobot/datasets/image_writer.py`（当前仅支持 3 通道图像写入）

关键结论（用于指导转换器设计）：
- **必须调用 `finalize()`**：否则 parquet footer 与 meta writer 可能未关闭，数据集不可加载。
- `add_frame(frame)`：
  - `frame` **必须包含 `task` 字段**（不是 features 的一部分，但必需）。
  - `timestamp` 可选，不传则自动用 `frame_index / fps`。
  - 图像特征 dtype 为 `video`/`image` 时，`frame` 里传入 `np.ndarray`（建议 HWC `uint8`）。
- depth（2D/uint16/单通道）当前无法直接写入 image/video pipeline：
  - `image_array_to_pil_image` 明确 TODO depth 支持，因此 v1 必须跳过或做可视化降维。

## 4. 方案 A：录制/回放 Web 化（原脚本零改动）

### 4.1 产品形态（URL 打开即可用）

访问 `http://<host>:<port>/`：
- `/record`：启动录制任务（固定长度），显示实时日志与进度（帧数），显示预计输出路径。
- `/replay`：启动回放任务（支持 stop），显示实时日志与状态。
- `/jobs`：查看历史任务（record/replay/convert），下载日志，查看参数与退出码。
- `/datasets`：浏览 `dataset_dir/task_name` 下 episode 列表（大小、mtime、可触发转换）。

### 4.2 Web 后端核心设计

技术栈建议：
- FastAPI（REST + WebSocket）
- 后端只做“控制面”，不 import 原脚本逻辑（避免 `rospy.init_node()` 同进程冲突）

任务进程模型：
- 每次 record/replay 都由后端启动一个子进程：
  - `python my_devs/cobot_magic/collect_data/collect_data.py ...`
  - `python my_devs/cobot_magic/collect_data/replay_data.py ...`
- stdout/stderr：
  - 同时落盘为 `jobs/<job_id>/stdout.log`、`jobs/<job_id>/stderr.log`
  - 通过 WebSocket 实时推送 tail

### 4.3 任务状态机与互斥策略

状态机：
- `pending -> running -> succeeded | failed | stopped`

互斥策略（必须做，否则会踩 ROS/topic/硬件冲突）：
- `record` 与 `replay` **互斥**（同一时刻只允许一个运行）。
- `replay`：强制单例（脚本 node name 固定）。
- `record`：也强制单例（固定话题/硬件资源占用）。
- `preview`（如果做）：允许与 record 并行，但必须降帧/限资源。

### 4.4 “停止”语义（必须在 UI/验收写清楚）

由于原脚本不可改：
- `record`：
  - “Stop”只能实现为 `Abort`（SIGINT/SIGTERM），**且很可能不保存文件**（见第 2.1 节）。
  - UI 应显示：录制是固定长度（`max_timesteps`），建议只提供“Abort”按钮，并在开始前明确提示。
- `replay`：
  - Stop 是正常行为（停止回放即可）。

### 4.5 参数策略（按代码现实做安全封装）

#### 4.5.1 录制（collect_data）建议暴露参数

必填（或强烈建议用户确认）：
- `dataset_dir`
- `task_name`
- `episode_idx`
- `max_timesteps`
- `frame_rate`
- `instruction`（自然语言指令，可选但强烈建议填写；用于后续 LeRobot v3 的 `task` 字段，原脚本不会保存）
- master/puppet topic（必须显式配置或选择 preset，避免脚本默认值错误）
- 相机 topic（三路 RGB）

建议默认隐藏/限制：
- `camera_names`：不允许用户自由输入（脚本 argparse 定义不可靠）；默认固定为脚本默认三路命名，并在 Web 层用 UI 显示其含义。
- `use_depth_image`：v1 Web MVP 建议默认禁用；若允许启用，必须强提示“脚本可能不稳定”，并在 Runner 中采用“只在启用时传参”的策略（不要传 `False`）。
- `use_robot_base`：同理（type=bool），只在启用时传参；禁用时不传。

#### 4.5.2 回放（replay_data）建议暴露参数

必填：
- `dataset_dir`（脚本 required）
- `task_name`
- `episode_idx`
- `frame_rate`
- 发布 topic（尤其是 `/cmd_vel`，要避免误发到真实机器人）

可选：
- `--only_pub_master`（store_true）

#### 4.5.3 语言指令（language/task）如何补齐（对齐 VLA 训练需求）

你提到的“language / 指令”在本仓库 LeRobot v3 里主要通过 **`task` 字段**表达：
- LeRobot writer 要求 `add_frame(frame)` 的 `frame` dict **必须包含 `task`（str）**，否则会校验失败（见 `src/lerobot/datasets/utils.py` 的 `validate_frame`）。
- 训练/推理侧如果用 VLA（例如 `pi0_fast`），会从 complementary_data 的 `task` 取文本并经过 tokenizer 变成 `observation.language_*` tokens（见 `src/lerobot/processor/tokenizer_processor.py` 与 `src/lerobot/policies/pi0_fast/processor_pi0_fast.py`）。
- 对于没有语言指令的环境，仓库代码也允许 `task=""`（见 `src/lerobot/envs/utils.py:add_envs_task` 的兜底逻辑）。

现实问题：
- `collect_data.py` / `replay_data.py` 并不会采集或存储“指令文本”，HDF5 里也没有 language 字段。
- 但 LeRobot v3 转换时仍必须为每一帧提供 `task`（哪怕为空字符串）。

最终策略（推荐 v1）：
1) **Web 录制时额外采集一个 `instruction` 文本（不传给脚本，只做元数据）**：
   - `/record` 页面增加输入框：`instruction`（例如“把红色方块放到碗里”）。
   - 后端把它写入 job 元数据（例如 `jobs/<job_id>/session.json`），并且在 episode 保存成功后，把同样的 instruction 固化到与 episode 同目录的 sidecar（例如 `.../episode_<idx>.json` 或 `instructions.json`）。
2) **转换器把 instruction 映射到 LeRobot 的 `task`**：
   - 每帧 frame dict 填 `task=<instruction>`（对该 episode 恒定）。
   - 若找不到 instruction（没有填写或缺失 sidecar），按降级策略：
     - 优先 `task=task_name`（至少非空且可追溯来源）
     - 或 `task=""`（严格表示“无语言指令”，与仓库 env fallback 语义一致）

这样做的好处：
- 你可以用同一套 LeRobot v3 结构同时容纳“带语言指令的 VLA 训练数据”与“采集时无指令、后处理补标注”的两类数据。
- 后续想训练语言条件策略时，只需要补齐/修正 instruction 文本，不需要重录数据。

### 4.6 Web 预览（MVP 推荐方案）

目标：录制时能在浏览器确认相机画面是否正常。

推荐 MVP：使用 ROS 的 `web_video_server`
- 优点：Web 后端不需要订阅 ROS image，也不会触发 `rospy.init_node()` 同进程冲突。
- 前端直接用 `<img src="http://host:8080/stream?topic=...">` 显示 MJPEG 流。

备选（后续增强）：后端独立预览进程订阅 image -> JPEG -> WebSocket
- 必须独立进程（避免与 record 子进程在同进程抢 ROS）。
- 必须降帧（例如 5 FPS）避免影响采集稳定性。

### 4.7 API 草案（后端）

REST：
- `POST /api/record/start`
- `POST /api/record/abort`
- `POST /api/replay/start`
- `POST /api/replay/stop`

## 5. 方案 B：Cobot Magic HDF5 -> LeRobotDataset v3.0 转换

### 5.1 转换目标与边界（v1）

目标：
- 将 `episode_*.hdf5` 批量转换为可被本仓库 `LeRobotDataset` 正常加载的数据集目录。
- 严格满足 v3.0 writer 的写入/校验约束，避免手写 meta 造成不一致。

边界（v1 强建议）：
- 必转：RGB 三路图像 + `qpos/qvel/effort` + `/action` + `/base_action`
- depth：v1 默认不转（D0），可选做“深度可视化 video”（D1）；不做原生 depth dtype（D2 留后续）。

### 5.2 写入策略（必须复用 LeRobot writer）

必须使用：
- `src/lerobot/datasets/lerobot_dataset.py:LeRobotDataset.create`
- `LeRobotDataset.add_frame`
- `LeRobotDataset.save_episode`
- `LeRobotDataset.finalize`

原因：
- `meta/info.json`、`meta/tasks.parquet`、`meta/episodes/*.parquet`、`meta/stats.json` 的维护逻辑已经在 writer 内封装；手写非常容易出错。

### 5.3 输出目录约定（避免踩 LeRobot create 的 root 语义）

注意：`LeRobotDataset.create(..., root=...)` 的 `root` 参数是“数据集根目录”，不会自动拼上 `repo_id`。

推荐约定：
- `repo_id = "local_cobot_magic/<task_name>"`（仅作为标识）
- `dataset_root = <output_root>/<repo_id>/`（真实写入路径）
- `LeRobotDataset.create(repo_id=repo_id, root=dataset_root, ...)`

要求：
- `dataset_root` 必须不存在（`LeRobotDatasetMetadata.create` 会 `mkdir(exist_ok=False)`）。

### 5.4 字段映射（v1 推荐，含语义说明）

以 HDF5 的一个时间步（索引 `i`）为单位，构造一个 LeRobot frame dict：

数值特征：
- `/observations/qpos[i]` -> `observation.state`（`float32`, shape `(14,)`）
- `/observations/qvel[i]` -> `observation.qvel`（`float32`, shape `(14,)`）
- `/observations/effort[i]` -> `observation.effort`（`float32`, shape `(14,)`）
- `/action[i]` -> `action`（`float32`, shape `(14,)`）
  - 重要：此处的 “action” 语义在当前脚本中是 master 关节位置拼接（但可能因默认 topic 错误而被污染），因此必须依赖 Web preset 或转换前校验确保它确实来自 master。
- `/base_action[i]` -> `action.base`（`float32`, shape `(2,)`）
  - 语义备注：录制时来自 `/odom` 的观测速度，但回放脚本把它当作 `/cmd_vel` 去发布；因此作为 `action.base`（动作标签）是合理的最短路径。若后续你明确要把它当 observation，可切换为 `observation.base_vel`。

图像特征：
- `/observations/images/<cam>[i]` -> `observation.images.<cam>`（dtype `video`，shape `(480, 640, 3)`）
  - `<cam>` 建议直接使用 HDF5 里的 group key（通常是 `cam_high/cam_left_wrist/cam_right_wrist`）。

任务字段：
- `task`：在本仓库里它既是“任务标签”也是“语言指令”的主要入口（会进入 tasks.parquet，并可被 tokenizer 转成语言 tokens）。
- 由于原始 HDF5 不含语言指令，本方案推荐：
  - 默认 `task=<instruction>`（来自 Web 录制时填写/后处理标注的 sidecar）
  - 无 instruction 时降级为 `task=task_name` 或 `task=""`
- 注意：`task` 必须出现在 `add_frame(frame)` 的 `frame` dict 中，否则会被 `validate_frame` 拒绝。

时间戳：
- HDF5 未保存显式时间戳，v1 使用 `timestamp = i / fps`（fps 来自录制参数或 session.json，默认 30）。
- 若 frame dict 不提供 `timestamp`，`add_frame` 会自动按 `frame_index / fps` 填充；但建议转换器显式写入，便于后续对齐与诊断。

### 5.5 features 字典（示例）

features 需要在 `LeRobotDataset.create` 时提供（shape 建议用 tuple）：

```python
features = {
    "observation.state": {"dtype": "float32", "shape": (14,), "names": None},
    "observation.qvel": {"dtype": "float32", "shape": (14,), "names": None},
    "observation.effort": {"dtype": "float32", "shape": (14,), "names": None},
    "action": {"dtype": "float32", "shape": (14,), "names": None},
    "action.base": {"dtype": "float32", "shape": (2,), "names": None},
    "observation.images.cam_high": {
        "dtype": "video",
        "shape": (480, 640, 3),
        "names": ["height", "width", "channels"],
    },
    "observation.images.cam_left_wrist": {
        "dtype": "video",
        "shape": (480, 640, 3),
        "names": ["height", "width", "channels"],
    },
    "observation.images.cam_right_wrist": {
        "dtype": "video",
        "shape": (480, 640, 3),
        "names": ["height", "width", "channels"],
    },
}
```

约束：
- feature key 不能包含 `/`（LeRobot 会校验）。
- 图像建议传 HWC `uint8`，以匹配 `validate_frame` 与 image writer。

### 5.6 转换流程（推荐实现步骤）

1) 扫描输入：
- 找到 `dataset_dir/task_name/episode_*.hdf5`
- 解析 episode_idx，按 idx 排序

2) 读取并校验每个 episode：
- 校验 keys 是否存在（qpos/qvel/effort/action/base_action/images）
- 校验 `T` 一致（所有数组第一维相同）
- 校验图像 shape 是否为 `(480, 640, 3)`，dtype 是否为 `uint8`
- 记录相机 key 列表（来自 `/observations/images/` group）

3) 创建 LeRobotDataset：
- 推荐 `use_videos=True`，`vcodec="h264"`（更快），必要时 `batch_encoding_size>1` 做批量编码。
- 如需最快跑通，也可用 dtype `"image"`（避免视频编码，但 parquet 会很大），此时验收标准要相应调整（不要求 videos/ 目录存在）。

4) 写入：
- 对每个 episode：
  - `dataset.episode_buffer = dataset.create_episode_buffer(episode_index=ep_idx)`
  - for `i in range(T)`：
    - 组装 frame dict：`{"task": ..., "timestamp": ..., <features>}`
    - 调 `dataset.add_frame(frame)`
  - 调 `dataset.save_episode()`

5) 完成：
- 调 `dataset.finalize()`
- 以读者方式验证：`LeRobotDataset(repo_id, root=dataset_root)` 可成功加载。

### 5.7 深度图策略（v1/v2/v3）

现实限制：
- `src/lerobot/datasets/image_writer.py` 当前只支持 3 通道图像写入；depth 通常是 2D uint16。

策略：
- D0（v1 推荐）：不转换 depth（最稳、最快打通）。
- D1（v2 推荐）：将 depth 做可视化（归一化 + colormap）得到 3 通道 `uint8`，以 `observation.depth_viz.<cam>` 的 video 存储（明确“不可度量深度”）。
- D2（后续）：扩展 LeRobot pipeline 支持 `video.is_depth_map=true` 与 1 通道/uint16 的存储与解码（工作量大，不与 Web MVP 同期）。

### 5.8 相机通道顺序（RGB/BGR）与校验

风险：
- 回放脚本里出现了 B/R swap，并发布 `"bgr8"`，不能保证 HDF5 内一定是 RGB。

转换器必须提供：
- `--image_channel_order {rgb,bgr}`（至少二选一）
- 当选 `bgr` 时，在写入 LeRobot 前做 `img = img[:, :, ::-1]` 转为 RGB（以便 PIL/训练侧一致）。

最小校验：
- 转换前从 HDF5 抽 3 张帧（每个相机各 1 张），保存为 PNG，肉眼确认颜色（或和 RViz/回放对比）。

## 6. 验收标准（可操作）

### 6.1 Web MVP 验收

- `/record` 能启动录制子进程，日志实时可见，且能在录制结束后看到 HDF5 文件存在于预期路径。
- `/replay` 能启动回放子进程，能 stop，stop 后进程退出且日志可追溯。
- `/jobs` 能查看历史任务：参数、开始/结束时间、退出码、stdout/stderr。
- 并发限制生效：尝试同时启动 record+replay 或两个 replay，会被拒绝并给出明确错误提示。

### 6.2 转换器 MVP 验收

最小验收（必须通过）：
- 转换成功后，可执行以下加载逻辑且不报错：
  - `from lerobot.datasets.lerobot_dataset import LeRobotDataset`
  - `ds = LeRobotDataset(repo_id, root=dataset_root)`
  - `len(ds) > 0`
  - `sample = ds[0]`，包含：
    - `observation.state` / `action` / `action.base`
    - `observation.images.*` 可被解码为图像 tensor

一致性验收（建议自动化脚本）：
- 每个 episode 的帧数与 HDF5 的 `T` 一致。
- 抽样帧做像素级近似对比（允许视频编码损失，但不能出现错相机/错顺序）。

## 7. 已知风险与规避策略（在不改脚本的前提下）

1) 录制无法“早停保存”
- 风险：Web UI 的“Stop”容易让用户误以为会保存 partial episode。
- 对策：UI 改为 “Abort（不保存）”；在开始前强提示“固定长度录制”。

2) `type=bool` 参数导致 CLI 传参不可靠
- 风险：Web Runner 拼参错误导致脚本行为与 UI 不一致。
- 对策：Runner 对 bool 参数采用“只在启用时传参”的约定；禁用时不传；并在 schema 层做显式说明。

3) master topic 默认值错误导致 `/action` 被污染
- 风险：转换后 `action` 语义错误，训练会学错标签。
- 对策：Web UI 强制用户选择 preset（实验室默认 topic 集）或手工填写 master/puppet topic；并在启动前做 topic 存在性检查（可选）。

4) depth 分支不稳定
- 风险：启用 depth 可能触发空指针或 shape 不一致。
- 对策：v1 禁用 depth；v2 再引入 depth_viz。

5) 视频编码耗时/CPU 占用
- 风险：转换耗时长，影响迭代效率。
- 对策：v1 默认 `vcodec="h264"`；提供 `--use_images`（dtype image）作为“最快跑通但体积大”的模式；转换任务放后台并可取消。

6) ROS 环境差异导致子进程启动失败
- 风险：Web 后端环境与手工终端不同，导致找不到 ROS 包或话题。
- 对策：后端启动子进程时显式设置 `cwd=/home/agilex/cqy/flexible_lerobot`，并允许在配置中指定启动命令前缀（例如 `bash -lc "source /opt/ros/noetic/setup.bash && ..."`)。

## 9. 需要你确认的关键决策（否则无法彻底“定稿”）

1) `base_action` 在 LeRobot 中落位：
- 推荐默认：`action.base`（因为 replay 脚本将其发布为 `/cmd_vel`）
- 备选：`observation.base_vel`（若你明确它只作为观测，不作为控制标签）

2) 图像通道顺序（RGB/BGR）：
- 需要你指定：HDF5 内保存的是 RGB 还是 BGR（或提供样本 episode 让我们用可视化确认）。

3) `/action` 语义一致性：
- 以当前 `collect_data.py` 的实现，`/action` 是 master 关节位置拼接；但默认 CLI topic 配置可能错误。
- 需要你确认：Web 层是否必须强制使用正确的 master topic preset（强烈建议 “必须”）。

4) language / 指令策略：
- A. 允许 `task=""`（完全无语言指令，训练 VLA 时等价于“无条件”）
- B. 默认 `task=task_name`（弱标注，至少能区分任务目录）
- C. 录制时必须填写 `instruction` 并写 sidecar（推荐，最适合 VLA 训练的使用方式）
