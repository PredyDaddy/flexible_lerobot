# PI05 第二阶段执行方案

文档日期：2026-03-15

文档位置：`my_devs/docs/pi05_engineering/PI05第二阶段执行方案_20260315.md`

适用范围：

- `my_devs/pi05_engineering/`
- Phase-1 完成后的后续架构推进

## 1. 阶段定位

第二阶段当前不应被理解成：

- 继续堆运行时修补
- 直接改通用 `src/lerobot/async_inference/`
- 直接把 RTC 打开并投入正式工作

第二阶段更准确的定位应当是：

> 在 Phase-1 本地 plain chunk runtime 已经完成契约冻结和实机验收后，进入“后端化与运行治理”阶段。

这里的核心目标不是改模型，而是把当前已经验证过的本地运行时语义，变成可迁移、可观测、可回退的系统能力。

## 2. 准入门槛

在进入第二阶段主开发前，建议先满足以下门槛。

## 2.1 plain chunk 实机准入门槛

至少满足：

1. `connect_smoke` 稳定通过
2. `load_connect_smoke` 稳定通过
3. plain chunk 30 秒通过
4. plain chunk 120 秒通过
5. 无 `Port is in use!`
6. 无持续 `queue_empty_events`
7. shutdown 可稳定收尾

如果这一步还没通过，就不应进入第二阶段主开发。

## 2.2 契约冻结门槛

以下语义需要在进入第二阶段前固定下来：

1. `original_actions` 与 `processed_actions` 的边界
2. `leftover` 的真实含义
3. `real_delay` / `inference_delay` / `action_index_delta` 的口径
4. queue merge / refill 规则
5. startup barrier
6. shutdown / disconnect ownership
7. plain 与 RTC 模式的差异边界

如果这些定义还在频繁变化，第二阶段会把本地问题直接放大到跨进程问题。

## 3. 第二阶段目标

第二阶段建议拆成四个目标。

## 3.1 目标一：冻结本地 runtime 契约

要做到：

- 本地 plain chunk runtime 的关键语义不再频繁修改
- 运行日志与指标已经足以支撑故障定位
- 回退路径明确

交付物建议：

- 一份契约说明文档
- 一份实机验收清单
- 一份固定的 smoke / live 验证矩阵

## 3.2 目标二：做 PI05 专用 server/client 语义映射层

这里强调“PI05 专用”，而不是直接泛化。

当前建议做法是：

1. 先在 `my_devs/` 下做 PI05 专用启动器
2. 先把 Phase-1 已经验证过的语义映射过去
3. 不直接改通用 `async_inference` 主路径

原因很直接：

- 当前 Phase-1 是本地 runtime
- 第二阶段如果直接去改通用层，很容易把还没冻结的行为扩散到更大的故障面

## 3.3 目标三：补运行治理

第二阶段必须把以下能力做成一等公民：

1. 分层 health check
2. connect / load / live 的故障域拆分
3. 结构化日志
4. 指标归档
5. 明确的回退路径

第二阶段不是“先做 server/client，再想怎么运维”，而是：

> 运行治理要和后端化一起设计。

## 3.4 目标四：建立灰度与回退机制

第二阶段完成后，系统至少应支持：

1. baseline 单步入口回退
2. Phase-1 本地 chunk runtime 回退
3. Phase-2 server/client 入口回退

没有回退路径的第二阶段，不建议上机做真实切换。

## 4. 工作分解

建议把第二阶段拆成下面六个工作包。

## 4.1 WP1：契约冻结

内容：

1. 明确 queue/state ownership
2. 明确 delay 相关指标定义
3. 明确 shutdown 边界
4. 明确 plain / RTC 模式分界

验收标准：

- 文档口径与当前代码实现一致
- 验收清单与日志口径一致

## 4.2 WP2：Phase-2 启动器骨架

内容：

1. PI05 policy server 启动器
2. PI05 robot client 启动器
3. 基础配置解析
4. 健康检查与启动顺序

约束：

- 放在 `my_devs/` 路径下
- 先不要直接泛化成多 policy 共用框架

验收标准：

- 本地可以 shadow 启动
- 不接管真实执行也能采样并打日志

## 4.3 WP3：跨进程 ownership 设计

内容：

1. 哪一侧维护 queue
2. 哪一侧维护 `action_index_before_inference`
3. 哪一侧维护 `prev_chunk_left_over`
4. 哪一侧负责 stop / teardown 协调

这一步如果不先做清楚，后面所有问题都会变成“到底谁是状态源”的争论。

验收标准：

- 每个关键状态只有一个明确 owner
- 没有双写或口径不一致

## 4.4 WP4：结构化观测

内容：

1. 结构化日志
2. server/client 两侧 latency
3. queue/delay 对齐检查
4. trace 回放最小能力

建议优先增加以下指标：

1. `delay_mismatch_ratio`
2. `signed_delay_error`
3. `executable_backlog`
4. `original_backlog`
5. `inference_overrun_ratio`
6. `producer_refill_lead`

补充说明：

- 截至 2026-03-16，Phase-1 本地 runtime 已经开始落地其中一部分：
  - `delay_mismatch_ratio`
  - `latest_delay_error`
  - `delay_error_mean`
  - `delay_error_max_abs`
  - `latest_original_backlog_length`
  - `latest_executable_backlog_length`
  - `latest_backlog_gap`
  - `latest_inference_overrun_ratio`
- 第二阶段不是“从零开始造指标”，而是把这些已落地口径继续结构化、跨进程化，并补齐仍未落地的指标

验收标准：

- 能从日志中清晰区分 connect、load、live 三类故障
- 能区分真实 backlog 与账本 leftover

## 4.5 WP5：shadow 对比与灰度

内容：

1. 先 shadow 跑，不接管真实执行
2. 比对本地 runtime 与 Phase-2 路径的指标漂移
3. 建立灰度切换方案

验收标准：

- shadow 运行结果与本地 runtime 关键语义一致
- 出现漂移时能快速定位来源

## 4.6 WP6：RTC 引入策略

RTC 不应作为第二阶段入口条件，而应作为第二阶段后置能力。

推荐顺序：

1. 先 plain
2. 再 shadow plain server/client
3. 再 RTC

验收标准：

- RTC on/off 差异可解释
- 出现异常时能一键回退到 plain local runtime

## 5. 风险前置实验

在第二阶段主开发前，建议先固定做四类前置实验。

## 5.1 分层冒烟矩阵

固定执行：

1. `connect_smoke`
2. `load_connect_smoke`
3. plain chunk 30 秒
4. plain chunk 120 秒

目的：

- 证明当前故障域已分开

## 5.2 生命周期注入

建议有意识验证：

1. connect 失败
2. worker 异常退出
3. shutdown 超时
4. disconnect 前 worker 未退出

目的：

- 提前把 teardown 风险暴露出来

## 5.3 指标漂移实验

建议在 Phase-2 shadow 模式下比对：

1. actor / producer 节奏
2. queue 深度
3. delay 相关指标
4. backlog 相关指标

目的：

- 确认后端化没有改坏当前语义

## 5.4 回退演练

至少固定演练一次：

1. Phase-2 -> Phase-1
2. Phase-1 -> baseline

目的：

- 避免发布后才发现回退路径不通

## 6. 验收标准

第二阶段建议分两层验收。

## 6.1 架构验收

通过条件：

1. 契约冻结完成
2. owner 划分清楚
3. 日志与指标足够支撑定位
4. 回退路径明确

## 6.2 运行验收

通过条件：

1. shadow 运行稳定
2. 关键指标与本地 runtime 无异常漂移
3. connect / load / live 三层故障能快速定位
4. small rollout 可灰度切换

## 7. 推荐先后顺序

当前建议不要并行推进所有东西，而应按下面顺序来。

1. 完成 plain chunk 实机验收
2. 冻结 Phase-1 契约
3. 补 connect 生命周期治理
4. 补 observability 语义
5. 做 PI05 专用 server/client 最小骨架
6. 先 shadow，对比漂移
7. 再考虑 RTC
8. 最后才决定是否抽回通用层

## 8. 当前结论

截至 2026-03-15，第二阶段已经可以开始规划，但还不建议直接全面开工。

更准确的判断应当是：

1. Phase-1 核心开发基本完成
2. 当前工作重点是 Phase-1.5：plain chunk 实机验收与运行治理
3. 第二阶段应当先做“PI05 专用后端化”，而不是直接做“通用异步框架改造”

因此，第二阶段最重要的原则是：

> 只迁移已经验证过的语义，不把还没冻结的问题扩散到更大系统里。
