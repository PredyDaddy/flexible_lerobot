## 1. 总体判断

当前 PI05 第一阶段工作可以定义为：

> 本地 plain chunk runtime 的核心架构已经成立，当前进入 Phase-1.5，也就是实机验收与运行治理阶段。

这意味着：

1. 当前问题已经不是“脚本能不能启动”。
2. 当前问题也不是“模型是否能够加载”。
3. 当前主要矛盾已经转为：
   - connect 生命周期是否足够稳
   - plain chunk 是否已经达到可重复验收门槛
   - delay/backlog 指标是否足够支撑后续 RTC 与 Phase-2 判断

## 2. 已确认成立的架构结论

### 2.1 第一阶段的方向是对的

最重要的正向结论不是“代码变多了”，而是 runtime 语义被显式化了：

- `predict_action_chunk()` 已经进入独立 runtime 路径
- `original_actions` / `processed_actions` 边界已被保留下来
- queue merge/refill、startup barrier、shutdown、metrics 都已有明确落点
- `plain` 与 `RTC` 两条路径的边界开始变得可讨论、可测试、可回退

### 2.2 ACT 能控机，不足以否定 chunk runtime；但它足以说明故障域

ACT 和旧 PI05 入口都是单线程同步链路：

- `get_observation -> predict_action -> send_action`

新 PI05 chunk runtime 才引入：

- producer / actor 双线程

因此：

- 早期 `Port is in use!` 更像 runtime ownership 设计问题
- 它不能被解释成“机器人本体不能控”
- 也不能被解释成“PI05 policy 本身不能工作”

### 2.3 当前版本已经不再停留在“完全跑不起来”

保守参数下的 plain chunk 30 秒运行已经表明：

- `first_chunk_ready=True`
- `queue_empty_events=0`
- `queue_actions_popped` 持续增长
- `actor_rate_hz` 基本贴近目标值

这说明当前版本已经具备：

- 连续获取 observation
- 连续推理
- 连续发动作
- 正常收尾

的最小可运行能力。

## 3. 已收敛的软件问题

截至 2026-03-16，至少有以下问题已经明确收敛：

1. robot I/O 串行化已经补上，不再让 producer/actor 直接并发打同一个 robot I/O。
2. `camera_fps` 与 control `fps` 已解耦，不再把相机档位问题误判成控制频率问题。
3. `run_time_s` 的起点已后移到 `connect()` 成功之后，不再让加载阶段吞掉运行预算。
4. plain 模式下的 `delay mismatch` warning 已收窄，不再把预期的 1-step 相位差过度告警。
5. plain 模式的 observability 已开始收敛，新增了：
   - `delay_mismatch_ratio`
   - `latest_delay_error`
   - `delay_error_mean`
   - `delay_error_max_abs`
   - `latest_original_backlog_length`
   - `latest_executable_backlog_length`
   - `latest_backlog_gap`
   - `latest_inference_overrun_ratio`

## 4. 尚未收敛的风险

### 4.1 connect 生命周期风险

当前仍需谨慎对待：

- `Incorrect status packet`
- `Torque_Enable` 相关 connect/configure 失败

这些错误更像：

- 总线初始化瞬态
- 接触或供电波动
- connect 阶段缺少 retry/backoff 与预检

### 4.2 短时稳定不等于长期稳定

当前已经证明：

- 30 秒保守 plain chunk 可以运行

但尚未证明：

- 120 秒级别是否稳定
- queue / delay / backlog 指标是否会慢性漂移
- shutdown 在更长运行后是否仍然稳定

### 4.3 第二阶段尚未具备直接开工的通用化条件

现在如果直接去改：

- `src/lerobot/async_inference/`

风险很高，因为会把尚未完全冻结的本地 runtime 语义直接扩散到更大的系统里。

更合理的做法应当是：

> 先做 PI05 专用 server/client 语义映射层，再决定哪些能力值得沉淀回通用层。

## 5. 当前阶段的架构定位

当前阶段不应再称为“第一阶段开发中”，更准确的叫法是：

- Phase-1：本地 chunk runtime 基础实现
- Phase-1.5：plain chunk 实机验收与运行治理
- Phase-2：PI05 专用 server/client 化与后端运行治理

这个划分的意义在于：

1. 避免把“代码写出来了”和“实机可验收”混为一谈。
2. 避免把“本地语义未冻结”和“跨进程后端化”同时推进。

## 6. 推荐推进顺序

建议按下面顺序继续：

1. 先完成 plain chunk 的分层验收矩阵：
   - `connect_smoke`
   - `load_connect_smoke`
   - plain chunk 30 秒
   - plain chunk 120 秒
2. 再收一轮 connect 生命周期治理：
   - 预检
   - retry/backoff
   - 更清晰的失败分类
3. 基于新增指标做 live 验证：
   - 区分相位差噪声与真实 delay 异常
   - 区分 original backlog 与 executable backlog
4. 只有在 plain chunk 通过门槛后，再保守开启 RTC。
5. RTC 与 plain 都稳定后，再启动第二阶段 PI05 专用 server/client 方案。

## 8. 最终结论

一句话总结当前状态：

> PI05 本地 plain chunk runtime 已经从“能不能做”进入“怎么把它做成可验收、可回退、可迁移”的阶段。

因此，后续最该做的不是盲目开 RTC，也不是直接跳第二阶段通用异步化，而是：

- 先把 plain chunk 的实机门槛跑扎实
- 再用已经收敛的语义去驱动 Phase-2
