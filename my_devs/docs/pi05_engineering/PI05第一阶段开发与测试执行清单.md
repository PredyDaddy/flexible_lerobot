# PI05 第一阶段开发与测试执行清单

## 0. 文档目的

这份文档不是技术方案，也不是实施 Plan 的重复版本。

它只做一件事：

- 把 **PI05 本地 Chunk Runtime 第一阶段** 的实际执行流程写成一份可以直接照着推进的清单

重点回答两个问题：

1. 第一步到最后一步，具体怎么做
2. 每一步怎么测试，哪些测试我可以做，哪些必须由用户来做

---

## 1. 第一阶段目标

第一阶段只交付：

- `my_devs/pi05_engineering/` 下的本地 PI05 chunk runtime
- 从 `select_action()` 切换到 `predict_action_chunk()`
- 本地双线程 producer / actor 运行时
- queue / latency / startup / safety / metrics
- RTC plumbing 先接上，但默认不开

第一阶段不交付：

- `src/lerobot/async_inference/` 改造
- gRPC server/client 版本
- `src/lerobot/policies/pi05/` 模型结构修改
- 未经用户确认的任何上机执行

## 3. 当前基线事实

当前 baseline 是：

- `my_devs/train/pi/so101/run_pi05_infer.py`

当前已确认的事实：

- baseline 走的是同步单循环
- 调用方走 `predict_action(...) -> policy.select_action(...)`
- PI05 的 RTC 不能走 `select_action()`，必须切到 `predict_action_chunk()`
- baseline 显式使用 `strict=False` 加载
- baseline 从 checkpoint 目录加载 `policy_preprocessor.json` / `policy_postprocessor.json`
- baseline 依赖 repo root 下的本地 tokenizer 目录兜底

当前还要特别记住一个容易误判的点：

- `run_pi05_infer.py --dry-run true` 只能验证路径、配置和入口解析
- 它不能证明 tokenizer、policy、pre/post processor、`predict_action_chunk()` 链路已经通

## 5. 全流程执行顺序

严格按下面顺序推进，不跳步。

### Phase 0：基线固化

目标：

- 不动 baseline 主行为
- 先把当前已知可用路径固定住

动作：

1. 记录默认 checkpoint 路径
2. 记录 baseline CLI 参数和默认值
3. 记录当前脚本依赖的 tokenizer / processor / robot wiring 事实
4. 运行 baseline 的离线 dry-run，确认入口解析正常

我可以执行：

```bash
conda run -n lerobot_flex python my_devs/train/pi/so101/run_pi05_infer.py --dry-run true
```

通过标准：

- repo root 可解析
- policy path 存在
- `config.json` 可读取
- CLI 参数解析正常
- 不触发真实硬件连接

注意：

- 这一步不验证 tokenizer / policy / processor 真正加载

---

### Phase 1：提取共享 helper

目标：

- 把 baseline 中必须保留的加载与初始化逻辑提取到共享模块

动作：

1. 新增 `runtime/common.py`
2. 提取以下 helper：
   - `resolve_repo_root(...)`
   - `parse_bool(...)`
   - `env_bool(...)`
   - `maybe_path(...)`
   - `ensure_local_tokenizer_dir(...)`
   - `load_pre_post_processors(...)`
   - robot config 组装逻辑
   - policy 加载逻辑
   - dataset feature 构建逻辑
3. 保留 baseline 当前显式使用的：
   - `policy_class.from_pretrained(str(policy_path), strict=False)`

我可以执行的测试：

```bash
conda run -n lerobot_flex python my_devs/train/pi/so101/run_pi05_infer.py --dry-run true
```

以及后续新增的**纯离线 load smoke**：

- 检查 tokenizer 目录
- 加载 policy
- 加载 pre/post processor
- 不执行 `robot.connect()`

通过标准：

- helper 不改变 baseline 的配置解析结果
- pure load smoke 能在不接硬件的情况下完成加载链验证

---

### Phase 2：搭建 chunk runtime 骨架

目标：

- 新入口切到 `predict_action_chunk()`
- 先不开 RTC

动作：

1. 新增 `run_pi05_chunk_infer.py`
2. 新增 `runtime_config.py`
3. 新增 `runtime_state.py`
4. 新增 `chunk_inference.py`
5. 新增 `actor_loop.py`
6. 新增 `producer_loop.py`

实现要求：

- 入口脚本负责 CLI、初始化、线程启动、统一 shutdown
- `chunk_inference.py` 负责 observation -> preprocessor -> `predict_action_chunk()` -> postprocessor
- 明确区分：
  - `original_actions`
  - `processed_actions`
- `original_actions` 只用于 leftover / delay / merge
- `processed_actions` 只用于机器人执行

我可以执行的测试：

```bash
conda run -n lerobot_flex python my_devs/pi05_engineering/run_pi05_chunk_infer.py --dry-run true
```

以及：

- fake observation 单测
- mock policy 单测
- `predict_action_chunk()` 离线 smoke

通过标准：

- 新入口 CLI 正常
- 不触发真实硬件连接
- `predict_action_chunk()` 路径能在 mock/fake 输入下打通

---

### Phase 3：接入显式 queue controller

目标：

- 把动作队列显式化
- 让 runtime 获得 queue 控制权

动作：

1. 新增 `queue_controller.py`
2. 底层优先复用：
   - `src/lerobot/policies/rtc/action_queue.py`
3. 外层补足第一阶段需要的 plain-mode 语义：
   - stale-prefix trimming
   - 空队列策略
   - 更细粒度日志
   - queue 指标采集

这里要明确：

- 现有 `ActionQueue` 在 RTC disabled 时只是 append
- 它不会自动修剪“新 chunk 中已过期的 prefix”
- 所以 plain-mode 额外语义必须由 `queue_controller.py` 实现

我可以执行的测试：

```bash
conda run -n lerobot_flex pytest -q \
  tests/policies/rtc/test_action_queue.py \
  tests/policies/rtc/test_latency_tracker.py
```

如果新增 runtime 自己的测试，再执行：

```bash
conda run -n lerobot_flex pytest -q tests/ -k pi05_runtime
```

通过标准：

- append 行为正确
- replace 行为正确
- leftover 行为正确
- 空队列行为符合配置
- plain-mode stale-prefix trimming 逻辑被单测覆盖

---

### Phase 4：启动安全与停机安全

目标：

- 把安全 contract 写死

必须落实：

1. 首个 chunk 未 ready 前不允许发送未定义动作
2. 空队列时行为必须配置化
3. 默认推荐：
   - `hold-last-action`
   - 若无 last action，则 `skip-send + warning`
4. 单步动作发送前预留 sanity check
5. Ctrl+C、线程异常、queue 异常时统一 shutdown

这里要特别说明：

- 这些是**第一阶段要新增实现的 contract**
- 不能假设 `examples/rtc/eval_with_real_robot.py` 已经完整提供这些行为

我可以执行的测试：

- actor loop mock test
- startup barrier test
- empty queue strategy test
- shutdown path test

通过标准：

- 首个 chunk 未 ready 前不会误发动作
- 空队列行为与配置一致
- 异常时能够走统一停机路径

---

### Phase 5：指标与日志

目标：

- 把“能跑”变成“可解释”

至少新增指标：

- chunk inference latency
- preprocess / model / postprocess 分段耗时
- queue depth
- queue starvation 次数
- refill 次数
- `action_index_before_inference`
- `real_delay`
- leftover 长度
- stale-prefix trimming 次数
- first chunk ready 时间
- first warmup latency
- actor loop 实际频率

这里的口径：

- 这些大多是**新 runtime 需要新增的 instrumentation**
- 不是 baseline 当前现成已有的日志

我可以执行的测试：

- mock 输入日志回放
- 单测验证计数器
- 结构化日志格式检查

通过标准：

- 指标非负且数值合理
- 启动期和稳态指标可区分
- starvation、queue depth、latency 都可见

---

### Phase 6：RTC plumbing 接通，但默认关闭

目标：

- 接上 RTC 所需状态链路
- 默认不启用 guidance

动作：

1. 维护以下 runtime 状态：
   - `prev_chunk_left_over`
   - `action_index_before_inference`
   - `real_delay`
   - `original_actions`
   - `processed_actions`
2. 若用户启用 RTC：
   - 显式设置 `policy.config.rtc_config`
   - 显式调用 `policy.init_rtc_processor()`

这里要明确当前仓库实现现实：

- 第一版若沿用仓库参考，merge 直接使用的仍然是 `ceil(new_latency / dt)` 计算出来的 `real_delay`
- `action_index_before_inference` 更适合作为校验和日志指标
- 如果未来要把 action index 差值升格为 authoritative merge 信号，那是后续新增实现，不是当前仓库现成事实

我可以执行的测试：

- RTC off 的离线 smoke
- RTC on 的 mock policy smoke
- RTC state logging test

通过标准：

- RTC off 时不影响 plain chunk 行为
- RTC on 时状态链路完整可见
- on/off 切换不破坏入口逻辑

## 7. 用户执行的上机清单

这一节只提供给用户执行。

我不会替用户执行。

### 7.1 上机前确认

用户在上机前确认：

- 机器人处于安全位置
- 周围无碰撞风险
- 串口、相机、供电状态正常
- 当前使用的是正确 checkpoint
- 已理解回退方式

### 7.2 用户执行 plain chunk 上机

用户执行：

```bash
conda run -n lerobot_flex python my_devs/pi05_engineering/run_pi05_chunk_infer.py --enable-rtc false
```

用户重点观察：

- 是否持续稳定输出动作
- 是否有明显停顿
- actor loop 频率是否稳定
- queue 是否频繁见底
- 首个 chunk 未 ready 前是否误发动作

### 7.3 用户执行 RTC 保守启用

只有 plain chunk 稳定后，用户再执行：

```bash
conda run -n lerobot_flex python my_devs/pi05_engineering/run_pi05_chunk_infer.py \
  --enable-rtc true \
  --actions-per-chunk 8 \
  --rtc-execution-horizon 8 \
  --rtc-max-guidance-weight 10.0 \
  --rtc-prefix-attention-schedule EXP
```

用户重点观察：

- chunk 切换是否更平滑
- 是否出现跳变
- 是否出现异常替换
- 是否出现 runaway motion
- RTC on/off 差异是否可解释

### 7.4 用户执行后的反馈模板

建议用户反馈以下内容：

- 执行命令
- 运行时长
- 是否出现停顿
- 是否出现误动作
- 是否触发 warning / error
- plain chunk 与 RTC 的主观差异
- 关键日志片段
