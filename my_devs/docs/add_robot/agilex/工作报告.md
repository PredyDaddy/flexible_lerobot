## 3. 我对当前架构的理解

### 3.1 AgileX 在当前仓库里的角色划分

当前 AgileX 接入在仓库里被拆成两部分：

1. 机器人端 `robot`
   - 负责读取从臂状态和相机观测。
   - 对应代码在 `src/lerobot/robots/agilex/`。
2. 遥操作端 `teleoperator`
   - 负责读取主臂动作。
   - 对应代码在 `src/lerobot/teleoperators/agilex_teleoperator/`。

关键源码位置如下：

- 机器人读取从臂状态与图像：
  - `src/lerobot/robots/agilex/agilex.py:47-57`
  - `src/lerobot/robots/agilex/agilex.py:101-112`
- 遥操作端读取主臂动作：
  - `src/lerobot/teleoperators/agilex_teleoperator/agilex_teleoperator.py:35-45`
  - `src/lerobot/teleoperators/agilex_teleoperator/agilex_teleoperator.py:77-79`

### 3.2 改造前的默认 topic 绑定关系

在脚本与配置中，当前默认绑定关系是：

- 观测状态：
  - `STATE_LEFT_TOPIC=/puppet/joint_left`
  - `STATE_RIGHT_TOPIC=/puppet/joint_right`
- 遥操作动作：
  - `ACTION_LEFT_TOPIC=/master/joint_left`
  - `ACTION_RIGHT_TOPIC=/master/joint_right`
- 回放命令：
  - `COMMAND_LEFT_TOPIC=/master/joint_left`
  - `COMMAND_RIGHT_TOPIC=/master/joint_right`

这意味着当前录制链路默认语义是：

- `observation.state` 来自从臂 `puppet`
- `action` 来自主臂 `master`

这个设计与遥操作数据集的一般定义一致，但是否符合当前用户想要的“数据中不要主臂、只要从臂”这一目标，需要单独审计。本报告后续会把这一点作为专门审核项。

### 3.3 LeRobot 数据格式中语言信息的正确入口

本轮一个关键问题是：如果后续要训练 VLA，语言信息应该怎么进入数据集。

我核对了当前仓库的数据流后，结论是：

1. 录制阶段应提供 `dataset.single_task`
2. 录制循环会把该文本写入每一帧的 `task`
3. 保存数据集时不会把原始字符串直接写进 `data/*.parquet`
4. 保存时会把 `task` 映射为 `task_index`
5. 任务字符串表写入 `meta/tasks.parquet`
6. 读取样本时，`task_index` 会被重新还原成 `item["task"]`
7. tokenizer 在训练前再把 `task` 转成 `observation.language.tokens`

对应源码如下：

- 录制阶段写入 `task`：
  - `src/lerobot/scripts/lerobot_record.py:389`
- 保存任务表与 `task_index`：
  - `src/lerobot/datasets/lerobot_dataset.py:307-320`
  - `src/lerobot/datasets/lerobot_dataset.py:1196-1209`
- 读取时恢复 `task`：
  - `src/lerobot/datasets/lerobot_dataset.py:1075-1077`
- tokenizer 生成语言 token：
  - `src/lerobot/processor/tokenizer_processor.py:155-177`

因此，本轮的正确实现方向不是新增一个磁盘字段叫 `observation.language`，而是把脚本层的语义文本输入，准确接到 `dataset.single_task` 上。

#### 5.1.3 为什么要用 Python 生成配置 JSON

这是本轮一个非常重要的稳定性改动。

原始写法如果用 shell 直接拼：

- 中文可能可以工作
- 空格可能勉强工作
- 双引号很容易把 JSON 写坏

用户后续明确要录制语义文本，而且语义文本很可能是自然语言句子，因此必须处理：

- 中文
- 空格
- 双引号
- 更长的任务描述

所以最终改法是：

1. 在 shell 中导出所有环境变量
2. 在 `lerobot_flex` 环境内运行 Python
3. 使用 `json.dump(..., ensure_ascii=False)` 写配置

这样 `single_task` 的内容不会再因为 shell 转义问题损坏。

## 7. 当前能力边界与已知限制

### 7.1 可视化只能逐条 `episode` 看

当前 `vis.sh` 的能力边界不是脚本人为限制，而是底层 `lerobot-dataset-viz` 决定的。

所以当前使用方式只能是：

```bash
bash my_devs/add_robot/agilex/vis.sh <dataset_name> 0
bash my_devs/add_robot/agilex/vis.sh <dataset_name> 1
```

不能直接打开一个“整数据集浏览器”。

### 7.2 当前语义是 episode 级，而不是 frame 级

`single_task` 是本次录制命令的任务文本，不是逐帧变化的语言标注。

这意味着：

- 一次 `record.sh` 录多个 episode 时，这些 episode 默认共享同一条语义文本
- 如果需要每个 episode 不同的语言描述，应该分多次调用脚本
- 如果需要同一 episode 内分阶段语义标注，当前方案不够，需要单独设计

### 7.3 本轮初始实现中，录制链路会同时保留主臂动作和从臂观测

根据现有实现：

- `observation.state` 来自 `puppet`
- `action` 来自 `master`

如果用户最终要求是“被录制下来的数据里不要主臂，只保留 follower / puppet”，那么当前实现很可能不满足最终目标。

这一点不是文档措辞问题，而是数据语义问题，必须做独立审核。

## 8. 本轮独立审核结果

本轮按照用户要求，除了主代理自身分析外，还做了两类独立审核：

1. 专审 `master/follower` 录制语义
2. 审核报告、脚本、现存证据是否一致

### 8.1 审核结果 A：`master/follower` 录制语义

独立审核结论非常明确：

1. 当前录制链路是混合源设计
   - `observation.state` 来自 follower / `puppet`
   - `action` 来自 master / `master`
2. 这与“只保留 follower 数据”的目标是冲突的
3. 当前 `action` 也不是“实际发给 follower 的命令确认值”，而是 teleop / master 侧读数

独立审核所依据的关键位置包括：

- `my_devs/add_robot/agilex/record.sh:66-72`
- `my_devs/add_robot/agilex/record.sh:188-213`
- `src/lerobot/robots/agilex/agilex.py:102-112`
- `src/lerobot/teleoperators/agilex_teleoperator/agilex_teleoperator.py:35-45`
- `src/lerobot/teleoperators/agilex_teleoperator/agilex_teleoperator.py:77-79`
- `src/lerobot/scripts/lerobot_record.py:377-389`

独立审核还直接读取了现存数据集 `test_prefix/replay_test` 的 parquet，认为其中 `action` 与 `observation.state` 并不相同，更符合“leader 与 follower 存在跟随偏差”的情况，而不是“同一份 follower 状态被重复记录”。

#### 8.1.1 对当前需求的含义

如果当前用户的真实目标是：

- “数据里不要主臂，只保留 follower”

那么当前实现不满足这个目标。

如果当前用户的真实目标其实是：

- “标准遥操作示教数据：follower 作为 observation，master 作为 action”

那么当前实现是自洽的。

所以这里最重要的不是代码是否能跑，而是要先把动作语义定义清楚。

#### 8.1.2 审核建议的最小改法

独立审核给出的最小改法建议是：

1. 在 `record.sh` 增加类似 `ACTION_SOURCE=master|follower` 的显式开关
2. 当需要“只保留 follower 数据”时，把 teleop 的 `action_*_topic` 切到 `/puppet/joint_*`

这个改法的优点是：

- 只改脚本
- 不需要改 Python 逻辑
- 可以快速形成“`observation.state` 和 `action` 都来自 follower”的数据集

但代价也很清楚：

- `action` 会和 `observation.state` 的关节位置部分高度重复

如果后续希望语义更清晰，还可以做更明确的 Python 级改法：录制时仍然读 master，但写盘时不要把 teleop/master 读数写入 `action_frame`，而是显式从 follower 观测中构造 `action_frame`。这个方案更清楚，但不是最小改法。
