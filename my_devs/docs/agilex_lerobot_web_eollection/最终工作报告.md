# AgileX LeRobot Web 采集页最终验收报告

## 1. 文档定位

本文是本轮 `AgileX LeRobot Web 采集页` 的最终验收报告。

它不是新的技术方案，也不是实施计划，而是在现有方案和实现基础上，对以下问题给出最终归档：

- 这次到底做了什么。
- 采用了什么协作方式推进。

## 2. 最终结论

当前版本已经完成本轮目标，可以作为 **可用的网页采集入口** 使用。

最终验收结论如下：

- 已实现 `record.sh` 的网页化封装。
- 已支持网页填写并启动 AgileX 数据采集。
- 已支持实时状态、日志、结果与历史任务展示。
- 已支持 `single_task_text` 任务语义字段。
- 已支持 `repo_prefix + dataset_name` 组合成最终数据集 repo_id。
- 已修复启动时终端无实时输出的问题。
- 已修复输入过程中表单抖动、提示区漂移的问题。
- 已修复“录制实际成功但页面误判 failed”的 exit code race。
- 最新最小测试集在 `lerobot_flex` 环境下通过。

验收结论不是“理论上可用”，而是“代码、页面、测试和关键缺陷修复都已经收口”。

需要明确的一点是：

- 如果你当前浏览器里还看不到最新字段，比如 `repo_prefix`，那通常不是“功能没做”，而是服务还没重启到新版本，或者浏览器还在使用旧静态资源缓存。

## 3. 本轮目标与边界

本轮工作的核心目标是：

- 不修改 `my_devs/add_robot/agilex/record.sh` 的前提下，给 AgileX 数据采集提供一个网页入口。
- 页面风格保持简单、克制、偏 Apple 风格。
- 页面能够驱动脚本参数，而不是重新实现录制逻辑。
- 页面能承担现场使用所需的最小闭环：
  - 参数输入
  - 预检查
  - 命令预览
  - 创建采集任务
  - 查看运行状态
  - 查看日志
  - 查看结果
  - 恢复历史任务

本轮明确不做的事情：

- 不改 `record.sh` 的核心职责。
- 不把 replay、vis、训练等能力混进首版主页面。
- 不做复杂多任务调度平台。
- 不做多实例并发录制支持。

## 4. 协作方式与推进方法

本轮按“架构师主代理 + 多个子代理并行执行”的方式推进。

主代理的职责只有四类：

- 拆分任务。
- 给实现代理下发责任边界。
- 给验收代理下发验证任务。
- 汇总结果并形成最终文档。

主代理不承担具体业务代码实现，不直接参与实现细节讨论。

整个过程按以下节奏推进：

1. 先做方案与实施计划收敛。
2. 再按模块拆成后端、前端、测试三条实现线。
3. 实现完成后，再独立派出验收代理做代码审查、测试复核、结构审计。
4. 对验收中发现的问题，再单独开修复代理进行第二轮或第三轮定点修复。
5. 所有问题收口后，再形成最终验收报告。

这种方式的目标是：

- 让实现与测试尽可能并行。
- 让审查和实现相互独立。
- 减少“自己实现、自己宣布没问题”的偏差。

## 7. 页面最终支持的参数

页面当前支持的录制参数分成两层。

### 7.1 脚本位置参数

当前网页已支持以下 7 个脚本参数：

- `dataset_name`
- `episode_time_s`
- `num_episodes`
- `fps`
- `reset_time_s`
- `resume`
- `single_task_text`

### 7.2 网页补充参数

当前网页额外补充了 1 个命名空间参数：

- `repo_prefix`

其作用是：

- 不改变脚本第 1 个参数的语义。
- 通过环境变量覆盖 `DATASET_REPO_ID`。
- 使最终数据集可写入：
  - `dummy/demo_Data`
  - `agilex_lab/demo_001`
  - 等等

最终真实语义是：

- `dataset_name` = 叶子名字
- `repo_prefix` = repo 前缀
- `repo_id = <repo_prefix>/<dataset_name>`

### 8.2 终端输出修复

用户现场反馈启动命令看起来像“卡住了”，终端没有输出。

排查后确认根因是：

- `conda run` 默认捕获 stdout/stderr，不实时输出。

最终处理方式：

- 把启动方式改成 `conda run --no-capture-output -n lerobot_flex ...`
- 补了启动脚本 [run.sh](/home/agilex/cqy/flexible_lerobot/my_devs/agilex_web_collection/run.sh)
- README 里同步更正启动命令

现在推荐启动方式是：

```bash
bash my_devs/agilex_web_collection/run.sh
```

### 8.3 表单抖动与漂移修复

用户现场反馈输入框在输入时会变形、漂移。

排查后确认根因不是输入框本体坏掉，而是：

- `dataset_name` 下方提示文本会随着输入不断变成长路径
- `resumeHint`、`resetHint` 长度也会动态波动
- 上方预检查区会实时改高度
- 这些区域都没有预留稳定高度，导致整块表单反复重排

最终处理方式：

- 把 `dataset_name` 下方长路径提示改成静态短文案
- 动态目录信息移到预览区
- 给 `.field-hint`、预检查区、状态区预留稳定高度
- 把 `resume` 开关改成顶部对齐

### 8.4 新增任务语义参数

后续用户又新增了这个脚本调用形式：

```bash

```

这意味着网页不能只支持 6 参数，而必须支持第 7 个语义参数。

最终处理方式：

- 后端模型新增 `single_task_text`
- 前端页面新增“任务语义”输入
- `command_text` 预览显示完整第 7 参数
- 测试、fake script、样例请求体全部升级

### 8.5 修复成功录制却误判失败的问题

在真实使用中发现一个关键问题：

- 数据已经录成功
- 数据目录和 parquet/meta 文件都存在
- 但页面显示 `failed`
- 状态文案是：
  - `Process is no longer running and no exit code was captured.`

排查后确认根因是：

- `service.py` 在收尾阶段会先看进程是否还活着
- 如果发现进程已经不在，就尝试用 `state.returncode` 决定成功/失败
- 但真实 `returncode` 是由 `supervisor.py` 的输出线程稍后才通过 `on_exit` 回写
- 所以存在一个窗口：录制已成功结束，但 `returncode` 还没写进 state
- 在这个窗口里就会被误判成 `failed`

最终修复分两轮完成：

第一轮：

- 在 `service.py` 中引入 `poll_returncode()` 抢救退出码

第二轮：

- 在 `supervisor.py` 中加入 `_finished_returncodes` 缓存
- 即使 `_running` 句柄已被移除，只要退出码已知，`poll_returncode(job_id)` 仍能返回真实码
- 从时序上封死最后那一小段窗口

并补了专项回归测试：

- [test_exit_code_race.py](/home/agilex/cqy/flexible_lerobot/tests/my_devs/agilex_web_collection/test_exit_code_race.py)

### 8.6 新增 repo_prefix 参数

用户后续又提出：

- 现在固定写到 `dummy/<dataset_name>`
- 希望支持类似 `dummy/demo_Data`
- 甚至 `agilex_lab/demo_001`

经过专门分析后，最终没有选择“放开 `dataset_name` 允许带 `/`”，因为那会把现有脚本语义和路径逻辑绕乱。

最终采用的设计是：

- 页面新增 `repo_prefix`
- `dataset_name` 保持叶子名
- 最终 `repo_id = <repo_prefix>/<dataset_name>`
- 通过环境变量注入：
  - `DATASET_REPO_ID=<repo_prefix>/<dataset_name>`

这样既保留脚本位置参数语义，又能支持命名空间扩展。

## 14. 对历史问题的说明

对于先前已经录完但页面曾误判失败的历史任务，例如 `demotest`：

- 数据本身通常是有效的
- 页面旧状态文件可能仍保留错误的 `failed`
- 这不影响已存在的 parquet 和 meta 文件
- 本轮修复保证的是“后续新任务不再出现同类误判”

也就是说：

- 历史坏状态不一定自动回写
- 但后续同类问题已被修复

## 15. 当前剩余边界与已知说明

当前版本已经可用，但仍有几条边界需要明确：

- 页面最终成功/失败仍以当前 live stack 状态机为准，虽然关键 race 已修掉，但这不是分布式任务平台。
- `repo_prefix` 当前按单段 namespace 处理，不支持再带 `/`。
- `dataset_name` 仍然是叶子名，不支持直接写成 `dummy/demo_Data`。
- 浏览器如果继续命中旧缓存，会导致看不到新字段或旧逻辑仍在生效。
- 历史已写坏状态不会自动批量回填。

这些边界不阻碍当前版本使用，但需要维护同学知道它们。
