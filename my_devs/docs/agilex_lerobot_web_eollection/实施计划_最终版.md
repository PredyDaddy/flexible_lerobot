# AgileX 数据采集网页化实施计划（最终版）

更新时间：2026-03-11

> 说明：本文是面向实施同学的落地计划，不是代码实现说明，不替代技术方案，也不引入技术方案之外的新设计。实施过程中应以 `技术方案_最终版.md` 作为唯一方案依据，且明确保持 `my_devs/add_robot/agilex/record.sh` 不变。

## 1. 总体实施策略

### 1.1 实施目标

围绕既有命令：

```bash
bash my_devs/add_robot/agilex/record.sh <dataset_name> <episode_time_s> <num_episodes> <fps> <reset_time_s> <resume>
```

落地一个 AgileX 专用轻量 Web 工具页，使实施结果满足以下目标：

1. 用户可通过网页完成 6 个核心参数录入与一次完整录制提交。
2. 页面能清楚展示命令预览、输出目录、目录冲突状态和运行状态。
3. 后端能稳定承担参数校验、作业启动、日志持久化、状态恢复和停止控制。
4. 系统保持单活跃任务互斥，不支持并发录制。
5. 整体实现必须是“网页包装层”，不是“重写采集脚本”。

### 1.2 实施原则

1. `record.sh` 视为黑盒执行器，所有实施工作都围绕包装层展开。
2. 后端是唯一事实来源，前端只做展示、交互和轮询，不持有权威任务状态。
3. 第一阶段只做单页工具，不扩展回放、训练、多设备调度等非目标。
4. 运行前提固定为 `lerobot_flex` 环境，不能在其他环境下定义为正式支持。
5. 所有接口、参数、环境变量都必须走白名单，不允许自由透传。
6. 优先落地“能安全运行、能恢复、能排障”的基础能力，再补体验增强。

### 1.3 阶段性推进策略

实施顺序按“三段式推进”执行：

1. 先打通主链路：预检查、参数校验、命令预览、启动作业、展示日志与结果。
2. 再补运行可靠性：停止链路、刷新恢复、服务重启恢复、错误分类。
3. 最后补体验增强：最近任务、快捷入口、更细环境健康检查。

### 1.4 交付边界

本计划要求交付的是以下能力，而不是代码层面的超范围扩写：

1. 独立 AgileX Web 页面与配套 REST API。
2. 单任务作业管理与本地持久化台账。
3. 参数白名单校验与安全命令调用。
4. 前端单页表单、确认弹窗、状态区、日志区、结果区。
5. 对应测试、验收项、上线前准备清单。

明确不纳入本次实施：

1. 改写 `record.sh`。
2. React/Vite、Gradio、数据库、消息队列、独立 Worker。
3. 多用户、多租户、多机器人并发调度。
4. 把 topic、camera key、video codec 等内部细节暴露到首版主表单。

### 2.4 必须修改或接入的系统点

这里的“修改”指接入点，不是改动采集脚本：

1. Web 服务启动入口需要新增 AgileX 路由与静态资源映射。
2. 服务启动流程需要增加 `runtime/jobs/` 恢复扫描。
3. 页面加载逻辑需要增加活跃任务恢复与日志轮询恢复。
4. 若已有统一导航入口，只增加 AgileX 页面入口，不复用或侵入现有其他采集逻辑。

### 2.5 明确禁止改动的部分

1. 不修改 `my_devs/add_robot/agilex/record.sh`。
2. 不把用户输入透传为任意脚本路径、任意 shell 片段、任意环境变量。
3. 不在首版中引入数据库、消息队列、WebSocket、React/Vite。

## 3. 前后端分阶段开发顺序

### 3.1 Phase 0：方案冻结

目标：冻结边界、状态机、接口轮廓、页面结构与非目标。

实施动作：

1. 以最终技术方案为准，确认 `record.sh` 黑盒边界。
2. 确认单活跃任务互斥为强约束。
3. 确认权威状态落在后端 `runtime/jobs/<job_id>/` 台账。
4. 确认前端只采用静态 HTML/CSS/JS。

交付物：

1. 本实施计划文档。
2. 任务拆分清单与开发顺序基线。

### 3.2 Phase 1：最小可用版

目标：跑通“参数录入 -> 校验 -> 预览 -> 启动作业 -> 查看日志 -> 结束反馈”的完整闭环。

后端开发顺序：

1. 建立 FastAPI 基础骨架与静态资源挂载。
2. 实现 `param_validator`、`command_builder`、`precheck_service`。
3. 实现 `GET /api/precheck` 与 `POST /api/validate`。
4. 实现 `job_store` 的目录结构与元数据落盘。
5. 实现 `job_manager`、`process_runner`，支持作业创建与单任务互斥。
6. 实现 `POST /api/jobs`、`GET /api/jobs/active`、`GET /api/jobs/{job_id}`。
7. 实现日志落盘与 `GET /api/jobs/{job_id}/logs?offset=...`。
8. 实现基础结果归档：`succeeded`、`failed`、`stopped` 的元数据记录。

前端开发顺序：

1. 落地单页骨架与顶部概要区、预检查区。
2. 落地 6 参数表单与 3 个快速模板。
3. 落地命令预览区与目录冲突提示。
4. 落地启动前确认弹窗。
5. 落地运行状态区、日志区、结果区。
6. 打通页面与 `precheck`、`validate`、`jobs`、`logs` 接口的轮询与切换逻辑。

Phase 1 完成标准：

1. 内部试用场景可跑通。
2. 不读 README 的用户也能理解参数含义并完成一次录制。
3. 页面故障时仍可回退到 CLI 手工执行。

### 3.3 Phase 2：运行可靠性加固

目标：把“能跑”提升为“现场可用、可恢复、可停止、可诊断”。

后端开发顺序：

1. 完成进程组停止链路。
2. 增加 `stop_requested`、`lost` 等状态的完整处理。
3. 实现 `POST /api/jobs/{job_id}/stop`。
4. 实现服务启动恢复扫描。
5. 实现错误分类与错误摘要。
6. 实现 `GET /api/jobs` 最近任务列表。

前端开发顺序：

1. 增加运行中停止按钮与状态切换。
2. 增加浏览器刷新恢复逻辑。
3. 增加错误摘要展示。
4. 增加最近任务列表与查看入口。

Phase 2 完成标准：

1. 刷新页面不丢当前任务状态与日志。
2. 服务重启后运行中任务可恢复或被明确标记为 `lost`。
3. 停止操作能验证进程树被终止，不残留孤儿进程。

### 3.4 Phase 3：体验补齐与后续扩展

目标：在不破坏简洁性的前提下补齐高价值功能。

建议顺序：

1. 结果页增加 `replay.sh` / `vis.sh` 的快捷入口提示。
2. 完善最近任务回看体验。
3. 增加更细粒度的设备或环境健康检查。
4. 仅在 Phase 1 与 Phase 2 稳定后，评估是否补充排队能力。

明确暂缓：

1. 多设备并发调度。
2. 大型平台化重构。
3. 重型前端工程化。

## 4. 接口、状态管理、页面模块的落地顺序

### 4.1 接口落地顺序

接口应按“先校验、后创建、再观察、最后控制”的顺序实现。

1. `GET /api/precheck`
   作用：先回答“环境是否可用、当前是否可执行”。
2. `POST /api/validate`
   作用：在真正启动前完成参数合法性、输出目录冲突和命令预览生成。
3. `POST /api/jobs`
   作用：在校验通过后创建并启动作业。
4. `GET /api/jobs/active`
   作用：支撑首屏恢复和运行态展示。
5. `GET /api/jobs/{job_id}`
   作用：获取详情、状态、退出信息、参数摘要。
6. `GET /api/jobs/{job_id}/logs?offset=...`
   作用：支持前端轮询获取增量日志。
7. `POST /api/jobs/{job_id}/stop`
   作用：在停止链路验证完成后开放。
8. `GET /api/jobs`
   作用：作为历史与回看能力在第二阶段补充。

### 4.2 状态管理落地顺序

#### 后端状态管理

1. 先实现状态机枚举：`pending`、`validating`、`starting`、`running`、`stop_requested`、`stopped`、`succeeded`、`failed`、`lost`。
2. 再实现 `meta.json` 权威状态写入。
3. 再实现 `events.jsonl` 结构化事件记录。
4. 再实现启动、退出、失败、停止的状态推进。
5. 最后实现服务重启恢复与 `lost` 状态识别。

#### 前端状态管理

1. 页面初始化时先调用 `GET /api/precheck`。
2. 如有活跃任务，再调用 `GET /api/jobs/active` 和任务详情接口。
3. 日志偏移量只作为前端体验态保存，不作为权威事实。
4. 任何按钮状态和运行状态都以后端返回为准。

### 4.3 页面模块落地顺序

页面模块应按“先认知，再操作，再反馈”的顺序实现。

1. 顶部概要区
   先让用户知道这是 AgileX 采集页，当前脚本路径、环境状态和全局状态是什么。
2. 采集前检查区
   让用户在填表前知道当前是否可运行，以及输出根目录和任务占用情况。
3. 快速模板区
   降低首次试录门槛，确保推荐路径先可用。
4. 核心参数表单区
   以 6 个核心参数为主，不放入额外内部参数。
5. 命令预览区
   与表单联动，始终可见或可快速访问。
6. 启动前确认弹窗
   作为安全提交阀门，在真正启动作业前完成二次确认。
7. 运行状态区
   任务启动后第一时间反馈状态、开始时间、运行时长和输出目录。
8. 日志区
   与运行状态同时上线，便于现场排障。
9. 结果区
   在成功、失败、停止后提供下一步动作建议。
10. 最近任务区
   作为第二阶段补充，不抢首屏主任务。

## 5. 测试与验收顺序

### 5.1 测试实施原则

测试按“先输入安全、再主链路、再异常链路、最后恢复能力”的顺序开展，避免一开始就投入复杂现场联调。

### 5.2 建议测试顺序

1. 参数校验测试
   覆盖 `dataset_name` 正则、数值范围、`resume` 语义、`num_episodes=1` 时 `reset_time_s=0` 的联动。
2. 预检查接口测试
   验证脚本路径、环境状态、输出根目录、活跃任务状态是否返回完整。
3. 命令预览测试
   验证前端展示命令与后端真实命令数组表达一致。
4. 目录冲突测试
   覆盖“目录存在 + `resume=false` 阻止提交”和“目录存在 + `resume=true` 高亮提醒”。
5. 作业创建与单任务互斥测试
   验证同一时刻只能有一个运行中的录制任务。
6. 日志与状态落盘测试
   验证 `meta.json`、`stdout.log`、`events.jsonl`、`record_config.json` 均正确生成。
7. 运行态页面轮询测试
   验证运行状态、输出目录、开始时间、日志增量刷新。
8. 停止链路测试
   验证 `stop_requested -> stopped` 流程以及进程组停止完整性。
9. 浏览器刷新恢复测试
   验证页面刷新后任务详情和日志恢复正常。
10. 服务重启恢复测试
   验证运行中任务可恢复或明确标记为 `lost`。
11. 环境异常测试
   覆盖 `lerobot_flex` 不存在、ROS 异常、输出目录不可写、磁盘空间不足等错误摘要。
12. 访问控制与回退测试
   验证可信网络限制或基础认证已具备，CLI 回退路径未被破坏。

### 5.3 验收执行顺序

验收建议分三轮进行：

1. 开发自验
   对照功能验收和最低实测场景逐项自测，先保证接口、状态机和日志台账正确。
2. 内部试用验收
   由实施同学或现场操作者完成至少一轮真实录制，重点验证交互理解、误操作防护和错误提示。
3. 上线前回归验收
   在目标运行环境上复测停止、恢复、单任务互斥、CLI 回退与访问控制。

### 5.4 必须覆盖的最低实测场景

以下场景必须按顺序完成，且都通过后才能认为首版可交付：

1. 新建数据集，单段录制，脚本默认值。
2. 新建数据集，单段录制，30 FPS。
3. 多段录制 + reset 时间。
4. 既有目录 + `resume=true`。
5. 既有目录 + `resume=false`。
6. 非法数据集名。
7. 非法数值输入。
8. 运行中刷新页面。
9. 运行中点击停止。
10. Conda 或 ROS 环境异常。

## 6. 依赖、环境与上线前准备

### 6.1 依赖与运行前提

实施同学在落地和联调时，应明确以下前提：

1. 所有运行必须使用 `lerobot_flex` 环境。
2. Web 后端采用 FastAPI + `uvicorn`。
3. 前端采用静态 HTML/CSS/原生 JavaScript。
4. 作业执行依赖 Python `subprocess.Popen(...)`，且必须走参数数组。
5. 状态持久化依赖本地文件系统，目录形态为 `runtime/jobs/<job_id>/`。

### 6.2 关键环境准备清单

上线前必须逐项确认：

1. `my_devs/add_robot/agilex/record.sh` 路径稳定可访问。
2. `lerobot_flex` 环境可用，且能满足录制脚本运行。
3. 输出根目录可写，磁盘空间充足。
4. 运行环境能继承或正确配置 `ROS_MASTER_URI`、`ROS_IP`、`ROS_HOSTNAME`、`DISPLAY`。
5. 服务进程对 `runtime/jobs/` 具有创建和写入权限。
6. 目标机器网络暴露范围受控，至少具备可信网络限制、网络隔离或基础认证中的一种。

### 6.3 运行时文件准备

上线前需要确保以下文件和目录策略明确：

1. 每个任务独立生成 `meta.json`。
2. 每个任务独立生成 `stdout.log`。
3. 每个任务独立生成 `events.jsonl`。
4. 每个任务独立生成 `record_config.json`。
5. 输出数据目录仍保持 `my_devs/add_robot/agilex/outputs/dummy/<dataset_name>` 习惯，不改变 CLI 认知。

### 6.4 上线前联调准备

建议按以下顺序准备上线前联调：

1. 先在目标机器上完成 `GET /api/precheck` 联调。
2. 再验证 `POST /api/validate` 的目录冲突和命令预览。
3. 再做一次短时单段录制作为冒烟验证。
4. 再验证刷新恢复、停止链路、服务重启恢复。
5. 最后确认 CLI 手工执行 `record.sh` 的兜底路径仍可用。

## 7. 哪些地方必须先做，哪些可以后补

### 7.1 必须先做

以下内容是实施前期的强依赖，必须优先完成，否则不应进入现场试用：

1. 固定命令模板与参数数组调用。
2. 参数白名单校验与目录冲突校验。
3. `GET /api/precheck` 和 `POST /api/validate`。
4. 单活跃任务互斥。
5. `job_store` 落盘与 `meta.json`/`stdout.log`/`events.jsonl`/`record_config.json` 生成。
6. `POST /api/jobs`、`GET /api/jobs/active`、`GET /api/jobs/{job_id}`、日志增量接口。
7. 前端单页表单、命令预览、确认弹窗、运行状态区、日志区、结果区。
8. 成功、失败、停止三类结束态的区分展示。
9. 页面故障时不破坏 CLI 回退路径。

### 7.2 应尽量在第一阶段完成

以下内容虽然可与主链路并行推进，但最好在首版内部试用前完成：

1. 停止链路的基本可用版本。
2. 基础错误分类与错误摘要。
3. 浏览器刷新恢复。
4. 预检查中的 Conda、ROS、目录可写性提示。

### 7.3 可以后补

以下内容可以放到第二阶段或第三阶段补充，不阻塞首版主链路交付：

1. 最近任务列表。
2. 结果页中的 `replay.sh` / `vis.sh` 快捷入口。
3. 更细的设备健康检查。
4. 更丰富的结果摘要与历史回看体验。
5. 可选排队能力。

### 7.4 明确不应提前扩展

以下内容即使有人提出需求，也不应在本轮实施中抢占优先级：

1. 多机器人或多用户并发。
2. 大型多页面平台化改造。
3. WebSocket 替换轮询。
4. 数据库、消息队列、独立 Worker。
5. React/Vite 或其他重型前端工程链路。

## 8. 实施落地提醒

### 8.1 给实施同学的执行口径

实施过程中必须始终用以下口径约束自己：

1. 这是一个“围绕 `record.sh` 的 Web 包装层”项目，不是“采集逻辑重构”项目。
2. 任何实现选择若与运行安全、信息清晰、工程可实现性冲突，应优先保证安全与可靠。
3. 页面体验可以逐步打磨，但参数安全、状态真实性、停止可靠性和回退路径不能后置。

### 8.2 交付完成判定

只有同时满足以下条件，才可视为本计划交付完成：

1. 用户无需手写 shell 命令即可完成一次录制。
2. 页面命令预览与后端真实执行命令一致。
3. 非法输入、目录冲突、并发启动都会被明确拦截。
4. 运行中能看到状态、日志、输出目录和结束结果。
5. 页面刷新和服务重启后的状态不会静默丢失。
6. CLI 原工作流仍然可作为故障回退路径使用。
