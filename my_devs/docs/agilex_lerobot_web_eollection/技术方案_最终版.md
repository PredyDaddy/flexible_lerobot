## 2. 项目目标、边界与原则

### 2.1 项目目标

第一阶段要交付的是一个面向 AgileX 数据采集的网页入口，用最少的表单参数安全触发一次录制，并向用户持续反馈：

- 录制前是否可执行。
- 实际会执行什么命令。
- 任务当前处于什么状态。
- 日志、结果和输出目录在哪里。
- 失败后下一步应该怎么处理。

### 2.2 范围边界

第一阶段范围明确限定为 `record.sh` 的网页封装，不扩展为通用机器人控制台。

当前唯一必须支持的命令边界是：

```bash
bash my_devs/add_robot/agilex/record.sh <dataset_name> <episode_time_s> <num_episodes> <fps> <reset_time_s> <resume>
```

第一阶段不纳入以下能力：

- 不重写录制逻辑，不重新实现 `lerobot-record` 调度。
- 不把 ROS topic、camera key、视频编码、push_to_hub 等底层内部项暴露到主表单。
- 不支持多任务并发录制。
- 不把训练、回放、可视化混入同一主流程。
- 不承诺“停止采集”等于“当前 episode 一定完整保存”。

### 2.3 不改动 `record.sh` 的原则

这是本方案的硬约束。

- 不修改 `my_devs/add_robot/agilex/record.sh` 的现有命令接口、默认行为和脚本职责。
- 网页层只做更安全的参数输入、前置校验、任务管理、日志展示和状态回显。
- 一切实现都必须保持命令行手工执行 `record.sh` 的路径仍然成立，网页故障时可以回退到 CLI。

## 3. 已确认事项

本节列出已从脑暴文档中收敛后的确定事实，后续实现默认以此为准。

### 3.1 命令与参数事实

- `record.sh` 的网页首版只应暴露 6 个核心参数：`dataset_name`、`episode_time_s`、`num_episodes`、`fps`、`reset_time_s`、`resume`。
- `resume` 在网页中必须是显式布尔开关，不允许用户自由输入文本。
- `num_episodes = 1` 时，`reset_time_s` 在业务语义上应自动收敛为 `0`。
- 网页层应始终生成完整 6 参数命令，不依赖脚本的缺省位参数语义。

### 3.2 运行环境与输出事实

- 默认 Conda 环境是 `lerobot_flex`。
- 默认输出根目录是 `my_devs/add_robot/agilex/outputs`。
- 实际数据集目录是 `my_devs/add_robot/agilex/outputs/dummy/<dataset_name>`。
- `record.sh` 默认共享配置路径是 `my_devs/add_robot/agilex/outputs/record_config.json`。
- 因为默认配置路径是共享文件，第一阶段不能支持并发录制。

### 3.3 已知脚本行为

- 当目标数据集目录已存在且 `resume=false` 时，脚本会直接拒绝执行。
- 脚本不会主动拦截 `resume=true` 且目录不存在的情况，因此网页后端必须补齐这一层预检查。
- 脚本内部还有固定项，如 `robot=agilex`、`control.mode=passive_follow`、`video=true`、`vcodec=h264`、`push_to_hub=false`、`display_data=true`、`play_sounds=true`，第一阶段不开放为可编辑项。

### 3.4 默认值策略

围绕 `fps` 默认值存在两个来源：

- 脚本真实默认值是 `10`。
- README 中最常用示例值是 `30`。

最终收敛方案如下：

- 页面基础默认值与脚本保持一致：`episode_time_s=8`、`num_episodes=1`、`fps=10`、`reset_time_s=0`、`resume=false`。
- `dataset_name` 默认留空，但展示脚本默认名作为占位提示，不直接预填固定值，避免撞名。
- 页面额外提供一个显式的 `首次试录` 预设，一键切换为 `8s / 1 段 / 30 FPS / reset=0 / resume=false`。

这样同时满足两件事：

- 不偏离脚本语义。
- 不牺牲新手首次试录的易用性。

### 3.5 第一阶段产品定位

第一阶段页面定位收敛为：

**面向实现现场和新手操作者的 `record.sh` 安全执行面板。**

它不是脚本参数总控台，也不是训练平台首页。

## 4. 推荐技术路线

### 4.1 总体路线

推荐采用以下 MVP 架构：

1. 前端采用单页式采集工作台。
2. 后端采用一个很薄的 Web API 层承接表单请求。
3. 后端将一次录制视为一个受控 `job`，而不是直接执行一条临时命令。
4. 任务运行时通过文件式 JobStore 保存状态、日志和配置快照。
5. 进程管理使用“新进程组 + 分级停止信号”的方式保证停止链路可靠。

推荐分层如下：

- `FastAPI Router`
- `RecordJobService`
- `ProcessSupervisor`
- `JobStore(file-based)`

### 4.2 推荐理由

选择这条路线的原因是：

- 满足“不改 `record.sh`”的前提下，仍可补齐网页所需的参数校验、状态恢复和日志可追踪能力。
- 当前录制链路的主要问题不是“能不能起 subprocess”，而是并发互斥、共享配置文件、进程树停止和页面刷新恢复。
- 第一阶段没有必要引入 MQ、Celery、Redis 或多进程 worker 架构，文件式台账足够支撑单机单操作者场景。
- `record`、未来 `replay`、未来 `vis` 都可以复用统一 `job` 模型，后续演进成本低。

### 4.3 第一阶段明确不采用的路线

- 不采用“前端直接拼 shell 命令并执行”的方案。
- 不采用 `shell=True` 的子进程启动方式。
- 不采用并发录制或复杂队列调度。
- 不采用多页、多标签后台式信息架构。

## 5. 网页信息架构与关键交互

### 5.1 页面结构

推荐页面采用单页式结构，按以下顺序组织：

1. 顶部轻导航与页面身份
2. 固定安全提示
3. 采集前检查区
4. 快速模板区
5. 采集参数区
6. 命令与输出预览区
7. 运行状态区
8. 结果与下一步区
9. 辅助帮助区

页面的主路径应保持极短：

`确认就绪 -> 选择模板或填参数 -> 预览 -> 确认开始 -> 查看运行状态 -> 查看结果`

### 5.2 页面主区块定义

| 区块 | 目标 | 必须展示的内容 |
| --- | --- | --- |
| 顶部区 | 建立页面身份 | 页面名、环境标签、主操作入口 |
| 安全提示区 | 明确现场操作前提 | 机器人工作区安全提醒、设备和 ROS 准备提醒 |
| 采集前检查区 | 让用户先确认可执行性 | 脚本路径、Conda 环境、输出根目录、目标数据目录、目录占用状态、当前服务状态 |
| 快速模板区 | 降低新手上手成本 | `首次试录`、`脚本默认值`、可选 `继续追加` 模板 |
| 采集参数区 | 暴露最少必要参数 | `dataset_name`、`episode_time_s`、`num_episodes`、`fps`、`reset_time_s`、`resume` |
| 预览区 | 保证网页行为可解释 | 命令预览、输出目录、预计总时长、运行环境摘要 |
| 运行区 | 运行时成为视觉中心 | 状态、阶段、当前 episode、剩余时间、最近日志、停止入口 |
| 结果区 | 结束后形成闭环 | 成功/失败/停止摘要、输出路径、下一步建议 |

### 5.3 关键交互规则

#### 1. `dataset_name` 规则

- 必填。
- 只允许字母、数字、下划线、中划线。
- 不允许空格、`/`、`\\`、`..`、引号、换行。
- 实时联动计算输出目录：`my_devs/add_robot/agilex/outputs/dummy/<dataset_name>`。
- 输入变化时同步刷新目录存在性校验结果。

#### 2. `resume` 规则

- 只能是开关或分段控件，不能是自由文本。
- 当目录存在且 `resume=false` 时，必须禁止提交。
- 当目录存在且 `resume=true` 时，允许提交，但必须明确提示“将追加到已有数据集”。
- 当目录不存在且 `resume=true` 时，前端应提示冲突，后端必须拒绝。

#### 3. `num_episodes` 与 `reset_time_s` 联动

- 当 `num_episodes = 1` 时，自动将 `reset_time_s` 设为 `0` 并禁用输入。
- 当 `num_episodes > 1` 时，才允许编辑 `reset_time_s`。

#### 4. 命令预览

- 命令预览必须始终可见且只读。
- 命令必须完整展示 6 个参数。
- 页面展示内容必须与后端实际调用一一对应，不能出现“页面预览一套、后端执行另一套”。

#### 5. 安全确认弹窗

点击“开始采集”后，不直接执行，必须弹出确认弹窗。

弹窗必须包含：

- 参数摘要。
- 输出目录摘要。
- `resume` 语义提示。
- 三项必须勾选的安全确认：
  - 已确认机器人周围无人处于危险区域。
  - 已确认相机、机械臂和 ROS 话题已正常工作。
  - 已确认数据集名称与输出目录正确。

#### 6. 运行态交互

- 运行中锁定主表单，避免中途改参数造成歧义。
- “开始采集”按钮禁用。
- 若停止能力已验收通过，则显示“停止采集”按钮。
- 若停止能力尚未完成整棵进程树验证，则不要先暴露该按钮。

#### 7. 刷新恢复

- 浏览器刷新后，页面必须优先查询当前活跃任务并恢复状态。
- 关闭页面不等于停止录制，页面文案中必须明确说明这一点。

### 5.4 页面推荐默认值与模板

页面初始默认值：

- `dataset_name`：留空
- `episode_time_s=8`
- `num_episodes=1`
- `fps=10`
- `reset_time_s=0`
- `resume=false`

推荐模板：

| 模板 | 用途 | 参数 |
| --- | --- | --- |
| 首次试录 | 新手首跑 | `8 / 1 / 30 / 0 / false` |
| 脚本默认值 | 与脚本语义对齐 | `8 / 1 / 10 / 0 / false` |
| 继续追加 | 已有数据集上补录 | `8 / 1 / 30 / 0 / true` |

## 6. API 与后端编排建议

### 6.1 总体原则

后端的职责不是“替前端执行命令”，而是“创建、管理并追踪一个受控录制任务”。

必须遵守以下原则：

- 白名单脚本调用。
- 结构化参数请求，不接受命令文本。
- 参数数组启动子进程，不使用 `shell=True`。
- 服务端固定注入运行环境，不允许前端透传脚本路径和任意环境变量。
- 单机单活跃录制任务，不支持并发。

### 6.2 请求模型

建议前端提交结构化请求：

```json
{
  "dataset_name": "test_001",
  "episode_time_s": 8.0,
  "num_episodes": 1,
  "fps": 30,
  "reset_time_s": 0.0,
  "resume": false,
  "safety_ack": true
}
```

### 6.3 服务端参数校验

建议后端在启动前做严格校验：

| 字段 | 规则 |
| --- | --- |
| `dataset_name` | `1-64` 字符，正则建议 `^[A-Za-z0-9][A-Za-z0-9_-]{0,63}$` |
| `episode_time_s` | 浮点数，`0 < value <= 3600` |
| `num_episodes` | 整数，`1 <= value <= 1000` |
| `fps` | 整数，`1 <= value <= 120` |
| `reset_time_s` | 浮点数，`0 <= value <= 600` |
| `resume` | 布尔值，只由后端转为 `true/false` 字符串 |
| `safety_ack` | 必须为 `true` 才允许提交 |

### 6.4 启动前预检查

后端在真正启动前至少做以下检查：

1. 当前是否已有活跃录制任务。
2. 目标数据集目录是否存在。
3. `resume` 与目录状态是否一致。
4. 输出根目录是否可写。
5. 必要服务环境是否存在，至少包括 `lerobot_flex` 环境与目标脚本路径。

强制规则：

- 目录存在且 `resume=false`：返回冲突，不启动。
- 目录不存在且 `resume=true`：返回冲突，不启动。
- 活跃任务存在：返回冲突，不启动第二个任务。

### 6.5 实际启动方式

后端实际调用命令应固定为参数数组：

```text
[
  "bash",
  "<repo_root>/my_devs/add_robot/agilex/record.sh",
  dataset_name,
  "<episode_time_s>",
  "<num_episodes>",
  "<fps>",
  "<reset_time_s>",
  "true" | "false"
]
```

服务端注入的关键环境变量建议为：

- `CONDA_ENV=lerobot_flex`
- `HF_LEROBOT_HOME=<repo_root>/my_devs/add_robot/agilex/outputs`
- `CONFIG_PATH=<job_dir>/record_config.json`
- `PYTHONUNBUFFERED=1`

其中最关键的是：

**必须为每个 job 指定独立 `CONFIG_PATH`，否则共享 `record_config.json` 会造成串台和并发冲突。**

### 6.6 进程编排与停止策略

录制链路不是单进程模型，因此需要按进程组管理。

推荐实现：

- 使用 `Popen(..., start_new_session=True)` 启动新进程组。
- 记录 `pid` 与 `pgid`。
- 停止时按整个进程组依次发送：
  1. `SIGINT`
  2. 超时后 `SIGTERM`
  3. 再超时后 `SIGKILL`

停止语义必须明确为：

- “停止录制”表示请求尽量优雅收尾。
- 不表示当前正在录制的 episode 一定已经完整提交。
- 只有日志或后验产物校验能证明已保存的数据，才能作为成功采集结果展示。

### 6.7 状态机设计

推荐 job 级状态机：

- `created`
- `validating`
- `rejected`
- `starting`
- `running`
- `stop_requested`
- `stopped`
- `succeeded`
- `failed`

同时增加一个 UI 可读的 `phase`：

- `preflight`
- `booting`
- `recording`
- `resetting`
- `saving`
- `finalizing`

要求：

- `status` 表示系统语义。
- `phase` 表示运行阶段。
- 两者不得混用。

### 6.8 日志、状态与文件式 JobStore

建议每个任务创建独立运行目录：

```text
runtime/agilex_jobs/<job_id>/
  request.json
  state.json
  stdout.log
  events.jsonl
  record_config.json
```

建议落盘内容：

- `request.json`：原始请求与规范化参数
- `state.json`：当前状态快照
- `stdout.log`：真实子进程输出
- `events.jsonl`：解析后的结构化事件
- `record_config.json`：该任务实际使用的配置快照

选择文件式 JobStore 的原因：

- 单机场景足够。
- 调试和追溯成本低。
- 页面刷新恢复与历史查询容易实现。
- 后续需要时可平滑升级为 SQLite。

### 6.9 建议 API 清单

| 接口 | 作用 |
| --- | --- |
| `GET /api/v1/runtime` | 返回页面初始化所需只读信息 |
| `POST /api/v1/records/validate` | 做参数与目录预检查，不真正启动 |
| `POST /api/v1/records` | 创建录制任务 |
| `GET /api/v1/jobs/active` | 查询当前活跃任务，支持刷新恢复 |
| `GET /api/v1/jobs/{job_id}` | 查询任务详情 |
| `GET /api/v1/jobs/{job_id}/logs?cursor=...` | 获取增量日志 |
| `GET /api/v1/jobs` | 查看最近任务历史 |
| `POST /api/v1/jobs/{job_id}/stop` | 请求停止当前任务 |

### 6.10 幂等、互斥与访问控制

建议第一阶段同步纳入：

- `Idempotency-Key`，用于拦截重复点击与网络重试。
- 全局互斥锁，保证同一时刻只有一个活跃录制任务。
- 访问控制或网络隔离，避免局域网任意访问触发录制。

后续如果扩展到 `replay` 或 `vis`，继续复用 `job kind + 资源锁` 模型，但这不是第一阶段交付内容。

## 7. Apple 风格视觉方向（仅设计原则）

本节只定义视觉与交互原则，不输出任何代码实现。

### 7.1 视觉目标

本项目需要的不是“像 Apple 官网”，而是借用其方法论，做出一个：

- 单任务聚焦
- 低噪声
- 状态清晰
- 高可信感
- 适合工业现场操作的工具页面

### 7.2 设计原则

#### 1. 单任务聚焦

- 首屏先讲“当前是否可采”和“如何开始”，不先堆 12 个表单项。
- 同一时刻只能有一个绝对主操作。
- 运行中页面视觉重心必须切换到状态区，而不是继续让表单抢焦点。

#### 2. 大留白与稳定节奏

- 通过留白、短标题和大卡片建立秩序感。
- 页面模块按“确认 -> 配置 -> 运行 -> 结果”分段，不做后台式密集拼盘。

#### 3. 卡片化但不过度玻璃化

- 主要内容采用大圆角卡片承载。
- 轻玻璃或半透明材质只用于顶部状态条、轻导航或弹窗，不用于整页大面积铺开。
- 不采用高噪声毛玻璃和强发光效果。

#### 4. 色彩克制

- 以浅色中性底为主。
- 只保留一种主强调色用于关键操作和激活态。
- 成功、警告、错误状态色要可读但不过饱和。
- 不做监控大屏式满屏红黄绿。

#### 5. 文案短而准

- 标题和按钮文案都应简短直接。
- 错误文案必须比品牌文案更直白，至少包含“发生了什么、可能原因、下一步建议”。
- 不使用“提交”“执行脚本”这类过于技术化且抽象的按钮文案。

#### 6. 数字与状态优先级高于日志

- `当前 episode`、`总段数`、`剩余时间`、`FPS` 等数字要明显大于一般说明文字。
- 实时日志默认只展示最近关键几行，可展开完整日志。
- 不允许让黑底终端日志成为页面主视觉。

#### 7. 动效服务于理解

- 只允许轻量级淡入、位移、状态切换动画。
- 动效用于帮助用户理解状态变化，而不是展示炫技。
- 需兼容“减少动态效果”偏好，低性能设备应自动降级。

### 7.3 明确不建议的视觉方向

- 不做传统后台系统式左侧重导航。
- 不做通篇深色工业控制台。
- 不做满屏玻璃材质。
- 不做复杂仪表盘和高频闪烁动效。
- 不为了“高级感”牺牲警告与错误的可读性。

## 8. 风险与验收口径

### 8.1 核心风险

| 风险 | 具体问题 | 对应控制措施 |
| --- | --- | --- |
| 参数映射错误 | 页面参数与脚本位置参数错位 | 命令预览与后端调用统一生成；接口层强类型校验 |
| `resume` 语义误用 | 用户误以为会覆盖旧数据 | 显式开关、目录预检查、确认弹窗提醒 |
| 共享配置文件冲突 | 多任务或残留任务覆盖 `record_config.json` | 每任务独立 `CONFIG_PATH`，单活跃锁 |
| 停止不彻底 | 只停父进程，子进程残留 | 进程组管理，按 `SIGINT -> SIGTERM -> SIGKILL` 停止 |
| 页面状态失真 | 页面显示运行中，进程已退出 | 以 `state.json + 退出码 + 日志尾部` 为单一事实来源 |
| 浏览器刷新丢上下文 | 刷新后回到初始页 | `GET /jobs/active` 恢复活跃任务 |
| 服务重启后失忆 | 后台仍在跑但页面不知道 | 依赖 JobStore 恢复，至少做到“可诊断而不是静默丢失” |
| 环境依赖不透明 | `lerobot_flex`、ROS、显示环境未准备好 | 预检查区显式展示环境前提，失败时给出分类错误 |
| 存储与权限问题 | 输出目录不可写或磁盘不足 | 启动前检查、失败归因与日志留存 |
| 未授权访问 | 任意访问端口即可发起录制 | 网络隔离、基本访问控制、必要时增加鉴权 |

### 8.2 页面层验收口径

页面层验收必须满足：

- 字段级校验在提交前即可发现明显错误。
- 命令预览、输出目录与实际调用一致。
- 目录冲突与 `resume` 关系可被用户直观看懂。
- 开始前必须完成安全确认。
- 运行中、成功、失败、停止四种状态在视觉与文案上明确可区分。
- 刷新页面后仍可恢复当前活跃任务状态与最近日志。

### 8.3 API 层验收口径

API 层验收必须满足：

- 只接受白名单字段，不允许任意环境变量或脚本路径透传。
- 非法参数返回明确 `4xx`，不触发脚本。
- 已有活跃任务时拒绝第二个录制请求。
- `start`、`status`、`logs`、`stop` 语义稳定且可追踪。
- 日志、退出码、开始结束时间、输出目录都能被查询和持久化。

### 8.4 脚本调用层验收口径

脚本调用层验收必须满足：

- 实际调用命令严格保持 `record.sh` 的 6 参数顺序。
- `resume` 只传小写 `true/false`。
- 每个 job 使用独立 `CONFIG_PATH`。
- 停止后不会留下录制残留进程。
- 失败原因能区分是参数问题、环境问题、运行期问题还是用户主动停止。

### 8.5 最小测试矩阵

上线前至少覆盖以下最小测试集：

| 用例 | 目标 |
| --- | --- |
| 新建数据集成功录制 | 验证基础 happy path |
| 30 FPS 首次试录 | 验证常用模板路径 |
| 多 episode + reset | 验证联动、预计总时长和运行阶段显示 |
| 已有目录 + `resume=true` | 验证追加录制路径 |
| 已有目录 + `resume=false` | 验证冲突拦截 |
| 非法数据集名 | 验证安全与输入校验 |
| 非法数值参数 | 验证边界校验 |
| 运行中点击停止 | 验证整棵进程树停止 |
| 运行中刷新页面 | 验证状态恢复 |
| 服务重启后查询任务 | 验证最基本恢复或可诊断能力 |
| 环境不存在或不可执行 | 验证环境失败归因 |
| 输出目录不可写 | 验证存储异常归因 |

## 9. 待实现阶段验证事项

以下事项不再作为方案争论点，但必须在实现阶段完成验证，并据此决定是否暴露相关能力。

### 9.1 停止能力验证

- `SIGINT` 对 `record.sh -> conda run -> lerobot-record` 整条进程链是否足够稳定。
- 用户停止后，当前 episode 的保存边界是否能被稳定判断。
- 若该能力未验证通过，前端不得提前上线“停止采集”按钮。

### 9.2 服务重启后的恢复策略

- 服务重启后是否能自动恢复活跃任务状态。
- 若不能完整恢复，是否至少能明确标记“状态未知，需要人工确认”。
- 上线说明必须写清这一行为，不能让操作者误解。

### 9.3 运行环境依赖

- Web 服务实际运行时是否需要显式继承 `DISPLAY`、`DISPLAY_DATA`、`PLAY_SOUNDS`。
- 当前现场是否依赖音频提示与图像显示窗口。
- 所需的 ROS 相关环境变量清单是否在部署方式中可稳定继承。

### 9.4 数据集合法性检查

- 当 `resume=true` 且目录存在时，是否需要进一步验证该目录确实是合法 LeRobot 数据集。
- 第一阶段若不做深校验，文案必须明确仅校验“目录存在性”，不承诺目录结构正确。

### 9.5 访问控制与审计

- 服务是否只暴露在受控网络。
- 是否需要登录、反向代理、来源限制或简单审计日志。
- 是否需要 CSRF 防护，取决于最终部署方式。

### 9.6 历史任务保留策略

- `stdout.log`、`events.jsonl` 和历史任务列表保留多久。
- 日志过大时如何截断或归档。
- 是否只保留最近 N 个任务供页面查询。

### 9.7 `replay` 与 `vis` 的接入时机

- 第一阶段不纳入交付范围。
- 但实现 `job` 模型时应预留 `kind` 字段与资源锁抽象，避免第二阶段推翻结构。

## 10. 实施建议

建议按以下顺序推进实现：

1. 先做 `record` 单链路闭环，不引入 `replay/vis`。
2. 先把参数校验、命令预览、目录冲突拦截和 JobStore 跑通。
3. 再补进程组停止、刷新恢复和增量日志。
4. 最后补历史任务、结构化事件和更细错误分类。

实施过程中，如果出现“为了兼容网页而修改 `record.sh`”的冲动，应优先回到本方案的基本原则重新审视，默认答案是不改脚本，而是在后端编排层补齐能力。

## 11. 结论

本方案的最终收敛点如下：

- 产品边界聚焦为 `record.sh` 的安全网页封装。
- 技术路线采用“单页前端 + 薄 API + job 模型 + 文件式 JobStore + 进程组管理”。
- 第一阶段坚持“单机单活跃录制任务”，不做并发。
- 视觉上采用 Apple 风格的方法论，但服务于工业采集工具的可读性和可信度，而不是追求展示感。
- 所有实现必须在不改动 `record.sh` 的前提下完成，并且始终保留 CLI 回退路径。

这份文档可直接作为后续原型、前后端实现、联调和验收的执行基线。
