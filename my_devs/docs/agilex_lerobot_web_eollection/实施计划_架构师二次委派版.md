# AgileX LeRobot Web 采集页实施计划（架构师二次委派版）

## 1. 文档定位

本文只基于《`技术方案_最终版.md`》反推出实施计划，不引用脚本源码或仓库实现细节。

目标是回答四件事：

- 要做什么。
- 应该改哪些代码区域。
- 按什么顺序推进。
- 每一阶段怎么验收。

本文默认继续遵守技术方案中的硬约束：

- 不修改 `my_devs/add_robot/agilex/record.sh`。
- 第一阶段只封装 `record.sh` 的 6 个核心参数。
- 第一阶段只支持单活跃录制任务。
- 运行与测试都应在 `lerobot_flex` 环境中完成。

## 2. 实施总目标

第一阶段要交付一个可上线试用的最小闭环系统：

1. 用户可以在网页中填写最少必要参数。
2. 页面可以在启动前完成参数校验和目录冲突检查。
3. 后端可以安全启动 `record.sh`，并将其作为可追踪任务管理。
4. 页面可以展示运行状态、最近日志、输出目录和结果摘要。
5. 页面刷新后可以恢复当前活跃任务状态。
6. 至少具备可验证的单任务互斥、失败归因和基础停止能力。

## 3. 建议改动的代码区域

由于本文不读取现有实现代码，以下按“模块区域”而非“既有文件名”给出实施范围。实现负责人可以根据仓库现状决定最终落点，但职责边界不应改变。

### 3.1 Web 应用入口区域

需要新增或整理一个 AgileX 专用 Web 应用入口，承载：

- 应用初始化。
- 静态页面挂载。
- API 路由注册。
- 运行时环境信息暴露。

职责要求：

- 与已有其他网页工具解耦。
- 不把 `replay`、`vis`、训练入口混入首版页面。
- 保持后续可扩展为多 job kind 的结构。

### 3.2 API 路由与请求模型区域

需要新增面向录制任务的 API 层，至少包括：

- 运行时初始化信息接口。
- 参数预检查接口。
- 创建录制任务接口。
- 查询活跃任务接口。
- 查询任务详情接口。
- 查询增量日志接口。
- 请求停止接口。
- 最近任务列表接口。

职责要求：

- 只接受结构化 JSON 请求。
- 不接受命令文本。
- 不接受脚本路径和任意环境变量透传。
- 对 6 个字段做强类型校验与边界校验。

### 3.3 录制任务服务区域

需要新增一个 `RecordJobService` 一类的业务层，负责：

- 将页面参数规范化。
- 执行目录存在性与 `resume` 语义校验。
- 计算命令预览与输出路径。
- 创建 job 元数据。
- 调用底层进程监督器启动任务。
- 更新任务状态与阶段。

职责要求：

- 所有和 `record.sh` 参数顺序有关的逻辑都集中在这一层。
- 页面预览命令与实际执行命令共用同一套构造逻辑。
- 单活跃任务互斥在这一层和更底层都要有保护。

### 3.4 进程监督与停止控制区域

需要新增一个 `ProcessSupervisor` 一类的运行时模块，负责：

- 使用参数数组启动命令。
- 使用新进程组管理整棵进程链。
- 记录 `pid`、`pgid`、开始时间、结束时间、退出码。
- 支持 `SIGINT -> SIGTERM -> SIGKILL` 分级停止。
- 将原始标准输出持续写入日志文件。

职责要求：

- 禁止 `shell=True`。
- 启动时固定脚本路径与运行环境。
- 必须为每个 job 提供独立 `CONFIG_PATH`。

### 3.5 JobStore 与运行时台账区域

需要新增文件式 JobStore，负责在每个任务目录中管理：

- `request.json`
- `state.json`
- `stdout.log`
- `events.jsonl`
- `record_config.json`

职责要求：

- 活跃任务与历史任务都可查询。
- 页面刷新后能靠 JobStore 恢复状态。
- 服务异常退出后至少能做到“状态可诊断”。

### 3.6 前端单页采集工作台区域

需要新增一张单页式采集界面，至少包含：

- 顶部页面身份区。
- 安全提示区。
- 采集前检查区。
- 快速模板区。
- 参数表单区。
- 命令与输出预览区。
- 运行状态区。
- 结果与下一步区。

职责要求：

- 风格遵循轻量 Apple 方法论，不做重后台布局。
- 运行中锁定主表单。
- 刷新后优先恢复活跃任务。
- 实时显示命令预览与输出路径。

### 3.7 部署与运行配置区域

需要补齐服务运行所需的配置与说明，至少包括：

- `lerobot_flex` 环境要求。
- 服务启动方式。
- 运行目录与日志目录约定。
- 访问范围与网络隔离建议。
- 现场安全 SOP 和上线前准备项。

## 4. 分阶段实施顺序

建议按五个阶段推进，而不是一次性把所有能力堆上去。

### 阶段 0：边界确认与骨架搭建

目标：

- 确定 Web 应用放置位置。
- 建立基础应用入口、静态页面挂载与 API 框架。
- 建立基础目录结构与运行时目录约定。

本阶段应完成：

- AgileX 专用 Web 应用骨架。
- 基础首页占位页。
- 基础 API 健康检查或运行时信息接口。
- 运行目录约定，例如 `runtime/agilex_jobs/<job_id>/`。

验收口径：

- 服务可以启动。
- 页面可以访问。
- 可以返回只读运行时信息。
- 不依赖真实录制任务即可跑通最小骨架。

### 阶段 1：参数模型与预检查闭环

目标：

- 在不真正启动录制的前提下，跑通“填写参数 -> 预检查 -> 命令预览 -> 冲突反馈”的闭环。

本阶段应完成：

- 6 个核心字段的前后端模型。
- `dataset_name` 白名单校验。
- 数值边界校验。
- `num_episodes` 与 `reset_time_s` 联动。
- `resume` 与目录存在性预检查。
- 命令预览与输出目录预览。
- 安全确认开关。

验收口径：

- 非法数据集名可被拦截。
- 非法数值参数可被拦截。
- `resume=true` 且目录不存在时后端拒绝。
- 目录存在且 `resume=false` 时后端拒绝。
- 页面展示的命令预览固定为 6 参数完整形式。

### 阶段 2：创建任务与运行态闭环

目标：

- 真正打通“提交 -> 创建 job -> 启动 `record.sh` -> 持续记录状态与日志”的主链路。

本阶段应完成：

- JobStore 基础实现。
- `RecordJobService` 的创建与状态管理。
- 参数数组方式启动脚本。
- 单活跃任务互斥锁。
- 独立 `CONFIG_PATH` 注入。
- 标准输出持续落盘。
- 任务详情和日志查询接口。

验收口径：

- 可以成功创建新录制任务。
- 运行中再次提交会被拒绝。
- 每个 job 都有独立任务目录。
- 页面可以持续看到状态变化和最近日志。
- 任务结束后能看到退出结果与输出目录。

### 阶段 3：停止能力与刷新恢复

目标：

- 补齐实际可用的运行时控制能力，避免页面只是一次性启动器。

本阶段应完成：

- 活跃任务查询接口。
- 页面刷新恢复逻辑。
- 进程组停止机制。
- 停止后的状态归档。
- “关闭页面不等于停止任务”的明示文案。

验收口径：

- 刷新页面后可恢复当前活跃任务状态。
- 运行中请求停止可以作用到整棵进程链。
- 停止后状态与日志能被查询。
- 若停止行为尚不能证明 episode 完整保存，结果区必须明确提示“不保证当前段已完整保存”。

### 阶段 4：历史任务、错误归因与体验收尾

目标：

- 在主闭环稳定后补齐第一版可交付所需的可用性与诊断能力。

本阶段应完成：

- 最近任务历史列表。
- 更清晰的成功/失败/停止摘要。
- 环境类错误、参数类错误、运行期错误的基本分类。
- 长日志查看与最近日志摘要。
- 模板区与帮助文案优化。
- 移动端可读性优化。

验收口径：

- 最近任务可查询。
- 失败原因至少能区分参数、环境、运行期、用户停止四大类。
- 页面在桌面端和移动端都可读。
- 不需要打开终端也能判断下一步该做什么。

## 5. 具体实施清单

### 5.1 后端优先项

必须先做：

- 请求模型与字段校验。
- 目录冲突预检查。
- 命令构造器。
- 单活跃任务锁。
- JobStore。
- 子进程启动与日志落盘。
- 任务状态接口。

可以后补：

- `events.jsonl` 的结构化解析。
- 更细粒度 `phase` 推断。
- 历史任务分页。
- 更完整的访问控制。

### 5.2 前端优先项

必须先做：

- 单页结构。
- 参数表单。
- 命令预览。
- 输出目录预览。
- 运行状态区。
- 最近日志摘要。
- 成功/失败摘要。

可以后补：

- 更丰富的动画。
- 更完整的历史记录页。
- 更细的移动端交互优化。

### 5.3 联调优先项

必须先做：

- 页面预览命令与后端真实执行命令一致性核对。
- `resume` 语义联调。
- 单任务互斥联调。
- 刷新恢复联调。
- 停止链路联调。

可以后补：

- 更细的日志结构化展示。
- 轻量性能优化。

## 6. 关键实现决策点

实施过程中有几项决策不能模糊处理。

### 6.1 是否先做停止按钮

决策规则：

- 如果整棵进程链停止能力在联调中未验证通过，前端不要暴露“停止采集”。
- 可以先交付“只支持开始与观察”的版本，再补停止能力。

### 6.2 是否允许并发或排队

决策规则：

- 第一阶段一律不开放并发录制。
- 是否做排队也应后置，优先交付单活跃闭环。

### 6.3 是否提前接入 `replay` / `vis`

决策规则：

- 第一阶段不接入。
- 只在底层模型中预留 `job kind` 和资源锁抽象。

### 6.4 是否做复杂前端框架

决策规则：

- 第一阶段目标是低风险闭环，不以重前端框架为优先。
- 只要单页工作台的状态管理能稳定支撑任务恢复和日志轮询，就不应为“现代化”而抬高工程复杂度。

## 7. 测试与验收顺序

建议按从低风险到高风险的顺序验收。

### 7.1 P0 必测

- 新建数据集成功录制。
- `8 / 1 / 30 / 0 / false` 的首次试录模板成功。
- 已有目录 + `resume=false` 被拒绝。
- 目录不存在 + `resume=true` 被拒绝。
- 非法 `dataset_name` 被拒绝。
- 非法数值参数被拒绝。
- 运行中第二次提交被拒绝。
- 刷新页面后活跃任务恢复成功。

### 7.2 P1 必测

- 多 episode + reset 的运行阶段展示。
- 停止链路是否能终止整棵进程树。
- 环境不存在时的失败归因。
- 输出目录不可写时的失败归因。

### 7.3 P2 建议测

- 服务重启后的可诊断能力。
- 长日志展示稳定性。
- 移动端可读性。
- 帮助文案可理解性。

## 8. 上线前准备清单

上线前应额外确认以下事项：

- Web 服务运行环境确实是 `lerobot_flex`。
- 服务启动方式和现场值守流程已明确。
- 输出目录有写权限且磁盘空间足够。
- 现场安全提示和确认弹窗文案已经审阅。
- 浏览器关闭、页面刷新、停止录制的行为边界已经写清楚。
- 若没有访问控制，服务必须至少处于受控网络环境中。

## 9. 实施完成定义

满足以下条件时，可认为第一阶段实施完成：

1. 不修改 `record.sh` 前提下，网页可稳定触发一次录制。
2. 参数、目录、`resume` 语义可被稳定校验。
3. 单活跃任务互斥生效。
4. 任务运行状态、最近日志、结果摘要、输出目录都能查询。
5. 页面刷新可恢复活跃任务。
6. 最小测试矩阵中的 P0 项全部通过。
7. 已知限制被清晰写入页面文案或部署说明。

## 10. 一句话执行建议

先把“参数预检查 + 单任务启动 + JobStore + 状态恢复”做成稳定闭环，再去补停止、历史和视觉打磨；任何需要通过修改 `record.sh` 才能完成的能力，都不应作为第一阶段默认实现路径。
