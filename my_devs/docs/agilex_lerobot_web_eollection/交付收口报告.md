# AgileX Web 采集页交付收口报告

更新时间：2026-03-11

## 1. 一页式结论摘要

基于当前已经完成的 6 份脑暴文档，可以直接收口为一个可交付的一阶段方案：

- 目标不是重写采集逻辑，而是在 **不修改** `my_devs/add_robot/agilex/record.sh` 的前提下，提供一个简单、克制、可用的网页采集入口。
- 第一阶段推荐路线是：
  - 后端：`FastAPI`
  - 前端：原生 `HTML + CSS + JavaScript`
  - 运行模型：后端把 `record.sh` 作为黑盒命令调度
  - 状态回传：REST + 轮询
  - 并发策略：**全局单活任务**
- 页面只暴露 6 个核心参数：
  - `dataset_name`
  - `episode_time_s`
  - `num_episodes`
  - `fps`
  - `reset_time_s`
  - `resume`
- 视觉风格采用“Apple 风格的秩序感”，不是做成营销页：
  - 大留白
  - 强层级
  - 轻材料
  - 短文案
  - 稳定反馈
- 当前最重要的不是“界面漂不漂亮”，而是 4 件事是否做扎实：
  - 参数校验
  - 单任务互斥
  - 日志与状态持久化
  - 故障与中断可归因

结论很明确：

**可以进入实现阶段，但第一阶段必须严格控制边界，不做并发录制、不做重前端工程、不做花哨实时通信层。**

## 2. 当前已知事实

以下事实来自当前已落地的调研与脑暴文档，已足够支撑一阶段架构决策：

- 底层真实执行入口是：
  - `bash my_devs/add_robot/agilex/record.sh <dataset_name> <episode_time_s> <num_episodes> <fps> <reset_time_s> <resume>`
- 默认 Conda 环境是：
  - `lerobot_flex`
- 默认输出根目录是：
  - `my_devs/add_robot/agilex/outputs`
- 实际数据目录是：
  - `my_devs/add_robot/agilex/outputs/dummy/<dataset_name>`
- 当同名数据集已存在且 `resume=false` 时，脚本会失败。
- `resume` 本质是布尔开关，网页不应让用户手填字符串。
- 现有仓库已经有可复用的 Web 形态样板：
  - `FastAPI + 静态页面 + 轮询状态/日志`
- 风险上最敏感的一点不是“能不能起 subprocess”，而是：
  - 固定路径
  - 共享资源
  - 中断语义
  - 日志恢复
  - 环境依赖

## 3. 推荐技术路线

### 3.1 推荐方案

推荐采用：

- `FastAPI` 作为单机后端
- 原生静态页承载前端
- 后端固定调用 `record.sh`
- 前端轮询任务状态和日志
- 后端维护持久化任务台账

### 3.2 为什么推荐这条路线

原因有 5 个：

1. 当前仓库已经证明这套模式能工作，复用成本最低。
2. 用户要求“不改脚本”，黑盒调度比重写逻辑更稳。
3. 页面目标单一，不值得为第一阶段引入 React/Vite 或复杂任务队列。
4. Apple 风格网页不依赖重前端框架，原生静态页完全可以做出克制的观感。
5. 风险控制主要在后端编排，而不是在前端技术选型。

### 3.3 第一阶段推荐页面结构

页面建议保持单页，控制在 3 个主区：

- 顶部与安全提示区
- 参数配置区
- 状态与日志区

页面主职责只有一件事：

**让用户安全地组织一次 `record.sh` 执行。**

## 4. 第一阶段建议的系统边界

### 4.1 应做

- 暴露 6 个核心参数表单
- 展示命令预览
- 展示预计输出目录
- 启动前做参数校验
- 启动前检测目录冲突
- 运行中展示状态、耗时、日志
- 支持刷新后恢复当前任务状态
- 支持最基础的停止能力
- 明确成功/失败后的下一步提示

### 4.2 暂不做

- 不支持并发录制
- 不支持多设备资源调度
- 不引入 React/Vite
- 不引入 WebSocket
- 不把回放、可视化、训练全部塞进同页
- 不暴露 topic、相机映射等内部高级配置到主表单
- 不承诺“精确进度”，只提供状态与预计耗时

## 5. 风险优先级、缓解策略与验收建议

## 5.1 高优先级风险

### 风险 H1：参数注入或非法输入导致执行失控

已知事实：

- 网页最终会组织 shell 脚本调用。
- `dataset_name` 等字段如果不做白名单校验，会成为风险入口。

缓解策略：

- 后端固定脚本路径，不接受用户自定义脚本路径。
- 后端使用参数数组启动，不使用字符串拼接。
- `dataset_name` 仅允许白名单字符。
- 数值字段做类型、范围、步长约束。
- `resume` 只用开关或严格布尔转换。

验收建议：

- 非法字符输入必须被后端拒绝。
- 前端拦截和后端拦截都应存在，但以后端为准。
- 启动日志中必须能看到结构化参数摘要，而不是原始拼接字符串。

### 风险 H2：并发录制导致目录、配置或机器人资源冲突

已知事实：

- 当前脚本语义天然更适合单任务。
- 共享资源与固定路径是当前已知敏感点。

缓解策略：

- 第一阶段强制单活任务。
- 后端启动前检查是否已有运行中任务。
- UI 在任务运行时禁用再次启动按钮。
- 如需后续扩展，多任务应优先做排队，不做并发。

验收建议：

- 同时发起 2 次启动请求时，只允许 1 个成功。
- 刷新页面后仍然只能看到 1 个活跃任务。
- 任务结束前再次点击启动，必须得到明确提示。

### 风险 H3：浏览器关闭、刷新或网络抖动后状态丢失

已知事实：

- 用户可能误以为页面关闭就代表任务停止。
- 现场使用时，标签页刷新和网络波动是高概率事件。

缓解策略：

- 后端保存任务元数据和日志文件。
- 前端刷新后主动恢复当前任务状态。
- 页面文案明确说明“关闭浏览器不等于停止任务”。

验收建议：

- 启动任务后刷新页面，状态与日志可以恢复。
- 关闭页面后重新进入，仍能看到当前任务。
- 结束后历史任务摘要仍可查询。

### 风险 H4：环境依赖未满足导致现场不可用

已知事实：

- 当前已知运行环境强依赖 `lerobot_flex`。
- 机器人、相机、ROS 节点状态会直接影响采集结果。

缓解策略：

- 页面展示只读环境信息与启动前检查项。
- 后端启动前记录当前环境摘要。
- 启动失败时优先输出环境类错误摘要。
- 上线说明必须固定使用 `lerobot_flex`。

验收建议：

- 在缺失环境或命令不可用时，页面给出清晰错误而不是沉默失败。
- 现场操作文档必须写明启动命令与环境要求。

## 5.2 中优先级风险

### 风险 M1：`resume` 语义不清导致误写数据集

缓解策略：

- `resume` 只做开关，不做自由输入。
- 检测到目录已存在时高亮警告。
- 启动前确认弹窗必须明确写明“本次是新建还是续录”。

验收建议：

- 已有目录且 `resume=false` 时，不允许静默启动。
- 已有目录且 `resume=true` 时，确认弹窗必须明显提示。

### 风险 M2：日志只存在内存，服务重启后不可追溯

缓解策略：

- 日志必须落盘。
- 任务元数据与日志文件使用独立任务目录。
- 页面展示摘要日志，必要时允许查看完整日志。

验收建议：

- 任务完成后日志文件可追溯。
- 失败后可查看最近关键日志，而不是只看到“失败”。

### 风险 M3：停止语义不清，用户误解“停止”能力

缓解策略：

- 第一阶段只提供最基础的停止能力。
- UI 文案要清楚说明停止是“请求停止任务”，不是瞬时强杀。
- 停止结果应区分：
  - 用户主动停止
  - 脚本异常失败
  - 正常完成

验收建议：

- 点击停止后，状态必须进入明确的停止中或已停止状态。
- 日志中能看到停止相关事件。

### 风险 M4：风格过度追求“苹果化”，损害可操作性

缓解策略：

- 视觉克制优先于装饰。
- 首屏先展示任务与状态，不堆大面积动态效果。
- 日志区不抢首屏主视觉。

验收建议：

- 用户首屏能在数秒内看懂“填哪里、点哪里、现在什么状态”。
- 页面不依赖复杂动画才能完成核心操作。

## 5.3 低优先级风险

### 风险 L1：默认值与实际使用习惯不完全一致

缓解策略：

- 保留脚本默认值语义。
- 同时提供常用快捷模板，例如首录模板。

验收建议：

- 新手能一键填入推荐参数组合。

### 风险 L2：文案过少导致新手不敢点，文案过多导致页面变重

缓解策略：

- 主界面只保留短帮助文案。
- 复杂说明放折叠帮助或确认弹窗。

验收建议：

- 用户无需翻外部文档也能完成首录。

## 6. 最小可执行验收建议

第一阶段建议至少通过以下验收门槛：

### A 类：核心流程

- 能通过页面成功启动 1 次新建录制任务。
- 能通过页面成功执行 1 次续录任务。
- 成功后页面能展示输出目录与下一步建议。

### B 类：风险兜底

- 非法参数被拦截。
- 目录冲突被拦截或强提示。
- 运行中无法再次启动第二个任务。
- 刷新页面后状态与日志可恢复。

### C 类：失败可归因

- 环境异常时页面能给出明确错误。
- 任务失败时页面能区分“参数问题/环境问题/用户停止/脚本失败”。

### D 类：可回退

- 即使网页失败，仍然可以回退到手工执行 `record.sh`。
- 页面上线不会破坏原有命令行使用路径。

## 7. 最小测试建议

按优先级建议至少做以下测试：

### P0 测试

- Happy path：新建任务成功完成。
- Resume path：已有目录下续录成功。
- Duplicate path：已有目录且未续录时被正确拒绝。
- Single-active path：运行中再次点击启动被拒绝。
- Refresh recovery：任务运行中刷新页面后状态恢复。

### P1 测试

- Stop path：运行中点击停止，状态与日志符合预期。
- Env failure：缺失环境或依赖时报错清晰。
- Invalid input：非法字符、空值、负数、非布尔值等输入被拒绝。

### P2 测试

- 移动端可读性验证。
- 长日志展示稳定性。
- 帮助文案与确认弹窗可理解性验证。

## 8. 第一阶段明确不做什么

为了保证交付质量，第一阶段建议明确不做以下内容：

- 不做多任务并发录制
- 不做复杂后台队列系统
- 不做 WebSocket
- 不做 React/Vite 重前端改造
- 不做同页整合 replay/vis/train 的大工作台
- 不把 Apple 风格理解成重动画和大面积视觉特效

## 9. 交付建议

如果马上进入实现，推荐按以下顺序推进：

1. 先搭出黑盒调度与单任务状态机。
2. 再补参数校验、目录冲突校验与日志持久化。
3. 再做单页 UI、确认弹窗和帮助文案。
4. 最后做视觉打磨与移动端适配。

这个顺序的核心原因是：

**网页的可靠性优先级高于视觉精修，风险控制优先级高于功能扩张。**

## 10. 最终结论

当前信息已经足够支持立项实现。一阶段最合理的交付目标是：

**做一个基于 FastAPI + 原生静态页的 AgileX 采集控制页，把 `record.sh` 稳定、安全、可恢复地包装起来；同时坚持单任务、强校验、可回退这三条底线。**

只要这三条底线不放松，这个网页就可以在不改底层脚本的前提下，成为一个可交付、可上线、可被新手使用的采集入口。
