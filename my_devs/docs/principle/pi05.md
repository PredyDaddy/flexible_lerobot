# PI0.5（pi05）原理与本仓库实现解读

## 策略背景
PI0.5 是 Physical Intelligence 在 π0 之后提出的 VLA 策略，目标是把视觉语言模型中的通用语义知识迁移到机器人控制。  

证据边界：公开博客、论文和 `openpi` 代码能说明总体方向，但没有完整披露全部训练数据和线上系统细节。下文会区分“公开原理”和“本仓库实现推断”。

## 核心原理
从本仓库实现看，`pi05` 的低层动作头属于连续动作 flow matching。训练时构造：

`x_t = t * eps + (1 - t) * a`

目标速度场为：

`u_t = eps - a`

损失为：

`L = ||v_theta(x_t, t, cond) - u_t||^2`

其中 `cond` 来自图像、文本和状态。按 README 与代码，PI0.5 相比 PI0 的关键差异是：状态离散后拼进文本、`tokenizer_max_length=200`、动作专家使用 AdaRMS。

## 在本仓库中的实现解读（模块与数据流）
`configuration_pi05.py` 定义默认骨干与超参数：`paligemma_variant=gemma_2b`、`action_expert_variant=gemma_300m`、`chunk_size=50`，状态/动作 pad 到 32 维，并默认做 quantile normalization。  

`processor_pi05.py` 把归一化后的状态离散到 256 个 bin，再与任务文本拼成 `Task: ... , State: ...; Action:`，然后用 PaliGemma tokenizer 编码。  

`modeling_pi05.py` 中，前缀是图像加文本，后缀是带时间条件的动作 token；`PI05Pytorch.forward()` 计算训练损失，`sample_actions()` 从噪声迭代得到动作 chunk，`PI05Policy` 负责图像预处理和动作队列。

## 训练与推理机制
训练流：观测重命名 -> 加 batch 维 -> 归一化 -> 状态离散并写入文本 prompt -> tokenizer -> 图像缩放/补边并映射到 `[-1,1]` -> flow matching 损失。  

推理流：先编码图像和文本前缀并缓存 `past_key_values`，再从随机噪声动作开始，用固定步数欧拉更新逐步去噪，输出长度为 `chunk_size` 的动作块，并按 `n_action_steps` 执行。  

边界说明：公开材料强调 PI0.5 的高层语义抽象能力，但本仓库这几个文件主要呈现的是 `openpi` 风格连续动作头，没有完整展示全部高层推理链路。

## 关键超参数及影响
`chunk_size/n_action_steps` 决定一次预测和执行的时间范围；更大 chunk 更前瞻，但累计误差也更大。`num_inference_steps` 影响采样质量与时延。`max_state_dim/max_action_dim` 决定内部 pad 维度。`time_sampling_beta_*` 影响训练时刻采样分布。

## 适用场景、优势与局限
适合多模态条件控制、任务语言重要、且希望具备一定开放世界泛化能力的机械臂任务。优势是统一视觉、语言和动作建模。局限是：公开证据有限；连续去噪推理有时延；状态离散文本化可能损失连续信息；本仓库实现不能直接等同于 PI 官方完整系统。
