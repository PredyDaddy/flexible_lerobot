## 5. 实现方式

### 5.1 总体策略

本次没有采用“直接改 parquet / meta / videos”的方式，而是采用“读取源数据后重建新数据集”的方式。

原因很直接：

1. LeRobot 数据集不是单一 parquet 文件
2. 数据、元数据、episode 信息、视频索引是联动的
3. 直接原地改字段风险高，容易造成 metadata 和真实数据不一致
4. 重建目标数据集虽然更慢，但更稳，验收也更简单

### 5.2 拆分脚本的主要逻辑

拆分主脚本在：

- [split_bimanual_lerobot_dataset.py](/home/agilex/cqy/flexible_lerobot/my_devs/split_datasets/split_bimanual_lerobot_dataset.py)

它做的事情可以概括为：

1. 读取源 `LeRobotDataset`
2. 检查源数据是否包含必须的字段
3. 为左臂和右臂构造各自的 feature schema
4. 逐帧读取源数据
5. 左臂帧切出前 7 维 action/state
6. 右臂帧切出后 7 维 action/state
7. 左臂只保留 `camera_front + camera_left`
8. 右臂只保留 `camera_front + camera_right`
9. 写入两个新的 `LeRobotDataset`
10. 在每个 episode 保存前回填 `timestamp`

### 5.3 CHW 到 HWC 的处理

真实数据在通过 `LeRobotDataset` 读取时，图像张量是 `CHW` 排列；但写回数据集时，LeRobot 的图像 feature 元信息是 `HWC` 语义。

因此拆分脚本里额外做了一层通道变换逻辑，把读取出来的图像统一转成 channel-last 再写盘。对应实现位置：

- [split_bimanual_lerobot_dataset.py#L216](/home/agilex/cqy/flexible_lerobot/my_devs/split_datasets/split_bimanual_lerobot_dataset.py#L216)

这个处理是必要的，否则真实 `video` 路径会在写盘或校验阶段出现 shape 不匹配。

### 5.4 timestamp 的处理

LeRobot 的 frame 写入流程里，`timestamp` 不是直接作为普通 frame feature 一起 add，而是需要在一个 episode 的 frame 都进入 buffer 之后，再把 timestamp 列回填进 `episode_buffer`，最后执行 `save_episode()`。

本次实现里做了专门处理：

1. 每帧先从 frame 字典中取出 `timestamp`
2. 单独暂存到当前 episode 的 `current_timestamps`
3. episode 切换或结束时回填到 `dataset.episode_buffer["timestamp"]`
4. 最后再保存 episode

对应实现位置：

- [split_bimanual_lerobot_dataset.py#L265](/home/agilex/cqy/flexible_lerobot/my_devs/split_datasets/split_bimanual_lerobot_dataset.py#L265)
- [split_bimanual_lerobot_dataset.py#L313](/home/agilex/cqy/flexible_lerobot/my_devs/split_datasets/split_bimanual_lerobot_dataset.py#L313)

### 5.5 task 文本策略

本次默认策略是：

- 直接复制源数据的 `task`

但同时预留了覆盖能力，支持：

- `--task-mode copy`
- `--task-mode override`
- `--task-text`
- `--left-task-text`
- `--right-task-text`

因此后续如果你要做 VLA，可以在拆分阶段就把左右臂语言换成更有语义的信息，而不需要再改代码。

## 6. 校验脚本的实现

校验脚本在：

- [validate_split_dataset.py](/home/agilex/cqy/flexible_lerobot/my_devs/split_datasets/validate_split_dataset.py)

它现在会检查下面这些内容：

1. 目标数据能否被 `LeRobotDataset` 正常加载
2. 左右输出的 `action.shape` 是否为 `[7]`
3. 左右输出的 `observation.state.shape` 是否为 `[7]`
4. 左右输出的图像键是否正确
5. episode 总数是否与源数据一致
6. frame 总数是否与源数据一致
7. 每个 episode 长度是否与源数据一致
8. 左右输出的 action/state 数值是否与源数据切片完全一致
9. timestamp 是否与源数据一致
10. task 文本是否与源数据一致
11. `lag+1` 是否为 0，也就是 `action_t = state_(t+1)` 是否仍然成立

## 9. 当前残余风险

### 9.1 task 语义偏弱

当前 `task` 仍然是：

- `agilex static record test`

这在数据格式上没有问题，但如果后续你想直接用于 VLA 训练，语言监督的语义价值偏低。

建议：

- 在正式整理训练数据时，使用 `--task-mode override`
- 把左右臂 task 写成更具体的动作描述

### 9.2 `--episode-indices` 还不建议作为主流程使用

当前拆分脚本已经支持 `--episode-indices`，也就是只拆部分 episode。

但校验脚本现在仍然默认按“目标数据的总 episode/frame 数必须与源全集一致”来检查。因此：

- 拆整份数据没有问题
- 如果只拆部分 episode，校验脚本会误报 mismatch

这属于后续增强项，不影响当前完整数据拆分。

### 9.3 真实视频路径的自动回归测试还不够强

当前单测已经覆盖拆分逻辑本身，但主要用的是合成小数据集。

真实 `video` 特征和 `CHW -> HWC` 路径，这次是通过真实数据实测验证通过的，但还没有额外补成一条更强的自动回归测试。
