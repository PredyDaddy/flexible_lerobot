# 模型导出方案（ACT → ONNX）

本文针对仓库内 ACT（Action Chunking Transformer）单臂模型（两路相机 + 7 维关节）给出一套“可落地”的 ONNX 导出方案，目标是为后续 TensorRT / ONNXRuntime 加速提供稳定的中间产物。

> 适用脚本：`agilex_infer_single_cc_vertical.py`  
> 典型 checkpoint：`outputs/act_agilex_left_box/checkpoints/last/pretrained_model`
>
> 导出脚本（方案 A 的落地实现）：`trt_act/export/export_single.py`

---

## 1. 导出目标与边界

### 1.1 你要“导出”的到底是哪一段？

当前推理链路（简化）：

1. 采集观测（相机图像 HWC uint8 + 关节 state float）
2. `prepare_observation_for_inference`：图像 `/255` + `HWC→CHW` + `add batch`
3. `policy_preprocessor`：mean/std 归一化（来自 checkpoint 的 `.safetensors` stats）
4. `ACTPolicy.select_action`：
   - queue 为空时：跑一次 `policy.model(batch)` 得到 action chunk（`(B, chunk_size, action_dim)`）
   - queue 非空时：直接从 queue 取下一步 action（避免每步都前向）
5. `policy_postprocessor`：action 反归一化（mean/std）

**ONNX 导出不建议直接导出 `ACTPolicy.select_action`**，原因：

- `select_action` 包含 Python 侧队列状态机（deque），ONNX 很难表达且不必要。
- 最稳定的导出粒度是：**只导出 `policy.model`（ACT 网络本体）**，输出 action chunk；队列逻辑保留在 Python 侧实现即可。

### 1.2 推荐导出产物（最小闭环）

导出一个 ONNX 模型：输入为“已对齐预处理 + 已归一化”的 Tensor，输出为“归一化 action chunk”：

- Inputs：
  - `obs_state_norm`: `(1, 7)` FP32
  - `img0_norm`: `(1, 3, 480, 640)` FP32
  - `img1_norm`: `(1, 3, 480, 640)` FP32
- Output：
  - `actions_norm`: `(1, chunk_size=100, 7)` FP32

后续推理侧再用 checkpoint stats 做 action 反归一化，并实现 action queue 语义。

## 2. 在导出前必须确认的“模型事实”

### 2.1 输入相机数量与顺序（非常关键）

ACT 在 `ACTPolicy.predict_action_chunk` 中会把多路相机图像组织为 `batch["observation.images"] = [img0, img1, ...]`，其顺序来自 `config.image_features`（本质上是 `config.json` 里 `input_features` 的视觉特征插入顺序）。

**导出/部署时必须保持相同顺序，否则动作会明显跑偏。**

建议用下面方式读取并固定顺序（以 left 模型为例）：

```bash
python - <<'PY'
import json
from pathlib import Path

ckpt = Path("outputs/act_agilex_left_box/checkpoints/last/pretrained_model")
cfg = json.loads((ckpt / "config.json").read_text())
image_keys = [k for k, v in cfg["input_features"].items() if v["type"] == "VISUAL"]
print("VISUAL keys in order:", image_keys)
PY
```

对单臂模型通常是：

- 左臂：`["observation.images.camera_left", "observation.images.camera_front"]`
- 右臂：`["observation.images.camera_right", "observation.images.camera_front"]`

但**以实际 `config.json` 为准**。

### 2.2 `use_vae=true` 但推理不走 VAE encoder（重要简化点）

虽然 `config.json` 里 `use_vae: true`，但在 `ACT.forward` 中只有在：

- `self.training == True` 且
- batch 里包含 `action`（训练目标）

才会走 VAE encoder 分支。

推理时模型处于 `eval()`，且 batch 不含 `action`，因此会走：

- `latent_sample = zeros([B, latent_dim])`

这意味着：

- 导出推理图不需要提供 action / action_is_pad；
- 推理图是确定性的（无采样/随机性）。

## 3. 预处理/后处理对齐（与 checkpoint 完全一致）

### 3.1 stats 文件位置

在 checkpoint 目录下（如 `.../pretrained_model/`）：

- `policy_preprocessor_step_3_normalizer_processor.safetensors`：输入归一化 stats
- `policy_postprocessor_step_0_unnormalizer_processor.safetensors`：输出反归一化 stats
- `policy_preprocessor.json` / `policy_postprocessor.json`：记录 eps、feature shape 等元信息（建议以此为准）

### 3.2 归一化公式（与仓库实现一致）

仓库 `NormalizerProcessorStep` 的 MEAN_STD 逻辑：

- normalize：`x_norm = (x - mean) / (std + eps)`，其中 `eps=1e-8`
- unnormalize：`y = y_norm * std + mean`

### 3.3 图像预处理对齐

推理脚本通过 `prepare_observation_for_inference` 做：

1. `uint8` → `float32`
2. `/255`
3. `HWC → CHW`
4. `unsqueeze(0)`（batch=1）

导出/部署时也必须严格一致，否则模型行为会漂移。

## 6. 导出过程中可能踩到的坑（结合本仓库 ACT 实现）

### 6.1 `ACT.forward` 的 Python list 展开会导致 ONNX 图膨胀

当前实现对每个相机 feature map 会做：

- `(B,C,H,W)` → `(H*W,B,C)` 后 `list(tensor)` 展开为 `H*W` 个 `(B,C)` token

这会导致 export 时 ONNX 图可能变得非常大，后续 TensorRT/ORT 解析也更慢。

### 6.3 torchvision backbone 的权重下载问题（离线环境）

如果你的环境是离线且缓存里没有 torchvision 权重，实例化模型时可能触发下载失败。

- 对应到仓库脚本：`trt_act/export/export_single.py` 默认会将 `pretrained_backbone_weights` 置为 `None`；如需保留，使用 `--keep-backbone-pretrained-weights`。
