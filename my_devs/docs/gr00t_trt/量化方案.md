# GROOT TensorRT 量化方案（上游 Isaac-GR00T N1.5）

本文记录上游 `Isaac-GR00T-n1.5-release` 的量化做法，作为后续把量化能力集成到本仓库的技术依据。

约束与共识：

- 当前优先级仍然是 **FP16 全链路跑通 + TRT 部署**；量化属于下一阶段工作。
- 上游的量化不是发生在 `trtexec` 构建 engine 时，而是发生在 **ONNX 导出前**（PyTorch 模型先校准/量化，再导出 ONNX）。

---

## 0. TL;DR（你需要记住的 6 件事）

1. 上游量化入口是 `deployment_scripts/export_onnx.py`，依赖 `modelopt`（NVIDIA ModelOpt）。
2. ViT/DiT 走 **FP8 量化**；LLM 支持 **NVFP4(W4A4)** 或 FP8。
3. LLM 的 NVFP4 导出后会做一次 **fp4 qdq -> 2dq** 的 ONNX 转换，并可能生成外部数据文件（`*.onnx_data`），打包产物时必须带上。
4. NVFP4 LLM 的 TRT engine 构建通常要求 **固定 shape / batch=1**（上游 `build_engine.sh` 内部会把 nvfp4 的 `LLM_MAX_BATCH` 设为 1）。
5. 量化是校准驱动的，`--calib-size` 太小会让精度风险不可控。上游默认 10 只是 demo 级别，真实部署建议从 100 起步（再视效果增大）。
6. 推荐路线是“逐模块、从低风险到高风险”：
   - `vit=fp8`，其余 fp16
   - `vit=fp8 + dit=fp8`，LLM 仍 fp16
   - 最后再尝试 `llm=nvfp4`（或 `llm=fp8`）

## 2. 量化发生在哪：导出 ONNX 之前

上游的工作流可以理解为：

1. 加载 PyTorch 模型（GR00T N1.5）
2. 根据 `--vit-dtype/--llm-dtype/--dit-dtype` 决定是否量化
3. 如果启用量化：用 `modelopt.torch.quantization` 做 calibration + quantize
4. 用 `torch.onnx.export(...)` 导出 ONNX（此时模型已经被量化）
5. 对 NVFP4 LLM：额外做一次 ONNX 图转换（qdq -> 2dq），并保存外部数据

对应上游关键代码位置（便于你将来回溯）：

- ViT：`export_eagle2_vit(...)` 内当 `vit_dtype=="fp8"` 时调用 `quantize_vit(...)`
  - 参考：`export_onnx.py:1043`
- LLM：`export_eagle2_llm(...)` 内当 `llm_dtype in ["nvfp4","fp8"]` 时调用 `quantize_llm(...)`
  - 参考：`export_onnx.py:1147`
  - NVFP4 的 qdq->2dq：`export_onnx.py:1215`
- DiT：`export_action_head(...)` 内当 `dit_dtype=="fp8"` 时调用 `quantize_dit(...)`
  - 参考：`export_onnx.py:1352`

## 3. 依赖与前置条件（量化必需）

### 3.1 必需依赖

上游量化直接 `import modelopt.torch.quantization as mtq`（`export_onnx.py:22`）。

结论：只要你打算用 `fp8/nvfp4`，`modelopt` 没装就一定会失败。

### 3.2 校准数据必需

上游的三种量化（ViT/DiT/LLM）都需要基于 LeRobot 数据集做校准，它们的函数都要求：

- `dataset_path`（或 `--calib-dataset-path`）
- `data_config`（决定 modality_config + transform）
- `embodiment_tag`
- `video_backend`（`decord` 或 `torchcodec`）
- `denoising_steps`（会影响 DiT 的校准 loop）

在上游 CLI 中对应参数（`export_onnx.py:1538` 起）：

- `--dataset-path`
- `--calib-dataset-path`（可选）
- `--data-config`
- `--embodiment-tag`
- `--denoising-steps`
- `--video-backend`
- `--calib-size`
- `--full-layer-quant`（仅 nvfp4 有意义）

## 4. ViT 的 FP8 量化（上游做法）

### 4.1 触发方式

把 `--vit-dtype` 设为 `fp8`：

- `--vit-dtype fp8`

上游 `export_eagle2_vit(...)` 检测到 `vit_dtype=="fp8"` 后，会在导出 ONNX 前做量化（`export_onnx.py:1043`）。

### 4.2 量化配置要点

`quantize_vit(...)` 的关键点（`export_onnx.py:532`）：

- 使用 `mtq.FP8_DEFAULT_CFG`
- **显式禁用 Conv2d 的量化**以避免精度下降：
  - `quant_cfg["quant_cfg"]["nn.Conv2d"] = {"*": {"enable": False}}`（`export_onnx.py:576`）
- 校准数据来自 `ViTCalibrationDataset`（基于 LeRobot dataset）

### 4.3 导出产物

- `gr00t_onnx/eagle2/vit_fp8.onnx`

---

## 5. DiT 的 FP8 量化（上游做法）

### 5.1 触发方式

把 `--dit-dtype` 设为 `fp8`：

- `--dit-dtype fp8`

上游 `export_action_head(...)` 在导出 action head 的 DiT 前做量化（`export_onnx.py:1352`）。

### 5.2 量化配置要点

`quantize_dit(...)` 的关键点（`export_onnx.py:620`）：

- 仍使用 `mtq.FP8_DEFAULT_CFG`，但额外针对 attention 量化器做了更激进的设置：
  - `*[qkv]_bmm_quantizer`、`*softmax_quantizer` 的 `num_bits` 设置为 `(4, 3)`（`export_onnx.py:668`）
- 更关键的是它的 calibration loop：上游实现了 `_quantize_dit_model(...)`，在校准时跑多步 denoising（`export_onnx.py:448` 附近）。
  - 这点很重要：DiT 是 diffusion 迭代模型，如果只做单步 forward 的校准，分布不匹配会更明显。

### 5.3 导出产物

- `gr00t_onnx/action_head/DiT_fp8.onnx`

## 6. LLM 的 NVFP4 / FP8 量化（上游做法）

### 6.1 触发方式

上游 CLI 的 LLM dtype 选择（`export_onnx.py:1573`）：

- `--llm-dtype fp16`
- `--llm-dtype fp8`
- `--llm-dtype nvfp4`

其中：

- `fp16`：不量化
- `fp8`：FP8 量化（ModelOpt）
- `nvfp4`：NVFP4(W4A4) 量化（ModelOpt）

### 6.2 selective vs full-layer（nvfp4 专属）

上游 `quantize_llm(...)`（`export_onnx.py:823`）提供两种策略：

1. selective layer quant（默认）
   - 会显式禁用两类层的量化，以减小精度损失：
   - `mlp.down_proj` 与 `self_attn.o_proj` 的 quantizer 被设置 `enable=False`（`export_onnx.py:857`）
2. full layer quant
   - 通过 CLI 传 `--full-layer-quant` 启用（`export_onnx.py:1626`）
   - 导出的 ONNX 会改名为 `llm_nvfp4_full.onnx`（`export_onnx.py:1186`）

### 6.3 NVFP4 的 ONNX 特殊处理（必须知道）

上游对 nvfp4 做了两个非常关键的特殊处理：

1. 为 nvfp4 ONNX 导出设置 quantizer 的 metadata（`export_onnx.py:1165`）
2. 在导出后把 fp4 qdq 图转换为 2dq（`export_onnx.py:1215`）：
   - 会调用 `fp4qdq_to_2dq(...)`
   - 保存时启用 external data（`onnx.save_model(..., save_as_external_data=True, location="llm_nvfp4.onnx_data")`）

这意味着：

- 你的 LLM ONNX 目录里可能会出现额外文件：`llm_nvfp4.onnx_data`
- engine 构建与部署打包时，这个文件必须与 `.onnx` 一起存在，否则模型不完整

### 6.4 导出产物

- `gr00t_onnx/eagle2/llm_fp16.onnx`
- `gr00t_onnx/eagle2/llm_fp8.onnx`
- `gr00t_onnx/eagle2/llm_nvfp4.onnx`
- `gr00t_onnx/eagle2/llm_nvfp4_full.onnx`（仅 `--full-layer-quant`）

## 7. 推荐“低风险到高风险”的量化路线（可直接照抄）

下面命令是上游 `export_onnx.py` 的用法示例，参数含义见 `export_onnx.py:1538` 起的 argument parser。

注意：以下命令只描述“上游方案怎么做”。你后续如果要在本仓库落地，应先确保环境依赖与路径适配。

### 7.1 只量化 ViT（风险最低）

目标：

- `vit=fp8`
- `llm=fp16`
- `dit=fp16`

示例：

```bash
python deployment_scripts/export_onnx.py \
  --dataset-path <LEROBOT_DATASET_PATH> \
  --onnx-model-path <OUT_DIR>/gr00t_onnx \
  --data-config <DATA_CONFIG> \
  --embodiment-tag <EMBODIMENT_TAG> \
  --denoising-steps 4 \
  --video-backend decord \
  --calib-size 100 \
  --vit-dtype fp8 \
  --llm-dtype fp16 \
  --dit-dtype fp16
```

### 7.2 量化 ViT + DiT（中风险）

目标：

- `vit=fp8`
- `llm=fp16`
- `dit=fp8`

```bash
python deployment_scripts/export_onnx.py \
  --dataset-path <LEROBOT_DATASET_PATH> \
  --onnx-model-path <OUT_DIR>/gr00t_onnx \
  --data-config <DATA_CONFIG> \
  --embodiment-tag <EMBODIMENT_TAG> \
  --denoising-steps 4 \
  --video-backend decord \
  --calib-size 100 \
  --vit-dtype fp8 \
  --llm-dtype fp16 \
  --dit-dtype fp8
```

### 7.3 最后再量化 LLM（高风险，且对 engine profile 约束更强）

目标：

- `vit=fp8`
- `llm=nvfp4`（或 fp8）
- `dit=fp8`

```bash
python deployment_scripts/export_onnx.py \
  --dataset-path <LEROBOT_DATASET_PATH> \
  --onnx-model-path <OUT_DIR>/gr00t_onnx \
  --data-config <DATA_CONFIG> \
  --embodiment-tag <EMBODIMENT_TAG> \
  --denoising-steps 4 \
  --video-backend decord \
  --calib-size 200 \
  --vit-dtype fp8 \
  --llm-dtype nvfp4 \
  --dit-dtype fp8
```

如果要做全层 nvfp4：

```bash
python deployment_scripts/export_onnx.py \
  ... \
  --llm-dtype nvfp4 \
  --full-layer-quant
```

## 8. TRT engine 构建（量化后与 FP16 的差异点）

上游 `build_engine.sh` 的使用方式是通过环境变量指定 dtype 与 profile（`build_engine.sh:38`）。

### 8.1 dtype 必须匹配 ONNX 文件名

例如你导出了：

- `vit_fp8.onnx`
- `llm_nvfp4.onnx`
- `DiT_fp8.onnx`

那么构建 engine 必须设置：

- `VIT_DTYPE=fp8`
- `LLM_DTYPE=nvfp4`
- `DIT_DTYPE=fp8`

否则 `build_engine.sh` 会去找不存在的 ONNX（或者构建出与你预期不同 dtype 的 engine）。

### 8.2 nvfp4 的 batch=1 限制

上游在构建 LLM engine 时对 nvfp4 做了强制处理：

- 当 `LLM_DTYPE` 是 `nvfp4/nvfp4_full` 时，将 `LLM_MAX_BATCH=1`（`build_engine.sh:96` 附近）

含义：

- 你的服务端如果想通过合 batch 提吞吐，nvfp4 路线会更容易受限
- 但如果你是单请求推理（很多机器人控制场景就是这样），batch=1 通常没问题

## 9. 验收建议（量化阶段更需要“每一步都可回退”）

建议你把验收分成三层：

1. 导出成功：ONNX 文件齐全，且（nvfp4 时）外部数据文件齐全
2. engine 构建成功：各 engine 对应 dtype 命名正确，log 无致命错误
3. 行为一致性：用 compare / replay / service healthcheck 做端到端验证

并且每次只改变一个变量（例如只把 `vit` 从 fp16 改成 fp8），不要一上来三件套全开。

## 10. 常见坑（上游方案视角）

1. `ImportError: modelopt is required for quantization`
   - 说明环境没装 modelopt；先回到 fp16 全链路，量化放后面。
2. nvfp4 导出后 engine 构建失败或加载失败
   - 优先检查是否缺失 `llm_nvfp4.onnx_data`（外部数据文件）。
3. engine dtype 与导出 dtype 不一致
   - 典型表现是 server 默认去找 `vit_fp8.engine / llm_nvfp4.engine / DiT_fp8.engine`，但你只导了 fp16。
4. shape profile 不匹配（尤其是 sequence length）
   - `build_engine.sh` 的 `MIN_LEN/OPT_LEN/MAX_LEN` 必须覆盖你真实运行时的长度范围；否则会报 profile 错误或隐性降速。
