## 2. 背景与上游方案（为什么这么做）

### 2.1 你提到的“上游方案”是什么

这里的上游指 Isaac-GR00T 发布包中的部署脚本，核心思路是：

- **分模块导出 ONNX**（而不是单一大图）
- **分模块构建 TensorRT engine**
- **分模块 + 流程级双层一致性校验**

上游参考位置：

- `my_devs/docs/gr00t_trt/Isaac-GR00T-n1.5-release/deployment_scripts/export_onnx.py`
- `my_devs/docs/gr00t_trt/Isaac-GR00T-n1.5-release/deployment_scripts/build_engine.sh`

### 2.2 为什么采用“多 ONNX”而非“单 ONNX”

对 GROOT 这类 VLA（视觉-语言-动作）模型，分模块导出的工程价值很直接：

1. **可定位性更强**：误差可以快速定位到 ViT / LLM / ActionHead 某一段。
2. **构建更稳**：某个模块 TRT 构建失败时，不阻塞全部链路排查。
3. **调优更灵活**：可按模块分别做 profile、精度策略与动态 shape 策略。
4. **审核更清晰**：便于第三方对脚本、输入输出签名和数值偏差做逐段审计。

### 2.3 本仓库中的推理链路（审核视角）

在当前实现里，推理可概括为：

1. **Backbone**
   - ViT：图像 -> 视觉 token
   - ViT 后处理：pixel-shuffle + mlp1
   - LLM：融合文本 token 与视觉 token，输出 backbone 特征
2. **Action Head**
   - `vlln + vl_self_attention` 做跨模态特征整合
   - `state_encoder`、`action_encoder` 编码状态与动作
   - `DiT` 进行条件建模
   - `action_decoder` 解码动作
   - 多步 denoising 迭代得到最终动作

这也是为什么一致性对比既要做“模块级”，也要做“流程级（action_denoising_pipeline）”。

## 5. 脚本级实现说明（代码审核重点）

这一节专门对应你提出的“要给别人审核代码”的要求。

### 5.1 `export_backbone_onnx.py`（导出 ViT + LLM）

脚本路径：`my_devs/groot_trt/export_backbone_onnx.py`

#### 5.1.1 主要职责

- 加载 GROOT policy checkpoint
- 导出：
  - `eagle2/vit_fp16.onnx`
  - `eagle2/llm_fp16.onnx`

#### 5.1.2 关键实现点

1. `VisionModelForOnnx`
   - 将视觉塔包装成 ONNX 友好的前向接口。
   - 接口保留 `position_ids` 输入，通过“`*0` 注入”确保 ONNX 图中输入签名可控。

2. `LanguageModelForOnnx`
   - 显式构造 causal mask（`_build_causal_mask`）。
   - 走 language model 的显式前向路径，输出指定 hidden layer，经 `eagle_linear` 投影得到 embeddings。

3. Attention 实现切换
   - 在导出前设置 `vision_model` 与 `language_model` 的 `_attn_implementation = "eager"`。
   - 目的是规避 FlashAttention 导出路径的不稳定性，提升 ONNX 导出可重复性。

4. 动态维度策略
   - ViT：batch 动态
   - LLM：batch + sequence 动态

#### 5.1.3 CLI 参数（审核常看）

- `--policy-path`：checkpoint 根目录
- `--onnx-out-dir`：输出目录（内部创建 `eagle2/`）
- `--seq-len`：LLM trace 使用的序列长度
- `--video-views`：ViT trace 视角数
- `--vit-dtype` / `--llm-dtype`：文件后缀标记
- `--opset`：默认 19
- `--device`：当前固定 `cuda`

---

### 5.2 `export_action_head_onnx.py`（导出 ActionHead 5 模块）

脚本路径：`my_devs/groot_trt/export_action_head_onnx.py`

#### 5.2.1 主要职责

从同一 checkpoint 中导出 action head 五个 ONNX：

1. `vlln_vl_self_attention.onnx`
2. `state_encoder.onnx`
3. `action_encoder.onnx`
4. `DiT_fp16.onnx`
5. `action_decoder.onnx`

#### 5.2.2 关键实现点

1. `VLLNVLSelfAttention` 包装器
   - 将 `vlln` 与 `vl_self_attention` 组合成一个独立导出模块，便于 TRT 侧单独构建与对比。

2. 输入 shape 设计
   - 依据 `action_head.config` 自动读取 embedding dim / action horizon / state dim / hidden size 等，避免手写常量漂移。

3. 动态维度策略
   - 对 batch 维全部开放动态；
   - 对与时间序列相关的输入（如 `backbone_features`、`vl_embs`）开放 sequence 动态。

4. 导出精度
   - 模块统一用 FP16 trace，保持与 TensorRT 目标配置一致。

---

### 5.3 `compare_torch_onnx.py`（一致性核心脚本）

脚本路径：`my_devs/groot_trt/compare_torch_onnx.py`

#### 5.3.1 主要职责

同时做两层一致性比较：

1. **模块级比较**
   - ViT
   - ViT 后处理（`pixel_shuffle + mlp1`）
   - LLM（`llm_direct` 与 `llm_from_vit_pipeline`）
   - ActionHead 五子模块
2. **流程级比较**
   - `action_denoising_pipeline`（多步迭代）

#### 5.3.2 关键函数与审核含义

1. `_metrics(ref, pred)`
   - 输出 `cosine / rmse / mean_abs / max_abs`，是报告主指标来源。

2. `_postprocess_vit(backbone, vit_embeds)`
   - 复现真实 ViT 后处理逻辑（含 pixel-shuffle 条件分支），确保 pipeline compare 不是“假链路”。

3. `_build_llm_inputs(...)`
   - 用统一 `input_ids` 构造 Torch/ONNX 两侧输入，控制变量，减少输入差异带来的伪误差。

4. denoising loop（流程级）
   - 按 time bucket 迭代更新动作，逐步调用 `action_encoder -> DiT -> action_decoder`。
   - 最终输出与 Torch 对比，直接反映动作闭环一致性。

#### 5.3.3 复现稳定性策略

- 固定随机种子：`20260303`
- 固定 profile：
  - 1-view：`seq_len=296`
  - 2-view：`seq_len=568`
- 输出完整 JSON，便于自动审计与复跑 diff。

## 8. 一致性验证方法、指标与判据

### 8.1 方法

本轮采用“模块级 + 流程级”双层对比：

1. 模块级：10 个节点（ViT/LLM/ActionHead 各子模块）
2. 流程级：`action_denoising_pipeline`

输出写入：

- `compare_metrics_1view.json`
- `compare_metrics_2view.json`

### 8.2 指标定义

- `cosine`：向量方向一致性（越接近 1 越好）
- `rmse`：均方根误差（越小越好）
- `mean_abs`：平均绝对误差（越小越好）
- `max_abs`：最大绝对误差（越小越好）

### 8.3 审核判据（建议）

结合本项目 FP16 导出目标，建议审核阈值：

- 模块级：`cosine >= 0.9999` 可视为高一致
- 流程级（最终动作）：`cosine >= 0.99999` 视为可进入 TRT 构建阶段
- `missing` 必须为空

## 12. 已知现象与解释（避免审核误判）

1. `llm_from_vit_pipeline` 的误差通常略高于 `llm_direct`
   - 原因：前者叠加了 ViT + 后处理 + token 注入 + LLM 的复合误差。
   - 但本轮 cosine 仍在 `0.99997+`，属于高一致。

2. compare 使用 ORT CPU provider
   - 这是数值一致性验证，不是性能测试。
   - 后续若做性能评估，应转到 TRT engine 测延迟与吞吐。

3. 导出阶段若出现 tracing 类 warning（环境相关）
   - 常见于 ONNX trace 的 Python 分支/常量提示；
