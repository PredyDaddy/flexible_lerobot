因此，这次工作不是简单“把参考代码复制过来”，而是要在当前仓库目录结构、当前 checkpoint 布局、当前 LeRobot 推理边界下，重新建立一条可实际使用的 ACT TRT 工作流。

## 2. 工作目标与验收标准

本次工作按四个层级定义验收标准：

### 2.1 导出层

- 能从 `pretrained_model` 导出 ACT core ONNX。
- 导出后能写出 metadata，明确输入输出名称、shape、相机顺序和 checkpoint 来源。

### 2.2 编译层

- 能从 ONNX 成功构建 TensorRT engine。
- 编译结果需要输出实际 engine I/O 描述，而不是只看 build 是否成功。

### 2.3 一致性层

要分两层做一致性验证：

- Core 一致性：
  比较 `torch core`、`onnx core`、`trt core`。
- Policy 一致性：
  比较完整策略推理路径的 `torch policy` 和 `trt policy adapter`。
  这里必须经过和真实运行相同的 preprocessor / postprocessor / action queue 逻辑，而不是只比较 core 网络。

### 2.4 推理层

- `my_devs/train/act/so101/run_act_infer.py` 可以在 `torch` 和 `trt` 两种 backend 之间切换。
- 新增 `my_devs/new_act_trt/scripts/run_trt_act.py`，作为 TRT 专用入口。
- 不重写机器人控制循环，只替换 policy backend。

## 3. 工作思路

总体思路是“保守替换，边界清晰”。

具体原则如下：

- 不重写 LeRobot 的控制主循环。
- 不改 `predict_action(...)` 的推理协议。
- 不改 checkpoint 里保存下来的 preprocessor / postprocessor。
- 只在 policy 这一层，把原本的 `ACTPolicy` 替换成 `TrtActPolicyAdapter`。
- 在导出和验证阶段，把 core 层和 policy 层分开验证，避免仅凭 engine 能 build 成功就误判“可以部署”。

这也是为什么最终方案不是把 TRT 直接塞进 `src/lerobot` 的通用工厂，而是先在 `my_devs/new_act_trt` 下做一条受控工作区，再通过 `run_act_infer.py --policy-backend trt` 进行接入。

## 4. 主要问题与处理方式

### 4.1 参考代码不能直接照搬

虽然 `my_devs/new_act_trt/reference/act_trt` 提供了可借鉴的脚本结构，但它不能直接作为当前仓库的正式入口，主要有几个原因：

- 它的路径解析是按参考目录布局写的，直接运行时不能保证正确定位当前仓库根目录。
- 它默认假设的输出目录不一定和当前仓库 checkpoint 布局严格一致。
- 它提供的是一套参考实现，而不是当前仓库下已经验证过的稳定工作区。

因此最终做法是：

- 参考脚本结构和接口设计。
- 重新在 `my_devs/new_act_trt/scripts` 下落地当前仓库可运行版本。
- 所有默认路径、checkpoint 解析、输出目录和推理入口，均按当前仓库实际情况重建。

### 4.2 不能信任旧产物

仓库里已经存在旧的 `outputs/deploy/act_trt/...` 产物，但这些历史文件本身不能证明当前代码仍然正确。

所以这次工作采取的是：

- 重新导出 ONNX。
- 重新 build TensorRT engine。
- 重新做 core 和 policy 两层一致性验证。
- 报告只引用这次重跑后的结果。

### 4.3 不能只比 core，需要比完整 policy

仅比较 `policy.model` 的输出是不够的，因为真实部署时还包含：

- `prepare_observation_for_inference(...)`
- checkpoint 保存下来的 `policy_preprocessor.json`
- ACT 的 action queue
- `policy_postprocessor.json`

如果这些边界没有一并校验，那么即使 core 网络误差很小，真实机器人推理仍然可能有问题。

所以这次工作把校验拆成两层：

- `verify_act_torch_onnx_trt.py` 负责 core 一致性。
- `verify_act_torch_trt_policy.py` 负责完整 policy 一致性。

## 6. 我是怎么做这件事的

下面按实际实施顺序说明。

### 6.1 先分析当前仓库与参考实现的差异

首先确认三部分代码：

- `my_devs/new_act_trt/reference/act_trt`
- `my_devs/train/act/so101/run_act_infer.py`
- 当前 checkpoint 布局

重点确认的是：

- 当前 checkpoint 的实际目录结构
- 当前 ACT policy 的输入特征顺序
- 机器人推理入口当前是如何加载 preprocessor / postprocessor 和 policy 的
- `predict_action(...)` 对 policy 的接口要求是什么

分析结果表明：

- `predict_action(...)` 并不关心底层是 torch 还是 TRT。
- 它只要求 policy 提供 `select_action(...)`，并有可用的 `config`。
- 因此最安全的改法，是保留 `predict_action(...)` 和控制循环，只替换 policy backend。

### 6.2 建立新的工作区

在 `my_devs/new_act_trt/scripts` 下重新整理脚本，不直接运行 reference 目录下的内容。

这么做是为了：

- 消除参考实现对目录结构的隐含假设
- 统一当前仓库的路径解析方式
- 把新的导出、验证、推理逻辑集中到一个可维护的工作区

### 6.3 先打通导出和编译

先完成两步：

1. 从 checkpoint 导出 ONNX
2. 从 ONNX build TRT engine

这一阶段生成的关键产物是：

- `act_single.onnx`
- `export_metadata.json`
- `act_single_fp32.plan`
- `trt_build_summary_fp32.json`

这里有两个关键点：

- ONNX 导出只导出 `policy.model`，不导出 Python 层 action queue 逻辑。
- `export_metadata.json` 记录输入输出名、camera 顺序、shape 和 export 设定，后续 TRT adapter 会依赖它来解析 engine I/O。

### 6.4 再做 core 一致性

导出和 build 成功之后，先做 `torch / onnx / trt` 三方 core 对比。

原因是：

- 如果 core 层已经不一致，就没有必要继续做完整 policy 校验。
- 这样可以把问题快速定位在 export 边界或 compile 边界，而不是混入 preprocessor / postprocessor / queue 等其他因素。

### 6.5 再做完整 policy 一致性

在 core 层通过之后，再做完整策略对比。

这个阶段重点验证的是：

- 同一份 mock observation 经过相同 preprocessor 之后
- `torch policy` 和 `trt policy adapter`
- 在 `predict_action_chunk`、`select_action`、postprocess 后动作上是否一致

这个验证更加接近真实部署情况，因为它包含了：

- 观测预处理
- 归一化
- policy 的 queue 行为
- 反归一化

### 6.6 最后接入真实推理入口

一致性通过后，再改 `run_act_infer.py`。

改法很克制，只做三件事：

- 增加 `--policy-backend`
- 增加 TRT engine / metadata / runtime device 参数
- 在实例化 policy 的地方，根据 backend 在 `ACTPolicy` 和 `TrtActPolicyAdapter` 间二选一

除此之外，机器人控制循环、观测构建、`predict_action(...)` 调用方式、动作发送逻辑都保持不变。

```bash
conda run -n lerobot_flex python my_devs/train/act/so101/run_act_infer.py \

  --task "Put the block in the bin"
```

这是主入口，支持 torch 和 trt 两种 backend。

### 8.2 TRT 专用入口

```bash
conda run -n lerobot_flex python my_devs/new_act_trt/scripts/run_trt_act.py \
  --policy-path outputs/train/20260305_190147_act_grasp_block_in_bin1_e15/checkpoints/last/pretrained_model \
  --task "Put the block in the bin"
```

这个脚本只是一个薄包装：

- 接收用户参数
- 自动追加 `--policy-backend trt`
- 调用 `run_act_infer.py` 的 `main(argv)`

因此它的运行原理和 `run_act_infer.py --policy-backend trt` 完全一致。

## 8.3 最终推理链路

最终真实推理链路如下：

1. 机器人读当前观测
2. `robot_observation_processor` 对观测做机器人到数据集特征对齐
3. `build_dataset_frame(...)` 生成 `observation.*` 格式的数据帧
4. `predict_action(...)` 开始统一推理流程
5. `prepare_observation_for_inference(...)` 把 numpy 观测转成 torch tensor，并完成图像 CHW / float32 / batch 化
6. checkpoint 保存的 `preprocessor` 对输入做 rename、to_batch、device、normalization
7. TRT backend 的 `TrtActPolicyAdapter.select_action(...)` 执行
8. adapter 内部通过 `TensorRTRunner.infer(...)` 调用 engine
9. adapter 返回 normalized action
10. checkpoint 保存的 `postprocessor` 执行 unnormalize 和 device 回传
11. `make_robot_action(...)` 把 tensor 转为机器人动作字典
12. `robot_action_processor(...)` 做机器人动作格式对齐
13. `robot.send_action(...)` 下发动作

这条链路最重要的特点是：

- 只替换 policy backend
- 不替换控制循环
- 不替换 preprocessor / postprocessor
- 不改 action queue 行为

## 9. 本次工作里实际用了哪些 API

下面把这次工作真正依赖的 API 分模块说明。

## 9.1 LeRobot / 当前仓库 API

### 配置与模型加载

- `PreTrainedConfig.from_pretrained(...)`
  用于从 checkpoint 读取 ACT 配置。

- `ACTPolicy.from_pretrained(...)`
  用于加载 torch 版本 ACT policy。

- `PolicyProcessorPipeline.from_pretrained(...)`
  用于直接加载 checkpoint 保存的 preprocessor / postprocessor。

### 推理协议

- `predict_action(...)`
  当前仓库统一的单步策略推理入口。

- `prepare_observation_for_inference(...)`
  把原始观测转成推理用 tensor，并完成图像维度和 dtype 处理。

- `make_default_processors()`
  获取 robot observation/action 的默认处理 pipeline。

- `build_dataset_frame(...)`
  把机器人观测组织成 policy 需要的 `observation.*` 数据格式。

- `make_robot_action(...)`
  把 policy 输出 tensor 转换成机器人动作字典。

### ACT 内部逻辑

- `ACTTemporalEnsembler`
  用于复现 temporal ensemble。

- `select_action(...)`
  用于执行 action queue 语义。

- `predict_action_chunk(...)`
  用于获取 action chunk。

## 9.2 PyTorch API

- `torch.onnx.export(...)`
  用于导出 ACT core ONNX。

- `torch.inference_mode()`
  在推理和对比阶段关闭梯度。

- `torch.cuda.Stream(...)`
  在 TRT runtime 中管理 CUDA stream。

- `torch.from_numpy(...)`
  用于把 numpy 输入转成 torch tensor。

- `torch.empty(...)`
  用于分配 TRT 输出 tensor。

## 9.3 ONNXRuntime API

- `onnxruntime.InferenceSession(...)`
  用于执行 ONNX 推理并与 torch / trt 做对比。

这里显式使用了 `CPUExecutionProvider`，目的是避免 ORT 的 CUDA 配置差异干扰核心数值对比。

## 9.4 TensorRT API

### Build 侧

- `trt.Builder(...)`
- `builder.create_network(...)`
- `trt.OnnxParser(...)`
- `builder.create_builder_config(...)`
- `config.set_memory_pool_limit(...)`
- `config.set_flag(...)`
- `config.clear_flag(...)`
- `builder.build_serialized_network(...)`

这些 API 用于把 ONNX 编译成 TensorRT engine。

### Runtime 侧

- `context.set_input_shape(...)`
